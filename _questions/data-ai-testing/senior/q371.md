## The Intent
The interviewer is checking whether you can think at programme level — not just "what tests do we write?" but "what does quality mean for this class of system, how do we operationalise it across teams, and how do we sustain it as the technology evolves?" At senior level, this is the question that separates a capable practitioner from a discipline leader.

## The STAR Script
**Situation:** I joined a scale-up that had shipped four ML-powered features in the previous 12 months: a recommendation engine, a fraud classifier, a demand forecasting model, and an LLM-powered customer support chatbot. Each had been built and tested in isolation, with no shared standards, and three of the four had experienced quality incidents post-launch.

**Task:** I was asked to design a unified data and AI quality strategy that would apply retroactively to the four existing products and be the standard for all future AI feature development.

**Action:** I structured the strategy across five domains. Domain 1 — data quality: a contract-based standard requiring schema contracts between all data producers and consumers, enforced in CI; a data quality framework using Great Expectations expectation suites gated in every pipeline deployment. Domain 2 — model evaluation: a mandatory evaluation report for every model release, including a labelled test set result, fairness assessment across defined demographic splits, and a comparison against the current production model baseline. Domain 3 — deployment gates: a model release checklist requiring evaluation report approval by a senior engineer outside the model's team, a canary deployment phase at 5% traffic with automated rollback if production metrics degraded within 24 hours. Domain 4 — production monitoring: standardised monitors for input drift (PSI), output distribution, business metric KPIs, and — for the LLM — weekly adversarial safety sampling. Domain 5 — model governance: a model registry documenting every deployed model's training data lineage, evaluation results, fairness findings, and monitoring status — the single source of truth for the AI portfolio.

**Result:** The strategy was adopted by the CTO as the AI quality standard. The first model released under the new gates was the fraud classifier retrain — the canary deployment triggered an automatic rollback after detecting a 3-percentage-point increase in false negatives within 6 hours. Without the gate, the degraded model would have gone to 100% traffic. The model registry became the most-referenced document in AI-related due diligence conversations with enterprise customers.

## Key Terms
| Term | Definition |
|------|------------|
| **Model registry** | A versioned catalogue of deployed and historical models, storing metadata including training data lineage, evaluation results, and deployment history |
| **Canary deployment** | A release strategy where a new model version is exposed to a small percentage of production traffic first, allowing quality signals to be observed before full rollout |
| **Automated rollback** | A system that automatically reverts to the previous model version when production quality metrics breach a defined threshold during a canary deployment |
| **Data lineage** | A record of a dataset's origin, transformations, and flow through the pipeline — essential for understanding what data a model was trained on and diagnosing quality issues |
| **Evaluation report** | A structured document summarising a model's performance on the test set, fairness assessment, and comparison to the production baseline — the gate artefact for a model release |
| **Adversarial safety sampling** | A scheduled exercise where a set of adversarial prompts is run against a production LLM to detect safety regressions introduced by model updates or prompt changes |
| **AI portfolio** | The full set of AI and ML-powered features deployed across an organisation's products — requiring coordinated governance rather than feature-by-feature management |
