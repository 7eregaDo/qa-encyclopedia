## The Intent
The interviewer is checking whether you can reason about quality assurance for systems you do not control. Most AI products today are built on foundation models from external providers — and a QA strategy that assumes internal model ownership will fail to address the unique risks of third-party dependency.

## The STAR Script
**Situation:** Our product used the OpenAI API (GPT-4) as the backbone for a legal document summarisation feature. We had no visibility into when OpenAI updated the model, no notification of behaviour changes, and no systematic way to know whether the model serving our users today was the same as the model that passed our pre-launch evaluation six months ago.

**Task:** I was asked to design a quality programme for our third-party model dependency that would give us reliable early warning of behaviour changes.

**Action:** I built the programme around three pillars. Pillar 1 — behavioural baseline: I constructed a golden evaluation set of 120 legal documents with human-annotated reference summaries and scored the model's outputs on accuracy, completeness, and format compliance at monthly intervals. I stored every evaluation run with its model version identifier (where available) and output hash so I could detect changes even when OpenAI did not announce them. Pillar 2 — automated regression monitoring: I ran a subset of 20 high-sensitivity test cases on every production deployment and on a weekly schedule — flagging when outputs changed materially from the stored baseline. I used an LLM-as-judge approach for semantic similarity scoring rather than exact-match, since identical meaning expressed differently was still correct. Pillar 3 — model version pinning and change management: I worked with the engineering team to pin the API calls to a specific model version identifier where the provider supported it, and defined an upgrade process — any model version change required a full evaluation set run and sign-off before the new version was promoted to production. I also built a fallback: if the provider's API was unavailable or latency exceeded 10 seconds, the feature degraded gracefully to a cached summary rather than a timeout error.

**Result:** Three months after implementing the baseline monitoring, a weekly regression check flagged that 14 of the 20 test cases had changed outputs. OpenAI had silently updated the model. Five of the changed outputs were lower quality by our scoring criteria. The new model version was rejected and the previous pinned version was retained until a full evaluation confirmed the quality regression was resolved in a subsequent update. Without the monitoring, the degradation would have gone undetected.

## Key Terms
| Term | Definition |
|------|------------|
| **Foundation model** | A large-scale AI model trained on broad data that can be adapted to many tasks — typically accessed via API from providers such as OpenAI, Anthropic, or Google |
| **Model version pinning** | Locking API calls to a specific named model version to prevent uncontrolled behaviour changes from provider-side updates |
| **Behavioural baseline** | A recorded set of model outputs against a defined input set, used as the reference for detecting changes in model behaviour over time |
| **Silent model update** | A change in model behaviour made by a provider without explicit versioning notification — a common quality risk for foundation model API consumers |
| **LLM-as-judge** | A technique using a second LLM to score the quality of a primary model's outputs — enabling semantic evaluation at scale without manual review of every output |
| **Graceful degradation** | A system design where a feature continues to function in a reduced but acceptable mode when its primary dependency is unavailable or degraded |
| **Change management** | A defined process for evaluating, approving, and rolling out changes to a production system — applied here to model version upgrades |
