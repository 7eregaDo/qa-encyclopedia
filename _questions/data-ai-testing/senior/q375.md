## The Intent
The interviewer is checking whether you can architect the visibility layer that makes a data platform trustworthy at scale. At senior level, reactive quality checks are insufficient — the goal is a platform where degradation is detected automatically, root cause is diagnosable in minutes, and downstream consumers are notified before they notice the problem.

## The STAR Script
**Situation:** Our data platform served 40 downstream consumers — dashboards, ML models, and product features — from 18 pipelines. Data incidents were consistently discovered by downstream consumers, not by the data team. Mean time to detection was 14 hours; mean time to resolution was 22 hours. The data team had no systematic visibility into what was healthy or degraded at any moment.

**Task:** I was asked to design and implement a data observability programme that would shift detection upstream and reduce both MTTD and MTTR.

**Action:** I designed the programme across four layers. Layer 1 — pipeline health monitors: I instrumented every pipeline with four standard monitors: freshness (was the pipeline output updated within the expected window?), volume (did the row count fall within the expected range based on historical patterns?), schema (did any column change type, name, or disappear?), and null rate (did the proportion of nulls in critical columns exceed the historical baseline by more than 10%?). These ran automatically after every pipeline execution and on a scheduled basis for slowly-changing pipelines. Layer 2 — statistical distribution monitoring: For the 8 pipelines feeding ML models, I added distribution monitoring using Evidently AI — tracking mean, standard deviation, and quartiles for numerical features, and value frequency for categorical features, alerting on PSI > 0.2. Layer 3 — end-to-end lineage: I implemented dbt's lineage graph combined with a custom metadata store tracking which downstream consumers depended on which pipeline outputs. When a monitor fired, the alert included the full downstream impact list — every dashboard and model affected. Layer 4 — automated incident notification: When a monitor breached its threshold, a Slack alert fired to both the owning data team and all affected downstream consumers simultaneously, with a standardised message: which monitor fired, which pipeline, which downstream consumers were affected, and a link to the lineage graph. Affected consumers no longer discovered incidents themselves — they received advance warning.

**Result:** MTTD dropped from 14 hours to 23 minutes (monitors detected the issue within one pipeline cycle). MTTR dropped from 22 hours to 6 hours (lineage information meant engineers knew exactly what was affected and could prioritise accordingly). In the first quarter, 11 data incidents were detected and communicated to downstream consumers before any of them noticed — a complete inversion of the previous pattern.

## Key Terms
| Term | Definition |
|------|------------|
| **Data observability** | The ability to understand the health, state, and lineage of data across a platform — the data equivalent of application observability (metrics, logs, traces) |
| **Freshness monitor** | An automated check that a pipeline's output was updated within the expected time window |
| **Volume monitor** | An automated check that a pipeline's output row count falls within expected historical bounds |
| **MTTD (Mean Time to Detection)** | The average time between a data quality incident occurring and the team becoming aware of it |
| **MTTR (Mean Time to Resolution)** | The average time between a data quality incident being detected and the affected data being restored to a correct state |
| **Data lineage** | A map of how data flows from source systems through transformations to downstream consumers — essential for impact assessment during incidents |
| **Downstream impact** | The set of consumers (dashboards, models, features) affected when a specific pipeline or data asset is degraded |
