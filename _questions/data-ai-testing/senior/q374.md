## The Intent
The interviewer is checking whether you understand that agentic AI introduces a qualitatively different class of testing challenge — because errors compound across steps, actions can have irreversible real-world consequences, and the system's decision to act or not act is itself a quality dimension. At senior level, this is the frontier of AI testing practice.

## The STAR Script
**Situation:** My team was building an AI agent for internal IT support: given a ticket, the agent could query a knowledge base, look up a user's account status, reset passwords, and escalate to a human engineer if it could not resolve the issue. The agent was LLM-powered with four tool integrations. Before any real user traffic, I was asked to design the testing approach.

**Task:** I needed to design a test suite that validated correct action execution, safe tool use, error handling, and appropriate escalation — none of which were covered by standard LLM evaluation techniques.

**Action:** I designed the test suite across five dimensions. Dimension 1 — action correctness: I created 60 scripted ticket scenarios with known correct resolution paths. I ran the agent against each and verified it took the right sequence of actions — the right tools, in the right order, with the right parameters. Incorrect tool calls (e.g., resetting the wrong account) were logged as defects. Dimension 2 — tool boundary safety: I verified that the agent could not call tools outside its defined permission set. I attempted to craft prompts that would lead the agent to use administrative tools it had no authorisation for. All 12 boundary-crossing attempts were correctly refused. Dimension 3 — intermediate state handling: I injected failures at each tool call stage (knowledge base timeout, account lookup returning null, password reset API error) and verified the agent handled each gracefully — retrying where appropriate, logging the failure, and escalating to a human rather than producing a confabulated resolution. Dimension 4 — escalation quality: I created 20 scenarios deliberately outside the agent's capability — ambiguous tickets, policy exceptions, emotionally distressed users. I verified the agent escalated rather than attempted resolution in all 20 cases, and that the escalation message included sufficient context for the human engineer. Dimension 5 — action irreversibility check: I verified that any irreversible action (password reset, account lock) was preceded by a confirmation step — either a user confirmation or a pre-condition check — and could not be triggered by a malformed intermediate state.

**Result:** Dimension 3 uncovered a critical defect: when the password reset API returned a timeout, the agent logged the action as completed and told the user their password had been reset — a false confirmation. This was fixed before any real users interacted with the agent. The test suite was documented and used as the baseline for all subsequent agent feature development.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that agentic AI systems take actions and that errors in action sequences are a quality concern |
| **Mid** | Tests individual tool calls and basic action sequences, validates error handling, and checks escalation behaviour |
| **Senior** | Designs a comprehensive agentic test suite covering action correctness, tool boundary safety, intermediate state handling, escalation quality, and irreversibility guards — and defines the safety principles governing agentic deployment |

## Key Terms
| Term | Definition |
|------|------------|
| **Agentic AI** | An AI system that autonomously plans and executes sequences of actions — using tools, APIs, or external services — to accomplish a goal |
| **Tool call** | An invocation of an external function or API by the AI agent as part of its task execution |
| **Action sequence** | The ordered set of tool calls and decisions made by an agent to resolve a given task — correctness is measured across the full sequence, not individual steps |
| **Tool boundary** | The defined set of tools and permissions an agent is authorised to use — preventing privilege escalation or unintended system access |
| **Intermediate state** | The system condition between steps in an agent's action sequence — unexpected intermediate states (errors, nulls, timeouts) are a primary source of agent failure |
| **Escalation** | The agent's decision to hand a task to a human when it cannot resolve it confidently — a critical safety behaviour that must be tested explicitly |
| **Irreversible action** | An agent action that cannot be undone (deleting a record, sending an email, making a payment) — requiring pre-condition checks or confirmation steps before execution |
