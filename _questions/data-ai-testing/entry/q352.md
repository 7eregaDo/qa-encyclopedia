## The Intent
The interviewer is checking whether you can categorise data problems systematically rather than treating them as a vague notion of "bad data." The ability to name the type of data quality issue you are looking for determines which test you write — and a QA engineer who can think in these dimensions adds real value to a data team.

## The STAR Script
**Situation:** My team received a bug report from the finance department: a daily revenue report was showing totals that did not match the figures in the transactional database. No one had been testing the ETL pipeline that generated the report.

**Task:** I was asked to investigate the discrepancy and propose a basic data quality test suite for the pipeline.

**Action:** I examined the pipeline output against the source database across each quality dimension. Accuracy: three records had incorrect currency conversion factors applied — a logic error in the transformation. Completeness: 47 transaction records from one payment provider were missing entirely — a source connector had silently failed for that provider. Consistency: the same transaction IDs appeared with different statuses in the report versus the operational database — the pipeline was reading a stale snapshot. Timeliness: the pipeline was scheduled to run at 06:00 but had been running at variable times between 05:45 and 08:30 — reports were not always current by the time the finance team opened them. Validity: three records had null transaction amounts, which the report treated as zero. I wrote SQL-based data quality checks for each dimension and scheduled them to run after every pipeline execution.

**Result:** The checks became a permanent part of the pipeline. The completeness check caught another connector failure two weeks later — before the finance team noticed. The team credited the exercise with giving them confidence in a system they had previously treated as a black box.

## Key Terms
| Term | Definition |
|------|------------|
| **Accuracy** | The degree to which data values correctly represent the real-world entity or event they describe |
| **Completeness** | The degree to which expected data is present — no nulls where values are required, no missing records |
| **Consistency** | The degree to which data is uniform across systems, tables, or time — the same entity should have the same attributes wherever it appears |
| **Timeliness** | The degree to which data is available and current at the time it is needed for its intended use |
| **Validity** | The degree to which data conforms to defined formats, types, and business rules — for example, a date field should contain a date, not a string |
| **ETL (Extract, Transform, Load)** | A data integration pattern that extracts data from source systems, transforms it into the required format, and loads it into a destination — a common source of data quality defects |
| **Data quality check** | An automated assertion run against a data set or pipeline output to detect violations of a defined quality dimension |
