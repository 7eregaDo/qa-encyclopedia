## The Intent
The interviewer is checking whether you can articulate the fundamental shift in thinking required when testing AI systems. Applying traditional pass/fail test thinking directly to ML models produces inadequate coverage — a QA engineer who understands the difference can design tests that are actually meaningful.

## The STAR Script
**Situation:** Our team added a machine learning sentiment classifier to our customer feedback tool. A developer asked me to write tests for it "like the other API tests." I had to explain why that approach would not work — and then design something that would.

**Task:** I needed to articulate the difference between testing a deterministic function and testing an ML model, and propose an appropriate test approach.

**Action:** I explained the core distinction with an example. For a traditional function like `calculateDiscount(tier, amount)`, I can write: given input A, assert output B — deterministic, always reproducible. For the sentiment classifier, the model might classify the sentence "not bad at all" as positive 92% of the time across thousands of similar inputs — but not necessarily on every single call. Testing it with one input and asserting one output would tell me almost nothing. Instead, I proposed three test categories. First, a labelled test set: 200 sentences with known correct labels, asserting that overall accuracy was above 85% and that precision and recall for each class were within defined thresholds. Second, boundary cases: inputs at the edge of sentiment (neutral sentences, sarcasm, very short inputs) to characterise model behaviour in ambiguous territory rather than assert correctness. Third, invariance tests: inputs where the classification should not change when an irrelevant word is substituted (e.g., swapping "the product" for "this product") — if it did change, that suggested the model was learning spurious correlations.

**Result:** The labelled test set approach was adopted as the standard for all ML features. The invariance tests caught a model version that had overfit to specific product names in the training data — a finding that would have been invisible to traditional API tests.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can explain that ML models are probabilistic and that testing them requires statistical rather than deterministic assertions |
| **Mid** | Designs labelled test sets, defines accuracy and fairness thresholds, and implements invariance and metamorphic tests |
| **Senior** | Owns the model evaluation framework, connects test design to model training practices, and introduces ongoing monitoring for model drift in production |

## Key Terms
| Term | Definition |
|------|------------|
| **Probabilistic output** | A model output that represents a likelihood or confidence score rather than a deterministic value — correct behaviour is defined statistically across many inputs |
| **Labelled test set** | A curated collection of inputs with known correct outputs, used to measure a model's accuracy against ground truth |
| **Accuracy** | The proportion of predictions a model gets correct across a test set |
| **Precision** | Of all the instances a model predicted as class X, the proportion that were actually class X |
| **Recall** | Of all the actual instances of class X, the proportion that the model correctly identified |
| **Invariance test** | A test verifying that a model's output does not change when an input characteristic that should be irrelevant is modified |
| **Metamorphic testing** | A testing technique where the relationship between outputs is asserted rather than the exact output — for example, adding a clearly negative sentence to a review should lower its positive sentiment score |
