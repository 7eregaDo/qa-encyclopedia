## The Intent
The interviewer is checking whether you recognise that quality extends into data systems and AI features, not just traditional application code. As AI becomes embedded in products, a QA engineer who cannot reason about data quality or model behaviour will have a shrinking scope. They want to see that you treat data and AI outputs as testable, not as black boxes.

## The STAR Script
**Situation:** I joined a team building a product that included a machine learning recommendation engine. During onboarding I noticed the QA test suite covered the UI and API thoroughly but had no tests touching the recommendation feature beyond "it returns a response."

**Task:** I wanted to understand what data and AI testing meant in practice, and where QA could add value beyond functional coverage.

**Action:** I researched the main categories: data pipeline testing (is the data correct, complete, and fresh?), model input validation (is the data fed to the model in the right format and range?), model output testing (are the predictions within expected ranges? are edge cases handled gracefully?), and fairness testing (does the model behave consistently across different user groups?). I documented a simple test checklist for our recommendation engine covering: null input handling, out-of-range input values, response schema validation, and a smoke check that recommendations changed when the input user changed (a basic sanity check that the model was not returning the same response for everyone).

**Result:** The checklist found two issues on first run: the model returned a 500 error on null user ID instead of a graceful fallback, and recommendations were identical for all new users with no purchase history. Both were fixed before the feature launched. I came away with a clear understanding that AI features need the same systematic test thinking as any other feature — the domain is different, but the discipline is the same.

## Key Terms
| Term | Definition |
|------|------------|
| **Data pipeline** | A sequence of automated processes that ingest, transform, and load data from source systems to a destination — subject to quality defects at every stage |
| **Model input validation** | Testing that data fed to a machine learning model is in the correct format, range, and completeness expected by the model |
| **Model output testing** | Verifying that a model's predictions or generated outputs are within expected ranges, handle edge cases gracefully, and meet defined quality thresholds |
| **Fairness testing** | Evaluating whether a model's behaviour is consistent and equitable across different demographic or user groups |
| **Data quality** | The degree to which data is accurate, complete, consistent, timely, and fit for its intended purpose |
| **Black box testing** | Testing a system based solely on inputs and outputs, without knowledge of internal implementation — the default approach for testing ML model behaviour |
| **Graceful fallback** | A system's ability to return a safe, defined response when it cannot produce its normal output — for example, returning popular items when no personalised recommendations are available |
