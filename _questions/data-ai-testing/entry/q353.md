## The Intent
The interviewer is checking whether you understand how data flows through a system and where quality defects can enter. Testing a pipeline is conceptually similar to testing any multi-step process — each stage is a function with inputs, outputs, and potential failure modes — but the domain requires specific techniques.

## The STAR Script
**Situation:** My team built a data pipeline that ingested customer event data from a Kafka topic, transformed it to calculate session durations, and loaded the results into a data warehouse for the analytics team. When the analytics team started using the data, they found session durations that were clearly incorrect — some showed zero seconds, others showed 47 hours.

**Task:** I was asked to investigate the pipeline and introduce testing to prevent recurrence.

**Action:** I decomposed the pipeline into three testable stages. Stage 1 — ingestion: I wrote tests that verified all expected event types were consumed from Kafka without message loss, and that the schema of incoming events matched the contract. I used a test Kafka topic seeded with known events to validate the consumer. Stage 2 — transformation: I wrote unit tests for the session duration calculation function using a set of known inputs and expected outputs — including edge cases like events with identical timestamps (zero duration), events spanning midnight (cross-day sessions), and a single orphaned session-start with no matching session-end. These tests caught the zero-duration bug: the function was subtracting timestamps in the wrong order for a specific timezone offset. Stage 3 — load: I wrote row count and null-check assertions that ran after each pipeline execution, comparing the output record count against the input event count and flagging any null session durations.

**Result:** The timezone bug was fixed. The assertions caught two further issues over the following month: a schema change in the upstream Kafka topic that broke ingestion, and a transformation bug introduced during a refactor. The pipeline now had a test suite that could be run in CI against test data before any change was deployed.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can decompose a pipeline into stages, describe what to test at each stage, and write basic assertions against known inputs and outputs |
| **Mid** | Designs a comprehensive pipeline test suite covering happy path, edge cases, schema validation, and failure handling — integrated into CI/CD |
| **Senior** | Owns the data testing strategy across multiple pipelines, defines reusable testing patterns, and introduces contract testing between pipeline stages and upstream producers |

## Key Terms
| Term | Definition |
|------|------------|
| **Data pipeline** | An automated sequence of data movement and transformation steps connecting a source system to a destination |
| **Ingestion** | The first stage of a pipeline: consuming raw data from a source system (database, API, message queue) |
| **Transformation** | The stage where raw data is cleaned, reshaped, enriched, or aggregated into the form required by the destination |
| **Load** | The stage where transformed data is written to the destination system (data warehouse, database, file store) |
| **Schema validation** | A test that verifies the structure of a data record — field names, types, and required presence — matches a defined contract |
| **Row count assertion** | A data quality check comparing the number of records in the output against the expected count derived from the input |
| **Edge case** | An input condition at the boundary of normal expected behaviour — in pipeline testing, examples include null values, empty batches, duplicate records, and out-of-order events |
