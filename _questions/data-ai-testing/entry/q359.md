## The Intent
The interviewer is checking whether you understand that data testing is not a privacy-free zone. Using real customer data to test a pipeline or evaluate a model carries legal and ethical obligations — and a QA engineer who handles data irresponsibly creates risk for the organisation and for the individuals whose data is involved.

## The STAR Script
**Situation:** When I joined a data team, I noticed that the engineers were routinely copying production database snapshots into a shared development environment to use as test data for pipeline development. The snapshots contained real customer names, email addresses, and purchase histories.

**Task:** I raised the concern with the team lead and proposed an alternative approach, but first needed to understand the compliance requirements and practical options well enough to make a credible case.

**Action:** I researched the relevant obligations under GDPR. Key points: personal data should only be processed for the purpose it was collected (production snapshots used for testing are a purpose mismatch), data should be minimised to what is necessary, and any processing must have a lawful basis. I then identified three practical alternatives the team could use. First, anonymisation: remove or irreversibly hash direct identifiers (name, email, national ID) so the data can no longer be attributed to an individual. Second, pseudonymisation: replace identifiers with tokens — more reversible than anonymisation but still reduces risk and may satisfy purpose limitation requirements with appropriate controls. Third, synthetic data generation: produce a structurally realistic dataset using tools like Faker or Mimesis, with no relationship to real individuals. I presented the options with their trade-offs and recommended synthetic data for new pipeline development and anonymisation for cases where real statistical distributions were essential.

**Result:** The team adopted a synthetic data generation script for standard pipeline testing and implemented an anonymisation pipeline for the cases where statistical realism was required. The production snapshot practice was discontinued. The data protection officer confirmed the new approach was compliant.

## Key Terms
| Term | Definition |
|------|------------|
| **GDPR (General Data Protection Regulation)** | EU regulation governing the collection, processing, and storage of personal data — applicable to any organisation processing data of EU residents |
| **Data minimisation** | The principle that only the minimum personal data necessary for the stated purpose should be collected and processed |
| **Purpose limitation** | The principle that personal data collected for one purpose should not be reprocessed for a different, incompatible purpose |
| **Anonymisation** | The irreversible removal or transformation of personal identifiers such that an individual can no longer be identified from the data |
| **Pseudonymisation** | The replacement of direct identifiers with tokens, reducing privacy risk while retaining the ability to re-identify under controlled conditions |
| **Synthetic data** | Artificially generated data that mimics the statistical properties of real data without containing any real individuals' information |
| **Data Protection Officer (DPO)** | The organisational role responsible for overseeing data protection compliance — a required appointment under GDPR for certain organisations |
