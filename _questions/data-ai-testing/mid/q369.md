## The Intent
The interviewer is checking whether you can bring software engineering discipline to data workflows — treating data pipeline changes with the same rigour as application code changes. At mid level, they want to see that you have done this in practice and can navigate the specific challenges of CI/CD in a data context.

## The STAR Script
**Situation:** Our data team deployed pipeline and dbt model changes by running them directly against the production data warehouse. There were no automated tests, no staging environment equivalent, and changes were frequently deployed on a Friday afternoon before anyone noticed a problem.

**Task:** I was asked to introduce a CI/CD practice that would catch issues before production deployment.

**Action:** I designed a three-stage pipeline using GitHub Actions and dbt. Stage 1 — PR validation: on every pull request, I ran dbt compile (syntax and dependency check) and dbt test --select state:modified+ (run only tests affected by the changed models, using dbt's state comparison feature). This gave fast feedback on the specific change without running the entire test suite. I also ran Great Expectations on a sample of fixture data to validate expectation suites. Stage 2 — staging deployment: on merge to main, the pipeline ran dbt run in a separate staging schema within the same warehouse, executing all modified models against a copy of recent production data. Great Expectations ran the full suite against the staging output. Any expectation failure or dbt test failure blocked promotion to production. Stage 3 — production deployment: a manual approval gate (one engineering lead sign-off) preceded the production run, which triggered automatically after approval. Post-production, a lightweight smoke check ran the critical expectation suites against the live output and alerted via Slack if any failed. I documented the "Friday deploy" risk and the team agreed to a soft freeze — no production deployments after 15:00 on Fridays without explicit approval.

**Result:** The first week the pipeline ran, Stage 2 caught a model change that introduced a column rename — which would have broken three downstream dashboards. The data analyst who made the change fixed it before it reached production. Deployment confidence increased visibly; the team's Friday freeze was observed without friction because the pipeline had demonstrated its value immediately.

## Key Terms
| Term | Definition |
|------|------------|
| **dbt state comparison** | A dbt feature that identifies which models have changed since the last production run, enabling selective test execution on only the affected models |
| **Staging schema** | A separate database schema used to run and validate pipeline changes before they affect the production tables |
| **Expectation suite** | A named collection of Great Expectations data assertions run as a test suite against a dataset |
| **Manual approval gate** | A CI/CD pipeline step requiring a human sign-off before the next stage executes — used to add human judgement at high-risk deployment points |
| **Smoke check** | A lightweight post-deployment validation confirming that the most critical outputs are correct — not a comprehensive test, but a fast signal that nothing is catastrophically wrong |
| **Soft freeze** | A team agreement to avoid deployments during high-risk time windows (e.g., Friday afternoons) — informal but effective at reducing incident frequency |
| **State:modified+** | A dbt selector syntax that targets changed models and all their downstream dependants — the minimum scope for impact-based test selection |
