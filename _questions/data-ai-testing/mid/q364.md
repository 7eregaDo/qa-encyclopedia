## The Intent
The interviewer is checking whether you understand the architecture of RAG systems well enough to design component-appropriate tests. RAG is an increasingly common pattern for LLM-powered products, and a tester who treats it as a single black box will miss the systematic failure modes specific to each layer.

## The STAR Script
**Situation:** My team built a RAG-powered internal knowledge base tool: users asked questions in natural language, the system retrieved relevant policy documents from a vector database, and an LLM generated an answer grounded in those documents. After launch, users reported that some answers were correct but not backed by the retrieved documents — the LLM was "hallucinating" content not in the retrieved context.

**Task:** I was asked to design a test suite that would detect this class of failure and more broadly validate the system's quality.

**Action:** I separated the test suite into two layers. Retrieval layer tests: I built a question-document relevance test set — 80 questions, each with a ground-truth list of documents that should be retrieved. I measured recall@k (were the right documents in the top k results?) and mean reciprocal rank (how highly ranked was the first relevant document?). I found that for 14 questions, none of the ground-truth documents appeared in the top 5 results — the embedding model was producing poor representations for technical policy language. Generation layer tests (using RAGAS): I measured three metrics against a sample of 100 question-answer-context triples. Faithfulness: was the answer supported only by the retrieved context? (We found 12% of answers contained statements not present in the context — the hallucination rate.) Answer relevance: did the answer address the question asked? Context precision: were the retrieved documents actually relevant to the question? I also added a safety check: 20 adversarial questions designed to elicit answers about topics outside the policy documents, asserting the model declined to answer rather than fabricating a response.

**Result:** The retrieval issues were fixed by re-chunking the policy documents into smaller segments and switching to a domain-tuned embedding model. The hallucination rate dropped from 12% to 2% after a system prompt change instructing the model to answer only from provided context. The two-layer test approach was adopted as the standard evaluation framework for all RAG features.

## Key Terms
| Term | Definition |
|------|------------|
| **RAG (Retrieval-Augmented Generation)** | An LLM architecture pattern where relevant documents are retrieved from a knowledge base and passed as context to the LLM, grounding its response in specific source material |
| **Vector database** | A database that stores and searches document embeddings — numerical representations of text — enabling semantic similarity search |
| **Recall@k** | A retrieval metric measuring the proportion of relevant documents present in the top k results returned by the retrieval system |
| **Mean reciprocal rank (MRR)** | A retrieval metric measuring the average reciprocal of the rank of the first relevant result across a set of queries |
| **Faithfulness** | A RAGAS metric measuring whether all claims in the generated answer are supported by the retrieved context |
| **Hallucination** | An LLM output that presents fabricated information as fact — a critical failure mode in RAG systems where answers should be grounded in retrieved documents |
| **Chunking** | The process of splitting documents into smaller segments before embedding, affecting both retrieval granularity and context quality |
