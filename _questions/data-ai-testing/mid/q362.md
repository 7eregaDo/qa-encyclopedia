## The Intent
The interviewer is checking whether you understand that data quality failures are often caused by undocumented assumptions between teams. At mid level, they want to see that you can formalise those assumptions into testable contracts, and build the enforcement mechanism that makes them durable.

## The STAR Script
**Situation:** Our data platform had a recurring class of incident: an upstream team would change the schema of an event or API response without notice, and two or three downstream pipelines would silently produce incorrect data until an analyst caught the anomaly — sometimes days later. There were no formal agreements between producers and consumers.

**Task:** I was asked to introduce a contract testing practice that would catch breaking schema changes before they reached the data warehouse.

**Action:** I implemented a lightweight data contract approach in three steps. First, formalisation: I worked with each producer team to document their output schema in a YAML contract file — field names, types, nullability, and value constraints. These files were committed to a shared Git repository. Second, producer-side enforcement: I added a validation step to each producer's CI pipeline that ran their contract file against a sample of their recent output using Great Expectations. If the output violated the contract, the pipeline failed. This gave producers visibility into whether they were breaking their own commitments before a change was merged. Third, consumer-side monitoring: Each consuming pipeline imported the relevant contract file and ran schema validation on ingested data at the start of every run. A contract violation triggered an alert to both the consuming team and the producing team's Slack channel, with a link to the specific contract clause that was violated.

**Result:** Within one quarter, the pattern of silent schema-change incidents dropped from an average of 2.4 per month to zero. The first contract violation alert was triggered by a producer team adding a new required field without updating their consumers — caught in staging, not production. Three producer teams updated their contracts proactively rather than waiting for downstream consumers to discover issues, because the contract file made their commitments explicit and visible.

## Seniority Shift

| **Mid** | Implements data contracts between specific producer/consumer pairs, enforces them in CI/CD, and alerts on violations |

## Key Terms
| Term | Definition |
|------|------------|
| **Data contract** | A formal, versioned agreement between a data producer and consumer specifying the schema, semantics, and quality expectations of the data being exchanged |
| **Producer** | The team or system that generates and emits a data asset — responsible for upholding the contract |
| **Consumer** | The team or pipeline that depends on a data asset — entitled to rely on the contract being maintained |
| **Breaking change** | A modification to a data schema or structure that causes downstream consumers to fail or produce incorrect results |
| **Schema evolution** | The process of modifying a data schema over time — ideally managed through backward-compatible additions and a formal deprecation process rather than silent breaking changes |
| **Contract violation** | An instance where a producer's actual output does not conform to the agreed contract — triggering an alert and requiring remediation |
| **Backward compatibility** | A property of a schema change that does not break existing consumers — adding optional fields is typically backward-compatible; removing or renaming fields is not |
