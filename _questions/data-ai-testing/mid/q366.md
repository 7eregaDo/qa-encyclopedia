## The Intent
The interviewer is checking whether you are familiar with dbt as a testing context, not just as a transformation tool. dbt is widely adopted in modern data teams, and understanding how to write and extend its tests demonstrates that you can contribute meaningfully to a data engineering workflow.

## The STAR Script
**Situation:** My team used dbt to build and maintain all data transformations in our analytics layer. The transformations had grown to 40+ models over 18 months, but fewer than a quarter had any tests attached — engineers had been writing models without writing tests, and there was no enforcement mechanism.

**Task:** I was asked to lead a test coverage initiative — assessing what tests were missing, adding critical tests, and establishing a standard that would prevent the coverage gap from recurring.

**Action:** I started with a coverage audit: I ran `dbt test` against the existing test suite and then reviewed the schema.yml files to identify models with no tests at all. 28 of 41 models had zero tests. I prioritised by impact — models feeding business-critical dashboards first. For each prioritised model I applied four layers of dbt testing. Built-in schema tests: `not_null` on all primary keys and required fields, `unique` on all primary keys, `accepted_values` on status and category columns, `relationships` to validate foreign key integrity across models. Custom SQL tests: I wrote `.sql` test files for business-rule assertions that built-in tests could not express — for example, asserting that total revenue in the aggregated model was within 0.1% of the sum in the source model (a reconciliation test). Freshness tests: I added `source freshness` checks in the sources.yml file to alert when source tables had not been updated within the expected schedule window. Test severity: I set critical tests to `severity: error` (pipeline fails) and advisory tests to `severity: warn` (pipeline continues but logs a warning), distinguishing must-fix from should-investigate.

**Result:** The test run immediately surfaced three issues that had been silently present: a non-unique primary key in the orders model (duplicate records from a pipeline bug), a foreign key violation in the user events model, and a source freshness failure on a supplier feed that had been stale for 11 days. I added a rule to the team's definition of done: no dbt model merged without at least not-null and unique tests on the primary key.

## Key Terms
| Term | Definition |
|------|------------|
| **dbt (data build tool)** | An open-source transformation framework that enables analysts to write, test, document, and version-control SQL transformations against a data warehouse |
| **Schema test** | A built-in dbt test applied to a column — including not_null, unique, accepted_values, and relationships |
| **Custom SQL test** | A user-defined dbt test written as a SQL query that returns rows when a condition is violated — flexible enough to express any business rule |
| **Source freshness** | A dbt feature that checks how recently a source table was updated against a defined maximum age, alerting when data is stale |
| **Test severity** | A dbt configuration that determines whether a failing test blocks pipeline execution (error) or logs a warning without blocking (warn) |
| **schema.yml** | The dbt configuration file where column-level tests, documentation, and metadata are defined for each model |
| **Reconciliation test** | A data quality check asserting that aggregated totals in a downstream model match the sum of corresponding records in the source — detecting data loss or duplication |
