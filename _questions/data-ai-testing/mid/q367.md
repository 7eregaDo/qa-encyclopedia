## The Intent
The interviewer is checking whether you understand that A/B tests are software — they have bugs, and those bugs produce bad decisions. At mid level, they want to see that you can contribute to experiment quality, not just read results.

## The STAR Script
**Situation:** Our product team ran an A/B test on a new checkout flow that showed a 9% improvement in conversion rate at 95% statistical significance. Before the team shipped the variant to 100% of users, I was asked to review the experiment's quality.

**Task:** I needed to assess whether the result was trustworthy — checking for implementation bugs and statistical validity issues.

**Action:** I examined five areas. Randomisation integrity: I verified that user assignment to control and variant was based on a stable user ID hash, not a session ID — session-based randomisation can assign the same user to both variants across sessions, polluting the comparison. I confirmed it was user-based. Sample Ratio Mismatch (SRM): I checked whether the control and variant groups had the expected 50/50 split. The actual split was 48.2% / 51.8% — a statistically significant deviation from 50/50 (chi-square test, p < 0.01). This indicated a bug in the assignment logic. I identified the cause: the variant page had a JavaScript error on Safari that caused some users to silently fall back to the control, skewing the variant group toward non-Safari users who historically convert better. Instrumentation check: I verified that the conversion event was firing correctly in both variants by sampling 100 sessions in each arm and confirming event logs matched observed behaviour. Multiple comparisons: the team had tracked 11 metrics and declared success on conversion rate. I checked whether any correction had been applied — it had not. With 11 metrics at α=0.05, the probability of at least one false positive was approximately 43%. Pre-registration: I checked whether the primary metric (conversion rate) had been declared before the test started — it had, which partially mitigated the multiple comparisons concern.

**Result:** The test was paused, the Safari bug was fixed, and it was restarted. The rerun showed a 5.1% conversion improvement — still positive, but materially lower than the original 9%. The team would have over-invested in the feature had the flawed result been accepted. I produced a one-page A/B test quality checklist that became part of the standard experiment review process.

## Key Terms
| Term | Definition |
|------|------------|
| **Sample Ratio Mismatch (SRM)** | A condition where the actual traffic split between experiment arms differs significantly from the intended split — indicating a bug in assignment or tracking |
| **Randomisation unit** | The entity (user, session, device) used to assign experiment variants — user-level randomisation prevents the same individual from being assigned to multiple arms |
| **Sample pollution** | The contamination of an experiment arm by users or events that should not be in that arm — causing the comparison to measure something other than the intended treatment |
| **Multiple comparisons inflation** | The increase in false positive probability when many metrics are tested simultaneously — corrected using methods such as Bonferroni correction or pre-registration |
| **Pre-registration** | The practice of declaring the primary metric and hypothesis before an experiment begins — reducing the temptation to retroactively select favourable outcomes |
| **Statistical significance** | The probability that an observed result is not due to chance, expressed as a p-value below a threshold (typically 0.05) |
| **Minimum detectable effect (MDE)** | The smallest true difference between variants that an experiment has sufficient sample size to detect reliably |
