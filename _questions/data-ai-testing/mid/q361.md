## The Intent
The interviewer is checking whether you can own the full testing scope for a data pipeline, not just write a few row count checks. At mid level, they expect you to design with depth — covering both the happy path and the failure modes that cause real incidents in data systems.

## The STAR Script
**Situation:** My team owned a pipeline that ingested raw clickstream events from S3, enriched them with user profile data from a Postgres database, aggregated them into hourly session summaries, and loaded the results into Snowflake. The pipeline had been running for six months with no formal test suite — issues were discovered when analysts noticed anomalies in dashboards.

**Task:** I was asked to design and implement a test suite that would catch pipeline defects before they affected downstream consumers.

**Action:** I structured the suite across five layers. Layer 1 — unit tests on transformation functions: I extracted the core Python transformation logic (session boundary detection, event deduplication, enrichment join) into pure functions and wrote pytest tests covering normal inputs, null handling, duplicate events, and out-of-order timestamps. Layer 2 — integration tests with fixture data: I created a set of known input records and expected output records, ran the full pipeline against them in a local Docker environment, and asserted the output matched exactly. This tested stage-to-stage data flow end to end. Layer 3 — schema contract tests: Using Great Expectations, I defined an expectation suite on the Snowflake output table — column presence, types, not-null constraints, and value ranges for session duration and event count. These ran after every pipeline execution. Layer 4 — data quality assertions: Row count reconciliation (output sessions ≥ 80% of input session-starts — accounting for legitimate incomplete sessions), uniqueness on session ID, and referential integrity checks between the sessions table and the user profile table. Layer 5 — idempotency test: I ran the pipeline twice against the same input and asserted that the output was identical — verifying that re-runs due to failures would not create duplicate records.

**Result:** The suite caught three issues in the first month: a transformation bug introduced by a refactor (caught by unit tests), a schema change in the upstream S3 event format (caught by contract tests), and a duplicate session ID bug triggered by a specific event ordering edge case (caught by the idempotency test). All were fixed before analysts noticed.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can describe the main types of pipeline test and write basic assertions against known outputs |
| **Mid** | Designs and implements a layered test suite covering unit, integration, contract, quality, and idempotency testing — integrated into CI/CD |
| **Senior** | Owns the pipeline testing framework across multiple pipelines, defines reusable patterns and standards, and introduces producer/consumer contract testing at organisational level |

## Key Terms
| Term | Definition |
|------|------------|
| **Idempotency** | The property that running a pipeline multiple times on the same input produces the same output — essential for safe re-runs after failures |
| **Schema contract test** | A test asserting that data conforms to a defined structure — column names, types, and constraints — at a pipeline stage boundary |
| **Referential integrity** | A data quality constraint requiring that foreign key values in one table correspond to existing records in another |
| **Fixture data** | A controlled, known dataset used as input in integration tests, enabling deterministic output assertions |
| **Row count reconciliation** | A check comparing the number of input records to output records to detect unexpected data loss or duplication |
| **Event deduplication** | The process of removing duplicate records from a dataset, typically using a unique event identifier |
| **Stage boundary** | The point between two pipeline stages where data is passed — a natural location for contract tests and quality checks |
