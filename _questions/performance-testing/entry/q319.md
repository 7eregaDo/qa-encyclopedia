## The Intent
The interviewer is checking whether you understand that a test with no think time does not simulate real users — it simulates a script hammering the server as fast as possible. Recognising the difference between realistic and synthetic workload is a marker of maturity in performance testing.

## The STAR Script
**Situation:** I was handed a 10-VU k6 script to run as a quick sanity check before a release. The script had no think time configured.

**Task:** Before running it, I needed to assess whether the results would be meaningful and, if not, how to improve them without a full redesign.

**Action:** I calculated the problem: with a five-step journey averaging 200 ms per request, 10 VUs with no think time would generate approximately 50 requests per second. Adding a realistic 3-second think time between steps reduced that to roughly 5 requests per second — which matched our analytics data showing an average of 4–6 requests per second per active user session. I added a `sleep(randomBetween(2, 4))` call between steps and documented the rationale.

**Result:** The test produced a throughput figure that matched production analytics, making the results credible. The team updated the script template to include think time as a default.

## Key Terms
| Term | Definition |
|------|------------|
| **Think time** | A simulated delay between virtual user actions representing the time a real user spends reading or interacting with a response |
| **Pacing** | The interval between successive iterations of a user journey for a single VU, controlling how frequently the journey repeats |
| **Request rate** | The number of HTTP requests generated per second; determined by VU count, think time, and server response time combined |
| **Workload model** | A description of the types of user journeys, their relative frequency, and the expected concurrency level — the specification a performance test is designed to replicate |
| **Randomisation** | Adding variability to think times (e.g., a random value within a range) to avoid synchronisation effects where all VUs hit the server simultaneously |
| **Synchronisation effect** | An artificial load spike caused by all virtual users completing a think time pause at exactly the same moment and submitting requests simultaneously |
| **Session duration** | The total time a simulated user spends in the system from first request to last, influenced by think time and journey length |
