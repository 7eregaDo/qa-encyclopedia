## The Intent
The interviewer is checking whether you understand the fundamental mechanism by which performance tools generate load. Without grasping what a virtual user is, you cannot reason about why 200 VUs behaves differently from 200 real users, or why think time matters.

## The STAR Script
**Situation:** During a knowledge-sharing session, a developer asked me to explain what "200 virtual users" actually meant — they assumed it was equivalent to 200 browser windows all clicking at once.

**Task:** I needed to explain the concept accurately, including the important differences between virtual users and real users, so that the team could interpret test results correctly.

**Action:** I explained that a virtual user is a scripted thread that sends HTTP requests in sequence with configurable pauses (think time). Unlike a real browser, a VU does not render CSS or JavaScript by default — it sends and receives HTTP traffic. I demonstrated with a short JMeter script: one VU hitting the homepage, pausing two seconds, then hitting a search endpoint, looping for the duration of the test. I also explained that 200 VUs with no think time would produce far more requests per second than 200 real users, and why that matters for realistic test design.

**Result:** The developer understood that VU count and think time together determine realistic throughput, and the team updated our scripts to include appropriate think times.

## Seniority Shift
| Level | Expected Depth |

| **Entry** | Can define a virtual user, explain think time, and understand that VU count alone does not equal realistic load |

| **Senior** | Designs user journey models from production telemetry and manages VU distribution across multiple test scenarios simultaneously |

## Key Terms
| Term | Definition |
|------|------------|
| **Virtual user (VU)** | A simulated user thread that executes a scripted sequence of requests against a target system |
| **Think time** | A delay inserted between actions in a VU script to simulate the time a real user spends reading or interacting with a page |
| **Thread group** | In JMeter, the configuration unit that defines how many VUs run, how they ramp up, and for how long |
| **Iteration** | One complete execution of the scripted user journey by a single VU |
| **Ramp-up period** | The time taken to bring all VUs to full concurrency; prevents an artificial thundering herd at test start |
| **Steady state** | The phase of a test where VU count is held constant, producing stable throughput data |
| **Scenario** | A scripted sequence of steps representing a specific user journey (e.g., browse, add to basket, checkout) |
