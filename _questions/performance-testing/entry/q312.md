## The Intent
The interviewer is checking whether you can distinguish between performance test types rather than treating them as interchangeable. Knowing which test to run for a given risk is a fundamental skill — choosing the wrong type produces data that does not answer the question being asked.

## The STAR Script
**Situation:** When onboarding to a new project, I was handed a JMeter script labelled "performance test" and asked to run it before a release.

**Task:** Before running it, I needed to understand what question it was actually answering and whether that was the right question for our release risk.

**Action:** I examined the script: it ramped to 200 concurrent users and held for ten minutes — a load test. I documented the four types in a one-page reference: load (expected volume), stress (beyond expected volume until failure), spike (sudden burst then return), and soak (sustained normal load over hours). I confirmed that a load test was correct for this release because we were launching a marketing campaign and needed to validate expected traffic — not find a breaking point.

**Result:** The test ran with a clear scope and success criteria. The reference document was adopted by the wider team and pinned to the project wiki.

## Key Terms
| Term | Definition |
|------|------------|
| **Load test** | Simulates the expected number of concurrent users to verify the system meets performance targets under normal conditions |
| **Stress test** | Applies load beyond the expected maximum to identify the system's breaking point and failure behaviour |
| **Spike test** | Introduces a sudden, sharp increase in load to assess how quickly the system responds and recovers |
| **Soak test** | Runs the system at a sustained normal load for an extended period to detect memory leaks, connection pool exhaustion, or gradual degradation |
| **Ramp-up** | A gradual increase in virtual user count at the start of a test to avoid an artificial spike at initialisation |
| **Think time** | A simulated pause between user actions that models realistic user behaviour rather than hammering the server at maximum speed |
| **Breaking point** | The load level at which the system begins to fail — returning errors, timing out, or crashing |
