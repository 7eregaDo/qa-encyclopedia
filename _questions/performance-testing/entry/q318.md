## The Intent
The interviewer is checking whether you understand that performance testing is only as useful as the environment it runs in. Running a load test on a developer laptop or an under-provisioned staging server produces numbers that are not actionable — and can give false confidence or false alarms.

## The STAR Script
**Situation:** My team ran a load test in the shared development environment and reported that the system handled 200 VUs comfortably. Three weeks later, the production release under real user load caused response times to spike above 10 seconds.

**Task:** I was asked to investigate why the test had not predicted the production issue.

**Action:** I compared the two environments. The dev environment had 2 vCPUs and 4 GB RAM; production had 8 vCPUs and 32 GB RAM per node — but crucially, dev also shared a database with three other teams' services, meaning the load test had contended for a database that production did not share. Additionally, the dev data set had 500 records; production had 14 million. I documented the discrepancies and made the case for a dedicated performance environment with production-representative data volumes.

**Result:** The team secured a dedicated staging environment provisioned to 50% of production spec (the agreed minimum). Subsequent test results correlated much more closely with production behaviour.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that environment spec and data volume affect results and can articulate why testing in dev/shared environments is unreliable |
| **Mid** | Assesses environment parity as part of test planning, documents known gaps, and adjusts interpretation of results accordingly |
| **Senior** | Defines environment standards for the performance programme, negotiates infrastructure budget, and implements production shadowing or traffic replay as alternatives where dedicated environments are impractical |

## Key Terms
| Term | Definition |
|------|------------|
| **Environment parity** | The degree to which a test environment matches production in hardware, configuration, data, and network characteristics |
| **Performance environment** | A dedicated environment provisioned specifically for performance testing, isolated from development and functional test activity |
| **Data volume** | The quantity of records in the database; low data volumes can mask performance problems caused by large table scans or inefficient queries |
| **Shared environment** | An environment used by multiple teams or services simultaneously, introducing noise into performance results due to uncontrolled concurrent activity |
| **Production shadowing** | A technique where a copy of live production traffic is replayed against a test environment to achieve realistic load without synthesising VU scripts |
| **Scaled-down environment** | A test environment provisioned at a fraction of production capacity, with results extrapolated — requires documented assumptions about linear scaling |
| **Noise** | Variability in test results caused by uncontrolled external factors such as other workloads running on the same hardware |
