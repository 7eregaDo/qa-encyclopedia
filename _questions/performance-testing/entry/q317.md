## The Intent
The interviewer is checking whether you understand that poor performance always has a cause — it does not happen randomly. They want to see that you can move from "the system is slow" to "this specific component is the constraint", which requires correlating test metrics with system-level observations.

## The STAR Script
**Situation:** During a load test at my first performance testing role, response times were climbing steeply once VU count exceeded 80, but the application servers showed only moderate CPU usage.

**Task:** I needed to identify which component was the actual constraint, so the team could fix the right thing.

**Action:** I checked the metrics layer by layer. Application CPU: 55% — not the issue. Database CPU: 92% — flagged as a candidate. I then queried the slow query log and found a single un-indexed foreign key lookup being called on every product page request. I also checked database connection pool metrics and found the pool was exhausted at peak load, causing requests to queue.

**Result:** The DBA added the missing index and the connection pool limit was raised from 20 to 50. A re-run at 100 VUs showed average response time drop from 4.2 s to 1.1 s. The bottleneck had been the database, not the application tier — and only looking at system metrics alongside test metrics revealed that.

## Key Terms
| Term | Definition |
|------|------------|
| **Bottleneck** | The single constrained resource that limits overall system throughput |
| **CPU saturation** | A condition where processor usage is sustained near 100%, causing request queuing and increased latency |
| **Connection pool exhaustion** | A state where all available database or service connections are in use, causing new requests to wait or fail |
| **Slow query** | A database query that takes disproportionately long to execute, often due to missing indexes or inefficient joins |
| **APM (Application Performance Monitoring)** | Tooling such as Datadog, New Relic, or Dynatrace that traces individual transactions through the system to pinpoint latency sources |
| **Thread starvation** | A condition where all worker threads in a thread pool are busy, causing incoming requests to queue |
| **Horizontal scaling** | Adding more instances of a component to distribute load — effective for stateless tiers, but ineffective if the bottleneck is in a shared resource like a database |
