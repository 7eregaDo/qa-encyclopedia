## The Intent
The interviewer is checking whether you understand that microservices introduce performance complexity that monolithic systems do not have — network hops, serialisation overhead, service mesh latency, and cascading failures. They want to see that you adapt your approach rather than applying a monolithic test strategy.

## The STAR Script
**Situation:** Our platform had recently migrated from a monolith to a microservices architecture comprising 14 services. Post-migration, end-to-end response times were 40% slower than the monolith, but no single service looked slow in isolation.

**Task:** I was asked to identify where the latency was accumulating across the service mesh.

**Action:** I structured the investigation in three layers. First, I ran isolated load tests against each of the five services in the critical checkout path to establish per-service baselines. Each service responded within 100–200 ms at the expected load — individually acceptable. Second, I used distributed tracing (Jaeger) to trace a complete checkout request across all services and measure the time spent in each hop, including serialisation and network transit. I found that the order service called the inventory service synchronously three times per request — 3 × 180 ms = 540 ms of serial network overhead. Third, I ran an end-to-end load test at 100 VUs and confirmed the cumulative latency pattern. The fix was to batch the three inventory calls into one.

**Result:** After the batching change, end-to-end p95 dropped from 2.9 s to 1.3 s. I documented the three-layer testing approach — unit service, distributed trace, end-to-end — as the standard for microservices performance work.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that microservices have multiple services and that each contributes to end-to-end latency |
| **Mid** | Tests at multiple layers (service isolation, trace analysis, end-to-end), uses distributed tracing to identify latency accumulation, and communicates findings across service teams |
| **Senior** | Designs the performance testing and observability strategy for the full service mesh, defines inter-service SLOs, and drives architectural decisions such as synchronous vs asynchronous communication patterns |

## Key Terms
| Term | Definition |
|------|------------|
| **Service mesh** | The infrastructure layer managing service-to-service communication in a microservices system, often including Istio or Linkerd |
| **Distributed tracing** | A technique that tracks a single request across multiple services, recording the time spent at each hop to enable end-to-end latency analysis |
| **Latency accumulation** | The compounding of response time as a request passes through multiple services in sequence, each adding its own processing and network transit time |
| **Synchronous call** | A service-to-service call where the caller waits for the response before proceeding — blocking the request thread for the duration |
| **Asynchronous messaging** | A communication pattern where services communicate via a message queue or event bus, decoupling the caller from the responder and avoiding synchronous blocking |
| **Service isolation test** | A performance test targeting a single microservice independently, to establish its baseline capacity separate from inter-service effects |
| **Jaeger / Zipkin** | Open-source distributed tracing systems that collect and visualise request traces across microservice architectures |
