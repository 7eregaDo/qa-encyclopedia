## The Intent
The interviewer is checking whether you understand that a cached system and an uncached system behave very differently under load, and that ignoring cache state is one of the most common sources of misleading performance results. At mid level, they expect you to design tests that account for this explicitly.

## The STAR Script
**Situation:** Two consecutive load test runs on our product catalogue service produced dramatically different results: the first run showed p95 response times of 3.8 s, the second showed 0.4 s. The team was confused about which result to trust.

**Task:** I needed to explain the discrepancy and design a test strategy that produced consistent, interpretable results.

**Action:** I investigated the caching stack: the application used Redis to cache product data with a 10-minute TTL, and Nginx cached full HTTP responses for 5 minutes. The first run hit a cold cache — Redis was empty, every request hit the database. By the second run, Redis and Nginx caches were warm from the first run's traffic. Both results were technically correct, but they answered different questions. I defined two explicit test scenarios: cold-cache (Redis flushed, Nginx cache cleared before test start — representing a deployment or cache eviction event) and warm-cache (cache pre-populated with a cache-warming script before test start — representing steady-state production). Each scenario had its own pass/fail thresholds.

**Result:** The team now ran both scenarios and reported them separately. The cold-cache result drove infrastructure sizing decisions (worst case); the warm-cache result validated steady-state SLA compliance.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can explain what caching is and that it affects performance test results |
| **Mid** | Controls cache state deliberately as part of test setup, designs cold and warm-cache scenarios, and interprets each result in its appropriate context |

## Key Terms
| Term | Definition |
|------|------------|
| **Cache warm** | A state where the cache has been populated with recently accessed data, producing fast response times as requests are served from memory rather than the origin |
| **Cache cold** | A state where the cache is empty, causing all requests to hit the origin system — producing response times representative of the uncached worst case |
| **TTL (Time to Live)** | The duration for which a cached item is considered valid before it expires and must be refreshed from the origin |
| **Cache hit rate** | The proportion of requests served from cache versus the origin; a high hit rate reduces origin load significantly |
| **Cache warming** | The deliberate pre-population of a cache before a test run, by replaying representative requests, to replicate steady-state production cache behaviour |
| **CDN (Content Delivery Network)** | A geographically distributed network of servers that caches and serves static or dynamic content closer to end users |
| **Cache invalidation** | The process of removing or expiring cached data when the underlying data changes, ensuring users receive fresh content |
