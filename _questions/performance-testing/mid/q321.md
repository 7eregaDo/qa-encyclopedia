## The Intent
The interviewer is checking whether you can move from "we should do some performance testing" to a structured, agreed plan that produces actionable results. At mid level, they expect you to own this artefact end to end — not wait for someone to hand you a script and a server.

## The STAR Script
**Situation:** My team was building a new real-time notifications feature expected to send push events to up to 10,000 connected WebSocket clients simultaneously. No performance testing had been scoped.

**Task:** I was asked to own the performance test plan for the feature before it went to production.

**Action:** I structured the plan across six sections. Objectives: validate that the system can sustain 10,000 concurrent WebSocket connections with message delivery latency under 500 ms. Scope: the notification service and its downstream message broker; out of scope: mobile client rendering. Workload model: derived from product analytics showing peak concurrency at 8,200 users on the previous notification release, with a 20% headroom buffer giving the 10,000 target. Test types: load test (10,000 VUs, 15-minute steady state), spike test (ramp from 2,000 to 10,000 in 30 seconds), soak test (5,000 VUs over 2 hours). Environment: dedicated staging at production parity. Exit criteria: all load test metrics within SLA, error rate below 0.5%, no memory growth trend during soak. I circulated the plan for sign-off from the engineering lead and product manager before execution.

**Result:** The plan was approved within a day. The spike test revealed message broker queue depth spiking to unsustainable levels under the burst scenario — a finding the team would not have looked for without the plan explicitly including a spike test.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands the purpose of a test plan and its key components |
| **Mid** | Authors the plan independently, derives the workload model from real data, and secures stakeholder sign-off before execution |
| **Senior** | Owns the performance strategy across multiple features and teams, standardises the planning artefact, and integrates it with the broader release governance process |

## Key Terms
| Term | Definition |
|------|------------|
| **Workload model** | A specification of user journeys, concurrency levels, and request distribution that defines the load a test will apply |
| **Exit criteria** | Pre-agreed conditions that determine when a test is considered passed or failed |
| **In scope / out of scope** | An explicit boundary declaration that prevents scope creep and focuses test effort on the highest-risk components |
| **Headroom buffer** | An additional percentage added to the expected peak load target to account for growth or measurement uncertainty |
| **Sign-off** | Formal approval from key stakeholders (engineering, product, operations) that the plan is agreed before test execution begins |
| **SLA (Service Level Agreement)** | The performance thresholds the system must meet, against which exit criteria are defined |
| **WebSocket** | A full-duplex communication protocol over a single TCP connection, commonly used for real-time features and requiring specific performance test tooling support |
