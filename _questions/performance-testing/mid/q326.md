## The Intent
The interviewer is checking whether you can distinguish between server-side and client-side performance concerns, and select the appropriate tools and metrics for each. Conflating the two leads to test designs that miss the actual user experience problem.

## The STAR Script
**Situation:** Our team had a solid API load testing practice using k6, but users were reporting that the application "felt slow" even when API response times were within SLA. The disconnect needed investigating.

**Task:** I was asked to assess the client-side performance of the web front end to determine whether the slowness was a browser-side problem.

**Action:** I used Google Lighthouse and WebPageTest to measure the UI performance of the three highest-traffic pages. The API calls returned in under 800 ms, but the Total Blocking Time on the dashboard page was 2.3 seconds — caused by a large, unoptimised JavaScript bundle being parsed on each page load. Largest Contentful Paint was 4.1 seconds, well above the 2.5-second Core Web Vitals threshold, because a hero image was not lazy-loaded and blocked the render. These were client-side issues invisible to our k6 API tests. I documented the two testing dimensions as complementary: API load testing (server capacity), UI performance testing (perceived user experience).

**Result:** The front-end team optimised the JS bundle and added lazy loading. LCP improved to 1.9 seconds and user complaints dropped. The team formalised a two-track performance practice: API load tests in CI, monthly Lighthouse audits on key pages.

## Key Terms
| Term | Definition |
|------|------------|
| **Server-side performance** | The time the server takes to process a request and return a response, exclusive of network transit and client rendering |
| **Client-side performance** | The time the browser takes to receive, parse, render, and make interactive the content returned by the server |
| **Largest Contentful Paint (LCP)** | A Core Web Vital measuring the time until the largest visible content element in the viewport is rendered |
| **Total Blocking Time (TBT)** | A measure of how long the main browser thread is blocked by JavaScript execution, delaying user interaction |
| **Real User Monitoring (RUM)** | The collection of performance data from actual users' browsers in production, capturing true client-side experience |
| **Lighthouse** | A Google open-source tool that audits web page performance, accessibility, and best practices using simulated browser conditions |
| **WebPageTest** | An online tool that measures and analyses web page load performance across real browsers and network conditions |
