## The Intent
The interviewer is checking whether you can operationalise performance testing — moving it from a manual, pre-release activity to a continuous, automated quality gate. At mid level, they want to see that you have done this, understand the practical constraints, and can make sensible decisions about what to run and when.

## The STAR Script
**Situation:** Our team ran performance tests manually, only before major releases. Between releases, performance regressions went undetected until a developer raised a complaint or a user filed a ticket.

**Task:** I was asked to integrate a lightweight performance check into our GitHub Actions pipeline so that regressions were caught at merge time.

**Action:** I chose k6 for its JavaScript syntax and native CI integration. I created a smoke-level performance script: 10 VUs, 60-second duration, covering the three highest-traffic endpoints. I set threshold assertions directly in the k6 script — p95 response time under 2 s, error rate under 1% — so the tool returned a non-zero exit code on failure, which GitHub Actions treated as a pipeline failure. I added the job to the PR merge workflow, running against a dedicated lightweight staging environment. For the nightly build, I kept the full load test (100 VUs, 10-minute run) as a separate scheduled job to avoid slowing down PR feedback loops.

**Result:** Within two weeks, the pipeline caught a regression introduced by a middleware change that added 800 ms to every authenticated request. It was caught and fixed before it reached the release branch. The team accepted the additional 90-second CI step without friction because the value was immediately visible.

## Key Terms
| Term | Definition |
|------|------------|
| **Pipeline gate** | An automated check that blocks the pipeline from proceeding if a defined condition is not met — in performance testing, a metric threshold breach |
| **Threshold assertion** | A pass/fail condition defined in the test script (e.g., p95 < 2000ms) that causes the tool to return a failure exit code if breached |
| **Smoke-level performance test** | A short, low-VU performance run designed for fast feedback in CI — validates that no severe regression exists without the cost of a full load test |
| **Scheduled job** | A CI/CD pipeline job triggered on a time basis (e.g., nightly) rather than on every commit — suitable for longer-running performance tests |
| **Exit code** | The numeric value returned by a process on completion; zero indicates success, non-zero indicates failure — used by CI/CD systems to determine whether to proceed or halt |
| **Feedback loop** | The time between a code change being made and a developer receiving test results; short feedback loops are essential for effective CI/CD |
| **Environment provisioning** | The automated or manual creation of a test environment for each pipeline run; a key practical challenge for performance test CI integration |
