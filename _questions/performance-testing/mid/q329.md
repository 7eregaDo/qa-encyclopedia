## The Intent
The interviewer is checking whether you can close the loop between testing and fixing — a performance finding that is poorly communicated gets deprioritised or misunderstood. At mid level, they expect you to take ownership of the defect report rather than pointing developers at a raw results file.

## The STAR Script
**Situation:** A load test on our reporting module failed — the p95 response time was 6.8 s against a 3-second threshold. I needed to communicate this to a backend team I had not worked with before.

**Task:** I needed to file a defect report that was specific enough for the team to investigate without needing to re-run the test or ask me for clarification.

**Action:** I structured the report with five components. Title: "Reporting module: p95 response time 6.8 s at 50 VUs — SLA breach (threshold: 3 s)." Context: test conditions (50 VUs, 10-minute steady state, warm-cache state, staging environment at production parity). Metrics: full percentile table and throughput graph from the test run, annotated with the threshold line. Correlation: server-side CPU was 34% (not a constraint); database slow query log showed a report generation query taking 4.1 s on average, with full table scans on the `report_events` table (276 million rows, no partition). Reproduction: a curl command to replicate the slow query directly, plus the k6 script and test data CSV attached. I assigned it P1 (blocks release) and tagged the database team.

**Result:** The database team responded within two hours, added a date-range partition to the table, and the query dropped to 280 ms. Re-run showed p95 at 1.4 s — passing. The report structure was adopted as the team's performance defect template.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can identify a threshold breach and describe which metric failed |
| **Mid** | Produces a complete, evidence-based defect report with correlated system data, reproduction steps, and a severity assessment — enabling developers to act without re-investigation |
| **Senior** | Manages the triage of multiple concurrent performance findings, facilitates cross-team root cause analysis, and tracks remediation to closure with regression verification |

## Key Terms
| Term | Definition |
|------|------------|
| **Defect report** | A structured document recording a test failure — including what failed, under what conditions, by how much, and the evidence supporting a probable cause |
| **Severity / priority** | Severity describes the impact of the issue (e.g., SLA breach); priority describes urgency of fix (e.g., P1 blocks release) |
| **Reproduction steps** | Instructions that allow a developer to replicate the observed issue without re-running the full load test |
| **Slow query log** | A database feature that records queries exceeding a configurable execution time threshold, used to identify expensive database operations |
| **Table partition** | A database technique that divides a large table into smaller, more manageable segments based on a column value (e.g., date range), improving query performance by limiting the rows scanned |
| **Annotation** | Marking a specific point on a graph or metric to highlight a threshold, event, or finding — improving readability of visual test output |
| **Remediation verification** | A re-run of the performance test after a fix has been applied, to confirm the issue is resolved and no regression has been introduced elsewhere |
