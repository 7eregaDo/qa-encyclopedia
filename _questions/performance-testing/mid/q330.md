## The Intent
The interviewer is checking whether you think about performance holistically — across the full delivery lifecycle, not just the pre-release test phase. At mid level, they expect you to understand that testing without monitoring leaves you blind after release, and monitoring without testing means you only learn about failures after users experience them.

## The STAR Script
**Situation:** After a strong performance testing practice was established in my team, we still experienced a production degradation incident — response times spiked on a Sunday morning when a batch job ran concurrently with peak user traffic. Our load tests had never modelled this combination.

**Task:** I was asked to propose how we could catch this class of issue before it affected users.

**Action:** I framed the problem as a gap between two disciplines: our load tests validated discrete scenarios in isolation, but production is messier — background jobs, third-party API slowdowns, and traffic spikes interact in ways that synthetic tests cannot fully anticipate. I proposed two additions: first, a soak test that included a simulated batch job running concurrently, to catch the interaction in staging. Second, production monitoring via Datadog with alerting thresholds: p95 latency alert at 2.5 s (warn) and 4 s (critical), error rate alert at 0.5%. I also proposed anomaly detection on throughput to flag unexpected drops indicating partial failures. I connected the monitoring alerts to our incident runbook so on-call engineers had a clear response path.

**Result:** Two weeks after the monitoring was configured, an alert fired for elevated p95 on a Thursday evening — a third-party payment provider was degraded. The team deprioritised the batch job and the issue resolved without a user-visible outage. Without monitoring, it would have been reported by a user the following morning.

## Key Terms
| Term | Definition |
|------|------------|
| **Production monitoring** | Continuous observation of a live system's performance metrics, availability, and error rates using telemetry from real user traffic |
| **Synthetic monitoring** | Scheduled, scripted requests made to production endpoints to simulate user journeys and detect availability or latency issues without waiting for real user traffic |
| **Anomaly detection** | Automated identification of metric behaviour that deviates from historical norms, triggering alerts for unexpected degradation |
| **Alerting threshold** | A metric value that, when breached, triggers a notification or incident — set based on SLOs to provide time to respond before an SLA is breached |
| **Incident runbook** | A documented response guide for a specific alert or incident type, enabling on-call engineers to diagnose and remediate quickly |
| **Observability** | The degree to which a system's internal state can be inferred from its external outputs — achieved through metrics, logs, and traces |
| **Continuous feedback loop** | The iterative cycle of testing, monitoring, finding, fixing, and re-testing that sustains performance quality across the full software lifecycle |
