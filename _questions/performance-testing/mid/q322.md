## The Intent
The interviewer is checking whether you can read a performance report critically, not just run a test and hand the output to a developer. At mid level, they expect you to triage results, correlate metrics across layers, and arrive at a specific, evidenced finding.

## The STAR Script
**Situation:** After a load test run on our search service, the average response time (1.4 s) was within SLA, but the product manager flagged that users were still complaining about slowness in production.

**Task:** I needed to investigate whether the test results were genuinely representative and whether there was a hidden issue the average was masking.

**Action:** I shifted my analysis from averages to percentiles. The 50th percentile was 0.9 s — well within SLA. But the 95th percentile was 5.2 s and the 99th was 9.8 s — both far above the 3-second threshold. The average looked acceptable because the majority of requests were fast, but a significant tail of slow requests was dragging the user experience. I correlated the slow requests with a specific query pattern: searches with more than three filter combinations. I then checked the APM trace for those requests and found a missing composite index on the search table. I filed a defect with the query pattern, the missing index, and the percentile data as evidence.

**Result:** The index was added in the following sprint. A re-run showed the 95th percentile drop to 1.8 s and the 99th to 2.6 s. User complaints ceased after the fix deployed to production. The analysis process — percentile-first, then correlation, then root cause — became the team's standard approach.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can read a summary report, identify which metrics are outside threshold, and describe what they mean |
| **Mid** | Analyses percentile distributions, correlates test metrics with system resource data, isolates root causes, and produces evidenced defect reports |
| **Senior** | Designs the analytical framework, sets up automated anomaly detection, and uses trend analysis across releases to identify systemic degradation |

## Key Terms
| Term | Definition |
|------|------------|
| **Percentile distribution** | A statistical view of response times showing the value at or below which a given percentage of requests fall, revealing tail latency that averages conceal |
| **Tail latency** | The response time experienced by the slowest fraction of requests (e.g., the 95th or 99th percentile); often more representative of real user experience than the average |
| **Correlation** | The process of comparing test metrics (e.g., slow response times) with system metrics (e.g., high CPU, slow queries) to identify a causal relationship |
| **APM trace** | A detailed record of a single request's journey through the system, showing time spent at each component — invaluable for isolating which layer caused the latency |
| **Composite index** | A database index covering multiple columns simultaneously, used to accelerate queries that filter on more than one field |
| **Outlier** | A data point that falls significantly outside the distribution; in performance testing, outliers can indicate transient infrastructure issues or specific request patterns that perform poorly |
| **Mean vs median** | The mean (average) is skewed by outliers; the median (50th percentile) is more robust — both should be reported, not just the mean |
