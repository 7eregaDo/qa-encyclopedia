## The Intent
The interviewer is checking whether you understand that performance test results are only valid if the data the system operates on is realistic. A database with 500 rows will behave very differently from one with 50 million — and naive test data strategies produce misleading results.

## The STAR Script
**Situation:** Our load test results consistently showed response times 60% faster than production. After ruling out environment differences, I suspected the database data volume was the issue.

**Task:** I needed to investigate and resolve the data volume problem without using real customer data in the test environment.

**Action:** I confirmed the hypothesis: the test database had 12,000 user records and 80,000 orders; production had 4.2 million users and 31 million orders. A table scan that was negligible at 12,000 rows was catastrophic at 31 million. I wrote a data generation script using Faker (Python) that produced synthetic but structurally realistic records — matching the production schema, value distributions (e.g., order status ratios, product category spread), and foreign key relationships. I seeded the performance environment with 5 million users and 38 million orders. I also parameterised the VU scripts to draw from a CSV of 50,000 unique user credentials, ensuring each VU authenticated as a different user rather than hammering a single cached session.

**Result:** Re-running the load test with the new data set produced p95 results within 8% of production — a credible match. Three previously undetected slow query patterns surfaced immediately.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that data volume affects results and can explain why testing on a small data set is risky |
| **Mid** | Generates representative synthetic data at production scale, parameterises VU scripts, and manages data state (setup and teardown) across test runs |
| **Senior** | Designs the data management strategy for the performance programme — including anonymisation pipelines from production, automated reseed scripts, and data governance for compliance |

## Key Terms
| Term | Definition |
|------|------------|
| **Synthetic data** | Artificially generated data that mirrors the structure, volume, and value distribution of production data without containing real user information |
| **Data parameterisation** | The practice of supplying each virtual user with unique input data (e.g., different credentials, search terms) to prevent artificial caching and ensure realistic query diversity |
| **Data seeding** | The process of populating a test database with a predefined data set before a test run to ensure consistent, representative starting conditions |
| **Cache warming** | The deliberate pre-population of application or database caches before a test run, to replicate the cache-warm state typical of production |
| **Data isolation** | Ensuring that test data does not interfere with other environments and that test runs do not corrupt each other's data state |
| **Value distribution** | The realistic spread of data values across categories (e.g., 70% completed orders, 20% pending, 10% cancelled) — matching production ratios prevents query plans from being unrealistically optimal |
| **Data teardown** | The cleanup of data created during a test run, restoring the environment to a known state for the next run |
