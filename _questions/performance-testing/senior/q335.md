## The Intent
The interviewer is checking whether you can adapt your performance testing approach to systems where the standard HTTP request/response model does not apply. Event-driven architectures are increasingly common, and a tester who only knows synchronous load testing will miss the performance risks specific to messaging systems.

## The STAR Script
**Situation:** Our order processing platform migrated from synchronous REST calls to an event-driven architecture using Apache Kafka. The existing k6 HTTP load tests were no longer relevant — they validated the producer API but said nothing about whether consumers could keep up with event volume at scale.

**Task:** I needed to design a performance test approach for the Kafka-based pipeline that validated both producer throughput and consumer processing lag.

**Action:** I designed a three-part test. First, producer throughput: I used the Kafka producer performance tool to generate 50,000 events per second against the order topic and measured end-to-end producer latency (time from produce call to broker acknowledgement). Second, consumer lag under load: I used Kafka's consumer group offset monitoring (via Burrow) to track consumer lag while producers were active at peak rate. I instrumented each consumer with OpenTelemetry to measure per-message processing time from consume to downstream write. Third, backlog recovery: I stopped consumers for 10 minutes while producers continued (simulating a consumer outage), then restarted and measured how long it took consumer lag to return to zero — the recovery throughput test. I set SLOs for each: producer p95 latency under 50 ms, consumer lag under 10,000 messages at steady state, backlog recovery under 5 minutes.

**Result:** The backlog recovery test revealed that consumers were idempotent but not parallelised — a single consumer partition per topic meant recovery was single-threaded and took 34 minutes to clear a 10-minute backlog. The team increased partition count from 1 to 12 and added consumer instances. Recovery time dropped to 3 minutes.

## Key Terms
| Term | Definition |
|------|------------|
| **Consumer lag** | The difference between the latest offset produced to a Kafka topic and the current offset consumed by a consumer group — a measure of how far behind the consumer is |
| **Producer throughput** | The rate at which a producer can write events to a message broker, measured in events or bytes per second |
| **Partition** | A Kafka topic subdivision that enables parallel consumption; increasing partitions allows more consumers to process messages simultaneously |
| **Backlog recovery** | The process of a consumer group catching up to the current producer offset after a period of downtime or reduced throughput |
| **Idempotency** | The property of an operation that produces the same result regardless of how many times it is executed — essential for safe message reprocessing in async systems |
| **Burrow** | A Kafka consumer lag monitoring tool from LinkedIn that tracks consumer group offsets and alerts on lag growth |
| **End-to-end latency** | In an async system, the total time from an event being produced to the business effect of that event being realised — spanning broker, consumer, and downstream processing |
