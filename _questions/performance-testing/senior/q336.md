## The Intent
The interviewer is checking whether you can shift performance accountability from QA-as-gatekeeper to a shared engineering responsibility. At senior level, the goal is not to catch problems at test time — it is to make performance violations impossible to accidentally ship by building the checks into the tools developers use every day.

## The STAR Script
**Situation:** Our web front end had accumulated significant performance debt — page weight had grown from 800 KB to 3.2 MB over 18 months as features were added with no weight discipline. No one had been explicitly responsible for performance, and by the time it was noticed, fixing it required a dedicated quarter of effort.

**Task:** I was asked to design an approach that would prevent this class of accumulation from happening again, across both front-end and API layers.

**Action:** I established performance budgets at two levels. Front-end: JavaScript bundle size under 250 KB (gzip), LCP under 2.5 seconds on a simulated 4G connection, Total Blocking Time under 200 ms. API: p95 response time under 1.5 seconds at the standard load profile, Apdex score above 0.9. I then embedded enforcement in three places. In the PR pipeline: Lighthouse CI ran on every front-end PR and blocked merge if any budget was breached, with a comment on the PR showing the delta from the budget. In the API pipeline: k6 smoke tests with threshold assertions ran on every backend PR merge. In the weekly team ritual: a Grafana dashboard was added to the team's Monday review meeting showing budget adherence trends over the past 7 days. I made the budget document public in the team wiki, with the rationale for each threshold so that developers understood the "why" rather than experiencing it as an arbitrary constraint.

**Result:** In the six months following implementation, bundle size reduced from 3.2 MB to 1.1 MB as existing debt was paid off incrementally under the new discipline. Zero new budget violations reached staging — all 14 attempted violations were caught and resolved at PR stage.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can define what a performance budget is and give an example |
| **Mid** | Implements budget checks in CI for a specific metric type and communicates violations to developers |
| **Senior** | Defines budgets across multiple metric dimensions, embeds enforcement in the developer workflow at PR stage, builds a visibility ritual, and manages the cultural change required for shared ownership |

## Key Terms
| Term | Definition |
|------|------------|
| **Performance budget** | A set of maximum allowed values for performance metrics that the team commits to maintaining across all releases |
| **Lighthouse CI** | A tool that runs Google Lighthouse audits in a CI pipeline and can fail builds when performance scores or metric values breach configured thresholds |
| **Bundle size** | The total byte weight of JavaScript delivered to the browser — a primary driver of parse time and Total Blocking Time |
| **Apdex score** | A standardised 0–1 performance satisfaction score based on the ratio of satisfactory, tolerable, and frustrating requests — useful as a single composite budget metric |
| **Performance debt** | Accumulated degradation in performance caused by incremental feature additions, each individually small but collectively significant |
| **PR gate** | An automated check that must pass before a pull request can be merged — used to enforce performance budgets at the point of code change |
| **Budget rationale** | The documented business and user experience justification for each threshold — essential for developer buy-in and for managing requests to adjust thresholds |
