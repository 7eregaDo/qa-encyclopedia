## The Intent
The interviewer is checking whether you think beyond happy-path performance validation. Production systems fail partially — a node goes down, a dependency becomes slow, a disk fills. They want to see that you test how the system performs under degraded conditions, not just under ideal ones.

## The STAR Script
**Situation:** Our platform had passed all load tests comfortably, but we experienced a painful incident where the loss of a single database replica during a deployment caused p95 response times to jump from 1.2 s to 18 s for 22 minutes. Our load tests had never simulated this scenario.

**Task:** I was asked to design a testing approach that would have caught this before it reached production.

**Action:** I proposed combining k6 load tests with Chaos Mesh (our Kubernetes-native chaos engineering tool) in a structured experiment. The hypothesis: if one of the two database replicas becomes unavailable during peak load, the application will re-route traffic to the remaining replica and p95 response times will remain under 3 seconds within 60 seconds of failure. The experiment: run a steady-state load test at 150 VUs (peak equivalent); at T+5 minutes, use Chaos Mesh to inject a network partition on one replica pod; observe response time, error rate, and recovery time. The steady state gave us clean baseline data; the chaos injection gave us the degraded-state behaviour. We monitored via Grafana dashboards in real time.

**Result:** The experiment showed that the application did not have connection pool retry logic — it held open connections to the failed replica for 90 seconds before timing out, causing a backlog. The engineering team added retry logic with exponential backoff. A repeat experiment showed recovery within 8 seconds. The experiment design was adopted as a quarterly resilience testing practice.

## Key Terms
| Term | Definition |
|------|------------|
| **Chaos engineering** | The discipline of experimenting on a system by introducing controlled failures to build confidence in its resilience |
| **Hypothesis** | A specific, testable prediction about system behaviour under a defined failure condition — the foundation of a rigorous chaos experiment |
| **Steady state** | A measurable normal behaviour of the system used as the baseline against which degraded behaviour is compared during a chaos experiment |
| **Chaos Mesh** | An open-source Kubernetes-native chaos engineering platform that can inject network faults, pod failures, CPU pressure, and other failure types |
| **Network partition** | A failure scenario where a service or node becomes unreachable due to a simulated network split |
| **Retry logic** | Application code that automatically retries a failed request after a configurable delay, improving resilience to transient failures |
| **Exponential backoff** | A retry strategy where the delay between successive retry attempts doubles, reducing thundering-herd load on a recovering dependency |
