## The Intent
The interviewer is checking whether you can translate test results into infrastructure and business decisions. At senior level, performance testing is not just about finding bugs — it is about producing the data that answers "how much does this cost to run at scale?" and "when will we need to add capacity?"

## The STAR Script
**Situation:** Our SaaS platform was growing at 15% month-on-month and the infrastructure team had no data-driven basis for provisioning decisions — servers were added reactively after production incidents.

**Task:** I was asked to use performance testing to produce a capacity model that would allow the team to provision proactively for the next six months of projected growth.

**Action:** I ran a stepped load test — 50, 100, 200, 400, and 600 VUs — measuring throughput, p95 response time, and CPU/memory utilisation at each step for the three most resource-intensive services. I plotted the throughput curve and identified the saturation point for each service: the order service saturated at around 380 VUs (CPU hitting 92%); the search service saturated at 280 VUs (database connection pool exhausted). I then correlated VU count with our analytics-derived active user definition to convert test VUs into real user equivalents. At current growth rates, the order service would reach saturation in approximately 14 weeks. I produced a two-page capacity report showing current headroom, projected saturation dates, and three infrastructure options (vertical scaling, horizontal scaling, read replica for the database) with estimated cost and complexity for each.

**Result:** The infrastructure team provisioned a read replica and increased the database connection pool limit ahead of the projected saturation point. No capacity-related incident occurred in the following six months. The stepped load test pattern became the standard capacity data-gathering methodology.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that performance tests produce data relevant to infrastructure decisions |
| **Mid** | Runs load tests at multiple levels to characterise system behaviour and identifies saturation points |
| **Senior** | Builds a capacity model from test data, correlates test VUs with real user equivalents, projects saturation timelines, and presents infrastructure options with cost and risk trade-offs |

## Key Terms
| Term | Definition |
|------|------------|
| **Capacity planning** | The process of determining the infrastructure required to sustain a target workload at acceptable performance, typically projecting future demand |
| **Saturation point** | The load level at which a system resource (CPU, memory, connection pool) is fully utilised and throughput cannot increase further without degradation |
| **Throughput curve** | A graph of requests per second plotted against VU count, showing how throughput scales and where it plateaus or collapses |
| **Stepped load test** | A test that increments VU count in discrete stages, holding each level long enough to observe stable throughput and resource utilisation |
| **Vertical scaling** | Increasing the capacity of an existing server (more CPU, RAM) — simpler but has a ceiling |
| **Horizontal scaling** | Adding more instances of a service to distribute load — more complex but theoretically unbounded |
| **Read replica** | A database instance that mirrors the primary and handles read queries, reducing load on the primary and increasing overall read throughput |
