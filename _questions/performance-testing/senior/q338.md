## The Intent
The interviewer is checking whether you can contribute to architecture and infrastructure decisions before a system is built or tested at scale. At senior level, performance engineers are expected to reason analytically about system behaviour, not just empirically — reducing the cost of discovering capacity problems.

## The STAR Script
**Situation:** Our engineering team was debating whether to horizontally scale the payment service (add instances) or vertically scale the database (increase RDS instance size) to handle a projected 3× traffic increase from an upcoming partnership launch. Both options had significant cost implications. There was no time to run exhaustive tests before the architecture decision needed to be made.

**Task:** I was asked to use existing performance data to model the two options and provide a recommendation before the sprint planning session.

**Action:** I applied Little's Law to the current system data: average concurrency (L) = throughput (λ) × average response time (W). From existing load tests: λ = 120 req/s, W = 0.8 s, therefore L = 96 concurrent requests in flight. At 3× traffic (360 req/s), if response time held constant, L would reach 288. I then checked resource utilisation at 120 req/s: the payment service instances were at 45% CPU — well within headroom. The database was at 78% CPU. The model suggested the bottleneck at 3× would be the database, not the application tier. Adding application instances would not help; increasing database capacity or adding a read replica would. I cross-validated by reviewing the slow query log at current peak — the top three queries were read-heavy product lookups, confirming a read replica would deflect significant load. I presented the model and recommendation with explicit assumptions and confidence levels.

**Result:** The team chose a read replica over vertical scaling (lower cost, lower risk). A subsequent load test at 2× traffic confirmed the database was the constraint — matching the model's prediction. The partnership launch handled 2.8× normal peak without incident.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that performance problems can be anticipated analytically, not only discovered empirically |
| **Mid** | Uses existing test data to extrapolate behaviour at higher loads and identifies likely bottleneck candidates |
| **Senior** | Applies queuing theory and Little's Law to produce quantitative predictions, articulates model assumptions and confidence levels, and drives architecture decisions ahead of test execution |

## Key Terms
| Term | Definition |
|------|------------|
| **Little's Law** | A queuing theory result stating that the average number of items in a system (L) equals the average arrival rate (λ) multiplied by the average time spent in the system (W): L = λW |
| **Queuing theory** | A branch of mathematics that models waiting lines (queues) in systems, used to predict response time and resource utilisation under load |
| **Performance modelling** | The use of mathematical or simulation techniques to predict system behaviour at a given load level without requiring full empirical test execution |
| **Extrapolation** | Using data from a known load range to predict behaviour at higher loads — valid within certain bounds but requires stated assumptions |
| **Confidence level** | An explicit statement of the uncertainty in a model's predictions — necessary for stakeholders to correctly weight the recommendation |
| **Read replica** | A database copy that serves read queries, reducing load on the primary write instance |
| **Bottleneck prediction** | Using resource utilisation data at current load to identify which component will saturate first as load increases |
