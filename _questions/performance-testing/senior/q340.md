## The Intent
The interviewer is checking whether you can apply performance testing expertise to a domain that is increasingly common but has characteristics conventional load testing does not anticipate. At senior level, they want to see that you can reason about the unique constraints of GPU-accelerated inference and adapt your methodology accordingly.

## The STAR Script
**Situation:** Our platform launched an AI-powered product recommendation feature served by a Python FastAPI inference service running a PyTorch model on GPU-backed instances. Initial user feedback was positive, but the engineering team had no visibility into how the service would behave at scale — they had only manual tested with single requests.

**Task:** I was asked to design and execute a performance test programme for the inference endpoint before the feature was promoted from 10% to 100% rollout.

**Action:** I identified four dimensions specific to ML inference that the standard test plan needed to cover. First, cold-start latency: the first request after a model load took 4–8 seconds while the model initialised on GPU — I tested this explicitly with a ramp that started from a cold service state, and measured time to first token separately from steady-state throughput. Second, batching behaviour: the service was configured to batch concurrent requests up to a batch size of 16. I tested at 1, 8, 16, and 32 concurrent requests to characterise how throughput and per-request latency changed with batch size — batch 16 gave 3.4× the throughput of batch 1 with only a 40% latency increase. Third, GPU utilisation under load: I instrumented nvidia-smi metrics via a Prometheus exporter to track GPU memory and compute utilisation alongside the k6 load profile. At 40 concurrent users, GPU compute saturated at 98% while CPU remained at 12% — confirming GPU was the constraint, not application logic. Fourth, graceful degradation: I tested what happened when requests exceeded the GPU's capacity — the service queued requests rather than returning errors, so I validated queue depth and confirmed the queue did not grow unboundedly.

**Result:** The test established that the service handled up to 35 concurrent requests within SLA (p95 under 2 s), and that request 36+ entered a growing queue. The engineering team configured a request queue limit of 40 with a 429 response beyond that, protecting the service from unbounded queue growth. The rollout to 100% proceeded without incident.

## Key Terms
| Term | Definition |
|------|------------|
| **Inference endpoint** | An API that accepts input data, passes it through a trained machine learning model, and returns a prediction or generated output |
| **Cold-start latency** | The delay experienced on the first request to an inference service after the model is loaded into memory or GPU VRAM — significantly higher than steady-state latency |
| **Batching** | The practice of grouping multiple concurrent inference requests into a single model forward pass, improving GPU utilisation and overall throughput at the cost of per-request latency |
| **GPU utilisation** | The percentage of GPU compute and memory capacity consumed by the inference workload — the primary resource constraint for ML serving |
| **Throughput-latency trade-off** | The inverse relationship in ML inference between batch size (higher = more throughput) and per-request latency (higher batch = longer wait per request) |
| **nvidia-smi** | NVIDIA's command-line utility for querying GPU status, used to monitor GPU memory usage and compute utilisation during performance tests |
| **Rate limiting (429)** | An HTTP response code indicating the server is refusing a request due to exceeding a configured request rate or queue limit — used to protect inference services from overload |
