## The Intent
The interviewer is checking whether you think about performance as an engineering discipline woven into delivery, not a test phase bolted on at the end. At senior level, they expect you to own the full strategy — tooling, process, environment, gates, and culture — across the teams you influence.

## The STAR Script
**Situation:** I joined an organisation where performance testing was a manual, pre-release activity run by a single QA engineer two weeks before each quarterly release. It routinely found critical issues too late to fix without delaying the release.

**Task:** I was asked to redesign the approach so that performance was validated continuously and releases were no longer blocked by late-stage findings.

**Action:** I designed a four-gate strategy. Gate 1 (requirements): I introduced performance acceptance criteria as a mandatory field in the definition of ready — every user story with a new database query, API endpoint, or high-traffic feature required a stated SLO before development began. Gate 2 (development): I standardised lightweight profiling during development using async-profiler (JVM services) to catch algorithmic inefficiencies before they reached integration. Gate 3 (CI/CD): I implemented k6 smoke tests (10 VUs, 60 s) on every PR merge and a nightly load test (200 VUs, 20 minutes) against staging, with automated threshold gates. Gate 4 (pre-release): A full soak test (production-parity environment, 48 hours) ran in the two weeks before release, with results reviewed in a performance readiness meeting attended by engineering leads. I also introduced a performance retrospective after each release — comparing test results against production monitoring data to calibrate the strategy.

**Result:** Within three quarters, the pre-release soak found zero critical issues — because Gates 1–3 had eliminated them earlier. Release cycle time dropped by 8 days. One team lead credited the definition-of-ready change as the single highest-value improvement because it stopped performance-hostile design decisions before any code was written.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that performance testing fits into the development process, not just pre-release |
| **Mid** | Implements performance gates at one or two pipeline stages and maintains a test plan per feature |
| **Senior** | Designs the end-to-end strategy across all SDLC phases, installs gates at every appropriate checkpoint, and retrospects continuously to calibrate the approach |

## Key Terms
| Term | Definition |
|------|------------|
| **Shift-left performance testing** | Moving performance validation earlier in the development cycle — into design, development, and CI — to detect issues when they are cheapest to fix |
| **Definition of ready** | A team agreement specifying the conditions a user story must meet before development begins — used here to require SLOs before coding starts |
| **Performance readiness review** | A structured gate meeting where test results are reviewed against SLOs by relevant stakeholders before a release is approved |
| **async-profiler** | A low-overhead sampling profiler for JVM applications that identifies CPU and memory hot spots in running code |
| **Soak test** | A long-duration test at sustained load used to detect gradual degradation — memory leaks, connection pool exhaustion, file handle accumulation |
| **Performance retrospective** | A structured review comparing pre-release test results with post-release production monitoring data to identify gaps in the testing strategy |
| **Performance acceptance criteria** | Explicit, measurable performance requirements attached to a user story or feature, evaluated as part of the definition of done |
