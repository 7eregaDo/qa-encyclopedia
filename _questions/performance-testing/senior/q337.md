## The Intent
The interviewer is checking whether you understand that auto-scaling changes the performance testing problem fundamentally — you are no longer just testing whether the system handles a fixed capacity, but whether the scaling mechanism itself works correctly and economically under realistic load patterns.

## The STAR Script
**Situation:** Our platform ran on AWS ECS with auto-scaling configured to add container instances when CPU exceeded 70%. Load tests showed the system handled 500 VUs — but in production, a flash sale caused a 90-second degradation window while scaling caught up. Our tests had not validated scaling behaviour.

**Task:** I needed to redesign the load test to validate the auto-scaling configuration, not just peak capacity.

**Action:** I redesigned the test with four phases specifically targeting scaling behaviour. Phase 1 (baseline): 50 VUs for 5 minutes — one container running, confirm stable p95 under 1 s. Phase 2 (ramp): 50 to 500 VUs over 90 seconds — the spike scenario that caused the production incident. I instrumented Grafana to track ECS task count alongside response time, so I could see exactly when scaling triggered, how many instances launched, and how long before response times recovered. Phase 3 (sustained peak): 500 VUs for 15 minutes at full scale — confirm the scaled state was stable. Phase 4 (scale-in): Ramp back to 50 VUs and observe how quickly instances were terminated and whether premature scale-in caused a second degradation dip. I also measured cost: the test reported total ECS task-minutes consumed, giving the engineering team data to evaluate whether the scaling trigger (70% CPU) was set optimally or whether a lower threshold would reduce the impact window at a modest cost increase.

**Result:** The ramp phase revealed a 75-second window where response times exceeded 4 s — the scaling trigger was correct but instance launch time was the bottleneck. The fix was to set a minimum instance count of 3 (instead of 1) during business hours, keeping a warm baseline and reducing the launch latency impact. The cost data showed this added £180/month — approved immediately given the flash sale revenue at risk.

## Key Terms
| Term | Definition |
|------|------------|
| **Auto-scaling** | A cloud capability that automatically adjusts the number of running instances of a service based on observed metrics (CPU, request count, queue depth) |
| **Scale-out latency** | The time between a scaling trigger being breached and new instances being ready to handle traffic — the degradation window during rapid load increases |
| **Scale-in** | The reduction in instance count as load decreases — can cause a second degradation dip if terminated too aggressively while requests are still in-flight |
| **Scaling trigger** | The metric threshold (e.g., 70% CPU) that initiates a scaling event — can be tuned to trade response time for cost |
| **Warm baseline** | A configured minimum instance count that keeps a floor of ready capacity available, reducing scale-out latency during demand spikes |
| **Task-minutes** | A cost metric for containerised workloads — total running containers multiplied by duration — used to quantify the infrastructure cost of a scaling configuration |
| **Cooldown period** | A configurable delay after a scaling event during which further scaling actions are suppressed, preventing oscillation |
