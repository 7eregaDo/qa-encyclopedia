## The Intent
The interviewer is checking whether you understand that running a load test without observability is like performing surgery in the dark — you can apply load, but you cannot see what the system is doing in response. At senior level, they expect you to architect the tooling that makes performance data meaningful, not just collect it.

## The STAR Script
**Situation:** Our team ran load tests and received JMeter summary reports, but when a test failed we had no visibility into what the system was doing during the failure — no CPU trends, no query timing, no trace data.

**Task:** I was asked to design and implement an observability stack that would allow us to correlate load test activity with system-level behaviour in real time.

**Action:** I designed a four-layer stack. Metrics: Prometheus scraped node exporter (CPU, memory, disk, network) and application-level metrics (JVM heap, connection pool size, request count) every 15 seconds, federated into a central Prometheus instance. Tracing: Jaeger integrated with the application via OpenTelemetry instrumentation — every request emitted a trace with spans for database queries, external API calls, and cache operations. Logs: Fluentd shipped application logs to Elasticsearch, enriched with trace IDs so logs could be correlated with traces. Visualisation: Grafana dashboards with three views — load test overview (k6 metrics via Prometheus), system health (node and JVM metrics), and slow request detail (Jaeger trace links embedded in the dashboard). I also added annotations: k6 sent a Grafana annotation at test start, ramp-up, steady state, and ramp-down, making it trivial to correlate test phases with system behaviour on the dashboard.

**Result:** The first full test run with the stack active identified a connection pool exhaustion event at T+7 minutes that had been invisible before. The annotation feature became the single most praised addition — engineers could see exactly which metric changes coincided with which test phase without cross-referencing timestamps manually.

## Key Terms
| Term | Definition |
|------|------------|
| **Observability** | The degree to which a system's internal state can be understood from its external outputs — achieved through the three pillars: metrics, logs, and traces |
| **Prometheus** | An open-source time-series metrics collection and storage system, widely used for infrastructure and application monitoring |
| **OpenTelemetry** | A vendor-neutral observability framework for generating, collecting, and exporting metrics, logs, and traces from applications |
| **Distributed tracing** | A technique for tracking a request across multiple services, recording the time spent at each component to attribute end-to-end latency |
| **Grafana** | An open-source visualisation and dashboarding tool that queries metrics sources (Prometheus, Elasticsearch, etc.) to produce graphs, heatmaps, and alerts |
| **Grafana annotation** | A vertical marker on a time-series graph that labels a point in time — used to mark test phase boundaries (ramp-up, steady state) for correlation analysis |
| **Log correlation** | The practice of attaching a shared identifier (trace ID or request ID) to both application logs and distributed traces, enabling navigation between them |
