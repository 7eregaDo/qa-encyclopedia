## The Intent
The interviewer wants to assess your systems thinking about the pipeline as a whole — not just whether you can write tests, but whether you can architect the execution strategy. They're looking for knowledge of trade-offs: fast feedback vs. full coverage, parallelisation vs. infrastructure cost, fail-fast vs. comprehensive reporting.

## The STAR Script
**Situation:** At a scale-up I joined, the CI pipeline ran all tests sequentially in a single job. The pipeline took 48 minutes on average. Developers were bypassing it by merging directly to main when under deadline pressure, which had caused two production incidents in the previous quarter.

**Task:** Redesign the pipeline to deliver useful feedback within 10 minutes for the common case, while maintaining full pre-release confidence, and remove the incentive to bypass it.

**Action:** I redesigned the pipeline in three stages. Stage one — fast feedback (target: under 5 minutes): unit tests, linting, and static analysis ran in parallel across multiple agents. A single unit test failure short-circuited the stage and reported immediately, so developers got signal before making coffee. This stage ran on every push. Stage two — integration confidence (target: under 15 minutes): component and contract tests ran in parallel, with the test suite split by module across agents using dynamic test allocation. This stage ran on every pull request. Stage three — pre-release validation (target: under 30 minutes): end-to-end journey tests, performance benchmarks against baseline thresholds, and security scans ran on merge to main. These were allowed to be slower because they ran asynchronously — a merge was permitted by stage two passing; stage three blocked promotion to staging. I also introduced test impact analysis for the unit stage: using code coverage data, the pipeline identified which unit tests were affected by the changed files and ran only those on push, falling back to the full suite on PR. This reduced the median push-to-signal time from 48 minutes to under 4 minutes.

**Result:** Pipeline bypass incidents dropped to zero within six weeks — developers stopped skipping because the pipeline was no longer an obstacle. Median time-to-feedback on a push dropped to 3.8 minutes. The three-stage model was adopted as the company standard for two new services onboarded in the following quarter.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows that CI runs tests automatically; may describe a flat sequential pipeline |
| **Mid** | Separates fast and slow tests into stages; understands parallelisation; advocates for fail-fast behaviour |
| **Senior** | Designs the full staged model, introduces test impact analysis, defines promotion gates per stage, balances infrastructure cost against feedback speed, and measures pipeline health as a quality metric |

## Key Terms
| Term | Definition |
|------|------------|
| **Test Impact Analysis** | A technique that identifies the subset of tests relevant to a given code change, enabling selective test execution rather than running the full suite |
| **Pipeline Stage** | A discrete phase in a CI/CD pipeline with a defined purpose, entry criteria, and set of jobs that must pass before the next stage begins |
| **Fail-Fast** | A pipeline design principle where the first test failure immediately stops execution and reports, rather than waiting for all tests to complete |
| **Dynamic Test Allocation** | The distribution of test files or suites across parallel agents based on historical execution times, ensuring agents finish at roughly the same time |
| **Promotion Gate** | A criterion that must be satisfied before a build artefact is allowed to advance to the next environment or pipeline stage |
| **Parallelisation** | The execution of multiple independent pipeline jobs simultaneously across separate agents to reduce total elapsed time |
| **Static Analysis** | Automated inspection of source code without executing it, identifying issues such as code style violations, security vulnerabilities, and type errors |
| **Pipeline Health** | A set of metrics — such as median duration, pass rate, and flakiness rate — that measure the effectiveness of the CI/CD pipeline as a quality instrument |
