## The Intent
The interviewer wants to know whether you've confronted the combinatorial complexity and operational debt that feature flags introduce. Many teams adopt flags without considering their testing implications. A senior QA should have a systematic approach to both the test coverage problem and the flag hygiene problem.

## The STAR Script
**Situation:** At a B2B SaaS platform, the team used LaunchDarkly extensively — at peak I counted 47 active flags in the codebase. Feature development had slowed because engineers were nervous about flag interactions, and a production incident had been traced to an untested flag combination where two independently safe flags produced unsafe behaviour when both were enabled.

**Task:** Design a strategy that provided adequate coverage of flag states without requiring exponential test growth, and introduce processes to prevent flag accumulation.

**Action:** I approached this in two parts. Coverage strategy: I introduced a flag inventory as part of the definition of done — every new flag required a documented matrix of its states (on/off, or variants) and which tests covered each. For the test suite, I applied a combinatorial testing principle: rather than testing all 2^n combinations, I identified flag pairs with shared code paths using a dependency analysis, then used pairwise testing to cover those interactions. This reduced the required combination tests from unmanageable to a tractable set. Critical paths (checkout, auth, billing) had explicit tests for every active flag state affecting them. For the rest, the default-on and default-off states were the minimum bar. Lifecycle strategy: flags were classified at creation — experiment (short-lived, max 30 days), operational (infrastructure control, longer-lived but documented), and permanent (kill switches). Each experiment flag had a removal date in the ticket. In sprint planning, "flag removal" was treated as a first-class task. I added a CI lint step that warned when a flag had exceeded its expiry date, making accumulation visible.

**Result:** The flag count dropped from 47 to 19 within two quarters through systematic removal. The incident-style flag interaction failure was not repeated. Engineers reported more confidence in releasing because flag states were explicitly covered in the test plan.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests the feature in its enabled state; may be unaware of the disabled path or flag interactions |
| **Mid** | Tests both enabled and disabled paths; documents which flags affect which tests; advocates for flag removal |
| **Senior** | Designs coverage using pairwise or combinatorial techniques, introduces flag lifecycle governance, builds CI enforcement of expiry, and performs dependency analysis to identify dangerous flag pairs |

## Key Terms
| Term | Definition |
|------|------------|
| **Feature Flag** | A configuration mechanism that enables or disables features at runtime without deploying new code, supporting gradual rollout and experimentation |
| **Flag State Matrix** | A document listing all possible states of one or more flags and the test coverage corresponding to each state |
| **Pairwise Testing** | A combinatorial test design technique that ensures every pair of input parameters is tested together at least once, dramatically reducing the number of test cases versus exhaustive combination testing |
| **Flag Interaction** | The situation where two or more flags share code paths such that enabling both produces behaviour not predictable from testing each flag independently |
| **LaunchDarkly** | A feature flag management platform that supports targeting, gradual rollouts, experimentation, and operational flag control |
| **Flag Lifecycle** | The stages a feature flag passes through from introduction (experiment), to operational use, to scheduled removal, governing how long it may exist in the codebase |
| **Definition of Done** | A team agreement specifying the criteria a piece of work must meet before it is considered complete, which can include flag documentation and test coverage requirements |
| **Technical Debt (Flag Debt)** | The accumulated cost of untested, undocumented, or long-lived feature flags that increase system complexity and reduce team confidence |
