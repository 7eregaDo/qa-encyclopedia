## The Intent
The interviewer is asking a meta-question: not just can you write tests, but can you evaluate and improve an existing test suite systematically? This is a hallmark of senior QA thinking — treating the test suite as a product with its own quality criteria. They want to hear about specific metrics, specific interventions, and the reasoning behind both.

## The STAR Script
**Situation:** I inherited a test suite at a platform company that had 1,800 tests and an 87% pass rate in CI. Engineers had low confidence in it — a common phrase was "the tests are just green noise." Post-release defects were still occurring at a rate that the suite should have caught.

**Task:** Assess the suite systematically, identify the root causes of low confidence, and implement a measurable improvement programme.

**Action:** I ran a three-part audit. First, flakiness analysis: I collected CI run data for the previous 30 days and identified tests that had passed and failed on the same commit. 11% of the suite was flaky. I quarantined flaky tests into a separate suite so they no longer polluted the main signal, then triaged them — root causes were async timing issues, shared state, and environment dependency. I fixed or deleted each one over four sprints. Second, mutation testing: I ran Stryker against the unit test suite. The mutation score was 41%, meaning most tests were asserting on outcomes but not actually verifying logic — many were testing that code ran without error rather than that it produced correct results. I ran workshops with engineers to improve assertion quality, targeting the 20% of classes with the lowest mutation scores first. Third, coverage gap analysis: I mapped code coverage against the defect log from the previous six months. Three modules with below-40% coverage had produced 60% of production defects. I prioritised test investment in those modules. I also introduced a test quality scorecard — monthly reporting on flakiness rate, mutation score, coverage of high-risk modules, and mean time for a test failure to lead to a bug fix.

**Result:** The flakiness rate dropped from 11% to under 1% within two quarters. The mutation score improved from 41% to 67%. Post-release defects from the three under-covered modules dropped by 55% over the following two release cycles. The scorecard became a standing agenda item in engineering all-hands.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows what code coverage is; writes tests to pass coverage thresholds |
| **Mid** | Identifies and fixes flaky tests; understands the limits of line coverage; prioritises test investment based on risk |
| **Senior** | Runs mutation testing, conducts flakiness audits using CI history, correlates coverage with defect data, defines a test quality scorecard, and runs improvement programmes with measurable outcomes |

## Key Terms
| Term | Definition |
|------|------------|
| **Flaky Test** | A test that produces inconsistent results (pass or fail) on the same code without any code change, typically caused by timing issues, shared state, or environment dependencies |
| **Mutation Testing** | A test quality technique that introduces small deliberate faults (mutations) into source code and checks whether the test suite detects them, producing a mutation score |
| **Mutation Score** | The percentage of introduced mutations that were detected (killed) by the test suite; a higher score indicates more meaningful assertions |
| **Stryker** | A mutation testing framework for JavaScript/TypeScript and .NET that automates the process of generating and evaluating code mutations |
| **Test Quarantine** | The practice of isolating known flaky tests into a separate suite so they do not block CI while being investigated and fixed |
| **Coverage Gap Analysis** | The process of identifying areas of the codebase with low test coverage and correlating them with defect history to prioritise testing investment |
| **Test Quality Scorecard** | A periodic report of metrics that assess the health and effectiveness of a test suite, used to track improvement over time |
| **Assertion Quality** | The degree to which a test's assertions verify the correctness of specific behaviour, as opposed to merely confirming that code executes without throwing an exception |
