## The Intent
The interviewer wants to see whether you understand the fundamental difference between testing synchronous and eventually consistent systems — and whether you've made concrete design decisions to handle this in practice. Vague answers about "waiting longer" indicate a junior mindset. They want deliberate strategy.

## The STAR Script
**Situation:** I worked on an e-commerce platform that used DynamoDB global tables for multi-region replication. The order confirmation flow wrote to the primary region and read from a replica. Intermittent test failures were occurring because tests asserted on read results immediately after writes, before replication had completed.

**Task:** Redesign the integration test assertions and introduce a broader strategy for testing eventual consistency across the platform.

**Action:** I introduced three practices. First, a polling assertion utility: rather than asserting immediately or adding fixed sleeps, all assertions on eventually consistent reads used a utility that polled on a configurable interval up to a maximum timeout, failing with a descriptive timeout message rather than a false negative. The timeout was set based on the measured 99th percentile replication lag from our monitoring, plus a safety margin. Second, consistency boundary documentation: for each data store and flow, I documented whether reads after writes were strongly or eventually consistent, so engineers knew which assertion strategy to apply. This lived in the team wiki and was referenced in PR templates. Third, conflict resolution tests: for data that could be written concurrently in two regions, I wrote tests that simulated concurrent writes with conflicting values and asserted that the conflict resolution policy (last-writer-wins in our case) was applied correctly and that no data was silently lost. I also added a monitoring alert on replication lag so any degradation was visible before tests began failing.

**Result:** Intermittent consistency-related failures were eliminated. The polling utility was adopted across the team and reduced flakiness in three other test suites that had been using sleep statements. The conflict resolution tests caught a misconfigured policy during a DynamoDB table migration that would have caused silent data loss in production.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that distributed systems can have stale reads; may use sleep statements as a workaround |
| **Mid** | Implements polling-based assertions with timeouts; distinguishes between strong and eventual consistency in tests |
| **Senior** | Designs the polling utility as a shared framework component, documents consistency boundaries, writes conflict resolution tests, and monitors replication lag to inform test timeout calibration |

## Key Terms
| Term | Definition |
|------|------------|
| **Eventual Consistency** | A consistency model guaranteeing that, in the absence of further updates, all replicas of a data item will converge to the same value |
| **Replication Lag** | The delay between a write being committed to a primary data store and becoming visible on a replica |
| **Polling Assertion** | A test assertion strategy that repeatedly checks a condition at intervals up to a maximum timeout, rather than asserting immediately or waiting a fixed duration |
| **Strong Consistency** | A consistency model in which a read always returns the most recently written value, regardless of which replica handles the request |
| **Conflict Resolution** | The mechanism by which a distributed system resolves conflicting concurrent writes to the same data item, such as last-writer-wins or application-level merge logic |
| **DynamoDB Global Tables** | An AWS DynamoDB feature that replicates table data across multiple AWS regions, providing multi-region active-active access with eventual consistency |
| **Consistency Boundary** | The documented point in a system beyond which reads are no longer guaranteed to reflect the latest writes, requiring consumers to account for potential staleness |
| **Compensating Transaction** | A follow-up operation that reverses or corrects the effects of a previous operation when eventual consistency or a failure has produced an incorrect state |
