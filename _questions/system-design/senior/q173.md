## The Intent
The interviewer wants to see how you balance test isolation with integration realism. Systems that depend on external APIs have a recurring tension: mocking too aggressively hides real failures; hitting live APIs makes tests slow, flaky, and cost-incurring. They want a senior answer that shows you've resolved this tension deliberately.

## The STAR Script
**Situation:** I worked on a travel booking platform that integrated with six third-party APIs — payment processors, hotel inventory, and airline availability. Tests against the live sandbox environments regularly caused CI failures due to rate limits, external downtime, and schema changes the third parties introduced without notice.

**Task:** Design a virtualisation strategy that kept the test suite fast and reliable while ensuring real integration issues were caught before production deployments.

**Action:** I implemented a three-tier approach. Tier one: recorded API stubs using WireMock. I wrote a capture mode that recorded real API responses into fixture files, versioned in the repo. Unit and component tests used these stubs — fast, offline, deterministic. Tier two: a weekly contract verification job that called the real sandbox APIs and compared responses against the recorded fixtures, flagging schema drift. This was a scheduled job, not blocking CI, but its failures triggered Slack alerts. Tier three: pre-production smoke tests against live sandbox APIs executed only during the release pipeline, covering the five most business-critical flows. I also introduced an API wrapper layer in the codebase — every third-party API had a single adapter class — so the seam for stubbing was explicit and consistent. This meant onboarding a seventh API followed the same pattern automatically.

**Result:** CI pass rate increased from ~82% to ~98% within a month. The contract verification job caught two third-party schema changes before they caused production incidents. The release smoke suite ran in under four minutes and gave the deployment team confidence before each go-live.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Uses mocks or stubs for external calls; may not distinguish between stub fidelity levels |
| **Mid** | Implements WireMock or similar; separates unit tests from live-API tests by environment; writes wrapper classes |
| **Senior** | Designs the full virtualisation tier, introduces scheduled contract verification against live sandboxes, and builds the seam architecture as a first-class design decision |

## Key Terms
| Term | Definition |
|------|------------|
| **API Virtualisation** | The practice of simulating the behaviour of a real API to enable testing without calling the live service |
| **WireMock** | A tool for stubbing and mocking HTTP-based APIs, supporting both static fixture-based responses and programmable behaviour |
| **Fixture** | A pre-recorded or manually authored API response payload stored as a file and served by a stub during test execution |
| **Contract Verification** | A process that checks whether a real API's current response structure matches the expectations (contracts) established during virtualisation |
| **Schema Drift** | The gradual divergence of an API's actual response format from the expected format, often introduced silently by third-party providers |
| **Adapter Pattern** | A structural design pattern that wraps a third-party interface behind an internal abstraction, creating a clean seam for testing and replacement |
| **Sandbox Environment** | A non-production environment provided by a third-party vendor for integration and testing purposes, typically simulating real behaviour |
| **Smoke Test** | A shallow test suite that verifies the most critical system paths are operational after a deployment, without exhaustive coverage |
