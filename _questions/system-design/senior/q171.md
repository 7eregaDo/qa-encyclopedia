## The Intent
The interviewer wants to assess whether you can think beyond individual service testing and address the systemic risks of distributed architecture — contract drift, cascading failures, observability gaps, and deployment autonomy. They're probing for strategic thinking, not just tool knowledge.

## The STAR Script
**Situation:** At a fintech company I joined, eight microservices had been built by separate squads over 18 months. Each had unit tests, but integration failures in production were increasing — services passed their own suites but broke when deployed together.

**Task:** I was asked to audit the testing approach and design a quality strategy that would scale with the team's planned growth to 20+ services.

**Action:** I mapped failure modes first: contract drift between producer/consumer pairs, shared database coupling, non-deterministic async behaviour, and missing chaos engineering. I then designed a four-layer strategy. Layer one: each service had isolated unit tests and component tests using mocked dependencies. Layer two: consumer-driven contract tests via Pact, enforced in CI so no producer could break a registered consumer. Layer three: integration tests in a dedicated composed environment covering critical user journeys end-to-end, scoped to the five highest-value flows only. Layer four: chaos and resilience tests using a fault-injection harness to verify timeout, retry, and circuit-breaker behaviour under degraded network conditions. I introduced distributed tracing (Jaeger) so failures in the integration layer produced actionable trace IDs rather than opaque errors.

**Result:** Contract-related production incidents dropped by 70% over two quarters. The composed integration suite ran in under 12 minutes because it was scoped to journeys, not exhaustive combinations. Onboarding a new service required following the contract test template — the strategy was self-propagating.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows unit and integration tests exist; may describe testing services individually |
| **Mid** | Understands the need for contract testing and environment management; can implement Pact for a pair of services |

## Key Terms
| Term | Definition |
|------|------------|
| **Consumer-Driven Contract Testing** | A testing pattern where the consumer defines the expectations of a provider's API, and those expectations are verified against the real provider in CI |
| **Pact** | An open-source consumer-driven contract testing framework that generates and verifies contract files (pacts) between service pairs |
| **Component Test** | A test that exercises a single service in isolation, with all external collaborators replaced by mocks or stubs |
| **Composed Environment** | A test environment where multiple real services are deployed together to validate inter-service behaviour without production traffic |
| **Circuit Breaker** | A resilience pattern that stops forwarding calls to a failing downstream service after a threshold is reached, preventing cascading failures |
| **Fault Injection** | Deliberately introducing failures (latency, errors, dropped connections) into a system to verify its resilience behaviour |
| **Distributed Tracing** | A technique for tracking a request as it propagates across multiple services, producing a correlated trace for debugging failures |
| **Contract Drift** | The divergence that occurs when a provider changes its API without the consumer being updated or notified, causing silent integration failures |
