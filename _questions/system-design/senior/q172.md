## The Intent
The interviewer is testing your ability to adapt testing strategy to non-request/response architectures. Event-driven systems introduce timing non-determinism, invisible coupling through event schemas, and failure modes that don't surface until consumers process messages. They want to know you've worked with this and can architect quality into it.

## The STAR Script
**Situation:** I was the senior QA on a platform that migrated a monolith to an event-driven architecture using Kafka. Twelve domain events were introduced in the first phase. The existing test suite was entirely synchronous and integration tests began producing intermittent failures as soon as events were introduced.

**Task:** Redesign the test architecture to handle asynchronous flows reliably and provide the same confidence the synchronous tests had previously offered.

**Action:** I introduced three changes. First, event schema contracts: each event schema was registered in a schema registry, and producer tests validated that emitted events always conformed. Consumer tests verified that services handled all registered schema versions, including backwards-compatible field additions. Second, async integration tests: I replaced polling-with-sleep patterns with a consumer harness that subscribed to the real Kafka topic in a test environment, applied a configurable timeout with a clear failure message on expiry, and asserted on the consumed message payload. This eliminated flakiness caused by arbitrary sleep durations. Third, end-to-end journey tests used an event inspector utility — a test consumer that recorded all events on a topic during a scenario — allowing assertion against the full event sequence, not just the final state. I also documented that tests should never assert on ordering unless ordering was a product requirement, since Kafka provides per-partition ordering only.

**Result:** Intermittent failures dropped from roughly 15% of CI runs to under 1%. The schema registry enforcement caught three producer-side breaking changes before they reached staging. New engineers onboarding to event-driven features had a clear testing template to follow.

## Key Terms
| Term | Definition |
|------|------------|
| **Event-Driven Architecture** | A design pattern where services communicate by emitting and consuming events via a message broker rather than direct synchronous calls |
| **Schema Registry** | A centralised store for event schemas (typically Avro or JSON Schema) that enforces compatibility rules between producers and consumers |
| **Consumer Harness** | A test utility that subscribes to a real or embedded message broker topic and surfaces received messages for assertion, replacing sleep-based polling |
| **Eventual Consistency** | A consistency model in which distributed data stores converge to the same state given sufficient time and no further updates |
| **Event Inspector** | A test-side consumer that records all events published during a test scenario, enabling sequence and payload assertions |
| **Schema Compatibility** | The property that ensures a new version of an event schema can be read by consumers using an older version, preventing breaking changes |
| **Idempotency** | The property of a consumer that ensures processing the same event more than once produces the same result as processing it once |
| **Dead Letter Queue (DLQ)** | A secondary queue where messages that cannot be processed after repeated attempts are sent, enabling error investigation without losing data |
