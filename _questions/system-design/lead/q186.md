## The Intent
The interviewer wants to see programme leadership capability — specifically your ability to manage risk, stakeholder confidence, and team coordination during a high-impact technical change. Many organisations have the right vision for their test infrastructure but fail the migration. They want to know you've navigated this and can do it again.

## The STAR Script
**Situation:** I led a migration from a legacy Selenium-based E2E suite (1,400 tests, running in a fragile on-premises grid) to a modern Playwright-based suite running in cloud infrastructure. The legacy suite was used by 12 squads as part of their release process, and the migration had to happen without blocking any team's ability to release.

**Task:** Plan and execute the migration over two quarters with no release freezes, no regression in quality coverage, and measurable confidence at each milestone.

**Action:** I structured the migration in four phases with explicit go/no-go criteria. Phase one — foundation: stood up the new Playwright infrastructure in parallel, wrote 50 high-value tests covering the five most critical user journeys, and validated them against the live staging environment. No legacy tests were removed. This phase established that the new infrastructure worked and gave early adopter squads something to reference. Phase two — incremental migration: each squad nominated a liaison. Liaisons migrated their squad's legacy tests in priority order — critical path first — using a co-located mapping document that tracked every legacy test to its Playwright equivalent. I held weekly migration office hours. Phase three — parallel run: for 30 days, both suites ran in CI for all squads. A parity dashboard tracked whether the Playwright suite was catching everything the legacy suite caught. Any gap was treated as a blocking issue. Phase four — cutover: squads decommissioned their legacy tests only after achieving parity and receiving sign-off. Legacy infrastructure was decommissioned module by module, not all at once. I communicated progress in the weekly engineering newsletter with a running migration scorecard.

**Result:** Migration completed in 11 weeks, one week ahead of schedule. Coverage parity was achieved across all 12 squads. CI pipeline duration dropped from 48 minutes to 18 minutes. No squad experienced a release block during the migration. The parity dashboard approach was adopted by the infrastructure team for a subsequent database migration.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Migrates individual tests when instructed; follows a migration guide |
| **Mid** | Leads migration for their squad; maps legacy tests to new equivalents; identifies gaps |
| **Senior** | Designs the migration approach for their service area; introduces parallel running; tracks parity |
| **Lead** | Owns the programme-level migration strategy, defines phased go/no-go criteria, coordinates across all squads, manages stakeholder communication, and decommissions legacy infrastructure with zero release impact |

## Key Terms
| Term | Definition |
|------|------------|
| **Parallel Run** | A migration phase in which old and new systems operate simultaneously, with output compared to verify parity before the old system is decommissioned |
| **Parity Dashboard** | A real-time view comparing the defect detection capability of a new test suite against the legacy suite it is replacing, used to confirm readiness for cutover |
| **Migration Liaison** | A squad-level representative responsible for coordinating and executing that squad's portion of a shared infrastructure migration |
| **Cutover** | The point at which a new system takes over fully from the old one, with the legacy system decommissioned |
| **Go/No-Go Criteria** | Explicitly defined conditions that must be met before a migration phase can proceed, preventing premature advancement under schedule pressure |
| **Playwright** | A modern browser automation framework that supports Chromium, Firefox, and WebKit, offering faster execution and more reliable selectors than legacy Selenium-based suites |
| **On-Premises Grid** | A Selenium test execution infrastructure hosted on physical or virtual servers managed internally, as opposed to a cloud-based execution service |
| **Migration Scorecard** | A periodic summary of migration progress shared with engineering stakeholders, showing completion percentage, open issues, and upcoming milestones |
