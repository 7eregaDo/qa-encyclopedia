## The Intent
The interviewer is probing for breadth of thinking: can you extend quality strategy to include the developer experience as a first-class product quality dimension? Platform APIs are a contract with external customers. Breaking them is a production incident that damages trust externally, not just internally. A lead should understand this asymmetry and design for it.

## The STAR Script
**Situation:** I was Head of QE at a company that had an internal platform used by 20 product squads and a public API used by approximately 300 external developer customers. The platform team treated both audiences identically — the same release process, the same changelog, the same communication approach. External developers were being hit by breaking changes with insufficient notice, and support tickets were increasing.

**Task:** Design a quality and release strategy that distinguished between the two audiences and met the standards appropriate to each.

**Action:** I introduced a tiered contract model. Internal APIs were treated as first-party contracts: changes could be coordinated directly with consuming squads, and the migration window could be a sprint. External APIs were treated as third-party contracts: changes required a deprecation period (minimum 90 days), a versioned API path, and a developer communication plan published in the changelog and emailed to registered API consumers. For testing, I introduced an external developer persona test suite — a suite written from the perspective of an external developer using only public documentation, with no knowledge of internal implementation. This suite ran against every release candidate and was maintained by a QE who was explicitly prohibited from accessing internal service code while writing it. For breaking change detection, I introduced a diff check in CI that compared the published OpenAPI specification against the previous version and flagged any breaking changes — removed fields, changed types, new required parameters — as build failures. A breaking change required a manual override with a documented justification.

**Result:** External developer support tickets related to unexpected API changes dropped by 65% within two quarters. The OpenAPI diff check caught four unintentional breaking changes before release in the first six months. The external persona test suite became a hiring artefact — candidates were shown it during interviews as evidence of the team's quality standards.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests API functionality against the specification; may not distinguish internal from external consumers |
| **Mid** | Understands the importance of API versioning and deprecation; writes consumer-perspective tests |
| **Senior** | Implements breaking change detection, versioning strategies, and consumer-driven contract tests for external APIs |
| **Lead** | Designs the tiered contract model for dual audiences, establishes deprecation policy, introduces the external persona test suite, and makes external developer experience a first-class quality dimension in release governance |

## Key Terms
| Term | Definition |
|------|------------|
| **Third-Party Contract** | An API contract with an external consumer over whom you have no direct coordination channel, requiring formal versioning, deprecation notices, and longer migration windows |
| **Deprecation Period** | A defined timeframe during which an old API version or field remains available alongside its replacement, giving consumers time to migrate before removal |
| **OpenAPI Specification** | A machine-readable description of an HTTP API's endpoints, parameters, request/response schemas, and authentication, used for documentation, client generation, and contract validation |
| **Breaking Change** | A modification to an API that is not backwards-compatible and would cause existing consumer code to fail without modification |
| **Breaking Change Detection** | An automated CI check that compares the current API specification against the published version and flags any incompatible changes before they are released |
| **External Developer Persona Test** | A test suite written strictly from the perspective of an external consumer using only public documentation, used to validate that the API is usable and consistent from the outside |
| **API Versioning** | The practice of assigning version identifiers to API contracts so that changes can be introduced on a new version path without breaking existing consumers |
| **Developer Experience (API DX)** | The quality of the experience an external developer has when discovering, integrating, and using a public API, treated as a product quality dimension |
