## The Intent

The interviewer is assessing your ability to design a high-stakes test strategy for an irreversible operation. Data migrations are one of the highest-risk activities in software delivery — mid-level testers must be able to design testing that gives genuine confidence before production execution, not just verify that the script runs without errors. They want to see structured pre-migration validation, post-migration verification, and explicit risk management.

---

## The STAR Script

**Situation:** The engineering team needed to migrate 85,000 customer records from a legacy table structure to a new schema as part of a platform modernisation. The migration involved splitting a single `full_name` field into separate `first_name` and `last_name` fields, converting date formats, and adding a new `account_tier` field derived from historical purchase data. Any error could corrupt customer records.

**Task:** I needed to design a test strategy that validated the migration script thoroughly before it ran on production data, and verified data integrity after migration.

**Action:** I structured the testing across four phases. First, **representative data sampling**: I worked with the data team to extract a 5,000-record sample from production (anonymised) that was stratified to include all edge cases I could identify — names with multiple spaces, names with special characters, names in non-Latin scripts, records with missing or null fields, records with duplicate entries, and records at each date format boundary. I also created 20 specific "golden records" with known, controlled values for precise assertion. Second, **dry-run migration on the sample**: I ran the migration script against the sample in an isolated environment and verified every golden record field by field against the expected transformation. I also ran automated checks across the full 5,000-record sample — null field counts, date format distribution, and account tier assignment counts — comparing against the expected distribution. Third, **rollback testing**: I ran the migration, then the rollback script, then verified the data was restored to its exact pre-migration state. The rollback had to be tested as rigorously as the migration itself. Fourth, **production go/no-go criteria**: I defined measurable thresholds before the production migration — the migration would be paused and investigated if more than 0.1% of records failed transformation, if any golden record produced an unexpected value, or if the migration took more than 2x the estimated duration (indicating a performance issue).

**Result:** The dry-run on the sample found three transformation defects: names with a middle initial (`J. Smith`) were being split incorrectly (first: `J.`, last: `Smith` instead of keeping as-is), names with only a single word (used by some legacy accounts) caused a null reference exception, and dates in a specific legacy format (`DD-MON-YY`) were not being converted. All three were fixed before the production migration. The rollback test confirmed the rollback script worked correctly. The production migration ran within the time estimate with a 0.03% error rate (all edge-case single-name accounts) — below the threshold, documented and resolved post-migration.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Verify that a simple data migration moves records correctly by comparing source and target data for a small sample. |
| **Mid** | Design a four-phase migration test strategy — representative sampling, dry-run validation, rollback testing, and go/no-go criteria — with golden records and automated distribution checks. |
| **Senior** | Design a full production migration programme — zero-downtime migration patterns (dual-write, gradual cutover), live data consistency monitoring during migration, automated anomaly detection, and a post-migration observation period with defined escalation procedures. |

## Key Terms

| Term | Definition |
|------|------------|
| **Data Migration** | The process of moving, transforming, or restructuring data from one schema or system to another. |
| **Golden Record** | A controlled data record with known values used to verify transformation correctness — a precise test fixture for migration validation. |
| **Dry-Run Migration** | Executing the migration script against a representative sample in an isolated environment before running against production data. |
| **Rollback Script** | A procedure for reverting a migration to the pre-migration state — must be tested as rigorously as the migration itself. |
| **Go/No-Go Criteria** | Pre-agreed measurable thresholds that determine whether a production migration proceeds or is paused for investigation. |
| **Data Distribution Check** | An automated verification that the distribution of values in migrated data matches expected patterns — detects systematic transformation errors. |
