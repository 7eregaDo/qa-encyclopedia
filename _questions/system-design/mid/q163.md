## The Intent

The interviewer is assessing your ability to design tests for systems you don't fully control. Mid-level testers are expected to go beyond "test the integration works when the third party is available" — they must test what happens when it isn't, and ensure the application behaves correctly under all realistic third-party conditions. This requires architectural awareness, service virtualisation skills, and contract thinking.

---

## The STAR Script

**Situation:** The application I was testing integrated with three third-party APIs: a payment gateway, a postcode lookup service, and a fraud detection API. Each was critical to the checkout flow. Testing exclusively against the live sandbox services meant tests were dependent on sandbox availability (not guaranteed), could incur costs for every test run, and could not simulate specific error conditions that the services might return in production.

**Task:** I needed to design a test strategy that gave comprehensive coverage of the third-party integrations without being dependent on live service availability and without sacrificing coverage of error and edge case scenarios.

**Action:** I structured the strategy across four areas. First, **service virtualisation for non-production environments**: I used WireMock to create mock servers for all three third-party APIs. Each mock was configured to return realistic, controlled responses for the scenarios I needed to test — successful responses, specific error codes, malformed responses, and timeout simulations. Tests in development and CI ran against the mocks, not the live services. Second, **contract monitoring**: I subscribed to each API provider's changelog and version announcements. I also implemented response schema validation in the integration layer — any response that didn't match the expected schema would log a warning. This gave early warning of API changes. Third, **live service tests in staging**: a small subset of tests ran against the real third-party sandbox in staging — specifically to verify the integration was correctly configured and that the live service was behaving as expected. These were tagged as `@live-service` and excluded from the main CI run. Fourth, **degraded behaviour testing**: using the WireMock mocks, I tested specific failure scenarios: payment gateway returning a 503 (service unavailable), fraud API timing out after 10 seconds, postcode service returning an unexpected error code. I verified the application responded gracefully in each case — displaying appropriate user-facing error messages without crashing or exposing raw error details.

**Result:** The WireMock-based approach gave complete control over third-party responses for CI testing, eliminating flakiness from sandbox instability. The degraded behaviour tests found that the application was exposing raw payment gateway error codes to users ("Error: PGW_DECLINED_05") instead of user-friendly messages. The contract monitoring caught a postcode API version update that changed a response field name — addressed before the API version we were using was deprecated.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Describe testing a third-party integration's happy path and basic error handling. |
| **Mid** | Design a complete third-party dependency test strategy — service virtualisation, contract monitoring, live service validation in staging, and degraded behaviour testing. |

## Key Terms

| Term | Definition |
|------|------------|
| **Third-Party API Dependency** | An external service integrated into the application that is not owned or controlled by the development team. |
| **Service Virtualisation** | Simulating third-party service behaviour with a controllable mock — enables testing without live service dependency. |
| **WireMock** | A popular service virtualisation tool for simulating HTTP services with configurable responses. |
| **Contract Monitoring** | Tracking third-party API changes through changelogs and version announcements — provides early warning of breaking changes. |
| **Degraded Behaviour Testing** | Testing the application's response to third-party failure scenarios — timeouts, errors, and unavailability. |
| **Response Schema Validation** | Automatically checking that third-party API responses match the expected structure — detects unannounced API changes. |
