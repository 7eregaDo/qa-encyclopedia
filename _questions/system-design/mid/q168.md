## The Intent

The interviewer is assessing your practical understanding of microservices infrastructure and its testing implications. Many entry-level testers treat microservices as a collection of APIs without understanding the runtime infrastructure that makes them work. Service discovery failures are a real class of production defect and environment configuration problem — mid-level testers are expected to understand them and account for them in their test strategy.

## The STAR Script

**Situation:** I was setting up a new test environment for a microservices platform running on Kubernetes. The platform had eight services that communicated with each other using Kubernetes DNS for service discovery. When I first deployed the test environment and ran the end-to-end tests, several tests were failing with connection errors — services were unable to reach each other.

**Task:** I needed to understand why service discovery was failing in the test environment and design a test approach that accounted for service discovery as a dependency.

**Action:** I diagnosed the service discovery failures and then designed a test strategy that addressed them. For the **diagnosis**: I inspected the Kubernetes deployments and found that two services had incorrect service names in their environment variables — they were looking for `payment-service` but the Kubernetes service was named `payment-svc`. In a production-like environment with a service registry, the service would have simply not been found. I also found that one service had a health check failing, preventing it from being registered as available — so other services that depended on it were timing out rather than receiving a clear "service unavailable" error. For the **test strategy**: I adopted a three-layer approach to service discovery in tests. First, **full service discovery tests** ran in the staging environment with all services deployed and registered — verifying end-to-end connectivity. Second, **component tests** ran with dependencies replaced by WireMock stubs that responded on predictable local addresses — removing service discovery from the test equation entirely for isolated testing. Third, **service registration tests** explicitly verified that each service registered itself correctly on startup — checking health check endpoints, verifying the service name in the registry, and confirming the service was reachable from a consumer's perspective before running any functional tests.

**Result:** The service registration tests became a prerequisite check before all other tests ran — if service discovery was broken, the tests failed immediately with a clear message rather than producing cryptic connection errors in the middle of a test run. The environment configuration errors were documented and added to the deployment checklist. Test reliability in the new environment improved significantly after making service discovery health an explicit, visible pre-condition.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understand that microservices find each other through service discovery and that test environments need this configured correctly. |
| **Mid** | Design a test strategy that separates service discovery concerns from functional concerns — full discovery tests in staging, stub-based component tests in CI, and service registration verification as a pre-condition. |
| **Senior** | Test service discovery failure modes — registry unavailability, stale registrations, health check flapping, and the impact of service discovery latency on request routing under load. |

## Key Terms

| Term | Definition |
|------|------------|
| **Service Discovery** | The mechanism by which services in a distributed system locate each other at runtime. |
| **Service Registry** | A directory of available services and their network locations — e.g., Consul, etcd, Kubernetes DNS. |
| **Kubernetes DNS** | The built-in service discovery mechanism in Kubernetes — services are discoverable by their service name within the cluster. |
| **Health Check** | An endpoint or probe indicating whether a service instance is healthy and ready to receive traffic. |
| **Service Registration** | The process of a service announcing its availability to the service registry on startup. |
| **Stale Registration** | A service registry entry for an instance that is no longer available — causes failed requests to be routed to dead instances. |
