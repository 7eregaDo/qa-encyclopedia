title: "How do you approach testing a system's failure and recovery behaviour?"
category: system-design
level: mid
theory: "Resilience testing verifies that a system degrades gracefully when components fail and recovers correctly when they come back. It covers three phases: failure detection (does the system identify the failure?), graceful degradation (does the system continue operating in a reduced capacity?), and recovery (does the system restore full functionality when the failed component is available again?). Testing this requires deliberate failure injection — not waiting for failures to happen accidentally."
---

## The Intent

The interviewer is assessing whether you proactively test system resilience rather than assuming failures won't happen. Mid-level testers working on production systems are expected to understand that failure is inevitable and that untested recovery behaviour is a significant production risk. They want to see a structured approach to failure injection and recovery verification — not just happy path testing.

---

## The STAR Script

**Situation:** I was responsible for testing a payment processing service that depended on three downstream services: a fraud detection API, a payment gateway, and an inventory service. Each could potentially be unavailable, and the business required clear, documented behaviour for each failure scenario.

**Task:** I needed to design a test strategy that verified the system's behaviour when each downstream service was unavailable — both what happened during the failure and how the system recovered.

**Action:** I designed a failure injection test suite using WireMock to simulate each downstream service's failure modes. For each service, I tested three failure types. First, **complete unavailability** (service returns connection refused): verified that the payment service returned a user-friendly error rather than a 500, that the user was not charged, and that no partial data was written to the database. Second, **slow response/timeout** (service takes longer than the configured timeout): verified that the timeout was enforced (not indefinitely blocking), the timeout error was handled gracefully, and the user received a clear message about the delay. Third, **error response** (service returns 500 or specific error codes): verified that the error was translated into an appropriate response — a fraud API rejection returned "unable to process payment at this time," not "FRAUD_SCORE_EXCEEDED." For recovery, after restoring each mock to normal operation, I verified that: queued or failed transactions were retried correctly, the system returned to full functionality without requiring a restart, and no data inconsistency had been introduced during the failure window.

**Result:** The failure injection tests found four defects: a timeout on the fraud API was causing the UI to hang indefinitely (no timeout enforced client-side), inventory failures were allowing orders to proceed without inventory checks (fail-open rather than fail-safe), payment gateway 500 errors were being propagated directly to the user as raw error messages, and recovery after fraud API restoration was not retrying transactions that had been queued during the outage. All four were critical defects that would have caused serious production issues.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Describe testing error handling for a single dependency failure — what should the user see when a service is unavailable? |
| **Mid** | Design a structured failure injection test strategy for all downstream dependencies — covering unavailability, timeout, and error response scenarios, plus recovery verification. |
| **Senior** | Design a chaos engineering programme — systematic, automated failure injection across the full system, including network partitions, database failures, and cascading failure scenarios — integrated into CI/CD as a resilience gate. |

## Key Terms

| Term | Definition |
|------|------------|
| **Failure Injection** | Deliberately introducing failures into a system to test its response — the foundation of resilience testing. |
| **Graceful Degradation** | A system's ability to continue operating in a reduced capacity when a component fails — rather than failing completely. |
| **Fail-Safe** | A failure mode where the system defaults to the safest option when uncertain — e.g., blocking an order rather than allowing it without an inventory check. |
| **Fail-Open** | A failure mode where the system defaults to allowing an operation when uncertain — often incorrect for security or data-critical operations. |
| **Circuit Breaker** | A design pattern that stops sending requests to a failing service after a threshold of failures — prevents cascading failures. |
| **Recovery Testing** | Verifying that a system correctly restores full functionality after a failure is resolved — including any queued or deferred work. |
