## The Intent

The interviewer is assessing both security and systems testing awareness. Rate limiting sits at the intersection of functional correctness (does the limit work?), security (does it prevent abuse?), and user experience (is the error helpful?). Mid-level testers are expected to design boundary and threshold tests for rate limits — applying BVA principles to a system-level mechanism rather than a form field.

---

## The STAR Script

**Situation:** I was testing a public REST API that allowed users to make up to 100 requests per minute per API key. The product team needed confidence that the rate limiting was correctly implemented before the API was made publicly available — both that legitimate use wasn't incorrectly blocked and that the limit actually prevented abuse.

**Task:** I needed to design a test suite that thoroughly verified the rate limiting behaviour — at the threshold, above it, and after reset — without over-testing the obvious happy path cases.

**Action:** I applied boundary value analysis to the rate limit threshold and designed tests across five scenarios. First, **below the limit**: send 99 requests within one minute and verify all return 200 responses. This confirmed legitimate usage was not being incorrectly throttled. Second, **at the limit**: send exactly 100 requests within one minute and verify the 100th request returns 200 — confirming the limit is inclusive. Third, **one over the limit**: send 101 requests within one minute and verify the 101st returns HTTP 429 (Too Many Requests) with a `Retry-After` header indicating when the client can retry, and a clear error message body. Fourth, **limit reset**: after receiving a 429, wait for the time window to expire, send one more request, and verify it returns 200 — confirming the counter resets correctly. Fifth, **isolation between API keys**: verify that requests from API key A do not count toward the limit for API key B — rate limits must be per-client, not global. I scripted these tests using a simple Node.js script that sent requests in a controlled loop, recording the response code and timestamp for each.

**Result:** Tests 1–3 all passed. The reset test (test 4) found a defect — after the 429 response, the time window reset was starting from the first request in the window, not from the time the 429 was received. This meant a client that had maxed out early in a window had to wait longer than the `Retry-After` header indicated. The isolation test (test 5) found a critical bug — rate limits were being applied globally per endpoint rather than per API key, meaning all users of the API shared a single 100-requests-per-minute limit. Both defects were severe and would have been immediately apparent to the first real API consumers.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define rate limiting and describe testing requests below, at, and above the defined limit. |
| **Mid** | Apply BVA to rate limit thresholds, test reset behaviour, verify per-client isolation, confirm HTTP 429 response format, and test the Retry-After header accuracy. |
| **Senior** | Test rate limiting under distributed load (multiple clients simultaneously hitting limits), verify rate limiting across multiple service instances (shared counter consistency), test rate limit bypass attempts (header manipulation, IP rotation), and design performance tests characterising throughput at the limit boundary. |

## Key Terms

| Term | Definition |
|------|------------|
| **Rate Limiting** | A mechanism restricting the number of requests a client can make within a defined time window. |
| **HTTP 429 (Too Many Requests)** | The correct HTTP status code returned when a rate limit is exceeded. |
| **Retry-After Header** | An HTTP header returned with a 429 response indicating when the client may retry — must be accurate for correct client behaviour. |
| **Time Window** | The period over which requests are counted for rate limiting purposes — e.g., 100 requests per minute. |
| **Per-Client Rate Limit** | A rate limit applied individually per API key, user, or IP address — as opposed to a global limit shared across all clients. |
| **Counter Reset** | The clearing of the request count at the end of each time window — must occur at the correct time for the limit to behave predictably. |
