title: "How do you test database transactions and what happens when they fail?"
category: system-design
level: mid
theory: "A database transaction is a unit of work that is executed atomically — either all operations within it succeed and are committed, or any failure causes all operations to be rolled back, leaving the database in its pre-transaction state. Testing transactions requires verifying atomicity (partial completion is impossible), isolation (concurrent transactions don't corrupt each other), and the rollback behaviour that restores data integrity when failures occur."
---

## The Intent

The interviewer is assessing your understanding of data integrity under failure conditions. Mid-level testers are expected to verify that systems handle transaction failures correctly — not just that the happy path commits successfully. This requires understanding ACID properties, designing failure-injection tests, and verifying database state after both success and failure scenarios.

---

## The STAR Script

**Situation:** I was testing an order placement feature that involved a multi-step database transaction: deduct inventory for each ordered item, create the order record, create individual order line items, and charge the payment method. All four operations had to succeed together — if any step failed, everything should roll back.

**Task:** I needed to design test cases that verified the transactional integrity — specifically that partial failures did not leave the database in an inconsistent state.

**Action:** I designed tests across three categories. First, **happy path transaction**: place a valid order and verify all four database changes are committed — inventory decremented, order created, line items created, payment charged. I verified by querying the database directly after the transaction to confirm all records existed with correct values. Second, **rollback on failure**: I needed to test that failures at each step triggered a full rollback. I worked with the developer to introduce controlled failure injection — a test flag that caused the payment step to fail after inventory and order records had been written. I verified that after the injected failure: the inventory was restored to its original level, no order record existed, no line items existed, and no payment had been charged. Third, **concurrent transaction isolation**: I tested two simultaneous orders for the same product when only one unit remained in inventory. I executed both orders concurrently from two different sessions and verified that exactly one succeeded and one failed cleanly — not both succeeding with negative inventory, and not a deadlock or data corruption.

**Result:** The rollback tests found a critical defect: when the payment step failed, the order record and line items were being rolled back correctly, but the inventory decrement was not — it persisted. This was caused by the inventory update running in a separate database connection outside the transaction boundary. The concurrent transaction test found that the inventory check was not using a row-level lock, meaning two simultaneous orders could both pass the inventory check before either committed — resulting in overselling. Both were severe data integrity defects caught before the feature reached production.

## Key Terms

| Term | Definition |
|------|------------|
| **Transaction** | A unit of database work executed atomically — either fully committed or fully rolled back. |
| **ACID** | Atomicity, Consistency, Isolation, Durability — the four properties that define a reliable database transaction. |
| **Atomicity** | The guarantee that all operations in a transaction succeed together or all are rolled back — no partial completion. |
| **Rollback** | The reversal of all database changes made during a transaction when any operation within it fails. |
| **Isolation** | The guarantee that concurrent transactions do not interfere with each other — as if they ran sequentially. |
| **Row-Level Lock** | A database mechanism preventing concurrent modification of the same row — required for correct inventory management under concurrent load. |
