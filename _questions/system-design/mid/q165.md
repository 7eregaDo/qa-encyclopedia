## The Intent

The interviewer is assessing both systems thinking and security awareness. Authentication and session management in distributed systems involve more failure modes than simple form-based login testing. Mid-level testers are expected to test token lifecycle, cross-service propagation, concurrent session behaviour, and the security edge cases that make authentication bugs among the most dangerous in any system.

---

## The STAR Script

**Situation:** I was testing a microservices platform where authentication was handled by a dedicated Auth Service using JWT (JSON Web Tokens). The JWT was issued on login and included in the `Authorization` header of every subsequent request to any service in the platform. Each service validated the JWT independently rather than making a central auth check on every request.

**Task:** I needed to design a test strategy that covered authentication and session management comprehensively — not just that login worked, but that the JWT was correctly validated across all services, that expiry and revocation worked, and that the security failure modes were properly handled.

**Action:** I organised the test strategy into five areas. First, **successful authentication flow**: login with valid credentials, receive a JWT, use it to access protected endpoints across multiple services, verify all services accepted the token and returned authorised responses. Second, **token validation**: attempt to access protected endpoints with no token (expect 401), an expired token (expect 401 with an appropriate message), a malformed token (expect 401), and a token with a tampered payload (expect 401 — testing that signature validation was in place). Third, **token expiry and refresh**: verify that the access token expired at the correct time, that a refresh token could obtain a new access token before expiry, and that an expired refresh token correctly forced re-authentication rather than silently failing. Fourth, **concurrent sessions**: log in from two different devices simultaneously and verify both sessions worked independently. Then log out from one device and verify only that device's session was invalidated — the other remained active. Fifth, **cross-service propagation**: verify that a user with a role that had access to Service A but not Service B received 403 (not 401) from Service B — confirming that authorisation (not just authentication) was being checked correctly per service.

**Result:** Tests 1–3 all passed. The concurrent session test found a defect — logging out from one device was invalidating all active sessions for that user, not just the logged-out session. This was because the revocation list was keyed by user ID, not by token. A user with multiple legitimate sessions (mobile and desktop) would lose both sessions on a single logout. The cross-service propagation test found that one internal service was returning 401 instead of 403 for authorisation failures — a security disclosure that revealed whether a resource existed rather than just denying access.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Test basic login/logout flows, verify that unauthenticated requests are rejected, and confirm session cookies or tokens are set correctly. |
| **Mid** | Test JWT lifecycle (issuance, validation, expiry, refresh, revocation), concurrent sessions, cross-service propagation, and security edge cases (tampered tokens, signature validation). |
| **Senior** | Design authentication testing for complex identity architectures — OAuth 2.0 flows, OpenID Connect, SAML federation, multi-factor authentication, privilege escalation prevention, and token security under distributed cache invalidation. |

## Key Terms

| Term | Definition |
|------|------------|
| **JWT (JSON Web Token)** | A compact, self-contained token encoding user identity and claims — signed to prevent tampering. |
| **Access Token** | A short-lived JWT used to authenticate requests — typically expires in minutes to hours. |
| **Refresh Token** | A longer-lived token used to obtain new access tokens without re-authentication — must be stored securely. |
| **Token Revocation** | The invalidation of a specific token before its natural expiry — required for logout and security incident response. |
| **401 Unauthorised** | HTTP status indicating the request lacks valid authentication credentials. |
| **403 Forbidden** | HTTP status indicating the request is authenticated but lacks permission to access the resource. |
