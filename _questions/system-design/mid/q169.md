## The Intent

The interviewer is assessing your understanding of feature flags as a testing complication that multiplies the state space of the system. Many testers test only the "feature enabled" state and miss defects in the disabled path or in the targeting logic. Mid-level testers are expected to design comprehensive feature flag test coverage — including interaction testing for multiple concurrent flags.

---

## The STAR Script

**Situation:** The product team was using LaunchDarkly to manage feature flags across the application. At any given time, there were 8–12 active flags — some for gradual rollouts, some for A/B experiments, some as emergency kill switches for high-risk features. A production incident had occurred where a feature appeared correctly for users with the flag enabled, but caused a JavaScript error that broke an unrelated part of the page for users with the flag disabled.

**Task:** I needed to design a test strategy that ensured both flag states were tested, that targeting logic was verified, and that flag combinations were considered — preventing the class of defect that had caused the production incident.

**Action:** I implemented a four-part feature flag testing approach. First, **both-states coverage rule**: I added a team standard that any story involving a feature flag must have test cases for both the enabled state and the disabled state. I added "flag state: enabled" and "flag state: disabled" as required fields in the test case template. Second, **targeting logic verification**: for flags with user-segment targeting (e.g., only enabled for beta users, or only for users in the UK), I tested three user types — a user who should see the flag enabled, a user who should see it disabled, and a user at the edge of the targeting criteria (e.g., a user who registered exactly on the beta cutoff date). Third, **flag combination testing**: I maintained a simple matrix of concurrent active flags and identified combinations that touched overlapping parts of the UI or shared code paths. For these combinations, I wrote interaction test cases verifying that enabling Flag A with Flag B also enabled produced correct behaviour — not just each flag in isolation. Fourth, **flag transition testing**: I tested the experience during a gradual rollout — specifically what happened if a user had the feature enabled, navigated away, and returned during a period when the flag was being rolled back to 0%.

**Result:** The both-states coverage rule immediately improved coverage of the disabled path. The combination testing found two flag interaction defects: Flag A (new navigation) and Flag C (new checkout) both modified the same CSS class in conflicting ways, causing a layout break when both were enabled simultaneously. The flag transition test found that a user's cached feature state was not cleared when the flag was rolled back, meaning some users continued seeing the disabled feature for up to 10 minutes after the rollback.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Test a feature with a feature flag in both the enabled and disabled states. |
| **Mid** | Design a comprehensive feature flag test strategy — both states, targeting logic, flag combinations, transition behaviour, and a team standard for flag coverage. |
| **Senior** | Design a feature flag testing framework — automated flag state injection in CI, flag inventory management, test coverage tracking per flag, and a cleanup process for removing flags after features are fully rolled out. |

## Key Terms

| Term | Definition |
|------|------------|
| **Feature Flag (Feature Toggle)** | A configuration mechanism enabling or disabling features at runtime without deployment. |
| **Gradual Rollout** | Enabling a feature flag for an increasing percentage of users over time — reduces risk of broad exposure to new features. |
| **Flag Targeting** | Logic determining which users see a flag as enabled — by segment, geography, account type, or percentage. |
| **Flag Combination** | The state of multiple concurrent feature flags — testing combinations prevents interaction defects. |
| **Kill Switch** | A feature flag used to quickly disable a problematic feature in production without a deployment. |
| **Flag Transition** | The change in a flag's state during a rollout or rollback — users may experience the transition mid-session. |
