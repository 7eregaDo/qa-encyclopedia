## The Intent

The interviewer is assessing whether you understand distributed data consistency as a concrete testing challenge. Mid-level testers working on modern distributed systems must be able to design test strategies that account for propagation delays, verify convergence, and catch the failure modes specific to eventually consistent systems — not just test synchronous, immediately consistent behaviour.

---

## The STAR Script

**Situation:** I was testing a collaborative document editing feature where multiple users could edit the same document simultaneously. The system used an eventually consistent distributed data store — changes made by one user would propagate to other users' views within seconds, but not instantaneously. My initial test approach asserted the updated content immediately after saving, which caused intermittent failures even when the feature was working correctly.

**Task:** I needed to design a test approach that correctly handled eventual consistency — verifying that the system converged to the correct state without producing false failures during the propagation window.

**Action:** I restructured my testing around three principles. First, **convergence verification instead of immediate assertion**: rather than asserting the state immediately after a write, I polled with a timeout — checking every 500ms for up to 15 seconds whether the expected state had been reached across all nodes. The test passed when convergence was confirmed, not when the write was acknowledged. Second, **read-your-writes testing**: I verified that a user who made a change could immediately read their own update — a common consistency guarantee that systems provide even when cross-user propagation is delayed. If User A saves a change, User A must see it immediately even if User B's view is still propagating. I wrote specific tests for this guarantee by reading back from the same session that made the write. Third, **conflict resolution testing**: I tested concurrent writes — two users saving changes to the same section of a document within milliseconds of each other. I verified that the system resolved the conflict deterministically (using the defined last-write-wins or merge strategy) rather than silently dropping one write or producing corrupted output. I used two browser sessions in Playwright to simulate simultaneous edits.

**Result:** The convergence-based assertion approach eliminated all false failures caused by propagation delay. The read-your-writes tests caught a defect where a user's own edits were not immediately visible after save — the system was routing the read to a replica that hadn't received the write yet. The conflict resolution tests revealed that simultaneous edits to adjacent paragraphs were occasionally merging incorrectly, producing a document with duplicated sentences.

## Key Terms

| Term | Definition |
|------|------------|
| **Eventual Consistency** | A distributed system property where all nodes will eventually reflect the same state, but may temporarily diverge. |
| **Convergence** | The point at which all nodes in an eventually consistent system reflect the same state — the correct test assertion target. |
| **Read-Your-Writes** | A consistency guarantee that a user who makes a write will always see their own update immediately — even in eventually consistent systems. |
| **Stale Read** | Reading data from a node that has not yet received the latest update — a normal, expected state in eventually consistent systems. |
| **Conflict Resolution** | The mechanism for determining the authoritative value when two nodes have received conflicting writes to the same data. |
| **Propagation Delay** | The time taken for a write to replicate across all nodes in a distributed system. |
