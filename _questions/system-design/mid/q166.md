category: system-design
level: mid
theory: "Vertical scaling increases the resources (CPU, memory) of a single server instance. Horizontal scaling adds more instances of a service to distribute load. Vertical scaling is simpler to test — the application runs the same way, just with more resources. Horizontal scaling introduces distributed system complexity — shared state management, session consistency, data synchronisation, and load balancer behaviour must all be tested explicitly."

## The Intent

The interviewer is assessing whether you can anticipate the testing implications of infrastructure decisions. Mid-level testers working on production systems must understand how scaling choices affect their test strategy — specifically that horizontal scaling introduces a class of defects that don't exist in single-instance deployments. They want to see that you can identify these defects and design tests to catch them.

---

## The STAR Script

**Situation:** Our application was being scaled horizontally from one instance to five to handle a planned marketing campaign expected to bring a significant traffic spike. The application had worked reliably at one instance for months, and the team assumed horizontal scaling would be straightforward. I was asked to ensure testing covered the new deployment configuration before the campaign launch.

**Task:** I needed to identify the specific defects that horizontal scaling could introduce and design targeted tests to verify the application behaved correctly across five instances.

**Action:** I identified five horizontal-scaling-specific risks and designed a test for each. First, **session state consistency**: the application stored user session data in memory. With five instances, a user's requests could be routed to different instances. I tested by logging in, adding items to a cart, then making requests that were forced to different instances (using direct instance IPs, bypassing the load balancer). Result: cart data was lost when requests hit a different instance — session state needed to be externalised to Redis. Second, **scheduled job deduplication**: the application had a scheduled job that ran every hour to send reminder emails. With five instances, all five would run the job simultaneously — sending five copies of every reminder email. I tested by observing job execution across instances. Result: confirmed — the job needed a distributed lock mechanism. Third, **file upload handling**: uploaded files were stored on the local filesystem of each instance. A file uploaded to instance 1 was not accessible from instance 2. I tested by uploading a file and attempting to retrieve it via a request routed to a different instance. Result: 404 — confirmed, required shared storage (S3). Fourth, **database connection pool exhaustion**: five instances each maintaining their own connection pool could exceed the database's maximum connection limit. I tested by observing connection count during load. Fifth, **log aggregation**: with five instances, logs needed to be aggregated to diagnose issues — tested by verifying the centralised logging configuration captured events from all instances.

**Result:** All five risks were confirmed as real defects — none had been caught by single-instance testing. The session state, scheduled job, and file storage issues were the most critical, all resolved before the campaign launch. The campaign ran successfully with no quality incidents attributable to scaling.

## Key Terms

| Term | Definition |
|------|------------|
| **Vertical Scaling** | Increasing the resources (CPU, memory) of a single server instance — simpler, no distributed complexity introduced. |
| **Horizontal Scaling** | Adding more instances of a service to distribute load — introduces distributed state management challenges. |
| **Session State Externalisation** | Storing session data in a shared cache (e.g., Redis) rather than in instance memory — required for correct horizontal scaling. |
| **Distributed Lock** | A mechanism ensuring only one instance executes a task at a time across a distributed system — required for scheduled jobs. |
| **Connection Pool Exhaustion** | A state where the number of database connections from all instances exceeds the database's limit — causes connection failures. |
| **Shared Storage** | A storage layer accessible from all instances — required for uploaded files and shared assets in a horizontally scaled system. |
