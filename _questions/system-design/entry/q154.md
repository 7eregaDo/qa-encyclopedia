theory: "A database is a structured storage system for persisting application data. Relational databases (SQL) store data in structured tables with defined schemas and enforce relationships between tables using foreign keys. Non-relational databases (NoSQL) store data in flexible formats — documents, key-value pairs, or graphs — without requiring a fixed schema. For testers, the key differences affect data integrity testing, schema change impact, test data setup, and the types of defects likely to appear at the data layer."

## The Intent

The interviewer is checking whether you understand the data layer of the systems you test. Testers who don't understand databases often miss data integrity defects, struggle to set up test data correctly, and cannot verify that the application is persisting data correctly. They want to see that you understand the basic differences between database types and can connect them to practical testing implications.

---

## The STAR Script

**Situation:** I was testing a product that had recently migrated part of its data storage from a PostgreSQL relational database to MongoDB (a document database) for user preferences and session data. The migration was intended to improve flexibility — user preferences had highly variable structures between different user types.

**Task:** I needed to understand how the different database types affected my testing approach for the migrated data.

**Action:** For the **relational database** (PostgreSQL, still used for orders and accounts): I knew the schema was fixed and enforced — every order record had the same fields, data types were validated by the database, and relationships between tables (orders to customers) were maintained by foreign keys. My test approach included: verifying that new records were created with all required fields, testing that relationship integrity was maintained (an order could not reference a non-existent customer ID), and checking that schema constraints (unique email addresses, not-null fields) were enforced. For the **document database** (MongoDB, used for preferences): there was no fixed schema — a preference record could contain any fields. My test approach shifted: I couldn't rely on schema enforcement at the database level, so I needed to verify that the application layer was validating and sanitising preference data before storing it. I also needed to check that missing or unexpected fields in a preference document were handled gracefully by the application, and that the flexible schema wasn't allowing invalid or dangerous data to be persisted (e.g., storing executable content in a preference field).

**Result:** Understanding the database differences helped me design targeted tests for each storage type. For the NoSQL preferences, I found a defect where the application stored preference keys with inconsistent casing (`"theme"` and `"Theme"` were treated as different preferences) — a class of defect that would have been impossible in a relational database with enforced column names. This was directly attributable to the flexible schema and would not have been found without understanding the storage model.

## Key Terms

| Term | Definition |
|------|------------|
| **Relational Database (SQL)** | A database storing data in structured tables with fixed schemas and enforced relationships — e.g., PostgreSQL, MySQL, SQL Server. |
| **Non-Relational Database (NoSQL)** | A database storing data in flexible formats (documents, key-value, graph) without requiring a fixed schema — e.g., MongoDB, Redis, DynamoDB. |
| **Schema** | The defined structure of data in a database — field names, data types, and constraints. |
| **Referential Integrity** | A database constraint ensuring that relationships between records remain valid — e.g., an order cannot reference a non-existent customer. |
| **Foreign Key** | A field in a relational database referencing the primary key of another table — enforces referential integrity. |
| **Schema Enforcement** | The database's ability to reject data that does not conform to the defined schema — present in relational databases, absent in schemaless NoSQL. |
