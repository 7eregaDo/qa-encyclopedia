## The Intent

The interviewer is checking whether you understand CDN caching as a testing surface — a practical concern that catches many teams off guard after deployments. CDN-related bugs are a common class of post-deployment issue, and testers with CDN awareness can design specific verification steps to catch them before or immediately after release. They want to see practical understanding of how CDNs work and what to test.

---

## The STAR Script

**Situation:** After a deployment that updated the application's JavaScript bundle, some users reported that the application was behaving as if the old version was still running — buttons were in the wrong place and a newly added feature was invisible. Other users saw the correct version. The application server was serving the new code, but many users were receiving the old JavaScript from the CDN cache.

**Task:** I needed to understand how the CDN had caused this inconsistency and design a post-deployment verification approach that would catch CDN cache issues before they affected users.

**Action:** I learned that the application was using a CDN (Cloudflare) to cache static assets. The CDN cached the JavaScript and CSS files at edge nodes around the world. When the deployment updated the files on the origin server, the CDN continued serving the old cached versions to users who were being routed to edge nodes that hadn't refreshed yet. The fix was to either use cache-busting (including a hash of the file content in the filename — so a new file version had a new URL that bypassed the CDN cache) or to explicitly purge the CDN cache after deployment. I added CDN-specific verification to the post-deployment smoke test checklist: verify the JavaScript bundle version hash in the network requests matches the version deployed (indicating the new file is being served), test from a different geographic location using a VPN or browser tool to simulate a user being served from a different CDN edge node, and verify that the CDN cache-control headers are configured to expire appropriately for each asset type.

**Result:** The post-deployment CDN verification caught a stale asset issue in the following release — the CSS had updated but the CDN edge nodes in Asia-Pacific hadn't refreshed because the cache purge had only been applied to European nodes. This was caught in the post-deployment smoke test rather than by user reports. The team also implemented content hashing in the build process, which eliminated the class of stale asset issue entirely for JavaScript and CSS bundles.

## Key Terms

| Term | Definition |
|------|------------|
| **CDN (Content Delivery Network)** | A distributed network of servers caching static assets geographically close to users — reduces latency. |
| **Edge Node** | A CDN server located geographically close to users — serves cached content without reaching the origin server. |
| **Cache Invalidation** | Removing or expiring cached content from CDN edge nodes — required when content is updated. |
| **Cache Busting** | Including a version hash in asset filenames so that updated files have new URLs — bypasses CDN caching automatically. |
| **Cache-Control Header** | An HTTP header specifying how long a CDN or browser should cache an asset. |
| **Origin Server** | The primary server that CDN edge nodes fetch content from when their cache is empty or expired. |
