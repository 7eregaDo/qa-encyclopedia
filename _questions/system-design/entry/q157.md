theory: "A load balancer is a component that distributes incoming requests across multiple instances of a service — preventing any single instance from being overwhelmed and improving availability. For testers, load balancers introduce challenges around session affinity (ensuring a user's requests consistently reach the same instance), state consistency (verifying that data is consistent across instances), and the need to test behaviour when one or more instances become unavailable."
---

## The Intent

The interviewer is checking whether you understand load balancing as a testing surface — not just a DevOps concept. Load balancer-related defects are a real class of production issue, and testers who don't understand load balancing cannot design tests that catch them. They want to see that you understand the key failure modes and can connect them to specific test cases.

---

## The STAR Script

**Situation:** After a new deployment that scaled the application from one to three instances behind a load balancer, users began reporting intermittent bugs — the shopping cart would appear empty despite items having been added, but only for some users and only sometimes. I was asked to investigate.

**Task:** I needed to understand how the load balancer was involved and design tests that could reproduce and verify the issue.

**Action:** I learned that the application was storing shopping cart state in memory on the server — not in a shared database or cache. When a user added an item to their cart, that data was stored on the specific server instance that handled their request. If the next request was routed to a different instance, that instance had no knowledge of the cart. This was a session affinity (or "sticky sessions") problem — the load balancer was not consistently routing each user's requests to the same instance. I reproduced the issue by making two sequential requests from the same session — first adding an item (handled by instance 1), then viewing the cart (handled by instance 2). The cart appeared empty because instance 2 had no cart state. I then designed tests specifically for load-balanced environments: tests that verified session data persisted correctly across multiple requests, tests that checked behaviour when the instance handling a session became unavailable, and smoke tests that verified the application behaved consistently when run across all instances simultaneously.

**Result:** The root cause was confirmed — the application needed to either use sticky sessions (routing all requests from a user to the same instance) or externalise session state to a shared store. The defect was architectural rather than a code bug. My load balancer-aware test approach was documented and added to the pre-deployment checklist for any future horizontal scaling changes.

## Key Terms

| Term | Definition |
|------|------------|
| **Load Balancer** | A component distributing incoming requests across multiple service instances — improves availability and prevents overload. |
| **Session Affinity (Sticky Sessions)** | A load balancer configuration routing all requests from a specific user to the same instance — required when session state is stored in memory. |
| **Horizontal Scaling** | Adding more instances of a service to handle increased load — as opposed to vertical scaling (making one instance more powerful). |
| **State Consistency** | The requirement that data is identical across all instances of a service — violated when state is stored in memory rather than a shared store. |
| **Health Check** | A load balancer mechanism testing whether an instance is healthy before routing traffic to it. |
| **Instance Failover** | The load balancer's behaviour when an instance becomes unavailable — routing traffic to remaining healthy instances. |
