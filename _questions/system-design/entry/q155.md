theory: "Caching is the practice of storing frequently accessed data in a fast, temporary storage layer — reducing the need to repeatedly fetch data from a slower source such as a database or external API. For testers, caching introduces specific failure modes: stale data (the cache serving outdated information), cache invalidation bugs (the cache not clearing when data changes), and cache bypass issues (the application bypassing the cache when it should use it or vice versa)."
---

## The Intent

The interviewer is checking whether you understand one of the most common and subtle sources of production defects that entry-level testers miss. Caching-related bugs are frequently invisible to testers who test only the happy path — the data looks correct when first loaded, and they never verify what happens after an update. They want to see that you actively consider cache behaviour in your test design.

---

## The STAR Script

**Situation:** I was testing a product catalogue feature that displayed product prices. All functional tests passed — the correct price was displayed for every product I tested. After release, customers reported seeing old prices for recently updated products. The prices in the database were correct, but the application was serving cached versions.

**Task:** I needed to understand why caching had caused this defect and how to build cache-aware testing into my standard practice.

**Action:** I discussed the architecture with the developer and learned that product prices were cached in Redis for 60 minutes to reduce database load. The cache was supposed to be invalidated when a price was updated, but the invalidation logic was only triggered through the admin UI — not through the bulk import tool that the business team used for price updates. My testing had not covered the cache layer because I hadn't known it existed. After learning about it, I redesigned my testing approach for any feature involving cached data: verify the data on first load (cache miss — data fetched from source), update the data through the application, verify the updated data is displayed immediately (cache invalidation), wait for the cache TTL and verify the data is still correct (cache refresh), and test updating through all pathways that could change the data (not just the primary UI).

**Result:** The new cache-aware test approach was incorporated into my test design checklist. In the following sprint, I caught a cache invalidation bug in the user profile feature — updating a user's display name through the account settings page invalidated the cache correctly, but updating it through the admin panel did not. The same class of bug, found before release this time.

## Key Terms

| Term | Definition |
|------|------------|
| **Cache** | A fast, temporary storage layer holding copies of frequently accessed data to reduce load on slower sources. |
| **Cache Hit** | A request served from the cache — no database query required. |
| **Cache Miss** | A request not found in the cache — fetches from the source and populates the cache. |
| **Cache Invalidation** | The process of removing or updating a cached entry when the underlying data changes. |
| **TTL (Time to Live)** | The duration a cache entry is kept before expiring and being refreshed from the source. |
| **Stale Data** | Outdated information served from the cache because invalidation did not occur — the most common cache-related defect. |
