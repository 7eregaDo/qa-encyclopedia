## The Intent

The interviewer is checking your understanding of APIs as a testing surface — not just as a thing the front end calls. Entry-level testers with API literacy can test more efficiently, find defects earlier, and communicate more precisely with developers. They want to see that you understand REST conventions, HTTP status codes, and request/response structure well enough to test at the API layer meaningfully.

---

## The STAR Script

**Situation:** I was testing a new user registration feature. The UI tests were passing — the registration form submitted correctly and the success message appeared. A week after release, a customer reported that registering with a non-ASCII character in their name caused a server error.

**Task:** I needed to understand why this had slipped through and how API-level knowledge would have helped me catch it.

**Action:** I investigated by sending the request directly to the registration API using Postman, bypassing the UI. The API endpoint accepted POST requests with a JSON body containing `firstName`, `lastName`, and `email`. When I sent a request with `"firstName": "Ångström"`, the API returned a 500 Internal Server Error — the encoding was being handled incorrectly at the API layer, not the UI layer. The UI had masked this because the browser's form submission was encoding the character in a way that happened to work in my browser but not others. Understanding the API structure meant I could test at the right layer — directly, with controlled inputs — and find the issue more precisely than UI testing allowed. I also learned that a well-designed API should return 400 (Bad Request) for invalid input, not 500 (Internal Server Error) — the 500 was itself a defect indicating unhandled exception rather than graceful error handling.

**Result:** API-level testing with non-ASCII inputs became a standard part of my test approach for any feature involving text input. Understanding HTTP semantics — specifically that 4xx means a client error and 5xx means a server error — made my defect reports more precise and helped developers immediately understand the nature of each failure. The registration defect, once found via API testing, was fixed and a regression test was added at the API layer where it could be verified directly.

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define an API, explain REST basics (HTTP methods, status codes, request/response format), and describe how API testing complements UI testing. |
| **Mid** | Design a comprehensive API test strategy — contract validation, schema testing, authentication, error handling, and performance baseline. |

## Key Terms

| Term | Definition |
|------|------------|
| **API (Application Programming Interface)** | A defined interface for software components to communicate — specifying request format, available operations, and response structure. |
| **REST** | Representational State Transfer — a widely used architectural style for APIs using HTTP methods (GET, POST, PUT, DELETE). |
| **HTTP Status Code** | A numeric code in an API response indicating the outcome — 2xx (success), 4xx (client error), 5xx (server error). |
| **Request/Response** | The fundamental unit of API communication — a structured request sent to an endpoint and a structured response returned. |
| **Endpoint** | A specific URL that an API exposes for a particular operation — e.g., `POST /users` for creating a user. |
| **Payload** | The data sent in an API request body or returned in a response body — typically JSON or XML. |
