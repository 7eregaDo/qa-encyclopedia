category: system-design

theory: "Environment tiers are distinct deployment environments serving different purposes in the software delivery lifecycle. Development environments are used by engineers for active coding and are unstable by nature. Staging environments closely mirror production and are used for integration testing, regression, and release validation. Production is the live environment serving real users. Each tier has different stability characteristics, data constraints, and testing purposes — a tester who treats all environments identically will produce unreliable results and miss environment-specific defects."
---

## The Intent

The interviewer is checking whether you understand the different characteristics and appropriate uses of each environment — a practical knowledge gap that causes real problems in teams. Testers who don't understand environment differences run tests in the wrong environment, draw incorrect conclusions from unstable environments, and miss environment-specific defects. They want to see clear, practical understanding of each tier.

---

## The STAR Script

**Situation:** During my first sprint on a new team, I reported three defects based on testing in the development environment. Two of the three were immediately closed by the developer as "environment-specific" — the dev environment was missing some configuration, and the issues did not exist in staging. I had wasted time on my investigation and the developer's time reviewing the tickets.

**Task:** I needed to understand the characteristics of each environment and adopt the right testing approach for each tier.

**Action:** I mapped each environment's characteristics and appropriate use. **Development environment**: unstable by design — developers were actively pushing code, breaking and fixing things constantly. I learned to use dev for quick smoke checks and exploratory sessions on features in active development, but never to record formal test results or raise formal defects based on dev-only observations. **Staging environment**: configured to closely mirror production — same infrastructure, same configuration, same external service integrations (or realistic mocks). This was my primary testing environment for formal regression, integration testing, and release sign-off. I learned to treat staging as the source of truth for test results. **Production environment**: real users, real data, no test data allowed. I used production only for post-release smoke testing and monitoring — never for functional testing that could affect real customer data. I also learned to understand the differences between environments at my team — for example, our staging used a payment sandbox while production used the live payment gateway. This meant I needed to specifically test payment gateway behaviour in production smoke testing after each release.

**Result:** After adopting environment-appropriate testing practices, my defect reports were almost never rejected as environment-specific. My staging-based testing also caught two environment configuration differences — a feature flag enabled in staging but not dev, and a third-party API key difference — that would have caused production incidents if not caught before release.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Describe the purpose and characteristics of each environment tier and explain which is appropriate for formal testing and defect reporting. |
| **Mid** | Design environment-specific test strategies — what to test in each environment, how to handle environment differences (mocks vs. live services), and how to validate environment configuration as part of the release process. |
| **Senior** | Design an environment management strategy — environment parity assurance, configuration drift detection, ephemeral environment provisioning, and production monitoring as a quality signal. |

## Key Terms

| Term | Definition |
|------|------------|
| **Development Environment (Dev)** | An unstable environment used by engineers during active development — not suitable for formal testing. |
| **Staging Environment** | A stable environment mirroring production configuration — the primary environment for formal regression and release validation. |
| **Production Environment** | The live environment serving real users — used only for post-release smoke testing and monitoring. |
| **Environment Parity** | The degree to which staging matches production configuration — higher parity produces more reliable pre-release test results. |
| **Environment-Specific Defect** | An issue that exists in one environment but not others due to configuration differences — not a real application defect. |
| **Configuration Drift** | The gradual divergence of environment configuration over time — a common source of environment-specific defects. |
