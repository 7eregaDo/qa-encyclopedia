## The Intent

The interviewer is checking whether you see testing as a narrow, UI-focused activity or as a discipline that requires genuine technical understanding of what you are testing. Entry-level testers who understand system design make better decisions about what to test, where to test it, and how to articulate defects precisely. They want to see that you value architectural literacy as a professional competency.

---

## The STAR Script

**Situation:** Early in my career I was testing a feature that displayed a user's order history. I was testing entirely through the UI — entering inputs and checking outputs on screen — and all my tests passed. A production incident later revealed that the order history was being fetched from a cache that was not being invalidated when orders were updated. My tests never caught it because I didn't know the cache existed.

**Task:** I needed to understand not just what the feature did but how it worked architecturally — so I could identify the components involved and design tests that covered the right layers.

**Action:** I started investing time in understanding the systems I tested. For each new feature, I asked the developer: what components does this involve? What external services does it call? Where is data stored and how is it retrieved? Is anything cached? Are there background jobs or async processes? This gave me a mental map of the system — not deep enough to build it, but deep enough to know where it could fail. For the order history feature, if I had known about the cache, I would have added a test case: update an order and verify the displayed history reflects the change immediately — testing cache invalidation, not just data display.

**Result:** My test coverage became significantly more comprehensive after developing system design literacy. I started finding defects at component boundaries — cache invalidation failures, API error handling gaps, race conditions in async flows — that my purely UI-focused testing had missed entirely. Engineers also began treating me as a more credible technical collaborator because I could discuss their architecture, not just report what I saw on screen.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understand enough system design to identify the components involved in a feature and ask the right questions about how it works. |
| **Mid** | Use system design knowledge to identify failure points at component boundaries and design targeted tests for each layer. |
| **Senior** | Evaluate system designs for testability, contribute to architecture discussions, and design test strategies that reflect the full component landscape of a complex system. |

## Key Terms

| Term | Definition |
|------|------------|
| **System Design** | The process of defining a software system's architecture, components, interfaces, and data flows. |
| **Architecture** | The high-level structure of a software system — how components are organised and how they interact. |
| **Component** | A discrete unit within a system — a service, database, cache, message queue, or external API. |
| **Component Boundary** | The interface between two components — a common location for defects, especially in distributed systems. |
| **Technical Literacy** | Sufficient understanding of how a system is built to test it effectively — not implementation knowledge, but architectural awareness. |
| **Testability** | The degree to which a system's design makes it easy to write tests that are meaningful, isolated, and reliable. |
