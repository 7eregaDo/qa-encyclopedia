## The Intent

The interviewer is assessing engineering maturity and long-term thinking. Many QA engineers focus on adding tests but rarely on maintaining or retiring them. At senior level, you are expected to own the health of the test suite as a strategic asset — not just a collection of scripts that grew over time.

---

## The STAR Script

**Situation:** I inherited a Selenium-based E2E suite of 800 tests that had been built over four years by multiple contractors. Pipeline runtime was 3.5 hours, flake rate was 22%, and the team had lost faith in the results — failing tests were routinely ignored because "it's probably just flaky."

**Task:** I needed to restore trust in the suite, reduce pipeline time, and establish a sustainable maintenance model — without stopping feature delivery.

**Action:** I approached it as a technical debt programme with four workstreams running in parallel. **Audit**: I ran three full suite executions and categorised every test: stable and high value, stable but low value, flaky, obsolete (testing features that no longer existed), or duplicate. 180 tests were immediately deleted — obsolete or exact duplicates. **Flake triage**: I quarantined all flaky tests into a separate non-blocking pipeline job and assigned each a 2-week resolution window. Tests not fixed within the window were deleted. This immediately removed 22% noise from the main pipeline. **Architecture refactoring**: the suite had no page object model — selectors were duplicated across hundreds of tests. I introduced a POM layer incrementally, starting with the highest-churn pages. This reduced maintenance effort by approximately 60% on those pages. **Ongoing hygiene**: I introduced a "test debt budget" — for every sprint, 20% of QA capacity was reserved for suite maintenance. I also added a pipeline metric tracking flake rate and test count over time, making debt accumulation visible before it became critical.

**Result:** After 12 weeks, pipeline runtime fell from 3.5 hours to 55 minutes. Flake rate dropped from 22% to 3%. Team trust in test results recovered — failing tests were investigated, not ignored. The suite shrank from 800 to 520 tests but coverage of high-risk areas actually increased.

## Key Terms

| Term | Definition |
|------|------------|
| **Test Technical Debt** | Accumulated cost of poor test design, outdated tests, and deferred maintenance. |
| **Flaky Test** | A test that produces inconsistent results without code changes, eroding trust in the suite. |
| **Test Quarantine** | Isolating flaky tests to a non-blocking pipeline job while they are fixed or retired. |
| **Page Object Model (POM)** | A design pattern that abstracts UI elements into reusable objects, reducing test maintenance cost. |
| **Test Retirement** | The deliberate deletion of tests that are obsolete, duplicated, or no longer provide value. |
| **Debt Budget** | A reserved portion of sprint capacity dedicated to test suite maintenance and improvement. |
| **Flake Rate** | The percentage of test runs that produce inconsistent results — a key health metric for a test suite. |
