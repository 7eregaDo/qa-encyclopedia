## The Intent

The interviewer is testing architectural awareness and the ability to adapt testing strategy to a complex, distributed system. Microservices introduce unique challenges — independent deployability, network dependencies, versioning, and distributed failures — that a monolith-focused strategy does not address. Senior engineers must demonstrate they understand these challenges and have concrete approaches to managing them.

---

## The STAR Script

**Situation:** Our platform migrated from a monolith to a microservices architecture over 18 months. The existing E2E test suite — designed for the monolith — began failing constantly as services deployed independently and interfaces changed without coordinated notice.

**Task:** I needed to redesign the test strategy to be compatible with independent service deployability while maintaining confidence in system behaviour.

**Action:** I restructured the strategy around four layers. **Layer 1 — Unit and component tests**: each service team owned tests for their service's internal logic, run in isolation with mocked dependencies. These were fast, deterministic, and the first line of defence. **Layer 2 — Contract testing with Pact**: instead of relying on integration tests that required all services to be running simultaneously, I introduced consumer-driven contract testing. Each service published a contract defining what it expected from its dependencies. Provider services ran contract verification tests on every build — catching interface breaks before they ever reached a shared environment. **Layer 3 — Integration tests**: for critical service pairs with complex interaction patterns, I maintained a small set of integration tests running in a composed environment. These were explicitly scoped and owned by the team responsible for the integration boundary. **Layer 4 — E2E tests**: I reduced the E2E suite dramatically — from 200 tests to 35 — keeping only the highest-value critical user journeys. These ran nightly rather than on every commit, accepting slower feedback for broader coverage. I also introduced chaos testing for resilience: deliberately killing individual services in staging to verify graceful degradation behaviour.

**Result:** The false failure rate in the pipeline dropped by 80%. Contract testing caught three breaking API changes before they reached staging, each of which would have caused multi-service failures in the old model. The team gained confidence in independent deployability — release frequency increased from weekly to multiple times per day.

## Key Terms

| Term | Definition |
|------|------------|
| **Contract Testing** | Verifying that a service honours the interface contract expected by its consumers. |
| **Consumer-Driven Contract** | A contract defined by the consuming service specifying what it needs from the provider. |
| **Pact** | A popular open-source contract testing framework. |
| **Component Testing** | Testing a single microservice in isolation with its dependencies mocked or stubbed. |
| **Testing Honeycomb** | A model suited to microservices: emphasises integration tests in the middle, with fewer unit and E2E tests. |
| **Chaos Engineering** | Deliberately injecting failures (e.g., service outages, latency) to verify system resilience. |
| **Independent Deployability** | The ability to deploy one microservice without requiring simultaneous deployment of others. |
