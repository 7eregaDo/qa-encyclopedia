## The Intent

The interviewer wants to know whether you treat coverage as a meaningful quality signal or as a vanity metric. Mid-level engineers are expected to understand that 100% code coverage does not mean 100% quality, and to articulate what meaningful coverage looks like in practice.

---

## The STAR Script

**Situation:** A developer told me we had 85% code coverage from unit tests and questioned why we needed further manual testing before release.

**Task:** I needed to explain why code coverage alone was an incomplete picture of test coverage, and what we were still missing.

**Action:** I pulled up the coverage report and showed that the 85% figure measured statement coverage — which confirmed lines of code were executed, but not that they were tested with meaningful assertions or realistic data. I then mapped our test cases against the requirements traceability matrix and found that three acceptance criteria had no corresponding test case at all — 0% requirements coverage in those areas. I also noted that all existing tests used ideal inputs; no boundary or negative scenarios were covered. I proposed a coverage model using three lenses: requirements coverage, risk coverage, and exploratory coverage for untested user journeys.

**Result:** We identified eight missing test cases before release. Two of them uncovered defects. The team adopted a multi-dimensional coverage checklist as part of the sprint exit criteria going forward.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define test coverage and describe what code coverage measures. |
| **Mid** | Distinguish between requirements, code, and risk coverage; explain why each matters and what each misses. |

---

## Key Terms

| Term | Definition |
|------|------------|
| **Requirements Coverage** | The percentage of documented requirements that have at least one corresponding test case. |
| **Code Coverage** | The percentage of code statements, branches, or paths executed during testing. |
| **Statement Coverage** | Every line of code is executed at least once. The weakest form of code coverage. |
| **Branch Coverage** | Every decision point (if/else) is exercised for both true and false outcomes. |
| **Risk Coverage** | The degree to which identified risk areas have been tested. |
| **Traceability Matrix** | A document mapping requirements to test cases to verify coverage completeness. |
