theory: "A Requirements Traceability Matrix (RTM) is a document that maps each requirement to its corresponding test cases, ensuring that every requirement has test coverage and that every test case has a traceable requirement. It provides bi-directional visibility: forward traceability (requirements → tests) and backward traceability (tests → requirements)."
---

## The Intent

The interviewer wants to see that you manage test coverage systematically, not by feel. The RTM is a professional-grade tool that demonstrates accountability — it answers the question "how do you know you've tested everything?" with evidence rather than confidence.

---

## The STAR Script

**Situation:** Approaching the release of a compliance-sensitive financial feature, the product owner asked: "Can you confirm that every acceptance criterion has been tested?" I had 60+ test cases across three testers and needed to answer that question definitively.

**Task:** I needed to provide evidence-based coverage confirmation, not just a verbal reassurance.

**Action:** I built a traceability matrix in a shared spreadsheet: rows were the acceptance criteria from each user story, columns were test case IDs. Each cell was marked: Covered, Partially Covered, or Not Covered. This took two hours to assemble but immediately surfaced two acceptance criteria that had no linked test case — they had been overlooked during test design. I created the missing test cases, executed them, and updated the matrix before sign-off.

**Result:** The release was signed off with full documented evidence of requirements coverage. When an auditor later requested test evidence for the compliance feature, the RTM provided a clear, structured answer in minutes rather than requiring a manual trawl through test reports.

## Key Terms

| Term | Definition |
|------|------------|
| **RTM (Requirements Traceability Matrix)** | A document mapping requirements to test cases to ensure full coverage. |
| **Forward Traceability** | Tracing from requirements → test cases (are all requirements tested?). |
| **Backward Traceability** | Tracing from test cases → requirements (does every test have a valid requirement?). |
| **Orphan Test Case** | A test case with no linked requirement — signals scope creep or a missing requirement. |
| **Uncovered Requirement** | A requirement with no linked test case — a direct coverage gap. |
| **Impact Analysis** | Using the RTM to identify which test cases are affected when a requirement changes. |
