theory: "Exploratory testing is a simultaneous process of test design, execution, and learning where the tester uses skill, intuition, and domain knowledge to investigate the system without predefined scripts. Scripted testing follows pre-written test cases with defined steps and expected results, optimised for repeatability and regression coverage."

## The Intent

The interviewer wants to know whether you treat exploratory testing as a structured professional skill or as "just clicking around." They also want to see that you understand when each approach is appropriate — and that the two are complementary, not competing.

---

## The STAR Script

**Situation:** We had completed full scripted regression on a new onboarding flow — all 45 test cases passed. The release was scheduled for the following morning.

**Task:** I had two hours remaining and no additional scripted tests to run. I decided to use that time for structured exploratory testing rather than signing off early.

**Action:** I used a **session-based exploratory approach**: I defined a charter ("Explore the onboarding flow as a user who makes unexpected choices — skips steps, uses back navigation, pastes unusual data into fields") and time-boxed the session to 90 minutes. I kept a running note of observations and questions. I tested with unusual but realistic inputs: emoji in the name field, a very long company name, navigating backwards mid-flow, and submitting with a slow network connection simulated via browser dev tools.

**Result:** I found two defects that scripted testing had missed: the company name field truncated silently at 50 characters with no error message, and navigating back mid-flow reset previously entered data without warning. Both were fixed before release. Neither would have been discovered by the scripted suite.

## Key Terms

| Term | Definition |
|------|------------|
| **Exploratory Testing** | Simultaneous test design and execution guided by tester learning and intuition. |
| **Scripted Testing** | Execution of pre-written test cases with defined steps and expected outcomes. |
| **Session-Based Testing** | Exploratory testing structured into time-boxed sessions with defined charters. |
| **Charter** | A brief mission statement for an exploratory session (e.g., "Explore the checkout flow under poor network conditions"). |
| **Debrief** | A post-session review documenting what was tested, what was found, and what remains. |
| **SFDIPOT** | A heuristic for structured exploration: Structure, Function, Data, Interfaces, Platform, Operations, Time. |
