## The Intent

The interviewer wants to see that you design tests systematically rather than by gut feel. Mid-level engineers are expected to select the appropriate technique for a given situation and explain their reasoning. Knowing the names is not enough — you need to demonstrate applied judgement.

---

## The STAR Script

**Situation:** I was designing tests for a pricing engine with tiered discount logic: 0% for orders under £50, 10% for £50–£199, and 20% for £200 and above.

**Task:** I needed to design the smallest set of test cases that would give meaningful coverage of the logic without exhaustively testing every possible value.

**Action:** I applied three techniques in combination. First, **equivalence partitioning** — I identified four partitions: below range (e.g., £0), tier 1 (e.g., £75), tier 2 (e.g., £150), tier 3 (e.g., £250), plus an invalid partition (e.g., negative value). One test per partition. Second, **boundary value analysis** — I tested at the exact edges: £49.99, £50.00, £199.99, £200.00. These are where off-by-one errors hide. Third, **decision table testing** — I mapped all combinations of discount tier and payment method to ensure no combination produced an unexpected result. For an onboarding flow, I also used **state transition testing** to verify each step in the multi-page form behaved correctly based on prior state.

**Result:** Using these techniques, 18 test cases covered what would have taken 60+ ad-hoc cases to approximate. Two boundary defects were found at £49.99 and £200.00 — exactly where the technique predicted risk.

## Key Terms

| Technique | Category | Best Used For |
|-----------|----------|---------------|
| **Equivalence Partitioning** | Black-box | Inputs that can be grouped into classes with identical behaviour. |
| **Boundary Value Analysis** | Black-box | Numeric ranges and limits where off-by-one errors are likely. |
| **Decision Table Testing** | Black-box | Complex business rules with multiple input combinations. |
| **State Transition Testing** | Black-box | Systems that behave differently based on current state (e.g., workflows, forms). |
| **Use Case Testing** | Black-box | End-to-end user journeys and real-world scenarios. |
| **Statement Coverage** | White-box | Ensuring every line of code is executed. |
| **Branch Coverage** | White-box | Ensuring every decision path (true/false) is tested. |
| **Exploratory Testing** | Experience-based | Unscripted investigation using tester intuition and domain knowledge. |
| **Error Guessing** | Experience-based | Targeting areas where experience suggests defects are likely. |
