category: qa-theory

theory: "Smoke testing is a broad, shallow test of a new build to verify that its core functionality is stable enough for further testing. Sanity testing is a narrow, focused check on a specific area following a bug fix or minor change, to confirm the fix works without running the full regression suite."
---

## The Intent

The interviewer wants to see that you use the right verification tool at the right time. Conflating these two — or running a full regression when a sanity check would suffice — wastes time. Using a smoke test when a detailed sanity check is needed risks missing a broken fix. Precision here signals testing maturity.

---

## The STAR Script

**Situation:** Our team deployed two builds in the same day: a morning deployment of a new feature release, and an afternoon hotfix for a defect in the promo code module.

**Task:** I needed to verify both deployments efficiently without running a full regression suite twice in one day.

**Action:** For the **morning release**, I ran a smoke test — a 20-minute suite of 15 high-level checks across all major areas: login, search, checkout, account management, and admin panel. The goal was not thoroughness but confidence that nothing was catastrophically broken before the broader testing team began work. For the **afternoon hotfix**, I ran a sanity test — a targeted 10-minute check of the promo code module specifically: valid codes applied correctly, expired codes rejected, edge cases around stacking discounts. No other areas were touched, as the fix was isolated and the morning smoke test had already confirmed broader stability.

**Result:** The smoke test caught a broken navigation link introduced by the morning deployment — resolved before the testing team was blocked. The sanity test confirmed the hotfix resolved the promo code defect without introducing side effects. Full regression was reserved for the end-of-week build, not burned on every deployment.

## Key Terms

| Term | Definition |
|------|------------|
| **Smoke Testing** | A broad, shallow build verification confirming core functionality is stable enough to test. Also called a "build acceptance test." |
| **Sanity Testing** | A narrow, focused check on a specific area after a fix or minor change. |
| **Build Acceptance Test** | Another name for a smoke test — confirming the build is worth investing further test effort in. |
| **Regression Testing** | Full re-execution of the test suite to confirm no existing functionality is broken. |
| **Depth vs. Breadth** | Smoke = broad coverage, low depth. Sanity = narrow coverage, high depth. |
