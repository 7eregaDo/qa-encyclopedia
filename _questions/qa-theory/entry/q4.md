## The Intent

The interviewer wants to confirm you can write and structure test cases properly — this is the most basic practical skill expected of any QA engineer. Vague or incomplete answers signal that you have been testing informally without rigour.

---

## The STAR Script

**Situation:** I was asked to test a registration form for a new web application. There were no existing test cases — just a brief requirements document.

**Task:** I needed to create structured, repeatable test cases that any team member could execute, not just ad-hoc exploratory checks.

**Action:** I wrote test cases covering happy path, boundary values, and negative scenarios. Each case included: a unique ID, a clear title, preconditions, step-by-step actions, test data, and an expected result. For example, one test case verified that submitting an email without an "@" symbol displays the error message "Please enter a valid email address."

**Result:** When a junior tester joined mid-project, they could execute the full suite independently without guidance. Two regressions were caught in a later sprint because the cases were precise enough to detect subtle UI changes.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | List and describe all core components of a test case with a concrete example. |
| **Mid** | Discuss test case design techniques (equivalence partitioning, boundary value analysis) and how to manage test cases in a TMS like TestRail or Zephyr. |
| **Senior** | Address test case maintenance at scale, metrics like test case effectiveness (defects found per case), and balancing formal test cases with exploratory testing. |

---

## Key Terms

| Component | Description |
|-----------|-------------|
| **Test Case ID** | A unique identifier for tracking and referencing the test case. |
| **Title / Objective** | A short description of what the test case verifies. |
| **Preconditions** | The state the system must be in before execution begins. |
| **Test Steps** | Sequential actions the tester must perform. |
| **Test Data** | Specific input values used during execution. |
| **Expected Result** | The exact outcome the system should produce if it behaves correctly. |
| **Actual Result** | Recorded during execution — what the system actually did. |
| **Status** | Pass, Fail, Blocked, or Skipped. |
