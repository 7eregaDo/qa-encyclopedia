## The Intent

The interviewer wants to confirm you understand the two fundamental quality checkpoints in software development. Mixing these up suggests a shallow understanding of the QA discipline. They also want to see whether you can connect abstract definitions to real testing activities.

---

## The STAR Script

**Situation:** While testing a login feature, I was handed a specification document that described exact field lengths, error messages, and redirect behaviour.

**Task:** I needed to both confirm the implementation matched the spec and that the overall login experience actually served the user's needs.

**Action:** First, I ran verification checks — comparing each implemented behaviour against the spec line by line (e.g., "does the password field accept exactly 8–20 characters as written?"). Then I performed validation — I asked whether the spec itself made sense for a real user, noting that the error message "ERR_401" was technically correct per the spec but meaningless to end users. I raised this as a usability defect.

**Result:** The spec was updated to use plain-language error messages. This demonstrated that verification alone is not enough — validation ensures the product is genuinely useful, not just technically compliant.

## Key Terms

| Term | Definition |
|------|------------|
| **Verification** | Checking that the product is built according to its specification. Static or dynamic. |
| **Validation** | Checking that the product meets real user needs. Typically dynamic. |
| **Static Testing** | Verification technique that does not execute code — e.g., reviews, inspections, walkthroughs. |
| **Dynamic Testing** | Testing that involves executing the software — e.g., functional tests, UAT. |
| **V-Model** | A development model that pairs each development phase with a corresponding testing phase. |
