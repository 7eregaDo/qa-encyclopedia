## The Intent

The interviewer is checking your understanding of fundamental test design perspectives. They want to know whether you can choose the right approach for a given situation and whether you understand that both techniques are complementary — not competing.

---

## The STAR Script

**Situation:** I was testing a discount calculation engine where the business rules were well-documented, but the internal implementation was complex with multiple conditional branches.

**Task:** I needed to choose a testing approach that would cover both what users experience and whether the internal logic was sound.

**Action:** I applied **black-box testing** first — using equivalence partitioning and boundary value analysis to test discount thresholds (e.g., 0%, 10%, 20%) purely from the user's perspective, using inputs and checking outputs. I then collaborated with a developer to apply **white-box testing** — reviewing the code paths and writing tests that specifically exercised each conditional branch to ensure full code coverage.

**Result:** The black-box tests caught three incorrect output values. The white-box review revealed an untested branch for negative discount values that would have caused a runtime exception in production.

## Key Terms

| Term | Definition |
|------|------------|
| **Black-Box Testing** | Testing based on inputs/outputs with no knowledge of internal code. |
| **White-Box Testing** | Testing based on internal code structure, logic, and paths. |
| **Grey-Box Testing** | Testing with partial knowledge of the internal structure. |
| **Equivalence Partitioning** | A black-box technique dividing inputs into groups that should behave identically. |
| **Boundary Value Analysis** | A black-box technique testing at the edges of input ranges. |
| **Code Coverage** | A white-box metric measuring what percentage of code paths are exercised by tests. |
