## The Intent

The interviewer wants to confirm you understand one of the most critical and time-consuming activities in QA. They are also listening for whether you understand *why* regressions happen — not just that you know the word. Bonus points for mentioning automation as the scalable solution.

---

## The STAR Script

**Situation:** Our team was releasing a new payment method feature. The developer assured us it was isolated and "wouldn't touch anything else."

**Task:** Despite the reassurance, I insisted on running the regression suite for the entire checkout flow before release.

**Action:** I executed the core regression suite covering: product search, cart management, promo code application, address entry, and order confirmation — not just the new payment section. This took two hours but covered the most critical user journeys.

**Result:** We found that the new payment code had inadvertently broken the promo code calculation — a completely separate area of the codebase. Without regression testing, this would have reached production and impacted every customer using discount codes. The defect was fixed before release.

## Key Terms

| Term | Definition |
|------|------------|
| **Regression Testing** | Re-running tests after a change to confirm existing functionality is unbroken. |
| **Regression Suite** | A curated set of test cases used specifically for regression runs. |
| **Side Effect** | An unintended change in behaviour caused by a code modification elsewhere. |
| **Smoke Test** | A quick, high-level regression check to confirm the build is stable enough for full testing. |
| **Sanity Test** | A narrow regression check focused on a specific area impacted by a recent change. |
| **Automated Regression** | Using test automation to execute regression suites faster and more frequently. |
