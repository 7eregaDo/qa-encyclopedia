## The Intent
The interviewer wants to know whether you treat performance as a deliberate test discipline with its own strategy, not just running k6 or JMeter once before a launch. Senior QEs should be able to design the test types, define meaningful thresholds, interpret results, and integrate performance checks into the delivery pipeline.

## The STAR Script
**Situation:** I was the QA lead for an e-commerce platform expecting a 10x traffic spike during a seasonal sale campaign. The platform had never been formally performance tested — response time targets existed in the SLAs but had never been measured under load.

**Task:** Design and execute a performance testing strategy that gave the engineering team confidence in the platform's behaviour under the expected sale traffic and identified any breaking points before the event.

**Action:** I defined four test types and ran them in sequence. Baseline test: 10 concurrent users for 5 minutes to establish normal p50, p95, and p99 latency and throughput without load. This gave us a comparison point. Load test: I modelled the expected sale traffic using historical analytics — peak was estimated at 500 concurrent users with a request distribution across endpoints (60% browse/search, 25% product detail, 10% cart, 5% checkout). I ran this for 30 minutes and asserted against SLA thresholds: p95 latency under 500ms, error rate under 0.1%. Stress test: I ramped users from 500 to 2,000 in increments to find the breaking point — the load level at which error rate exceeded 1% or latency degraded beyond an acceptable threshold. This identified that the checkout service began failing at approximately 1,200 concurrent users due to a database connection pool exhaustion. Soak test: I ran 400 concurrent users for 4 hours to detect memory leaks and gradual performance degradation. I used k6 for test execution with a JavaScript test script modelling the realistic user journey and used Grafana to visualise throughput, latency percentiles, and error rates in real time. Findings were presented in a performance report with explicit go/no-go recommendations for each service.

**Result:** The connection pool issue was found and fixed three weeks before the sale. The soak test revealed a memory leak in the search service that caused latency to increase 40% over four hours. Both were addressed. The sale ran without incident. The k6 scripts were committed to the repo and became the standard pre-release performance suite.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that performance testing exists; may run a basic load test with a tool following a tutorial |
| **Mid** | Writes and executes load tests; interprets latency percentiles; identifies obvious bottlenecks from test results |
| **Senior** | Designs the full performance test strategy (baseline, load, stress, soak); models realistic traffic distributions; defines and enforces SLA thresholds; integrates performance gates into CI; presents findings as actionable engineering recommendations |

## Key Terms
| Term | Definition |
|------|------------|
| **Baseline Test** | A performance test run at minimal load to establish normal latency and throughput as a reference point for comparison under load |
| **Load Test** | A performance test that applies the expected production load to verify the system meets its performance SLAs under normal operating conditions |
| **Stress Test** | A performance test that increases load beyond the expected peak to identify the system's breaking point and failure behaviour |
| **Soak Test** | A performance test that applies sustained moderate load over an extended period (hours) to detect memory leaks and gradual performance degradation |
| **p95 Latency** | The 95th percentile response time — 95% of requests completed faster than this value; a more useful SLA metric than average latency as it captures tail behaviour |
| **Throughput** | The number of requests a system can process per unit of time, typically expressed as requests per second (RPS) |
| **Connection Pool Exhaustion** | A performance failure where all database connections in the pool are in use, causing new requests to queue or fail |
| **k6** | An open-source performance testing tool by Grafana Labs that uses JavaScript test scripts to simulate virtual users and measure API response times and error rates |
