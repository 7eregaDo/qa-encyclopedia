## The Intent
The interviewer is testing whether you think about resilience as a testable property. A service that works perfectly in isolation can still fail catastrophically in production when a downstream dependency becomes slow or unavailable. Senior QEs should have concrete experience simulating failure conditions and verifying resilience patterns.

## The STAR Script
**Situation:** A payments service I was testing depended on three downstream APIs: a fraud detection service, a card processing gateway, and a notification service. In production, a slowdown in the fraud detection service had previously caused the payments service to queue requests until it ran out of threads and began rejecting all payment requests.

**Task:** Design and implement failure condition tests that verified the payments service handled downstream failures in isolation, without cascading to its consumers.

**Action:** I used WireMock as a dependency simulator, running it in the test environment with stub mappings for all three downstream services. I tested four failure scenarios per dependency. Timeout: I configured WireMock to add a fixed delay exceeding the documented timeout threshold and verified the payments service responded with a 503 within the expected window — not after an indefinite hang. Connection refused: I configured WireMock to refuse connections entirely and verified the same 503 behaviour with a distinct error code. Slow degradation: I introduced a delay just below the timeout threshold and verified that repeated calls triggered the circuit breaker — subsequent requests should fast-fail with a 503 for a configurable period, then allow a probe request through (half-open state). Partial failure: I verified that a failure in the notification service (non-critical path) did not affect the payment creation response — the payment should succeed with a 201, and the notification failure should be logged and retried asynchronously.

**Result:** Circuit breaker testing revealed the fraud detection circuit breaker was never entering the half-open state — it was staying open indefinitely, meaning the service would never recover automatically after a transient outage. This was a configuration defect fixed before the next release.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests the happy path assuming all dependencies are available; aware that dependencies can fail |
| **Mid** | Uses WireMock to stub dependencies; tests timeout and connection error scenarios for individual dependencies |
| **Senior** | Tests circuit breaker state transitions, cascading failure isolation, partial dependency failure handling, and recovery behaviour; designs failure simulation as a reusable test infrastructure component |

## Key Terms
| Term | Definition |
|------|------------|
| **Circuit Breaker** | A resilience pattern that stops forwarding requests to a failing dependency after a threshold of failures, fast-failing subsequent requests until the dependency recovers |
| **Circuit Breaker States** | The three states of a circuit breaker: closed (normal operation), open (fast-failing all requests), and half-open (allowing a probe request to test recovery) |
| **Cascading Failure** | A failure mode where the unavailability of one service causes dependent services to fail in turn, propagating through the system |
| **WireMock** | An HTTP mock server that can simulate dependency behaviour including fixed delays, connection refusals, and fault injection |
| **Timeout Configuration** | The maximum duration a service will wait for a response from a dependency before treating the request as failed |
| **503 Service Unavailable** | An HTTP status code indicating the server is temporarily unable to handle the request, commonly used to signal dependency failure or circuit breaker open state |
| **Half-Open State** | The circuit breaker state in which a single probe request is allowed through to test whether the dependency has recovered |
| **Fault Injection** | The deliberate introduction of failures — delays, errors, dropped connections — into a dependency to test a service's resilience behaviour |
