## The Intent
The interviewer wants to see whether you test the properties that matter most in distributed, high-throughput systems — properties invisible under sequential happy-path testing but that cause real data corruption and duplicate transactions in production. Senior QEs should be able to design concurrent test scenarios and verify idempotency guarantees programmatically.

## The STAR Script
**Situation:** I was testing a payments API where network retries were an expected part of the consumer integration pattern. The API documented support for idempotency keys on `POST /payments`. A production incident had previously created duplicate charges because a client retried after a timeout without understanding the idempotency mechanism.

**Task:** Design tests for idempotency key behaviour and concurrent payment creation to verify the API's guarantees held under realistic retry and concurrency conditions.

**Action:** Idempotency key tests: I sent a `POST /payments` with a specific `Idempotency-Key` value and recorded the response (201, with a payment ID). I then sent the identical request a second time with the same key and asserted that: the response was 200 (not 201), the returned payment ID was identical to the first response, and no new payment record had been created. I also tested edge cases: same key but different request body (expected 422), expired key after 24 hours (treated as a new request), and missing key on a POST endpoint that documented key support as optional. Concurrency tests: I wrote a test using Python's `asyncio` and `httpx` to fire 10 simultaneous `POST /payments` requests for the same customer with the same amount but no idempotency key. I also ran concurrent updates to the same resource — 5 simultaneous `PATCH /accounts/{id}` with different field values — and asserted that the final state reflected exactly one of the updates, not a corrupted merge.

**Result:** Concurrency testing revealed a race condition in the account update endpoint: two simultaneous PATCH requests could result in a partially merged state where some fields from each request were applied, producing a corrupted record. This was a lost-update problem caused by missing optimistic locking, fixed with an ETag-based conditional update pattern.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests sequential duplicate requests manually; aware that retries can cause duplicate records |
| **Mid** | Tests idempotency key behaviour for happy path and reuse; writes basic duplicate-request tests |
| **Senior** | Tests concurrent request scenarios using async tooling, verifies idempotency key edge cases, identifies race conditions and lost-update problems, and recommends locking or versioning solutions |

## Key Terms
| Term | Definition |
|------|------------|
| **Idempotency Key** | A client-provided unique identifier in a request header that enables the server to detect and deduplicate retried requests, returning the original response without processing twice |
| **Race Condition** | A concurrency defect where the outcome of an operation depends on the relative timing of concurrent requests, producing inconsistent or incorrect results |
| **Lost Update** | A concurrency defect where two simultaneous writes to the same resource overwrite each other, with the final state reflecting neither write correctly |
| **Optimistic Locking** | A concurrency control strategy where a resource version is checked at update time, rejecting updates based on a stale version |
| **ETag** | An HTTP response header containing a version identifier for a resource; clients include it in `If-Match` headers on updates to enable conditional, race-condition-safe modifications |
| **asyncio** | A Python library for writing concurrent code using async/await syntax, enabling multiple simultaneous HTTP requests in test code |
| **Duplicate Request** | A repeated API call with identical parameters, typically caused by a network timeout and client retry, which idempotency mechanisms are designed to handle safely |
| **Conditional Update** | An API pattern where an update request includes an `If-Match` header with the current ETag; the server rejects the update with 412 if the resource has changed since the client last read it |
