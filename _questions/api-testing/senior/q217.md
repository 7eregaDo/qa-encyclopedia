## The Intent
The interviewer wants to see whether you think about test code with the same engineering discipline as production code. At senior level, you're not just writing tests — you're building infrastructure other engineers depend on. They want to see deliberate architectural decisions, a consideration of maintainability, and evidence that teams actually adopted what you built.

## The STAR Script
**Situation:** Across eight squads at a platform company, each squad had its own API test approach — different HTTP libraries, different authentication patterns, different assertion styles. When a shared service changed its auth mechanism, eight test suites needed to be updated independently.

**Task:** Design a shared API testing framework that could be adopted by all squads, reducing duplication and standardising the approach without removing squad autonomy over their test logic.

**Action:** I designed the framework as a Python package published to our internal PyPI registry, versioned with semantic versioning. The package had three layers. Core layer: an `ApiClient` base class wrapping `httpx` with built-in retry logic, configurable base URL from environment, and structured logging of all requests and responses. Auth layer: pluggable authenticators implementing a common interface — `ClientCredentialsAuth`, `ApiKeyAuth`, and `JwtAuth` — so squads instantiated the right authenticator for their service without writing auth logic themselves. When the platform changed its token endpoint, I updated the authenticator in one place and all squads got the fix on the next dependency update. Validation layer: a `SchemaValidator` class that loaded schemas from the shared OpenAPI spec registry and provided `assert_response(response, schema_name)` in one call. I also included a `DataFactory` base class that squads could subclass to add their entity-creation methods while inheriting the teardown registry pattern. I ran framework adoption workshops with each squad and created a `#api-testing-framework` Slack channel for support.

**Result:** Six of eight squads adopted the framework within one quarter. The auth mechanism change was subsequently handled with a single one-line version bump across all suites. New QE onboarding time for API testing reduced from approximately two days to half a day. The framework accumulated 12 internal contributors across squads within six months.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Uses the framework provided by others; follows the getting-started guide |
| **Mid** | Contributes to a shared framework; adds new authenticators or schema validators; writes framework documentation |
| **Senior** | Designs the full framework architecture; makes library and versioning decisions; builds the plugin/extension model; drives adoption across teams; owns the framework roadmap |

## Key Terms
| Term | Definition |
|------|------------|
| **Internal PyPI Registry** | A private package repository that hosts Python libraries for internal use, enabling versioned distribution of shared test framework code |
| **Semantic Versioning** | A versioning scheme (MAJOR.MINOR.PATCH) that communicates the nature of changes: breaking changes increment MAJOR, new features MINOR, bug fixes PATCH |
| **httpx** | A modern Python HTTP client supporting both synchronous and asynchronous requests, used as the underlying HTTP layer in API test frameworks |
| **Pluggable Authenticator** | A pattern where authentication logic is encapsulated in interchangeable classes implementing a common interface, allowing the framework to support multiple auth schemes without coupling |
| **Teardown Registry** | A mechanism that records all resources created during a test session and deletes them during teardown, even if individual tests fail |
| **Framework Adoption** | The process of getting teams to use a shared library, requiring documentation, onboarding, support channels, and demonstrated value |
| **Dependency Version Bump** | Updating the version of a shared library dependency, which propagates framework fixes and improvements to all consumers |
| **Getting-Started Guide** | Documentation providing a minimal working example of how to use a framework, designed to reduce time from installation to first passing test |
