## The Intent
The interviewer wants to confirm that you understand OAuth 2.0 as a protocol with multiple flows and testable states, not just as "the thing that gives you a bearer token." At senior level, they expect you to test the security properties of the flow itself — scope enforcement, token lifecycle, and the attack surfaces introduced by each grant type.

## The STAR Script
**Situation:** I was the QA lead for a developer platform implementing OAuth 2.0 with three grant types: authorisation code (for user-facing apps), client credentials (for service-to-service), and refresh tokens. The security team had flagged that the OAuth implementation had not been formally tested beyond basic happy paths.

**Task:** Design a comprehensive OAuth test strategy covering all three grant types and their security properties.

**Action:** I structured tests across four dimensions. Token issuance correctness: for each grant type, I verified that a valid request issued a token with the correct format (JWT), the correct expiry, and the scopes matching what was requested. For client credentials, I verified that requesting a scope the client was not registered for returned a 400 with `invalid_scope`. Scope enforcement: I obtained tokens with different scope combinations and verified each API endpoint enforced the required scope — a token with `read:orders` scope was rejected (403) by the `POST /orders` endpoint requiring `write:orders`. Token lifecycle: I tested expiry by setting a short-lived token in the test environment, waiting for it to expire, then using the refresh token to obtain a new access token and confirming the original expired token was rejected. I tested token revocation by revoking a token via the revocation endpoint and confirming subsequent API calls with that token returned 401. Security edge cases: I tested that the authorisation code flow rejected a mismatched `redirect_uri`, that PKCE was enforced for public clients, and that the `state` parameter was required to prevent CSRF.

**Result:** Scope enforcement testing found that one endpoint was checking for the `admin` scope using a prefix match rather than exact match — a token with `admin_readonly` scope was incorrectly granted access to admin write endpoints. This was a significant security defect, fixed before the platform launched publicly.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Obtains a bearer token and uses it in API requests; aware that tokens expire |
| **Mid** | Tests token expiry, refresh flows, and basic scope rejection; understands the difference between client credentials and authorisation code flows |
| **Senior** | Tests all grant types end-to-end, scope enforcement precision, token revocation, PKCE enforcement, redirect_uri validation, and CSRF protection via the state parameter |

## Key Terms
| Term | Definition |
|------|------------|
| **OAuth 2.0** | An authorisation framework that enables applications to obtain limited access to user accounts or services via delegated permission, using access tokens |
| **Authorisation Code Flow** | An OAuth grant type for user-facing applications where the user authenticates directly and the app exchanges an authorisation code for tokens |
| **Client Credentials Flow** | An OAuth grant type for service-to-service authentication where the client authenticates using its client ID and secret, with no user involved |
| **Refresh Token** | A long-lived token used to obtain a new access token after the original expires, without requiring the user to re-authenticate |
| **Scope** | A string that defines the specific permissions an OAuth token grants, used to limit what actions the token holder can perform |
| **Token Revocation** | The act of invalidating an access or refresh token before its natural expiry, which the API must honour by rejecting revoked tokens with a 401 |
| **PKCE (Proof Key for Code Exchange)** | A security extension to the authorisation code flow that prevents authorisation code interception attacks, required for public clients |
| **CSRF (Cross-Site Request Forgery)** | An attack where a malicious site triggers authenticated requests on behalf of a user; the `state` parameter in OAuth mitigates this |
