## The Intent
The interviewer wants to see whether you can test asynchronous outbound integrations — a fundamentally different challenge from testing synchronous inbound APIs. Webhooks invert the request/response model: the system under test is now the caller, and your test infrastructure must act as the receiver. Senior QEs should have a concrete approach to this.

## The STAR Script
**Situation:** I was testing a payment platform that sent webhooks to merchant systems on payment events — `payment.completed`, `payment.failed`, and `payment.refunded`. Merchants depended on these webhooks to update their order management systems. The webhook delivery mechanism had no test coverage at all.

**Task:** Design and implement a test strategy for the outbound webhook system, covering payload correctness, security signature verification, retry behaviour, and delivery ordering.

**Action:** I built the test infrastructure first: I deployed a lightweight webhook receiver service (using a simple Flask app) in the test environment that recorded all incoming requests — headers, body, timestamp — to an in-memory store queryable by tests. I then designed tests across five areas. Payload correctness: I triggered a payment event via the payment API, polled my receiver until the webhook arrived (with a timeout), and asserted on the payload structure — event type, payment ID, amount, currency, and timestamp format. Signature verification: the platform signed each webhook with an HMAC-SHA256 signature in the `X-Webhook-Signature` header. I verified the signature was present and valid, then wrote a negative test confirming my receiver correctly rejected requests with an invalid signature. Retry behaviour: I temporarily returned HTTP 500 from my receiver and verified the platform retried delivery with exponential backoff, eventually marking the webhook as failed after the documented maximum attempts. Deduplication: I verified that retried deliveries used the same `X-Webhook-ID` header value. Event ordering: I triggered three sequential events and verified they arrived with monotonically increasing timestamps and were not reordered.

**Result:** Testing revealed that the retry mechanism was retrying immediately rather than with backoff — all retries fired within two seconds, which would have overwhelmed a recovering merchant endpoint. This was fixed to use exponential backoff. The Flask receiver was open-sourced internally and adopted by two other teams testing webhook integrations.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that webhooks are outbound events; may manually inspect webhook payloads using a tool like webhook.site |
| **Mid** | Tests payload correctness and basic delivery; uses a receiver endpoint to capture webhooks; verifies signature headers |
| **Senior** | Builds a controllable test receiver; tests retry and backoff behaviour, deduplication identifiers, delivery ordering, and signature verification; designs the webhook test infrastructure as a reusable component |

## Key Terms
| Term | Definition |
|------|------------|
| **Webhook** | An outbound HTTP callback sent by a system to a registered consumer URL when a specified event occurs, enabling event-driven integrations |
| **HMAC-SHA256** | A cryptographic signature algorithm used to sign webhook payloads, enabling the receiver to verify the request originated from the expected sender |
| **Webhook Signature** | A hash of the request body and a shared secret, sent in a request header, that the receiver verifies to authenticate the webhook sender |
| **Exponential Backoff** | A retry strategy where the wait time between attempts increases exponentially (e.g. 1s, 2s, 4s, 8s) to reduce load on a recovering recipient |
| **Deduplication** | The practice of assigning a unique ID to each webhook delivery so consumers can detect and ignore duplicate deliveries caused by retries |
| **Webhook Receiver** | A test infrastructure component that accepts incoming webhook requests and stores them for assertion, acting as a controllable consumer endpoint |
| **Event Ordering** | The property that webhook events for a given resource are delivered in the sequence they occurred |
| **Delivery Failure** | The condition where a webhook delivery attempt receives a non-2xx response or times out, triggering the retry mechanism |
