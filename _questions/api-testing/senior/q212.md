## The Intent
The interviewer wants to know whether your security testing goes beyond "does the 401 work?" to genuine threat modelling and attack simulation. This is a senior-level concern because it requires understanding attacker perspectives, not just specification coverage. They're looking for structured methodology and concrete examples of vulnerabilities you've found.

## The STAR Script
**Situation:** I was the senior QE for an API platform handling sensitive customer financial data. A security audit had flagged that the team had no formal API security testing practice beyond checking auth headers. The CISO asked QE to close this gap before the next compliance review.

**Task:** Build a repeatable API security testing practice based on a recognised threat model and demonstrate its effectiveness with findings.

**Action:** I used the OWASP API Security Top 10 as the framework. I'll describe the three areas where we found the most significant issues. Broken Object Level Authorisation (BOLA/IDOR): I wrote tests that authenticated as User A, created a resource, then attempted to access and modify it using User B's token by substituting the resource ID in the URL. The API had no ownership check on the `GET /accounts/{id}/transactions` endpoint — any authenticated user could view any account's transactions by guessing or enumerating IDs. Mass Assignment: I sent POST and PUT requests with additional fields beyond the documented schema — specifically `role`, `is_admin`, and `account_balance` — to test whether the server was whitelisting accepted fields. One endpoint accepted `is_admin: true` and applied it, granting privilege escalation. Excessive Data Exposure: I compared every response payload against what the API documentation said should be returned and what the consuming UI actually displayed. Three endpoints returned full database objects including internal fields (`internal_notes`, `cost_price`) that were filtered client-side rather than server-side. I used OWASP ZAP in API scan mode to supplement manual testing, importing the OpenAPI spec to guide its crawl.

**Result:** Three high-severity findings were raised: the BOLA vulnerability, the mass assignment privilege escalation, and the excessive data exposure. All three were fixed before the compliance review. The BOLA finding in particular would have been a reportable data breach under GDPR if exploited. The OWASP API Top 10 checklist became the mandatory pre-release security test checklist for all API features.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests authentication (401, 403) and basic authorisation scenarios |
| **Mid** | Tests horizontal privilege escalation (IDOR/BOLA) and error information leakage; aware of OWASP API Top 10 |
| **Senior** | Applies the full OWASP API Security Top 10 systematically; tests mass assignment, excessive data exposure, injection vectors, and resource consumption; integrates DAST tooling; builds a repeatable security testing practice |

## Key Terms
| Term | Definition |
|------|------------|
| **OWASP API Security Top 10** | A ranked list of the most critical API security risks published by the Open Web Application Security Project, used as a structured threat model for API security testing |
| **BOLA (Broken Object Level Authorisation)** | An API vulnerability where a user can access or modify another user's resource by manipulating the resource ID in the request, also known as IDOR |
| **IDOR (Insecure Direct Object Reference)** | See BOLA — a vulnerability where predictable resource identifiers allow unauthorised access to objects belonging to other users |
| **Mass Assignment** | A vulnerability where an API accepts and applies request fields that were not intended to be user-controlled, such as `role` or `is_admin`, because the server does not whitelist accepted fields |
| **Excessive Data Exposure** | A vulnerability where an API returns more data than the consumer needs, relying on the client to filter sensitive fields rather than restricting them server-side |
| **OWASP ZAP** | An open-source web and API security scanner that performs dynamic analysis (DAST) to identify common vulnerabilities by probing running applications |
| **DAST (Dynamic Application Security Testing)** | Security testing performed against a running application by sending crafted inputs and observing responses, as opposed to analysing source code statically |
| **Privilege Escalation** | A vulnerability where a user gains access to permissions or resources beyond those their role is entitled to, either vertically (gaining admin rights) or horizontally (accessing another user's data) |
