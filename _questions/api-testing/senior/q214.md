## The Intent
The interviewer wants to know whether you understand that GraphQL's flexibility is also a testing surface area problem — and whether you've adapted your strategy accordingly. Many testers apply REST thinking to GraphQL and miss the field-level auth, query complexity, and introspection risks that are unique to it. They want concrete experience, not just awareness.

## The STAR Script
**Situation:** I joined a team that had migrated a product catalogue service from REST to GraphQL. The existing API test suite was Postman-based and oriented around URL paths. The new GraphQL API had a single endpoint (`/graphql`) and the test suite was essentially empty — the team had no mental model for what to test.

**Task:** Design and implement a GraphQL-specific test strategy and onboard the team to it.

**Action:** I identified five test areas unique to GraphQL. Query correctness: I wrote tests for each query and mutation covering valid field selections, asserting that the returned data matched the expected shape and values. I used the `gql` Python library with `requests` to construct and execute queries programmatically. Field-level authorisation: unlike REST where auth is typically endpoint-level, GraphQL can return different fields based on the caller's role. I tested that a viewer-role token could not retrieve `cost_price` or `internal_sku` fields — the API should return null or an authorisation error for those fields, not simply omit them silently. Query complexity and depth limiting: I sent deeply nested queries (a product with its category, with its parent category, recursively 10 levels deep) and verified the server rejected them with a complexity error rather than timing out or crashing. I also tested the documented max query depth. Introspection: in production mode, introspection should be disabled — I verified that sending an introspection query returned an error rather than exposing the full schema. Error handling: GraphQL returns HTTP 200 even for errors, placing error detail in the response body's `errors` array. I wrote tests that asserted on the `errors` field structure, not just the status code — a pattern completely different from REST.

**Result:** Field-level auth testing found that `cost_price` was returned to all authenticated users regardless of role — it had been protected at REST endpoint level previously but the GraphQL resolver had no equivalent check. This was a data exposure vulnerability fixed before the API went to production. The test patterns I documented became the team's GraphQL testing standard.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can send a GraphQL query in Postman; tests the happy path response |
| **Mid** | Tests query and mutation correctness, basic error handling via the `errors` array, and variable passing |
| **Senior** | Tests field-level authorisation, query complexity and depth limits, introspection exposure, N+1 query behaviour, and designs a GraphQL-specific test strategy distinct from REST approaches |

## Key Terms
| Term | Definition |
|------|------------|
| **GraphQL** | A query language for APIs that allows clients to request exactly the fields they need from a single endpoint, with the server resolving each field independently |
| **Resolver** | A function on the GraphQL server responsible for returning the data for a specific field in a query |
| **Introspection** | A GraphQL feature that allows clients to query the API's schema structure; should be disabled in production to prevent schema exposure to attackers |
| **Field-Level Authorisation** | An access control mechanism in GraphQL that restricts which fields a caller can retrieve based on their role, as distinct from endpoint-level authorisation in REST |
| **Query Complexity** | A measure of the computational cost of a GraphQL query based on its depth and breadth; APIs should enforce a maximum complexity to prevent resource exhaustion |
| **Query Depth Limiting** | A GraphQL server protection that rejects queries nested beyond a defined depth, preventing recursive or deeply nested queries from consuming excessive resources |
| **N+1 Query Problem** | A performance issue in GraphQL where resolving a list of N items triggers N additional database queries for a nested field, causing exponential load |
| **errors Array** | The field in a GraphQL response body that contains error details when one or more resolvers fail; GraphQL always returns HTTP 200, making this the primary error assertion point |
