## The Intent
The interviewer is testing whether you understand the limitations of integration environments for verifying API compatibility and have designed a scalable alternative. Contract testing is a senior-level concern because it requires architectural thinking: you're changing how teams coordinate, not just adding tests. They want to see that you've implemented this end-to-end, not just described the theory.

## The STAR Script
**Situation:** At a company with 14 microservices, integration failures between services were consistently appearing in the shared staging environment — too late and too expensively. Teams were afraid to deploy because they didn't know if their changes would break a consumer they weren't aware of. The staging environment was perpetually partially broken, which was delaying releases.

**Task:** Design and implement a contract testing strategy that gave teams deployment confidence without depending on a shared environment.

**Action:** I introduced Pact as the contract testing framework. I ran workshops with each squad to explain consumer-driven contracts and identify which services were consumers of which providers. I then implemented contracts in two phases. Consumer side: each consumer team wrote Pact tests in their own test suite that defined exactly the fields and response structures they used from each provider — not the full response, just their slice. Running these tests generated a pact file, which was published to a Pact Broker hosted on our infrastructure. Provider side: each provider's CI pipeline included a Pact verification step that fetched all pact files from the broker for its consumers, ran them against the real provider implementation, and reported results. A provider could not deploy if any consumer's contract was broken. I introduced the concept of "can I deploy?" checks — before any service deployed to staging, it queried the Pact Broker to confirm all its contracts were verified for the target environment. I also ran a mapping session to identify the ten highest-risk service pairs and prioritised those for initial contract coverage.

**Result:** Within two quarters, the ten highest-risk service pairs had full contract coverage. Integration failures in staging caused by contract drift dropped by 80%. Teams reported significantly higher deployment confidence. The Pact Broker's network diagram became the authoritative map of service dependencies, which had previously not been documented anywhere.

| Level | Expected Depth |

## Key Terms
| Term | Definition |
|------|------------|
| **Consumer-Driven Contract Testing** | A testing pattern where the consumer defines its expectations of a provider API as an executable contract, which the provider verifies independently |
| **Pact** | An open-source consumer-driven contract testing framework that generates pact files from consumer tests and verifies them against the real provider |
| **Pact Broker** | A service that stores and manages pact files, tracks verification results, and answers "can I deploy?" queries based on contract status |
| **Pact File** | A JSON file generated by running consumer-side Pact tests, containing the consumer's expectations of the provider's request and response structure |
| **Provider Verification** | The CI step in which a provider runs all pact files from registered consumers against its real implementation to confirm it meets their expectations |
| **Can I Deploy?** | A Pact Broker query that checks whether a given service version is compatible with all its consumers and providers in a target environment, used as a deployment gate |
| **Contract Drift** | The divergence between a provider's actual API behaviour and the expectations defined in a consumer's contract, typically introduced by uncoordinated provider changes |
| **Service Dependency Map** | A diagram showing which services consume which providers, derived from contract testing relationships and used for deployment impact analysis |
