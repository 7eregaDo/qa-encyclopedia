## The Intent
The interviewer is checking foundational understanding — whether the candidate knows what an API is in a testable sense, not just a conceptual one. They want to hear that you understand APIs as a distinct test layer with its own value, not just a backend detail behind the UI.

## The STAR Script
**Situation:** During my first project at a QA internship, the team was testing a retail web application. The senior QA explained that the checkout button on the screen was calling an API to create an order, and that if I only tested the button, I was missing the actual business logic.

**Task:** I needed to understand what APIs were, why they needed their own tests, and how to start interacting with them directly.

**Action:** I read the team's API documentation and learned that the application had a REST API with endpoints for creating, reading, updating, and deleting orders. I installed Postman and began sending requests manually — first a GET to retrieve an existing order, then a POST to create a new one. I noticed that the UI had validation preventing me from submitting an order with a negative quantity, but when I sent the POST request directly through Postman with a negative quantity, the API accepted it and created the record. This was a defect the UI tests would never have found. I documented the finding and discussed it with the developer, who added server-side validation. I wrote this up as a concrete example of why API testing adds value beyond UI testing alone.

**Result:** The defect was fixed before release. The senior QA used my example in a team knowledge-sharing session to explain the testing pyramid to two newer joiners. I began consistently including API test cases in my test planning from that point on.

## Key Terms
| Term | Definition |
|------|------------|
| **API (Application Programming Interface)** | A defined contract specifying how one software component can request services from another, typically via structured requests and responses |
| **REST (Representational State Transfer)** | An architectural style for APIs that uses HTTP methods and resource-based URLs to enable stateless communication between client and server |
| **Endpoint** | A specific URL at which an API accepts requests, representing a resource or action (e.g. `/orders`, `/users/{id}`) |
| **HTTP Method** | A verb that indicates the intended action of an API request: GET (retrieve), POST (create), PUT/PATCH (update), DELETE (remove) |
| **Request** | A message sent to an API containing the HTTP method, URL, optional headers, and optional body describing the desired operation |
| **Response** | The API's reply to a request, containing a status code, headers, and typically a body with the requested data or an error message |
| **Postman** | A widely used API testing tool that provides a graphical interface for constructing requests, inspecting responses, and organising test collections |
| **Testing Pyramid** | A model that recommends a large base of unit tests, a middle layer of API/integration tests, and a smaller top layer of UI tests to achieve efficient, reliable coverage |
