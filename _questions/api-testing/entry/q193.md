## The Intent
The interviewer wants to confirm you have practical, hands-on experience with the most common API testing tool. They're not looking for advanced scripting knowledge at entry level — they want to see that you understand the anatomy of a request, know what to look at in a response, and test systematically rather than just hitting the happy path once.

## The STAR Script
**Situation:** On my first API testing task, I was asked to verify a new `POST /users` endpoint that allowed an admin to create a new user account. There was no UI for this feature yet — the API was the only way to interact with it.

**Task:** Manually test the endpoint thoroughly using Postman, covering both the happy path and error scenarios.

**Action:** I started by reading the API documentation to understand the required request structure: a JSON body with `email`, `name`, and `role` fields, an `Authorization` header with a bearer token, and a `Content-Type: application/json` header. In Postman, I set the method to POST, entered the endpoint URL, added both headers in the Headers tab, and pasted a valid JSON body in the Body tab set to raw/JSON. I ran the happy path first — a valid new user — and asserted that the response was 201, the body contained the new user's ID, and the `email` matched what I sent. I then tested the following scenarios: omitting the `email` field (expected 400), sending a duplicate email (expected 409 Conflict), sending an invalid role value (expected 422), sending the request without the Authorization header (expected 401), and sending with a non-admin token (expected 403). I saved each request as a named item in a Postman collection so they could be reused. I found that the duplicate email check was returning a 500 instead of 409 — a server bug — and raised it as a defect.

**Result:** Six test cases were documented and saved in the shared Postman collection. One defect was found. The developer fixed the 500 error, and the endpoint passed all cases on retest.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests happy path and basic error cases manually in Postman; saves requests in collections; asserts on status code and key response fields |
| **Mid** | Uses Postman environments and variables to manage multiple environments; writes test scripts in Postman for automated assertion; chains requests using variables |
| **Senior** | Integrates Postman collections into CI via Newman; uses pre-request scripts and dynamic variables; designs collections as reusable test suites |

## Key Terms
| Term | Definition |
|------|------------|
| **Postman Collection** | A saved group of API requests organised into folders, used to manage and share a set of related API tests |
| **Bearer Token** | An authentication token passed in the Authorization header of an HTTP request, granting the holder access to protected API resources |
| **Content-Type Header** | An HTTP header that tells the server the media type of the request body; `application/json` indicates the body is JSON-formatted |
| **Happy Path** | The test scenario that exercises the expected successful flow with valid inputs, confirming the API works as specified under normal conditions |
| **Request Body** | The data payload sent with an API request, typically a JSON object for POST and PUT requests |
| **409 Conflict** | An HTTP status code indicating that the request could not be completed due to a conflict with the current state of the resource, such as a duplicate unique field |
| **Authorization Header** | An HTTP header used to pass credentials (such as a bearer token or API key) to authenticate a request |
| **Defect** | A documented deviation between the actual API behaviour and the expected behaviour defined in the specification |
