## The Intent
The interviewer wants to know whether you test against a specification or just poke at an API empirically. Testing without a spec means you may not know what correct behaviour looks like. Testing against a spec means your test cases are traceable, complete, and defensible. They want to see that you read and use documentation actively.

## The STAR Script
**Situation:** I joined a team mid-project where an e-commerce API was partially built. I was handed a Swagger UI link and asked to begin testing immediately, before the frontend was ready.

**Task:** Use the API specification to derive test cases and validate the API's behaviour against it.

**Action:** I opened the Swagger UI and worked through it methodically. For each endpoint, I noted the HTTP method, URL pattern, required and optional parameters, the schema of the request body, all documented response status codes, and the schema of each response body. I used the spec as a checklist: every documented status code needed at least one test case, every required field needed a test for its absence, and every field with a defined type or format needed a boundary value test. For example, the `POST /products` endpoint documented a `price` field as a number with a minimum of 0.01. I tested exactly 0.01 (valid), 0.00 (invalid, expected 422), and a negative value (invalid, expected 422). I also used the spec to identify undocumented behaviour — I found two status codes returned in practice (a 409 for duplicate SKU and a 413 for oversized image payload) that were not listed in the spec. I raised these as documentation gaps and added them to the spec after confirmation with the developer.

**Result:** My test cases had full traceability to specification entries. The two undocumented status codes were added to the spec. The developer commented that testing against the spec caught two schema mismatches — response fields that existed in code but were named differently from the spec — before any frontend integration began.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Reads Swagger/OpenAPI specs to derive test cases; maps test cases to specification entries; flags spec gaps |
| **Mid** | Uses spec-driven testing tools; validates response schemas programmatically against the OpenAPI spec; raises spec discrepancies as formal issues |
| **Senior** | Treats the spec as a living contract; implements spec validation in CI; owns the spec review process as part of the definition of done |

## Key Terms
| Term | Definition |
|------|------------|
| **OpenAPI Specification** | A standardised, machine-readable format for describing REST APIs, including endpoints, parameters, schemas, and authentication, formerly known as Swagger |
| **Swagger UI** | A web-based interface that renders an OpenAPI specification as interactive documentation, allowing developers and testers to explore and call endpoints directly |
| **Schema** | A definition of the structure, field names, types, and constraints of a JSON object used in an API request or response |
| **Required Field** | A field that must be present in a request body or response; its absence should trigger a 400 or 422 error response |
| **Boundary Value** | A test input at the edge of an allowed range (e.g. the minimum or maximum permitted value), where defects are statistically more likely to occur |
| **Traceability** | The property of test cases that links each case to a specific requirement or specification entry, enabling coverage reporting and impact analysis |
| **Schema Mismatch** | A discrepancy between the field names, types, or structure defined in the API specification and those returned by the actual implementation |
| **413 Payload Too Large** | An HTTP status code indicating that the request body exceeds the server's configured size limit |
