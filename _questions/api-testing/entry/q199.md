## The Intent
The interviewer wants to see that your manual API testing is structured and repeatable, not just ad-hoc requests you run once and discard. A Postman collection that a teammate can pick up and run is significantly more valuable than a set of undocumented one-off requests. They're assessing your professionalism and attention to maintainability even at entry level.

## The STAR Script
**Situation:** My team had a Postman collection for a payments API that had grown organically over six months into an unorganised folder of 60 requests with no consistent naming, hardcoded URLs, and no test scripts — only manual inspection of responses. When I joined, it took me a full day just to understand which requests did what.

**Task:** Reorganise and improve the collection so it could be used reliably by any team member.

**Action:** I restructured the collection in four steps. First, organisation: I created top-level folders by resource (`/payments`, `/refunds`, `/webhooks`) with sub-folders for happy path and error cases. I renamed every request to a consistent pattern: `[METHOD] [endpoint] - [scenario]`, for example `POST /payments - valid card`. Second, environments and variables: I created two Postman environments — `dev` and `staging` — each storing `base_url`, `admin_token`, and `user_token` as variables. All requests used `{{base_url}}/payments` instead of hardcoded URLs, so switching environments took one click. Third, test scripts: I added JavaScript test scripts to each request using Postman's Tests tab. Every request had at minimum: `pm.test("Status is 201", () => pm.response.to.have.status(201))` and a check on a key response field. Fourth, documentation: I added a description to each folder explaining what it covered and which token (admin or user) was required.

**Result:** New team members could run the entire collection against any environment within five minutes of being given access. The collection was used in the next sprint's regression cycle and a previously undocumented 500 error was caught by a test script that had previously required manual response inspection to spot.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Organises requests into folders; uses environments and variables; writes basic test scripts for status code and field assertions |
| **Mid** | Uses pre-request scripts and chained variables to pass IDs between requests; runs collections via Newman in CI; manages collection versioning |
| **Senior** | Designs collections as reusable test suites integrated into the CI pipeline; implements data-driven testing; uses Postman monitors for scheduled API health checks |

## Key Terms
| Term | Definition |
|------|------------|
| **Postman Environment** | A named set of key-value variables (e.g. base URL, tokens) that can be switched to run the same collection against different target environments |
| **Postman Variable** | A placeholder (`{{variable_name}}`) used in requests to store reusable values, avoiding hardcoded strings and enabling environment switching |
| **Test Script** | JavaScript code written in Postman's Tests tab that runs after a response is received and asserts on its properties using the `pm.test` and `pm.expect` APIs |
| **Pre-Request Script** | JavaScript code in Postman that runs before a request is sent, used to set dynamic variables, generate tokens, or prepare request data |
| **Newman** | A command-line tool for running Postman collections in CI/CD pipelines or from a terminal without the Postman GUI |
| **Chained Request** | A sequence of Postman requests where the response from one request (e.g. a created resource's ID) is stored as a variable and used in a subsequent request |
| **Collection Runner** | Postman's built-in feature for executing all requests in a collection in sequence and reporting pass/fail results for each test script |
| **Data-Driven Testing** | A testing approach where the same request is executed multiple times with different input values loaded from a CSV or JSON file |
