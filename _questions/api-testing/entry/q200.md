## The Intent
The interviewer is testing adaptability and curiosity. Not every API has perfect documentation or a prepared test suite. They want to see that you can think on your feet, use the tools available (spec, responses, error messages) to understand a system, and apply structured thinking to unstructured situations — rather than waiting to be told exactly what to test.

## The STAR Script
**Situation:** I was asked to perform exploratory testing on a third-party logistics API that the company had just licensed. There was a Swagger spec and a sandbox environment, but no internal test cases, no prior testing history, and I had two hours to produce findings before a go/no-go decision on the integration.

**Task:** Explore the API systematically within the time constraint, focusing on the endpoints most critical to the planned integration: shipment creation, status tracking, and cancellation.

**Action:** I started by reading the spec for the three target endpoints to understand the intended behaviour, then opened Postman and began exploring. I followed a loose charter: "Explore the shipment lifecycle — create, track, cancel — with focus on edge cases and error handling." First, I ran the documented happy paths to confirm the sandbox was functional and I understood the basic flow. Then I began deviating deliberately: I sent a shipment creation request without the `recipient_address` field (the spec said required but the API returned 200 and silently created a shipment with a null address — a serious bug). I tested state transitions: could I cancel a shipment that was already in transit? (Yes, unexpectedly.) Could I re-cancel a cancelled shipment? (500 — another bug.) I checked whether the tracking endpoint returned consistent data immediately after creation or needed time to propagate. I noted every unexpected behaviour, response time that felt slow, and response body that differed from the spec. I documented findings in a shared notes doc in real time, tagging each as a bug, a spec gap, or a question.

**Result:** In two hours I found three defects (missing required field validation, unexpected cancel-in-transit permission, 500 on re-cancellation) and two spec gaps. The team used the findings to negotiate a fix timeline with the vendor before committing to the integration.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Uses a charter to focus exploration; deviates from the happy path deliberately; documents findings in real time |
| **Mid** | Applies heuristics (CRUD, state transitions, boundary values, auth bypass) systematically; time-boxes sessions and produces structured session notes |
| **Senior** | Designs exploratory charters for a team; uses historical defect data to inform risk focus; integrates exploratory findings into formal test coverage decisions |

## Key Terms
| Term | Definition |
|------|------------|
| **Exploratory Testing** | A simultaneous learning, design, and execution approach to testing where the tester uses skill and curiosity to investigate system behaviour without a predefined script |
| **Charter** | A brief statement that focuses an exploratory testing session, describing what to explore, with what tools, and for what information |
| **Happy Path** | The primary success scenario for an API endpoint with all valid inputs, used as a baseline before exploring deviations |
| **State Transition** | A change in the status of a resource (e.g. a shipment moving from `pending` to `in_transit` to `delivered`), which defines valid and invalid operation sequences to explore |
| **Session-Based Testing** | A structured approach to exploratory testing using time-boxed sessions with a charter, producing session notes that document coverage and findings |
| **Null Value Bug** | A defect where an API silently accepts a missing required field and stores a null value rather than rejecting the request with a validation error |
| **Sandbox Environment** | A non-production environment provided by a vendor for integration testing, simulating the real API without affecting live data or incurring real costs |
| **Heuristic** | A mental shortcut or rule of thumb used to guide exploratory test investigation, such as "test all CRUD operations" or "test every state transition boundary" |
