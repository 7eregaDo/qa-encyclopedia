## The Intent
The interviewer is checking whether you look beyond the status code and verify the actual content of what comes back â€” and whether you test what happens when you send bad content in. Many entry-level testers check the happy path response body but skip malformed input testing entirely. They want to see you cover both directions.

## The STAR Script
**Situation:** I was testing a `POST /invoices` endpoint that accepted a JSON body to create a new invoice record with fields including `client_id`, `amount`, `currency`, `due_date`, and optional `notes`.

**Task:** Design and execute payload tests for both the request (what we send) and the response (what we get back).

**Action:** For request payload testing, I covered three categories. Valid variations: I tested the minimum valid payload (required fields only), the full payload including optional `notes`, and boundary values for `amount` â€” the minimum (0.01), a large value (999999.99), and zero (expected 422). Field type violations: I sent `amount` as a string ("fifty"), `due_date` as an invalid date format ("32-13-2024"), and `client_id` as a negative integer â€” expecting 400 or 422 for each. Structural problems: I sent a completely empty body (expected 400), a body that was valid JSON but missing all required fields (expected 422), and a body that was not JSON at all â€” plain text â€” with a wrong Content-Type header (expected 415 Unsupported Media Type). For response payload testing, I verified: all documented fields were present in the success response (201), field types matched the spec (e.g. `id` was an integer, `created_at` was an ISO 8601 datetime), the `amount` returned matched what I sent, and the error responses included a structured error body with a `message` field rather than a raw server trace.

**Result:** I found two issues: sending a non-JSON body returned a 500 instead of a 415 (server was not handling the content type check), and the error response for a missing required field returned a plain string instead of the documented JSON error object. Both were raised and fixed.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests valid and invalid payloads; verifies required fields and type constraints; checks response body structure manually |
| **Mid** | Writes automated schema validation against the OpenAPI spec; tests pagination, nested objects, and array fields; validates error response structure consistency |
| **Senior** | Designs payload test strategy including fuzz testing inputs, schema regression checks in CI, and response contract validation |

## Key Terms
| Term | Definition |
|------|------------|
| **Request Payload** | The data sent in the body of an API request, typically formatted as JSON, describing the resource to be created or updated |
| **Response Payload** | The data returned in the body of an API response, containing the requested resource, a created resource, or an error description |
| **Schema Validation** | The process of verifying that a JSON object conforms to a defined schema in terms of field names, data types, required fields, and value constraints |
| **415 Unsupported Media Type** | An HTTP status code indicating that the server refuses to accept the request because the payload format is not supported |
| **ISO 8601** | An international standard for representing dates and times as strings (e.g. `2024-06-15T10:30:00Z`), commonly used in API datetime fields |
| **Fuzz Testing** | A testing technique that sends unexpected, malformed, or random inputs to an API to discover unhandled edge cases and security vulnerabilities |
| **Structured Error Response** | An API error response that returns a consistent JSON object with defined fields (such as `message`, `code`, `field`) rather than a raw server exception or plain text |
| **Null Value** | The explicit absence of a value for a field, distinct from a missing field; APIs may treat null differently from an omitted field |
