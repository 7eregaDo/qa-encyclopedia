## The Intent
The interviewer wants to know whether you test security as a first-class concern rather than assuming authentication works if the happy path succeeds. Missing authentication and authorisation test cases is a common entry-level gap that can leave real security vulnerabilities uncovered. They want to see that you test the negative cases deliberately.

## The STAR Script
**Situation:** I was testing a project management API that used JWT-based authentication. The application had three user roles: viewer, editor, and admin. Editors could create and update projects but not delete them; admins could do everything.

**Task:** Write test cases that verified both authentication (are you who you say you are?) and authorisation (can you do what you're trying to do?) across all role combinations.

**Action:** I structured my test cases into two groups. Authentication tests: I tested each protected endpoint with no Authorization header (expected 401), with a malformed token (expected 401), with an expired token (I modified the expiry in a test JWT — expected 401), and with a token signed with the wrong secret (expected 401). I also confirmed that public endpoints did not require authentication by calling them without a token. Authorisation tests: I generated tokens for each role using the test environment's token issuer. I then tested every sensitive operation — create, update, delete — with each role. Editor creating a project: 201. Editor deleting a project: 403. Viewer updating a project: 403. Admin deleting a project: 204. Crucially, I tested horizontal authorisation too: I created a project as User A (editor) and then attempted to update it using User B's editor token (a different user's valid token). The API correctly returned 403, confirming it was checking resource ownership, not just role.

**Result:** All authentication cases passed. One authorisation gap was found: viewers could call GET on admin-only reporting endpoints (expected 403, got 200). This was raised, confirmed, and patched before release.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests missing/invalid tokens (401) and role-based access (403); verifies that public endpoints are accessible without auth |
| **Mid** | Tests token expiry, refresh flows, horizontal privilege escalation, and role boundary edge cases; understands JWT structure |
| **Senior** | Designs authentication test strategy across OAuth flows, API key schemes, and JWT; identifies and tests privilege escalation vectors systematically |

## Key Terms
| Term | Definition |
|------|------------|
| **Authentication** | The process of verifying the identity of a caller, typically via a token, API key, or credentials |
| **Authorisation** | The process of determining whether an authenticated caller has permission to perform a specific action on a specific resource |
| **JWT (JSON Web Token)** | A compact, self-contained token format used to transmit authentication and claims information between parties, consisting of a header, payload, and signature |
| **Bearer Token** | An access token passed in the Authorization header (`Authorization: Bearer <token>`) to authenticate API requests |
| **401 Unauthorized** | An HTTP status code indicating that the request lacks valid authentication credentials |
| **403 Forbidden** | An HTTP status code indicating that the caller is authenticated but does not have permission to perform the requested action |
| **Horizontal Privilege Escalation** | A security vulnerability where a user can access or modify another user's resources of the same privilege level |
| **Role-Based Access Control (RBAC)** | An authorisation model that assigns permissions to roles, and roles to users, controlling what actions each user may perform |
