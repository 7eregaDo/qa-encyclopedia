## The Intent
The interviewer is checking whether the candidate treats API testing as more than functional testing — specifically, whether they assert on the full response including the status code. Missing status code assertions is one of the most common entry-level gaps: tests that pass when the server returns a 200 with an error message in the body, because nobody checked the code.

## The STAR Script
**Situation:** While writing API tests for a booking service, I noticed that two of my Postman tests were marked as passing even though the booking was not being created correctly. I had only asserted on part of the response body.

**Task:** Investigate why the tests were passing despite incorrect behaviour, and fix the assertions.

**Action:** I inspected the full response and discovered the server was returning a 422 Unprocessable Entity status code with a validation error message in the body. My tests had not asserted on the status code at all — they only checked that the response contained a field called `booking_id`, which happened to be present in the error response body as null. I updated every test to assert on the expected status code as the first assertion, before checking the body. I then reviewed all the status codes the API could return — 200 for successful retrieval, 201 for successful creation, 400 for malformed requests, 401 for missing authentication, 403 for insufficient permissions, 404 for missing resources, 422 for validation errors, and 500 for server errors — and wrote at least one test covering each expected code. I documented the status code map in the test plan.

**Result:** Several additional gaps were discovered during this review — the 401 and 403 cases had no test coverage at all. These were added before release. The team adopted the pattern of asserting status code first in all subsequent API tests.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows the major status code families (2xx, 4xx, 5xx); asserts on status codes in every API test; covers the common codes for the API under test |
| **Mid** | Tests all documented status codes systematically; distinguishes between 401 and 403 semantically; verifies error response bodies as well as codes |
| **Senior** | Designs status code coverage as part of API test strategy; identifies undocumented status codes through exploratory testing; validates correct codes in contract tests |

## Key Terms
| Term | Definition |
|------|------------|
| **HTTP Status Code** | A three-digit code in an API response indicating the outcome of the request, grouped into families: 1xx informational, 2xx success, 3xx redirection, 4xx client error, 5xx server error |
| **200 OK** | The standard success response for a request that returned data, typically used for GET requests |
| **201 Created** | A success response indicating that a new resource was created, typically returned for successful POST requests |
| **400 Bad Request** | A client error indicating that the request was malformed or missing required fields |
| **401 Unauthorized** | A client error indicating that the request lacks valid authentication credentials |
| **403 Forbidden** | A client error indicating that authentication was provided but the authenticated user does not have permission to perform the requested action |
| **404 Not Found** | A client error indicating that the requested resource does not exist at the given URL |
| **500 Internal Server Error** | A server error indicating that the server encountered an unexpected condition that prevented it from fulfilling the request |
| **422 Unprocessable Entity** | A client error indicating that the request was well-formed but contained semantic validation errors that prevented processing |
