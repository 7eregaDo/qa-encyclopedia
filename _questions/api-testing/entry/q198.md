## The Intent
The interviewer is checking whether you understand URL parameter types and their distinct testing implications. Path parameters are often mandatory resource identifiers — their absence or invalidity changes which resource is targeted. Query parameters are usually optional modifiers — missing them should have a defined default behaviour, not an error. These are distinct and both need deliberate test coverage.

## The STAR Script
**Situation:** I was testing a `GET /articles/{id}` endpoint and a `GET /articles` list endpoint that supported filtering by `category`, sorting by `sort_by`, and pagination via `page` and `limit` query parameters.

**Task:** Design test cases for both path and query parameters.

**Action:** For path parameters, I tested: a valid integer ID (200 with the correct article), a non-existent ID (404), a string where an integer was expected such as `GET /articles/abc` (400 or 404 depending on the framework), zero (400 or 404), and a negative integer (400). I also tested the boundary of the URL structure — what happened with a trailing slash (`/articles/123/`) — to verify the server handled it gracefully. For query parameters, I tested: no query params at all (expected to return the default page with the default limit), each filter independently, combinations of filters, an invalid `category` value (expected either an empty result set or a 400 depending on the spec), `page=0` and `page=-1` (expected 400), `limit` exceeding the documented maximum (expected either a cap at the maximum or a 400), and a completely unrecognised query parameter (`?foo=bar`) to check whether the API ignored it silently or returned an error. I documented what the API's actual default behaviour was for each parameter, since the spec only described the parameters partially.

**Result:** I found that `limit` values above 100 were accepted without a cap — the API returned 500 when `limit=10000` caused a database timeout. This was an undocumented upper limit that became a documented constraint after I raised it. The query parameter tests were used as regression cases in subsequent sprints.

## Key Terms
| Term | Definition |
|------|------------|
| **Path Parameter** | A variable segment embedded in a URL path that identifies a specific resource, denoted by curly braces in API specs (e.g. `/users/{id}`) |
| **Query Parameter** | A key-value pair appended to a URL after a `?` to filter, sort, or paginate a resource collection (e.g. `?status=active&page=2`) |
| **Pagination** | A mechanism for returning large result sets in discrete pages, controlled by `page`/`offset` and `limit`/`per_page` query parameters |
| **Default Behaviour** | The API's response when an optional parameter is omitted, which should be defined in the spec and verified in tests |
| **Resource Identifier** | A value — typically an integer or UUID — used to uniquely identify a specific instance of a resource in a path parameter |
| **Filter Parameter** | A query parameter that restricts the returned resource set to items matching a specified criterion (e.g. `?category=news`) |
| **Sort Parameter** | A query parameter that controls the ordering of a returned resource list (e.g. `?sort_by=created_at&order=desc`) |
| **Upper Bound** | The maximum permitted value for a numeric parameter such as `limit`, beyond which the API should return an error or cap the value |
