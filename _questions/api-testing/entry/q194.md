## The Intent
The interviewer wants to confirm that you understand HTTP method semantics, not just that different methods exist. The testing implications differ: GET should be idempotent, POST creates state, PUT and PATCH have different partial/full update semantics. A candidate who can explain these differences and their test implications shows genuine understanding.

## The STAR Script
**Situation:** While testing a task management API, I was assigned to write test cases for all five methods on the `/tasks` and `/tasks/{id}` endpoints.

**Task:** Understand the expected behaviour of each HTTP method and design appropriate test cases for each.

**Action:** I worked through each method systematically. For GET `/tasks`, I tested that it returned a list of tasks with a 200, that it returned an empty array (not an error) when no tasks existed, and that repeated calls returned the same data without creating side effects. For GET `/tasks/{id}`, I tested a valid ID (200 with the task), a non-existent ID (404), and a malformed ID (400). For POST `/tasks`, I tested a valid new task (201 with the created object and ID), a missing required field (400), and a duplicate task if uniqueness was enforced (409). For PUT `/tasks/{id}`, I tested that sending the complete updated object replaced all fields (200), that omitting a previously set optional field cleared it, and that a non-existent ID returned 404. For PATCH `/tasks/{id}`, I tested that sending only the `status` field updated just that field and left all others unchanged — verifying partial update semantics. For DELETE `/tasks/{id}`, I tested a valid delete (204 No Content), that attempting to GET the deleted resource returned 404, and that sending DELETE on the same ID again also returned 404 (not a 500).

**Result:** I found that the PUT endpoint was behaving like PATCH — it was not clearing omitted optional fields. This was a specification violation. The developer confirmed it was a bug and corrected the implementation. My test cases were retained in the collection for regression use.

## Key Terms
| Term | Definition |
|------|------------|
| **GET** | An HTTP method for retrieving a resource without modifying server state; should be safe and idempotent |
| **POST** | An HTTP method for creating a new resource; not idempotent — calling it twice creates two resources |
| **PUT** | An HTTP method for replacing an entire resource with the provided payload; idempotent — calling it multiple times with the same payload produces the same result |
| **PATCH** | An HTTP method for partially updating a resource, modifying only the fields included in the request body |
| **DELETE** | An HTTP method for removing a resource; idempotent — deleting an already-deleted resource should not cause an error (or should return 404 consistently) |
| **Idempotency** | The property of an operation whereby making the same request multiple times produces the same result as making it once |
| **204 No Content** | An HTTP status code returned for successful requests that have no response body to return, commonly used for DELETE and some PUT/PATCH operations |
| **Safe Method** | An HTTP method that does not modify server state; GET and HEAD are safe methods |
