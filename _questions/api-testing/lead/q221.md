## The Intent
The interviewer is testing executive-scale thinking about API quality as an organisational capability. At lead level, you can't test every API yourself — you must design the systems, standards, and incentives that cause good API quality decisions to happen across dozens of teams without your direct involvement. They want to see strategic design, not individual execution.

## The STAR Script
**Situation:** I joined a company as Principal QE where 40+ internal APIs existed across 18 product squads. Each team had a different testing approach — some had Postman collections, some had code-based tests, most had nothing systematic. API-related production incidents accounted for 55% of all P1s in the prior year. There was no shared standard, no contract testing, and no API quality visibility at the leadership level.

**Task:** Define and drive an org-wide API quality strategy that reduced API-related incidents and gave engineering leadership a coherent quality signal — without creating a centralised bottleneck.

**Action:** I built the strategy around four pillars. Standards: I convened a cross-squad API guild to define a minimal API quality standard — every API must have: a versioned OpenAPI spec committed to the API registry, at least one consumer-driven contract test per registered consumer, authentication tests covering 401/403 scenarios, and a synthetic monitor for its three most critical endpoints. These were expressed as fitness functions in the CI pipeline — automated, not manual. Platform: I worked with the platform team to build an API registry (a centralised repo of all OpenAPI specs) and a contract testing infrastructure (Pact Broker). These lowered the cost of compliance — adding contract tests became a template exercise, not a bespoke effort. Governance: I introduced a quarterly API health review — a 30-minute engineering leadership meeting reviewing the API quality dashboard: contract coverage, synthetic monitor uptime, incident attribution by API. Teams with degraded metrics had a visible flag and a conversation, not a penalty. Culture: I ran an internal "API quality week" with workshops on contract testing, schema design, and security testing, producing reference implementations that teams could copy.

**Result:** Within 18 months, contract test coverage across registered API pairs rose from 12% to 78%. API-related P1 incidents dropped by 48% year-on-year. The API registry became the company's authoritative service catalogue, used for dependency analysis and incident routing. Three squads that had initially resisted the standard became its most active contributors after seeing it catch production incidents on other teams.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests APIs within their squad; unaware of org-wide API landscape |
| **Mid** | Applies consistent standards within their squad; may advocate for contract testing beyond their team |
| **Senior** | Designs API quality strategy for a product area; introduces contract testing and schema governance for their services |
| **Lead** | Defines the org-wide API quality standard, builds the platform that makes compliance easy, establishes governance through dashboards and reviews rather than gatekeeping, and drives cultural adoption through community and reference implementations |

## Key Terms
| Term | Definition |
|------|------------|
| **API Registry** | A centralised repository of all OpenAPI specifications across an organisation, providing a single source of truth for API contracts and enabling dependency analysis |
| **API Guild** | A cross-squad community of practice focused on API design and quality standards, responsible for defining and evolving shared conventions |
| **API Fitness Function** | An automated CI check that verifies an API meets a specific quality standard — such as having a committed OpenAPI spec or passing contract tests — as a deployment gate |
| **Contract Coverage** | The proportion of registered consumer-provider API pairs that have an active, verified consumer-driven contract test |
| **API Health Review** | A recurring leadership meeting that reviews API quality metrics across the organisation, creating visibility and accountability without centralised gatekeeping |
| **API Quality Standard** | A defined minimum set of quality practices every API must implement, expressed as testable criteria rather than subjective guidelines |
| **Reference Implementation** | A worked example of a quality practice (such as contract testing or synthetic monitoring) that teams can copy directly, reducing the effort of adoption |
| **Service Catalogue** | An organisational inventory of all services, their owners, dependencies, and API contracts, used for incident routing, dependency analysis, and onboarding |
