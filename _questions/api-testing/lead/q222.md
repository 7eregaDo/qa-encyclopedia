## The Intent
The interviewer wants to know whether you operate upstream of testing — influencing how APIs are designed so they are easier to test, monitor, and integrate. This is a lead-level contribution: preventing testing problems at the design stage rather than solving them after implementation. They want to see that you've influenced API design as a quality advocate, not just tested what was handed to you.

## The STAR Script
**Situation:** At a platform company, I observed a recurring pattern: QEs spent significant time working around poorly designed APIs — inconsistent error formats, ambiguous state transitions, no idempotency on mutation endpoints, and missing correlation IDs that made distributed debugging nearly impossible. Every squad was making these design decisions independently and making the same mistakes.

**Task:** Define and introduce API design standards that embedded testability and observability as first-class design properties, and get them adopted across the organisation.

**Action:** I drafted an API Design Standard document in collaboration with senior engineers and the API guild. I focused on six testability-relevant properties. Consistent error contract: every error response must return a JSON object with `error_code` (enum), `message` (string), and optional `details` (array) — no plain text, no stack traces, no variation. This made error assertion code reusable across all consumers. Idempotency on mutations: every POST endpoint that creates a resource must support an `Idempotency-Key` header. This made retry testing deterministic. Explicit state machines: every resource with a lifecycle must document its valid states and transitions in the spec. This gave testers a complete state transition test matrix without reverse-engineering behaviour. Correlation ID propagation: every request must accept and propagate an `X-Correlation-ID` header, returning it in responses. This made distributed trace assembly reliable in tests and production. Pagination standard: all list endpoints must use the same pagination parameter names (`page`, `limit`) and return the same metadata fields (`total_count`, `total_pages`). This meant pagination tests were written once and reused. Versioning policy: semantic versioning on URL paths with a minimum 90-day deprecation window and a `Deprecation` header on sunset versions. I presented the standard to the CTO, got it adopted as a company standard, and built a linting tool that validated new OpenAPI specs against it in CI.

**Result:** New APIs built after the standard was adopted required 30% fewer test cases for equivalent coverage because shared assertion utilities could be applied consistently. The linting tool caught 23 standard violations in the first quarter of operation, preventing them from shipping. Senior engineers reported that the standard reduced their API design decision fatigue.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests APIs as designed; may flag inconsistencies informally |
| **Mid** | Advocates for testable design in design reviews; raises specific concerns about error formats or missing idempotency |
| **Senior** | Proposes testability improvements in design reviews; writes design guidelines for their service area; implements linting for specific rules |
| **Lead** | Defines the org-wide API design standard with testability as a first-class principle, builds automated enforcement via spec linting, presents the standard to executive leadership, and drives adoption as a cultural and process change |

## Key Terms
| Term | Definition |
|------|------------|
| **API Design Standard** | A documented set of conventions and requirements governing how APIs at an organisation are designed, covering error formats, versioning, pagination, and other cross-cutting concerns |
| **Spec Linting** | Automated validation of an OpenAPI specification against a set of design rules, executed in CI to catch standard violations before an API is implemented |
| **Testability** | The property of a system that makes it easy to write deterministic, maintainable tests — including consistent interfaces, predictable state, and observable behaviour |
| **Idempotency-Key Header** | An HTTP request header enabling clients to safely retry POST requests without creating duplicate resources, a requirement for testable and resilient mutation endpoints |
| **Correlation ID** | A unique identifier passed through all services in a request chain, enabling distributed trace assembly and reliable multi-service test debugging |
| **State Machine Documentation** | The explicit definition in an API spec of a resource's valid states and the transitions between them, providing a complete and unambiguous test matrix |
| **Deprecation Header** | An HTTP response header signalling that an API endpoint or version is deprecated, including the date it will be removed |
| **Design Review** | A structured process in which a proposed API design is reviewed against standards and testability criteria before implementation begins |
