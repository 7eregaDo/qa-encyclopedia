## The Intent
The interviewer wants to see whether you can manage quality risk across a high-stakes, multi-month engineering programme — not just within a sprint. Re-architectures are where quality strategies most commonly fail: teams focus on building the new thing and assume it works, then discover at migration time that subtle behavioural differences break consumers. They want to see programme-level thinking.

## The STAR Script
**Situation:** I was the QA lead for a platform re-architecture programme at a logistics company. The core order management API was being rewritten from a legacy monolith to a new microservices-based implementation. The new API had to be behaviourally identical to the old one — 35 internal consumer teams depended on it. The programme had a 9-month timeline with a hard cutover date driven by the contract expiry of the legacy infrastructure.

**Task:** Design the quality strategy for the migration that ensured the new API was behaviourally equivalent to the old one and that consumer teams could migrate safely.

**Action:** I organised quality work across three phases. Equivalence testing phase (months 1–5): I built a comparison harness — a tool that sent the same request to both the legacy and new API simultaneously and compared responses field-by-field. I ran the harness against the full production request log (anonymised and replayed) to discover behavioural differences. Differences were classified as intended (documented improvements) or unintended (bugs in the new implementation). This caught 47 unintended behavioural differences in the first three months, including three where the new implementation silently dropped optional response fields that consumers actually relied on. Consumer migration support phase (months 4–7): I worked with consumer team QEs to build migration test kits — a set of scenario tests each team could run against their own integration to verify their code worked with the new API. I ran weekly office hours for consumer QEs. Contract testing coverage: all 35 consumer teams registered Pact contracts against the new API, giving the programme a live compatibility signal. Production parallel-run phase (months 7–9): the new API ran in shadow mode — receiving a copy of production traffic and logging responses, but not serving consumers. The comparison harness ran continuously against the shadow traffic, monitoring for divergence. When equivalence confidence exceeded 99.5% over a 30-day window, the cutover gate was approved.

**Result:** The cutover was executed on schedule with zero consumer-facing incidents. The equivalence harness had caught 61 bugs in the new implementation before any consumer migrated. The Pact contract coverage gave the programme leadership real-time confidence in compatibility. The parallel-run approach was subsequently adopted as the company standard for any major API replacement.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests new API functionality against acceptance criteria; not involved in migration strategy |
| **Mid** | Runs regression tests against the new implementation; supports consumer teams with their migration tests |
| **Senior** | Designs equivalence testing for specific service pairs; implements shadow mode comparison; manages contract test coverage for the new implementation |
| **Lead** | Owns the programme-level quality strategy across all phases, designs the equivalence harness, establishes consumer migration support infrastructure, defines the cutover quality gate, and presents confidence metrics to programme leadership |

## Key Terms
| Term | Definition |
|------|------------|
| **Equivalence Testing** | A testing approach that verifies a new implementation produces identical or intentionally improved behaviour compared to the system it replaces |
| **Comparison Harness** | A tool that sends the same request to two implementations simultaneously and compares their responses to identify behavioural differences |
| **Shadow Mode** | A deployment pattern in which a new service receives a copy of production traffic and processes it without serving responses, enabling risk-free production-scale validation |
| **Production Traffic Replay** | The technique of capturing real production requests and replaying them against a new implementation to discover behavioural differences at scale |
| **Migration Test Kit** | A set of scenario tests provided to consumer teams that verifies their integration works correctly with the new API implementation |
| **Cutover Gate** | A defined quality criterion — such as 99.5% equivalence over 30 days — that must be met before the legacy system is decommissioned and the new one takes full production traffic |
| **Behavioural Equivalence** | The property that two implementations produce identical outputs for identical inputs, including edge cases, error responses, and optional field handling |
| **Programme-Level Quality** | Quality assurance that spans an entire multi-team, multi-month engineering programme, as opposed to quality within a single squad or sprint |
