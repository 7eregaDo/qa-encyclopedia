## The Intent
The interviewer wants to see whether you think about quality beyond the boundary of your own systems. In most organisations, third-party API failures are one of the most common root causes of production incidents — and they're often treated as force majeure rather than a governable risk. A lead QE should have a systematic approach to this.

## The STAR Script
**Situation:** At a retail platform, third-party API integrations accounted for 38% of production incidents in a 12-month period — payment processors, logistics carriers, tax calculation services, and product enrichment APIs. Each integration had been built independently by different squads with no shared standards for how failures were handled, how the API was tested, or how degradation was detected. When a payment processor had a partial outage, the platform had no automatic degradation path — it simply failed for all affected customers.

**Task:** As Head of QE, design an organisational governance framework for third-party API integrations that reduced incident risk and improved resilience.

**Action:** I introduced a third-party API governance lifecycle with four stages. Onboarding assessment: before any new third-party API integration was built, a QE review assessed the vendor's sandbox quality (was it representative of production?), the availability of test credentials, the API's documented error codes and retry guidance, and the vendor's SLA and incident communication process. Integrations with poor sandbox fidelity or no SLA documentation were escalated for a risk decision before engineering work began. Integration standards: every third-party integration was required to implement a circuit breaker, a timeout, structured logging of all requests and responses, and a graceful degradation path (what the user experience should be if the integration is unavailable). These were encoded as a checklist in the definition of done and verified in code review. Acceptance testing: before a new integration went to production, a QE-led acceptance test covered: happy path, vendor error responses (simulated via WireMock), timeout behaviour, and the degradation path. Ongoing monitoring: every third-party integration had a synthetic monitor calling a lightweight probe endpoint or performing a minimal real transaction, with an alert if it failed for more than two consecutive runs. A vendor incident dashboard gave the on-call team instant visibility of which third-party APIs were degraded.

**Result:** Third-party-related incidents dropped from 38% to 19% of all incidents within four quarters. The onboarding assessment blocked two integrations where the vendor's sandbox was not representative — both vendors subsequently invested in improving their sandbox quality as a condition of the contract. The degradation path requirement meant that when a logistics API had a two-hour outage, the platform gracefully displayed estimated delivery times from cache rather than failing the checkout flow.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests happy path integrations with third-party APIs; uses sandboxes as provided |
| **Mid** | Tests error responses and timeout behaviour; advocates for graceful degradation; raises sandbox quality concerns |
| **Senior** | Designs integration test strategy including WireMock stubs for vendor failure simulation; introduces synthetic monitoring for third-party APIs |
| **Lead** | Designs the full third-party API governance lifecycle — onboarding assessment, integration standards, acceptance testing, and ongoing monitoring — and drives adoption as an organisational policy |

## Key Terms
| Term | Definition |
|------|------------|
| **Third-Party API Governance** | A framework defining the standards and process for evaluating, integrating, testing, and monitoring external API dependencies |
| **Sandbox Fidelity** | The degree to which a vendor's sandbox environment accurately represents the behaviour of their production API, including error codes and edge cases |
| **Graceful Degradation** | The ability of a system to continue operating in a reduced-functionality mode when an integration is unavailable, rather than failing entirely |
| **Onboarding Assessment** | A structured evaluation of a new third-party API integration's testability, documentation quality, and vendor SLA before engineering work begins |
| **Vendor Incident Dashboard** | A real-time view of the health of all third-party API integrations, showing synthetic monitor status and recent incidents for each dependency |
| **Integration Standards** | The defined technical requirements — circuit breaker, timeout, structured logging, degradation path — that every third-party API integration must implement |
| **SLA (Service Level Agreement)** | A contractual commitment from a vendor about their API's availability, latency, and support response times, used to assess risk and set monitoring thresholds |
| **Acceptance Test (Integration)** | A QE-led test run before a new third-party integration goes to production, covering happy path, vendor error simulation, and degradation path verification |
