## The Intent
The interviewer wants to see whether you can operate at the intersection of security and quality engineering at organisational scale. Security testing is often siloed — either entirely absent or owned by a separate security team with no engineering integration. A lead QE who can bridge these worlds and run a programme is a significant organisational asset.

## The STAR Script
**Situation:** Following a security audit at a fintech company, the CISO presented findings that 60% of the organisation's internal APIs had never been tested for the OWASP API Security Top 10 vulnerabilities. The existing security team was small and focused on perimeter security, not application-layer API testing. The CTO asked the QE organisation to co-own API security testing.

**Task:** As Director of QE, design and lead an API security testing programme that systematically addressed the OWASP API Top 10 across the organisation's API portfolio within two quarters.

**Action:** I structured the programme in three workstreams. Automated baseline: I worked with the platform team to integrate OWASP ZAP's API scan mode into the CI pipeline for every API, importing the OpenAPI spec to guide the scan. The scan ran on every PR to main and produced a security findings report. Critical and high findings were pipeline-blocking; medium findings created tracked tickets. This gave every API a continuous automated security baseline without manual effort. Manual red-team sprints: I hired two senior QEs with security specialisation and ran six-week red-team sprints against the highest-risk APIs (those handling financial transactions and PII). Each sprint focused on the OWASP top 10 categories not reliably caught by automated scanning — BOLA, mass assignment, and business logic flaws. Findings were severity-rated and entered into the engineering backlog as security defects with SLA targets (critical: 24 hours, high: 7 days, medium: 30 days). Developer education: I ran a quarterly "API Security Clinic" — a two-hour workshop where real anonymised findings from the programme were used as teaching examples. I also produced a one-page "API Security Checklist" that developers completed as part of the definition of done for any new API feature.

**Result:** Within two quarters, automated scanning covered 100% of APIs in CI. The red-team sprints surfaced 14 high-severity findings, all remediated within SLA. Developer survey responses showed a 28-point improvement in self-reported confidence in writing secure APIs. No API-related security incidents were reported in the 12 months following the programme's launch, compared to two in the prior year.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests authentication and basic authorisation; aware of common API vulnerabilities |
| **Mid** | Applies OWASP API Top 10 systematically on their assigned APIs; uses DAST tooling |
| **Senior** | Designs and executes API security testing for a product area; integrates automated scanning into CI; conducts manual security testing for high-risk endpoints |
| **Lead** | Designs and runs the org-wide API security testing programme, integrates automated scanning across all APIs, manages a red-team capability, establishes remediation SLAs, and drives security education as a cultural investment |

## Key Terms
| Term | Definition |
|------|------------|
| **OWASP API Security Top 10** | A ranked list of the most critical API security risks, used as a structured threat model for systematic API security testing |
| **Red-Team Sprint** | A time-boxed, adversarial testing exercise in which a dedicated team attempts to find security vulnerabilities in a target system by simulating real attack techniques |
| **OWASP ZAP API Scan** | A mode of the OWASP ZAP scanner that uses an OpenAPI specification to guide automated security testing of an API's endpoints |
| **Business Logic Flaw** | A security vulnerability arising from incorrect implementation of business rules — such as applying a discount more than once — that automated scanners cannot detect |
| **Remediation SLA** | A defined target time within which a security finding of a given severity must be fixed, creating accountability for engineering teams |
| **Security Defect** | A finding from security testing entered into the engineering backlog and tracked with the same rigour as functional defects, with severity-based prioritisation |
| **API Security Checklist** | A developer-facing reference document summarising the key security checks to apply to any new API feature, used as part of the definition of done |
| **BOLA (Broken Object Level Authorisation)** | The most common API security vulnerability, where a user can access another user's resources by manipulating resource IDs in requests |
