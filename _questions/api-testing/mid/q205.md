## The Intent
The interviewer wants to assess whether you treat error handling as a first-class test concern, not an afterthought. Good error responses are a contract too. Poor error handling — inconsistent formats, leaked stack traces, vague messages — causes integration failures for consumers and can expose security vulnerabilities. Mid-level QEs should test error responses as rigorously as success responses.

## The STAR Script
**Situation:** While testing a healthcare data API, I noticed during exploratory testing that error responses were inconsistent — some returned a JSON object with a `message` field, others returned plain text, and a 500 error on one endpoint returned a full Java stack trace.

**Task:** Audit the error handling across the API and define and execute a systematic test plan covering all error response types.

**Action:** I first mapped out the error contract from the API spec: all error responses were documented to return a JSON object with `error_code` (a string enum), `message` (human-readable), and `details` (an optional array of field-level validation errors). I then tested three categories. Format consistency: for every documented error status code on every endpoint, I verified the response body conformed to the documented error schema — correct fields, correct types, no extra fields, no plain text. Content correctness: I verified that the `error_code` values were specific and meaningful — for example, a missing required field returned `VALIDATION_ERROR` with a `details` entry naming the field, not a generic `BAD_REQUEST`. I also verified that error messages were descriptive enough to act on without exposing internals. Information leakage: I deliberately triggered 500 errors by sending edge-case inputs and checked that no stack traces, database error messages, or internal class names appeared in the response. I documented every inconsistency as a defect with the expected and actual response body.

**Result:** Eight defects were filed. The stack trace leak was treated as a security issue and prioritised for immediate fix. The team agreed on a shared error response schema as a standard, and I wrote a shared pytest assertion utility that validated error responses against this schema in a single line — adopted across the suite.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Checks that error scenarios return non-200 status codes; may verify the presence of a message field |
| **Mid** | Tests error response schema consistency, field-level validation messages, information leakage, and error code specificity across all documented failure modes |
| **Senior** | Defines the error response contract as part of the API standard; enforces it via schema validation in CI; identifies and tests undocumented error paths |

## Key Terms
| Term | Definition |
|------|------------|
| **Error Response Schema** | The defined structure of an API's error response body, specifying required fields such as error code, message, and optional field-level details |
| **Error Code** | A machine-readable string or numeric identifier in an error response that categorises the failure type, enabling programmatic handling by API consumers |
| **Information Leakage** | A security vulnerability where an API error response exposes internal implementation details such as stack traces, database queries, or class names |
| **Stack Trace** | A debug output showing the sequence of function calls that led to an exception; must never appear in production API error responses |
| **Field-Level Validation Error** | An error response detail that specifies which request field failed validation and why, enabling consumers to correct their request |
| **Validation Error (422)** | A response indicating the request was syntactically valid JSON but contained semantic errors such as missing required fields or invalid value constraints |
| **Schema Assertion Utility** | A reusable test helper function that validates a response body against a defined schema in a single call, reducing repetition across test cases |
| **Error Contract** | The documented agreement about what error responses an API will return for each failure condition, treated as a binding specification for consumers |
