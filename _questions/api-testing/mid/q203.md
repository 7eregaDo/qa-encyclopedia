## The Intent
The interviewer is checking whether you treat pagination as a distinct and non-trivial testing concern. Pagination bugs are common and often subtle: off-by-one errors on the last page, duplicate items across page boundaries, incorrect total counts, and inconsistent behaviour when the last page is partially filled. A mid-level QE should have a systematic approach to all of these.

## The STAR Script
**Situation:** I was testing a `GET /products` endpoint that returned paginated results with `page`, `limit`, `total_count`, and `total_pages` fields in the response alongside the data array. The endpoint supported a maximum of 100 items per page.

**Task:** Design and execute test cases that fully covered the pagination behaviour.

**Action:** I structured pagination tests around six scenarios. Default behaviour: calling the endpoint with no pagination parameters returned the first page with the default limit (I verified what the default was and documented it). Explicit first page: `page=1&limit=10` returned the first 10 items and `total_count` reflected the full dataset size. Middle page: `page=2&limit=10` returned items 11–20, with no overlap with page 1 (I compared item IDs across pages). Last page: with 25 total items and `limit=10`, `page=3` should return 5 items — I verified the response contained exactly 5, not 10 with nulls or errors. Page beyond the last: `page=4` with 25 items should return an empty data array and a 200, not a 404 or 500. Boundary values: `limit=1` (minimum), `limit=100` (maximum), `limit=101` (expected 400), `page=0` (expected 400), `page=-1` (expected 400). I also verified that `total_pages` was calculated correctly as `ceil(total_count / limit)` and wrote a parametrized test that checked this formula across multiple combinations. Finally, I tested pagination stability: I fetched page 1, added a new product, then fetched page 2, and verified the new product appeared on the correct page without displacing items from page 1.

**Result:** I found that `page=0` returned a 200 with the same results as `page=1` rather than a 400. I also found that `total_pages` was calculated using integer division rather than ceiling division, so a dataset of 11 items with `limit=10` returned `total_pages=1` instead of 2, making the second page unreachable through the metadata. Both were raised and fixed.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests the happy path pagination flow; checks that pages contain data; may not test boundary pages or metadata accuracy |
| **Mid** | Tests all boundary pages, metadata correctness, out-of-range parameters, and pagination stability across data changes |
| **Senior** | Tests cursor-based pagination, pagination under concurrent writes, and pagination consistency in distributed read replicas |

## Key Terms
| Term | Definition |
|------|------------|
| **Offset-Based Pagination** | A pagination style where `page` and `limit` (or `offset` and `limit`) parameters determine which slice of results to return |
| **Cursor-Based Pagination** | A pagination style that uses an opaque cursor value from the previous response to fetch the next page, providing stable pagination under concurrent data changes |
| **total_count** | A response metadata field reporting the total number of items in the collection across all pages |
| **total_pages** | A response metadata field reporting how many pages exist for the current query and limit, calculated as `ceil(total_count / limit)` |
| **Off-by-One Error** | A common defect where a calculation is incorrect by exactly one unit, often occurring at pagination boundaries (e.g. returning 9 items on a page of 10) |
| **Pagination Stability** | The property that items do not shift between pages when the underlying dataset changes, which offset-based pagination does not guarantee |
| **Ceiling Division** | Division that rounds up to the nearest integer, required for correct `total_pages` calculation when the total count is not evenly divisible by the limit |
| **Empty Page** | A valid response for a page number beyond the last page of results, which should return a 200 with an empty data array rather than an error |
