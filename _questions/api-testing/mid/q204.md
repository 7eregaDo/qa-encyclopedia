## The Intent
The interviewer wants to assess whether you can test realistic business flows, not just individual endpoints in isolation. Real systems require sequences: create a resource, update it, use its ID to create a related resource, then verify the combined state. They want to see that you understand state management in tests and can handle inter-request dependencies cleanly.

## The STAR Script
**Situation:** I was testing an order management API where placing a full order required a four-step flow: create a cart, add items to the cart, apply a discount code, and then submit the cart to create an order. Each step depended on the ID returned by the previous one.

**Task:** Design automated tests for the end-to-end flow and the failure scenarios at each step.

**Action:** I structured the tests in two tiers. Flow tests — the full sequence: I wrote a test that executed all four steps in order, storing the `cart_id` from step one and passing it to steps two, three, and four. After step four, I asserted on the created order: correct items, correct total after discount, status of `pending`, and that the cart was marked as `submitted` (not still `active`). I used pytest fixtures with `yield` to handle teardown — if the test created a cart, the fixture deleted it after the test completed, even on failure. Failure path tests — each step independently: I wrote a test for each failure point: submitting an empty cart (expected 422), applying an invalid discount code (expected 404 with a specific error message), adding an out-of-stock item (expected 409), and attempting to submit an already-submitted cart (expected 409). For each failure test, I set up only the required preconditions for that step — not the full chain — so the tests were fast and independent. I avoided global shared state by using unique data per test run (random email addresses, unique product SKUs seeded in test setup).

**Result:** The test suite caught a defect where a submitted cart could be re-submitted, creating a duplicate order. It also caught that the discount code error message returned "not found" for both invalid codes and expired codes — the spec required distinct messages — which was fixed before release.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests individual endpoints; may manually chain requests in Postman but does not structure automated chain tests |
| **Mid** | Writes automated multi-step tests with proper ID chaining and teardown; tests failure at each step independently; manages test isolation to avoid shared state |
| **Senior** | Designs workflow test frameworks supporting complex branching flows; tests concurrent workflow execution; validates distributed saga patterns and compensating transactions |

## Key Terms
| Term | Definition |
|------|------------|
| **Chained Request** | An API request that uses a value (such as a resource ID) returned by a previous request as an input, creating a dependency between test steps |
| **Test Fixture** | A setup and teardown mechanism in a test framework that ensures preconditions are met before a test and state is cleaned up afterwards, even on failure |
| **Precondition** | The state that must exist before a specific test step can execute, achieved by running prior API calls as part of test setup |
| **Test Isolation** | The property that each test creates its own data and does not depend on or modify the state of other tests, enabling tests to run in any order |
| **Teardown** | The cleanup logic that runs after a test completes, removing any data or state created during the test to prevent contamination of subsequent tests |
| **Yield Fixture** | A pytest pattern where setup code runs before `yield` and teardown code runs after it, ensuring cleanup occurs regardless of test pass or failure |
| **Saga Pattern** | A distributed transaction pattern where a multi-step workflow is broken into independent steps with compensating transactions to handle failures |
| **Idempotency Key** | A client-provided unique identifier for a request that prevents duplicate operations when the same request is retried, important to verify in order and payment workflows |
