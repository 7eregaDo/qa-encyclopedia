## The Intent
The interviewer is checking whether you treat infrastructure-level API behaviour as a test concern — not just functional correctness. Rate limiting is a common source of production incidents for API consumers, and its behaviour is often poorly specified. Mid-level QEs should know how to test it and how to read the response headers that communicate quota state.

## The STAR Script
**Situation:** I was testing a public API that documented a rate limit of 100 requests per minute per API key, returning a 429 status with `Retry-After` and `X-RateLimit-Remaining` headers. The documentation said the counter reset every 60 seconds.

**Task:** Write test cases that verified the rate limiting behaviour matched the documentation.

**Action:** I designed four test scenarios. Header presence: I sent a single valid request and asserted that `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers were present and had correct values — `Limit` matched the documented 100, `Remaining` was 99 after one request. Counter decrement: I sent 10 requests in sequence and verified that `X-RateLimit-Remaining` decremented by 1 each time. Limit enforcement: I wrote a test that sent 101 requests rapidly in a loop using `asyncio` for concurrency, asserting that by request 101 the response was 429 with a `Retry-After` header present. I stored the `Retry-After` value in seconds. Reset behaviour: after receiving the 429, I waited the number of seconds indicated by `Retry-After` plus a two-second buffer, then sent one more request and asserted it returned 200 with `X-RateLimit-Remaining` back at 99. I also tested isolation: I verified that two different API keys had independent counters — exhausting one key's limit did not affect the other. I flagged in my test notes that the rapid-fire test was environment-specific and should not run against production.

**Result:** I found that the `Retry-After` header was missing from the 429 response — only the status code was returned. This meant clients had no programmatic way to know when to retry. I also found that after the reset window, `Remaining` was reset to 100, not 99, meaning the reset endpoint wasn't accounting for the request that triggered the reset. Both were raised. The first was fixed as a P2 defect; the second was accepted as a known minor deviation and documented.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows that rate limiting exists; may manually send requests until hitting a 429 |
| **Mid** | Tests header correctness, counter decrement, limit enforcement, reset behaviour, and per-key isolation; uses concurrency to reliably trigger the limit in automation |
| **Senior** | Designs rate limit testing strategy across multiple limit tiers (per-user, per-IP, per-endpoint); tests burst vs. sustained rate limits; validates behaviour under distributed API gateway deployments |

## Key Terms
| Term | Definition |
|------|------------|
| **Rate Limiting** | A mechanism that restricts the number of API requests a client can make within a defined time window, protecting the server from overload |
| **429 Too Many Requests** | The HTTP status code returned when a client has exceeded the API's rate limit |
| **Retry-After Header** | An HTTP response header accompanying a 429 or 503 that tells the client how many seconds to wait before retrying |
| **X-RateLimit-Remaining** | A standard HTTP response header indicating how many requests the client has left in the current rate limit window |
| **X-RateLimit-Reset** | A response header containing the Unix timestamp or seconds until the rate limit counter resets |
| **Throttling** | A related mechanism that slows down responses (e.g. adding latency) rather than rejecting them outright, used to degrade gracefully under load |
| **Burst Limit** | A short-window rate limit that permits a higher request rate for brief periods, in contrast to a sustained rate limit applied over a longer window |
| **Per-Key Isolation** | The property that rate limit counters are tracked independently per API key (or user/IP), so one client exhausting their quota does not affect others |
