## The Intent
The interviewer wants to see that you've moved beyond hand-written field assertions to systematic schema enforcement. Manually asserting on three fields out of a twenty-field response is common at entry level but fragile — it misses regressions on the other seventeen fields. Programmatic schema validation catches all of them automatically. They want to know you've implemented this and understand why it matters.

## The STAR Script
**Situation:** I was maintaining an API test suite for a financial data platform with complex response schemas — some responses had 30+ fields, nested objects, and arrays of sub-objects. The test suite only asserted on five or six fields per response. A refactoring sprint renamed several fields in the response without updating the API spec, which went undetected until a downstream consumer integration broke in staging.

**Task:** Introduce programmatic schema validation so that any response field change — addition, removal, rename, or type change — was caught automatically.

**Action:** I implemented JSON Schema validation using the `jsonschema` Python library. I extracted the response schemas from the OpenAPI spec using `pyyaml` and stored them as JSON Schema files in a `schemas/` directory in the test repo. I wrote a shared utility function `assert_schema(response_body, schema_name)` that loaded the relevant schema and called `jsonschema.validate()` — raising an assertion error with a descriptive message if validation failed, listing the specific field and violation. I added this utility call to every test that verified a success response, replacing the previous hand-written field checks or supplementing them. I also wrote a CI job that compared the checked-in schemas against the live OpenAPI spec published by the service, alerting if they drifted — this meant the schemas in the test repo stayed in sync with the real spec. For responses that intentionally varied (e.g. different fields returned for different user roles), I created role-specific schema variants and used the appropriate one based on the test's auth context.

**Result:** In the first week after rollout, the schema validation caught two undocumented field removals introduced in a hotfix. The downstream consumer incident was not repeated. The utility function reduced the average test case length by approximately 30% because a single schema assertion replaced multiple individual field checks.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Manually asserts on a subset of response fields by name; may not verify field types |
| **Mid** | Implements JSON Schema validation against the full response structure; maintains schema files in the test repo; catches type changes, missing fields, and renames automatically |
| **Senior** | Designs the schema validation strategy across the full API surface; enforces schema sync with the live OpenAPI spec in CI; defines schema versioning policy |

## Key Terms
| Term | Definition |
|------|------------|
| **JSON Schema** | A vocabulary for describing the structure and constraints of JSON data, including field names, types, required fields, and nested object definitions |
| **jsonschema (library)** | A Python library that validates a JSON object against a JSON Schema definition, raising a `ValidationError` with a descriptive message on failure |
| **OpenAPI Specification** | A machine-readable API description format that includes response schemas, from which JSON Schema files can be extracted for validation |
| **Schema Drift** | The divergence between a documented schema and the actual response structure returned by the API, typically caused by code changes that are not reflected in the spec |
| **Schema Assertion Utility** | A reusable test helper function that validates a response body against a named schema, providing a single-line API for schema verification in test cases |
| **pyyaml** | A Python library for parsing YAML files, used to read OpenAPI specifications stored in YAML format and extract schema definitions |
| **Type Change** | A breaking schema modification where a field's data type changes (e.g. from integer to string), which would break consumer code that relied on the original type |
| **Role-Specific Schema** | A schema variant that describes the response structure returned for a specific user role or permission level, used when the API returns different fields depending on the caller's authorisation |
