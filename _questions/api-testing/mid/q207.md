## The Intent
The interviewer wants to know whether you understand the contract obligations of a versioned API and can translate them into test cases. Versioning mistakes silently break consumer integrations, sometimes in ways that aren't immediately visible. A mid-level QE should be able to test across versions and verify backwards compatibility systematically.

## The STAR Script
**Situation:** I was working on a public API that was releasing v2 of its `/orders` endpoint. The v2 introduced a new required field (`shipping_method`) and changed the format of the `address` field from a flat structure to a nested object. Existing consumers on v1 needed to continue working without modification.

**Task:** Design test cases that verified v2 behaved correctly and that v1 remained fully functional and unaffected by the v2 changes.

**Action:** I structured tests across three areas. Version routing: I verified that requests to `/v1/orders` were served by the v1 implementation and `/v2/orders` by v2 — not by checking an internal flag but by sending a request that would behave differently under each version (the `address` field format) and asserting on the actual response structure. v1 regression: I ran the full existing v1 test suite against the updated codebase to confirm no v1 behaviour had changed. I paid particular attention to the `address` response field — it should still return a flat structure on v1, even though v2 used a nested object. v2 correctness: I tested that `shipping_method` was required (missing it returned 422), that the nested `address` object was correctly structured in responses, and that v2 consumers could not call v1 endpoints and receive v2-structured data. I also tested the version header approach as an alternative routing mechanism (the API supported both URL versioning and an `API-Version` header), verifying both routing methods worked consistently. Finally, I tested deprecation headers: v1 responses should include a `Deprecation` header with the sunset date — I verified this was present and correctly formatted.

**Result:** I found that the `API-Version` header routing was not implemented for the new `/orders` endpoint — v1 header requests were falling through to v2. This was a breaking change for header-based consumers and was fixed before the v2 launch. The v1 regression suite passed clean, confirming the v1 contract was intact.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests a single API version; may be aware that versioning exists but does not test cross-version behaviour |
| **Mid** | Tests version routing correctness, v1 regression under v2 deployment, backwards compatibility of field changes, and deprecation header presence |
| **Senior** | Designs version compatibility test strategy; introduces automated breaking change detection in CI; defines the versioning policy and its test implications as part of the API standard |

## Key Terms
| Term | Definition |
|------|------------|
| **API Versioning** | A strategy for introducing breaking changes to an API while maintaining support for existing consumers, typically via URL path versioning (`/v1/`, `/v2/`) or request headers |
| **Backwards Compatibility** | The property that a new API version or deployment does not break existing consumers built against an older version |
| **Breaking Change** | A modification that changes the API contract in a way that requires consumers to update their code — such as removing a field, changing a field type, or adding a required parameter |
| **Non-Breaking Change** | A modification that does not require consumer updates — such as adding an optional response field or a new optional request parameter |
| **Deprecation Header** | An HTTP response header that signals to consumers that the current API version or endpoint is deprecated and will be removed on a stated date |
| **Sunset Date** | The date after which a deprecated API version will no longer be available, communicated to consumers via deprecation headers or documentation |
| **URL Path Versioning** | A versioning strategy where the version identifier is embedded in the URL path (e.g. `/api/v2/orders`) |
| **Version Routing** | The mechanism by which an API gateway or server directs incoming requests to the correct versioned implementation based on the URL path or request header |
