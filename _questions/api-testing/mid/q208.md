## The Intent
The interviewer wants to see whether you can move beyond JSON-body testing and handle the mechanics of multipart requests — a distinct and often neglected test area. File upload endpoints have a unique set of failure modes: wrong MIME type, oversized files, corrupted files, empty files, and storage failures. Mid-level QEs should know how to test these in both tools and code.

## The STAR Script
**Situation:** I was testing a document management API with a `POST /documents` endpoint that accepted PDF and DOCX file uploads up to 10MB, processed them to extract metadata, and stored them in cloud storage. The endpoint used `multipart/form-data` encoding.

**Task:** Design and execute a comprehensive test suite for the file upload endpoint.

**Action:** I covered five test categories. Valid file types: I uploaded a valid PDF and a valid DOCX, asserting that both returned 201, the correct `content_type` in the response, and that the extracted metadata (filename, page count) was accurate. Invalid file types: I uploaded a JPEG, a plain text file renamed with a `.pdf` extension, and a ZIP file — expecting 422 with a specific error code for each. The renamed `.pdf` test was important: I wanted to confirm the server was inspecting the file's magic bytes, not just the extension. Size boundaries: I uploaded a file of exactly 10MB (valid), 10MB + 1 byte (expected 413), and an empty file of 0 bytes (expected 422). Corrupted files: I sent a file with a valid `.pdf` extension but corrupted binary content that could not be parsed — expecting a 422 rather than a 500. Multipart structure: I sent a request with the file field missing entirely (expected 400), with the wrong field name, and with an extra unexpected field (expecting it to be ignored). In Postman I used the form-data body type for manual tests; in automated tests I used Python's `requests` library with the `files` parameter to construct the multipart payload programmatically.

**Result:** The renamed file test revealed the server was only checking the file extension, not the MIME type detected from the file content — a security gap where a disguised executable could be uploaded. The corrupted file test returned a 500 instead of 422, indicating the parser was not handling malformed input defensively. Both were raised as defects.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Uploads a valid file and checks the happy path; may test an obviously wrong file type |
| **Mid** | Tests file type validation by content inspection, size limits with boundary values, corrupted files, empty files, and multipart structure errors; automates using the `files` parameter in code |
| **Senior** | Tests storage integration, file retrieval integrity (hash comparison), concurrent upload handling, and security controls including malware scanning and path traversal |

## Key Terms
| Term | Definition |
|------|------------|
| **Multipart Form Data** | An HTTP encoding format used for file uploads that packages multiple parts (fields and files) into a single request body, indicated by `Content-Type: multipart/form-data` |
| **Magic Bytes** | The first few bytes of a file that identify its true format, independent of its file extension; used for secure file type validation |
| **MIME Type** | A label identifying the nature and format of a file (e.g. `application/pdf`, `image/jpeg`), declared by the client and verified by the server |
| **413 Payload Too Large** | An HTTP status code returned when a request body (including a file upload) exceeds the server's configured maximum size |
| **Corrupted File** | A file whose binary content does not conform to the expected format despite having a valid extension, used to test whether the server parses defensively |
| **Multipart Part** | An individual section within a multipart request body, containing headers (including `Content-Disposition`) and content for a single field or file |
| **Hash Comparison** | A file integrity verification technique that computes a checksum (e.g. MD5 or SHA-256) of the uploaded and stored file to confirm no corruption occurred during transfer or storage |
| **Path Traversal** | A security vulnerability where a malicious filename (e.g. `../../etc/passwd`) attempts to access files outside the intended storage directory |
