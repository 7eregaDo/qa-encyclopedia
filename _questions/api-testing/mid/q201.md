## The Intent
The interviewer is testing whether you've moved beyond happy-path coverage and can think structurally about what completeness means for an API. Mid-level candidates should be able to justify their test case selection, not just list what they tested. They want to see a method, not just a list.

## The STAR Script
**Situation:** I was assigned ownership of the API test suite for a subscription management service. The previous suite had 30 test cases covering only successful scenarios. Post-release defects had been traced to validation gaps and incorrect error handling — areas the existing suite did not address.

**Task:** Redesign the suite to achieve comprehensive coverage without creating an unmanageable number of cases.

**Action:** I applied five coverage dimensions to each endpoint. Positive cases: the documented happy path plus valid variations (e.g. optional fields included and excluded, minimum and maximum valid values). Negative input cases: missing required fields, wrong types, values outside permitted ranges, empty strings for required string fields, and extremely long strings. Authentication and authorisation cases: no token, expired token, valid token for a user without the required permission, and valid token for a different tenant's resource. State-based cases: for the subscription lifecycle (active → paused → cancelled), I tested every valid transition and every invalid one — for example, attempting to pause an already-cancelled subscription. Dependency cases: I tested what happened when a downstream dependency (payment processor) returned an error, using WireMock stubs to simulate failure responses. I organised all cases into a test matrix linked to the API spec, with each case mapped to a specification entry and a risk level. High-risk cases (auth, state transitions, payment errors) were automated; low-risk exploratory cases were documented as manual-run checklists.

**Result:** The suite grew from 30 to 94 cases, of which 71 were automated. Four defects were found during the redesign — including a state transition bug where a cancelled subscription could be reactivated — and fixed before the next release. The test matrix became the handover document for new team members.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Covers happy path and obvious negative cases; tests cases are largely intuition-driven |
| **Mid** | Applies equivalence partitioning, boundary analysis, and state transition techniques; organises cases into a structured matrix with risk ratings; separates automated from manual cases |
| **Senior** | Designs the coverage strategy for the entire API surface; introduces contract testing and dependency failure simulation; defines coverage completeness criteria |

## Key Terms
| Term | Definition |
|------|------------|
| **Equivalence Partitioning** | A test design technique that divides input values into groups (partitions) expected to behave identically, requiring only one representative test per partition |
| **Boundary Value Analysis** | A technique that tests values at the edges of valid and invalid ranges, where defects are statistically more common |
| **State Transition Testing** | A technique that identifies all valid and invalid transitions between states of a resource and creates test cases for each |
| **Negative Testing** | Testing that verifies the system handles invalid, unexpected, or out-of-range inputs correctly rather than crashing or behaving incorrectly |
| **Test Matrix** | A structured document that maps test cases to coverage dimensions (endpoint, method, scenario type, risk level), enabling coverage gap analysis |
| **WireMock** | An HTTP mock server used to simulate downstream API dependencies, enabling testing of a service's behaviour under controlled dependency failure conditions |
| **Risk Rating** | A classification of a test case by its associated business or technical risk, used to prioritise automation and execution effort |
| **Dependency Failure Simulation** | The practice of using stubs or mocks to return error responses from downstream services, verifying that the system under test handles upstream failures gracefully |
