## The Intent
The interviewer wants to know that you can move beyond Postman manual testing into durable, maintainable automation. At mid level, you should have hands-on experience choosing a framework, structuring tests for maintainability, and wiring them into CI so they run automatically. They'll probe for specifics — what framework, how structured, what happens on failure.

## The STAR Script
**Situation:** My team had a solid set of Postman manual tests for a logistics API, but they were only run when a QE remembered to run them before a release. Twice in one quarter, defects reached staging because the Postman suite wasn't run after a backend change.

**Task:** Automate the API tests so they ran on every pull request, with results visible in the CI pipeline without any manual step.

**Action:** I chose to rewrite the tests in Python using the `requests` library and `pytest`, rather than exporting the Postman collection to Newman, because the team was more comfortable maintaining Python than JSON collection files and `pytest` gave us a better reporting and parameterisation model. I structured the test code into three layers: a client module with functions for each API endpoint (hiding the HTTP details), a fixtures module for authentication token generation and test data setup/teardown, and the test files themselves containing only assertions. I used `pytest` parametrize to run the same validation test across multiple input combinations without duplicating code. For CI, I added a `test:api` job to the GitHub Actions workflow that installed dependencies, ran `pytest tests/api/` with a JUnit XML reporter, and published the results as a CI test report. The job ran on every pull request targeting main. I also added a test environment variable set in GitHub Actions secrets so the base URL and credentials were not hardcoded.

**Result:** API tests ran automatically on every PR from day one of rollout. The first week, a PR that changed a field name in the response was caught immediately — a failure that would previously have reached staging. The JUnit report format meant failures were displayed inline in the PR with file and line numbers, reducing triage time.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Runs Postman collections manually; may use Newman for basic CLI execution |
| **Mid** | Writes API tests in a code-based framework; structures tests with a client layer and fixtures; integrates into CI with environment configuration and test reporting |
| **Senior** | Designs the framework architecture for the whole team; introduces parallel execution, test data management, and API test coverage reporting in CI |

## Key Terms
| Term | Definition |
|------|------------|
| **pytest** | A Python testing framework that provides test discovery, fixtures, parametrization, and plugin support for writing and running automated tests |
| **requests (library)** | A Python HTTP library used to make API calls in test code, providing a clean interface for constructing and sending requests and inspecting responses |
| **Fixture** | In pytest, a reusable function that provides setup and teardown logic — such as authentication tokens or test data — to tests that depend on it |
| **Parametrize** | A pytest decorator that runs the same test function multiple times with different input arguments, eliminating duplicated test code |
| **Client Layer** | A module in an API test framework that wraps all HTTP calls behind named functions, separating the mechanics of requests from the assertions in test cases |
| **JUnit XML** | A standard XML format for test results supported by most CI platforms, enabling pass/fail reporting inline in pull requests and pipelines |
| **GitHub Actions** | A CI/CD platform built into GitHub that executes workflow jobs defined in YAML files on events such as pull request creation or merge |
| **Newman** | The command-line runner for Postman collections, enabling their execution in CI without the Postman GUI |
