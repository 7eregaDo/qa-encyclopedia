## The Intent
The interviewer is assessing whether your API tests are fragile and environment-dependent or robust and portable. Mid-level QEs frequently encounter the problem of tests that pass locally but fail in CI, or tests that fail because another test dirtied shared state. They want to see that you've solved these problems deliberately.

## The STAR Script
**Situation:** The API test suite I inherited was unreliable in CI — tests would pass locally but fail in the pipeline, and would pass or fail in different orders depending on which tests ran first. The root cause was shared test data: tests were relying on records seeded into a shared database at the start of the week, and other tests were mutating or deleting them.

**Task:** Redesign the test data strategy so tests were isolated and reproducible in any environment.

**Action:** I made four changes. Environment configuration: I extracted all environment-specific values — base URL, credentials, database connection — into a `.env` file pattern loaded via `python-dotenv` locally and via CI secrets in GitHub Actions. No URLs or credentials existed in code. Test data ownership: I replaced all references to shared seed data with per-test data creation. Each test used a setup step that called the API to create the data it needed (a user, a product, an order) and a teardown step to delete it — using pytest fixtures with function scope. This meant tests were completely independent of each other and of the environment's initial state. Data namespacing: to avoid ID collisions in shared environments, all test-created entities used a naming prefix (`test_auto_` + a UUID suffix) so they were identifiable and could be bulk-deleted by a cleanup script if teardown failed. For read-only reference data (e.g. product categories), I maintained a seed script versioned in the repo that could be run to bootstrap any environment to the required baseline.

**Result:** The CI flakiness rate dropped from approximately 25% to under 2% within two weeks. Tests ran reliably in local, CI, and staging environments without modification. The UUID-prefixed naming convention allowed the team to run a weekly cleanup job in staging that removed any orphaned test data from teardown failures.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Uses a shared environment; may rely on manually seeded data; configures base URL manually |
| **Mid** | Implements per-test data creation and teardown; externalises environment configuration; introduces data namespacing to prevent conflicts in shared environments |
| **Senior** | Designs the data management strategy across all environments; introduces data factories; manages PII masking for production-derived datasets; defines environment bootstrap procedures |

## Key Terms
| Term | Definition |
|------|------------|
| **Test Data Isolation** | The property that each test creates and owns its own data, preventing interference from other tests or pre-existing environment state |
| **Function-Scoped Fixture** | A pytest fixture that runs setup and teardown for each individual test function, ensuring a clean state for every test |
| **python-dotenv** | A Python library that loads environment variables from a `.env` file, enabling local configuration without hardcoding values in source code |
| **Data Namespacing** | The practice of prefixing test-created data with an identifiable string (e.g. `test_auto_`) to distinguish it from production data and enable bulk cleanup |
| **Seed Script** | A script that populates an environment with the baseline reference data required for the application and tests to function |
| **Orphaned Test Data** | Test records that were created during a test run but not deleted due to a teardown failure, accumulating in shared environments over time |
| **Teardown Failure** | The situation where a test's cleanup step fails or is not reached (e.g. due to an exception), leaving test data in the environment |
| **CI Secrets** | Encrypted environment variables stored in a CI platform (e.g. GitHub Actions) and injected into pipeline jobs at runtime, preventing credentials from appearing in code |
