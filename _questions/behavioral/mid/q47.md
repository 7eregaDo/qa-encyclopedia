## The Intent

The interviewer is assessing initiative, influence without authority, and change management capability. At mid level, you are expected to improve your team's practice — not just follow existing process. They want to see that you can take an idea from identification through to sustained adoption, and that you understand the human side of change as well as the technical side.

---

## The STAR Script

**Situation:** Our API testing was entirely manual — Postman collections run by QA before each release, with no automation and no integration into the CI pipeline. Every release required 3–4 hours of manual API testing that was repetitive, error-prone, and frequently delayed deployments when the QA engineer was unavailable.

**Task:** I wanted to introduce automated API testing using Playwright's API testing capabilities, which the team was already using for E2E tests — meaning no new tooling to learn. But the team had tried and abandoned an API automation initiative 18 months earlier due to high maintenance cost, and there was scepticism.

**Action:** I addressed the adoption challenge in four stages. First, **understand the resistance**: I had one-to-one conversations with the two people most sceptical — both had been burned by the previous initiative. Their concern was maintenance cost, not the concept itself. I noted this and made it the centrepiece of my approach. Second, **start with a proof of value, not a proposal**: instead of writing a proposal document, I spent two days building automated tests for the three most painful manual API tests — the ones that consumed the most time and broke the most often. I ran them in CI and showed the team a green pipeline with 3-minute execution instead of 45-minute manual runs. Third, **address the maintenance concern directly**: I built the tests using a data-driven pattern with a shared request builder — changes to an endpoint only required updating one place. I demonstrated this live by changing a request parameter and showing that all affected tests updated automatically. Fourth, **propose a bounded trial**: I asked for a four-sprint trial covering only the top 15 highest-value API tests, with a retrospective at the end. No commitment to full adoption — just an honest evaluation.

**Result:** The trial was approved. By sprint four, the 15 automated tests had caught two regressions that manual testing had missed. The retrospective was unanimous — the team voted to continue and expand. Within three months, 80% of the Postman collection had been automated. The previous sceptics became two of the strongest advocates.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Propose a tool or practice improvement, demonstrate it, and ask for permission to try it. |
| **Mid** | Research resistance, build a proof of value, address the specific concerns, and manage a bounded trial through to sustained adoption. |
| **Senior** | Lead organisation-wide tooling strategy — tool evaluation frameworks, standardisation decisions, onboarding programmes, and long-term maintenance ownership models. |

## Key Terms

| Term | Definition |
|------|------------|
| **Change Management** | The structured approach to transitioning a team from a current state to a desired future state. |
| **Proof of Value** | A working demonstration that shows the benefit of a proposed change before full commitment is required. |
| **Adoption Friction** | The effort or discomfort required to start using a new practice — reducing it increases uptake rate. |
| **Bounded Trial** | A time-limited, scope-limited experiment that allows evaluation without full commitment. |
| **Data-Driven Testing** | A test design pattern where test inputs are separated from test logic, reducing maintenance cost when inputs change. |
