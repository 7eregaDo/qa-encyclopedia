## The Intent

The interviewer wants to see honesty, analytical depth, and a bias toward structural improvement. This question is a trap for candidates who either refuse to admit strategic failures or who identify the missing test but stop there. The distinguishing answer goes beyond "I added a test case" to "I changed the way we think about this category of testing."

---

## The STAR Script

**Situation:** A production incident occurred when a third-party payment gateway began returning a new error code that our application had never encountered. Instead of displaying a user-friendly error message, the application threw an unhandled exception and crashed the checkout page. When I traced the defect back through our test strategy, I found a clear gap: we tested the happy path and a small set of known error codes, but we had never tested for unknown or unexpected API responses.

**Task:** I needed to understand not just that we missed this specific error code, but why our strategy had no mechanism for catching this class of problem — and fix the strategy, not just the test.

**Action:** I ran a post-mortem focused on the strategy gap. The root cause was that our API testing assumed a closed set of possible responses — we tested what we expected, and nothing else. We had no tests for malformed responses, unexpected status codes, gateway timeouts, or responses that didn't match the documented schema. This was a systemic blind spot, not a single missed case. I proposed three changes. First, I added a negative API testing layer: for every third-party integration, we would now write explicit tests for out-of-spec responses — unknown error codes, empty bodies, schema mismatches, and timeout scenarios. Second, I introduced contract testing with Pact for our critical third-party integrations, so changes in provider behaviour would be detected at the contract level before reaching our test environment. Third, I added an error handling review step to our story definition of done: any story touching an external integration required explicit acceptance criteria for failure scenarios, not just success paths.

**Result:** The three changes were implemented over the following sprint. In the next quarter, contract testing detected two provider schema changes before they reached staging — both would have caused similar production incidents. The negative API testing layer caught one unhandled error code in a different integration during regression. The error handling review became a standing part of our refinement process.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Identify the missing test case, add it, and reflect on what was missed. |
| **Mid** | Identify the structural gap in the strategy that allowed the class of defect to escape, and implement changes that prevent recurrence at the strategy level. |
| **Senior** | Build monitoring and alerting that detects novel failure modes in production before users report them, and create a feedback loop from production incidents to test strategy evolution. |

---

## Key Terms

| Term | Definition |
|------|------------|
| **Test Strategy Gap** | A category of scenario or risk that the test strategy does not address — distinct from a single missing test case. |
| **Negative Testing** | Testing for invalid inputs, unexpected responses, and error conditions — as opposed to happy path testing. |
| **Contract Testing** | Verifying that a service honours the interface contract expected by its consumers — catches provider changes early. |
| **Error Handling** | Application logic that manages unexpected or invalid states gracefully, rather than crashing or exposing raw errors. |
| **Post-Mortem** | A structured retrospective after an incident or failure, focused on systemic causes and improvements. |
