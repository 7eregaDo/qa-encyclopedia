## The Intent
The interviewer is assessing programme leadership capability in a high-stakes, high-complexity context. Migrating CI/CD platforms affects every engineering team's daily workflow and delivery capability. Done poorly, it causes release delays, team friction, and loss of trust. Done well, it's a reference for how large technical changes can be executed cleanly. They want to see that you've led one of these and have a method.

## The STAR Script
**Situation:** I led the migration of 25 engineering teams from a self-hosted Jenkins installation to GitHub Actions. The Jenkins instance was reaching end-of-life, had mounting maintenance overhead, and could not support the auto-scaling or ephemeral environment patterns the organisation needed. The migration had a hard deadline of six months due to the Jenkins licence expiry.

**Task:** Plan and execute the migration without disrupting any team's delivery capability, within six months, across 25 teams with widely varying pipeline complexity.

**Action:** I treated the migration as a product programme with four phases. Discovery and tooling (weeks 1–4): I audited all 25 Jenkins pipelines, categorised them by complexity (simple build/test, complex multi-stage with custom steps, pipelines with external system integrations), and mapped each to an estimated migration effort. I built a migration toolkit: a CLI that scaffolded a GitHub Actions equivalent for common Jenkins patterns, a migration guide with worked examples for the top ten patterns, and a shared workflow library covering the standard stages. Simple migrations were now self-service. Pilot cohort (weeks 5–8): I selected three teams — one simple, one medium, one complex — as pilot migrants. I worked closely with each, documenting every friction point and updating the toolkit and guide. Broad rollout in cohorts (weeks 9–20): teams migrated in four cohorts of five to six teams, ordered by risk (lowest complexity first). I ran weekly office hours throughout. Each team had a migration ticket with defined done criteria: all pipeline stages replicated, all tests passing, the Jenkins pipeline decommissioned. Parallel running was permitted for up to two weeks per team. Final cohort and decommission (weeks 21–24): the most complex teams migrated with dedicated platform engineer support. Jenkins was decommissioned on schedule.

**Result:** All 25 teams migrated on schedule. Zero release freezes or delivery interruptions were reported. Post-migration, median pipeline duration improved by 38% due to auto-scaling runners. The migration toolkit was reused by an acquired company six months later, completing their migration in three weeks instead of the projected six months.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Migrates their own pipeline when given a guide; flags blockers to the migration lead |
| **Mid** | Leads migration for their squad; contributes to the migration guide; helps adjacent teams |
| **Senior** | Leads migration for a cohort of teams; builds migration tooling for common patterns; tracks progress and escalates blockers |
| **Lead** | Owns the programme — discovery, tooling, cohort strategy, stakeholder communication, and decommission — treating it as a product delivery with milestones, risk management, and measurable outcomes |

## Key Terms
| Term | Definition |
|------|------------|
| **Migration Cohort** | A group of teams migrating together in the same phase, ordered by complexity or risk to manage the support load and learn from lower-risk migrations before tackling harder ones |
| **Parallel Running** | A transition period where both the old and new pipelines are active for the same service, providing a safety net during migration validation |
| **Migration Toolkit** | A collection of CLI tools, templates, and documentation that make the migration self-service for teams with standard pipeline patterns |
| **Done Criteria (Migration)** | The explicit, verifiable conditions that confirm a team's migration is complete — all stages replicated, all tests passing, old pipeline decommissioned |
| **Pipeline Complexity Audit** | A structured assessment of existing pipelines categorised by the effort required to migrate, used to plan cohort ordering and resource allocation |
| **Platform Programme** | A time-bounded engineering initiative with defined scope, milestones, and risk management, applied to infrastructure changes that affect multiple teams |
| **Office Hours** | A scheduled, open-access support session where migrating teams can ask questions and get real-time help from the migration team |
| **Decommission** | The planned removal of the legacy system after all consumers have migrated, releasing the resources it consumed and eliminating the maintenance burden |
