## The Intent
The interviewer is testing whether you can operate at the intersection of engineering delivery speed and regulatory obligation. Regulated industries — finance, healthcare, government — have specific requirements for change control, evidence of testing, and access segregation that naive CI/CD implementations violate. A lead QE in these environments must design pipelines that are both fast and compliant.

## The STAR Script
**Situation:** I was Director of QE at a payments company subject to PCI-DSS and SOC 2 Type II audit requirements. The compliance team was maintaining a separate manual change management process — engineers submitted change tickets, approval was sought via email, and evidence was assembled in spreadsheets post-deployment. This process took 2–3 days per release and was creating a delivery bottleneck. The auditors were also expressing concerns about the reliability of manually assembled evidence.

**Task:** Redesign the change management process to embed compliance controls directly in the CI/CD pipeline, eliminating the manual overhead while producing tamper-evident audit evidence automatically.

**Action:** I designed the compliant pipeline around four regulatory requirements. Change traceability: every deployment was linked to a JIRA change ticket via a required pipeline input. The pipeline verified the ticket existed, was in "Approved" status, and was linked to the relevant PRs — rejecting deployments with no approved ticket. The deployment event was written back to JIRA automatically. Segregation of duties: no engineer could both write code and approve their own deployment to production. I implemented this using GitHub environment protection rules requiring production deployment approval from a member of a separate "release approvers" group, with the approver's identity logged in the pipeline run. Test evidence: the pipeline generated a signed test evidence report — containing test run ID, timestamp, pass/fail counts, and artefact digest — stored immutably in S3 with versioning enabled. This replaced the spreadsheet evidence with a cryptographically-linked artefact. Access controls: the production deployment job ran under a dedicated service account with the minimum required permissions, separate from the CI service account. Credentials were rotated quarterly via an automated rotation job. I packaged these controls into the shared reusable workflow template so all services inherited them automatically.

**Result:** The manual change management process was eliminated for standard releases. Release cycle time dropped from 2–3 days to under 4 hours. The SOC 2 auditor reviewed the pipeline architecture and accepted the automated evidence as equivalent to manual records. The PCI-DSS assessment cited the pipeline's change control implementation as a strong control. The compliance team's workload for release evidence reduced by approximately 70%.

## Key Terms
| Term | Definition |
|------|------------|
| **Change Traceability** | The ability to link every production deployment to an approved change ticket, PRs, test evidence, and deploying engineer — a core requirement in regulated industries |
| **Segregation of Duties** | A control requiring that the person who writes and approves code cannot also be the sole approver of its production deployment, preventing a single individual from introducing and deploying malicious or erroneous changes |
| **SOC 2 Type II** | A compliance framework auditing an organisation's security, availability, and confidentiality controls over a defined period, requiring evidence that controls operated continuously |
| **PCI-DSS** | The Payment Card Industry Data Security Standard, requiring specific change management and testing controls for systems handling payment card data |
| **Tamper-Evident Artefact** | A compliance evidence file stored immutably with cryptographic verification (e.g. a signed S3 object with versioning), making any after-the-fact modification detectable |
| **Environment Protection Rule** | A GitHub Actions configuration requiring specific status checks and named approvers before a deployment to a protected environment proceeds |
| **Service Account Rotation** | The scheduled replacement of CI/CD service account credentials, reducing the window of exposure if credentials are compromised |
| **Immutable Evidence Store** | A storage system configured to prevent modification of previously written records, ensuring compliance evidence cannot be altered after a deployment |
