## The Intent
The interviewer is checking whether you understand the concept of build immutability and why it matters for quality assurance. If a build is rebuilt for each environment, you can't guarantee that what you tested is what you deployed. Entry-level QEs don't always understand this, but it's foundational to trustworthy continuous delivery.

## The STAR Script
**Situation:** A senior engineer mentioned in a retrospective that a recent production defect had been caused by a "build-per-environment" practice — the application was being rebuilt from source code for each environment, meaning the staging and production builds were slightly different. I didn't fully understand why this was a problem at first.

**Task:** Understand what build artefacts are, why rebuilding per environment is risky, and how the team planned to fix it.

**Action:** The engineer explained it to me using a concrete example: two developers had merged changes in the hours between the staging build and the production build. The staging tests had passed on a build that didn't include those changes; the production build did include them, untested. I then researched the concept of immutable artefacts. In a correct CD pipeline, the build happens once — producing a Docker image tagged with the commit SHA. That exact image is deployed to staging, tested, and if it passes, the same image (not a new build) is promoted to production. The tag or digest guarantees it's byte-for-byte identical. I also learned about artefact registries — places where these images or packages are stored between pipeline stages (e.g. Docker Hub, AWS ECR, GitHub Packages). I started checking, in the pipelines I worked with, whether the same artefact tag was used across environments or whether a new build was happening at each stage.

**Result:** I was able to identify a pipeline on another service where the build step was duplicated in both the staging and production deploy jobs. I raised it with the team as a quality risk, which was corrected. My understanding of artefacts made me a more informed participant in deployment discussions.

## Key Terms
| Term | Definition |
|------|------------|
| **Build Artefact** | The versioned, immutable output of a CI build — such as a Docker image, JAR file, or compiled binary — produced once and deployed unchanged across environments |
| **Immutable Artefact** | A build output that cannot be modified after creation; the same bit-for-bit file or image is used in every environment it is deployed to |
| **Docker Image** | A portable, layered container image that packages an application and its dependencies, used as a common build artefact in modern CI/CD pipelines |
| **Artefact Registry** | A storage service for build artefacts (e.g. Docker Hub, AWS ECR, GitHub Packages) that holds versioned images or packages for promotion through pipeline stages |
| **Commit SHA** | A unique hash identifying a specific commit in a version control system, commonly used to tag build artefacts for traceability |
| **Artefact Promotion** | The act of deploying the same artefact to a higher environment (e.g. staging → production) without rebuilding, ensuring what was tested is what is deployed |
| **Build-Per-Environment** | An anti-pattern in which a new build is created for each deployment environment, breaking the guarantee that the tested artefact is the one deployed to production |
| **Artefact Digest** | A cryptographic hash of a container image's content that uniquely identifies it, used to verify that the exact same image is being deployed as was tested |
