## The Intent
The interviewer wants to see whether you write tests with CI reliability in mind from the start. Flaky tests are one of the most common quality debt problems teams accumulate — and they almost always stem from the same few root causes. They want an entry-level candidate who already understands these causes and writes defensively against them.

## The STAR Script
**Situation:** I inherited a small suite of API tests that a previous QE had written. When I added them to CI, approximately 20% of runs were failing intermittently. The tests passed locally every time, which confused me initially.

**Task:** Diagnose why the tests were flaky in CI and fix them to make them suitable for a reliable pipeline.

**Action:** I investigated each flaky test and found three root causes. Fixed sleep statements: one test used `time.sleep(3)` to wait for an async operation to complete. In CI, the environment was slower than local, so the operation took more than 3 seconds and the test asserted before it was ready. I replaced the sleep with a polling assertion that checked the condition repeatedly up to a 10-second maximum with a clear timeout message. Shared test data: several tests relied on a specific user record that was created in a one-time setup and shared between tests. When tests ran in parallel in CI, two tests were modifying the same record simultaneously, causing race conditions. I refactored each test to create its own user in setup and delete it in teardown. Hardcoded port: one test was calling `localhost:8080` directly — the CI environment ran the service on a different port. I replaced the hardcoded URL with an environment variable read from the CI configuration. After these three fixes, the tests ran green on 50 consecutive CI runs without a single flake.

**Result:** The suite was reliably included in the PR pipeline. The three root-cause patterns I identified — sleep statements, shared state, and hardcoded values — became a personal checklist I applied to every new test I wrote.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows the common causes of flakiness (sleeps, shared state, hardcoded values); replaces sleeps with polling assertions; creates test data per-test and cleans up; uses environment variables for configuration |
| **Mid** | Systematically audits for flakiness using CI history; implements retry policies as a temporary quarantine measure; tracks and reports flakiness rate as a metric |
| **Senior** | Designs flakiness prevention standards for the team; builds CI tooling to detect and quarantine flaky tests automatically; runs periodic flakiness elimination sprints |

## Key Terms
| Term | Definition |
|------|------------|
| **Flaky Test** | A test that produces inconsistent results on the same code, undermining CI reliability by generating false failures that mask genuine build signal |
| **Fixed Sleep** | A `time.sleep(N)` call used to wait for an asynchronous operation; fragile in CI because environment speed varies — should be replaced with a polling assertion |
| **Polling Assertion** | A test assertion that repeatedly checks a condition at intervals until it is true or a timeout expires, replacing fixed sleeps for async operation verification |
| **Shared Test Data** | Data created once and used by multiple tests; a common flakiness cause when tests run concurrently and mutate the shared record simultaneously |
| **Test Isolation** | The property that each test creates its own state, does not depend on other tests, and cleans up after itself — essential for deterministic CI execution |
| **Environment Variable** | A configurable value injected into a process at runtime, used to externalise test configuration (base URLs, credentials) so tests work across different CI and local environments |
| **Race Condition** | A defect where the outcome of an operation depends on the relative timing of concurrent actions — a common cause of flakiness in tests that share state |
| **Determinism** | The property of a test that always produces the same result when run on the same code, regardless of environment, ordering, or timing |
