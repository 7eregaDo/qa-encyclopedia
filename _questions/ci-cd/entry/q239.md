## The Intent
The interviewer is checking whether you use version control as an active quality tool, not just a place where code lives. QEs who understand branching, commit history, and pull requests can trace defects more efficiently, participate meaningfully in code reviews, and understand the change context that caused a pipeline failure.

## The STAR Script
**Situation:** Early in my role I was investigating a regression — a feature that had worked two weeks ago was now failing. I had no idea how to find what had changed.

**Task:** Learn to use version control effectively as a quality investigation tool.

**Action:** I learned four Git practices that directly improved my QA work. Git log and blame: I used `git log --oneline` to see recent commits to the affected file and `git blame` to identify which commit had last changed the specific lines of code where the failure was occurring. This led me to a commit from five days ago that had changed a validation rule. Git diff: I used `git diff <old-commit> <new-commit>` to see exactly what had changed between the last known good state and the current state — this was much faster than reading the full file. Branch strategy awareness: I learned that our team used feature branches merged via pull requests. Each PR triggered a CI run. This meant the CI history on a branch was a record of the quality state at each step — I could see exactly which commit had introduced a test failure. Reverting safely: I learned that if a defect was confirmed as introduced by a specific commit, a `git revert` created a new commit undoing the change without losing history — this was the safe way to roll back, not deleting commits. I also started including relevant commit SHAs in defect reports, which developers found much more useful than "it stopped working recently."

**Result:** My defect investigation time decreased noticeably because I could pinpoint introducing commits in minutes rather than hours. Two defect reports I filed with commit SHA references were resolved same-day because the developer could go directly to the relevant change.

## Key Terms
| Term | Definition |
|------|------------|
| **Version Control** | A system (commonly Git) that records every change to a codebase, enabling history navigation, defect tracing, safe reverting, and collaborative development |
| **git log** | A Git command that displays the commit history of a repository or file, showing author, timestamp, and commit message for each change |
| **git blame** | A Git command that annotates each line of a file with the commit that last modified it, used to identify the introducing commit for a specific code change |
| **git diff** | A Git command that shows the differences between two commits, branches, or a commit and the working directory, used to understand what changed between two states |
| **git revert** | A Git command that creates a new commit undoing the changes of a specified commit, providing a safe rollback that preserves history |
| **Feature Branch** | A short-lived branch created for a specific feature or fix, merged back to the main branch via a pull request after review and CI checks pass |
| **Pull Request (PR)** | A proposed merge of a feature branch into the main branch, which triggers CI checks and enables peer code review before integration |
| **git bisect** | A Git command that performs a binary search through commit history to find the specific commit that introduced a regression |
