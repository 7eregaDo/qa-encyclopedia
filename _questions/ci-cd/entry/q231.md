## The Intent
The interviewer is checking foundational understanding of CI/CD from a quality perspective — not just what the acronyms mean, but why the practice matters for QA specifically. They want to hear that you understand CI/CD as a quality mechanism, not just a DevOps tool.

## The STAR Script
**Situation:** When I started my first QA role, the team was doing weekly releases. Tests were run manually at the end of each sprint, and it was common to discover three or four days before release that a change made early in the sprint had broken something that everything else depended on.

**Task:** I wanted to understand why integration problems were so expensive to fix late and how the CI pipeline the team was starting to introduce would help.

**Action:** I spent time with a senior engineer understanding the CI setup. They explained that every time a developer pushed code, a pipeline ran: the code compiled, unit tests ran, and a set of integration tests ran — all automatically, with results visible in the pull request within minutes. I then traced a recent defect backwards: a developer had changed a shared utility function on day two of the sprint; the regression had only been found on day nine when QE ran the full suite manually. With CI, the same pipeline run would have caught it within minutes of the push, while the developer still had full context on the change. I began treating the CI pipeline as my first line of feedback — before I opened the application, I checked whether the pipeline was green. I also learned the difference between CI (always running) and CD (deploying to staging or production automatically when CI passes), and started asking questions about which environments were gated by what quality checks.

**Result:** I developed a habit of monitoring the pipeline and treating a failing CI build as a blocking concern — not something to leave until the end of the sprint. My understanding of CI/CD became a genuine working tool rather than background knowledge.

## Key Terms
| Term | Definition |
|------|------------|
| **Continuous Integration (CI)** | The practice of automatically building and running tests against every code change pushed to a shared repository, providing fast feedback on integration correctness |
| **Continuous Delivery (CD)** | The practice of ensuring every passing build is in a deployable state and can be released to production at any time, typically with a manual approval step |
| **Continuous Deployment** | An extension of continuous delivery where every passing build is automatically deployed to production without manual intervention |
| **Pipeline** | An automated sequence of stages — build, test, analyse, deploy — that a code change passes through from commit to production |
| **Build** | The process of compiling source code, resolving dependencies, and producing deployable artefacts from a code change |
| **Integration Risk** | The probability and impact of defects caused by combining independently developed code changes, which CI reduces by making integration continuous rather than periodic |
| **Green Build** | A CI pipeline run in which all stages pass, indicating the code change does not break any automated quality gates |
| **Pull Request (PR)** | A proposed code change submitted for review, against which CI pipeline results are typically displayed to inform reviewers of quality status |
