## The Intent
The interviewer is checking whether you understand the testing pyramid in the context of a pipeline — not just as a concept, but as a practical design decision. They want to know you can justify why certain tests belong in CI and others don't, based on speed, reliability, and feedback value.

## The STAR Script
**Situation:** On a project I was assigned to, the CI pipeline ran the full test suite on every push — including 200 end-to-end browser tests. The pipeline took 55 minutes and developers had stopped waiting for it, merging regardless of whether it passed.

**Task:** I was asked to help review what tests were in the pipeline and make recommendations about what should stay, move, or be removed.

**Action:** I categorised the tests by type and evaluated each against three criteria: speed (did it run in under a few seconds?), determinism (did it pass consistently on the same code?), and feedback value (did a failure point clearly to the cause?). Unit tests — all 400 of them — ran in 90 seconds, were deterministic, and gave precise failure messages. These belonged on every push. API integration tests — 80 tests — ran in about 4 minutes, were largely deterministic, and had good diagnostic value. These belonged on every pull request. The 200 E2E browser tests were the problem: they took 45 minutes, had a 15% flakiness rate, and when they failed it was often a test infrastructure issue rather than a code defect. I recommended moving them to a scheduled nightly run rather than per-push, with only a curated set of 20 critical-path E2E tests remaining in the PR pipeline. I also recommended fixing or removing the flaky tests before including them anywhere. The remaining pipeline — unit tests plus API tests plus 20 E2E — ran in under 12 minutes.

**Result:** Developers started waiting for the pipeline again because it was fast and trustworthy. The nightly full E2E run gave the team a comprehensive signal each morning. Pipeline bypass behaviour stopped within two weeks.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Can categorise test types by speed and determinism; understands why E2E tests don't always belong on every push; knows the testing pyramid |
| **Mid** | Designs the test stage structure for a pipeline; makes data-driven decisions about which tests run where based on duration and flakiness metrics |
| **Senior** | Architects the full pipeline test strategy with parallelisation, test impact analysis, and tiered gates; defines and enforces pipeline duration SLAs |

## Key Terms
| Term | Definition |
|------|------------|
| **Testing Pyramid** | A model recommending many fast unit tests at the base, fewer integration tests in the middle, and a small number of slow E2E tests at the top — applied to pipeline design to balance speed and coverage |
| **Determinism** | The property of a test that always produces the same result on the same code, making it reliable as a CI quality signal |
| **Unit Test** | A fast, isolated test of a single function or class with all dependencies mocked, typically running in milliseconds and highly suitable for per-push CI |
| **End-to-End (E2E) Test** | A test that exercises a complete user journey through the full stack, typically slow and more prone to flakiness — often better suited to scheduled runs than per-commit CI |
| **Scheduled Run** | A CI pipeline execution triggered by a time schedule (e.g. nightly) rather than a code change, suitable for slow or comprehensive test suites |
| **Pipeline Duration SLA** | A defined maximum acceptable runtime for a pipeline stage, used to prevent slow tests from degrading developer feedback loops |
| **Feedback Loop** | The time between a developer making a change and receiving information about whether that change is correct — shorter loops enable faster, cheaper defect resolution |
| **Pipeline Bypass** | The behaviour of developers merging code without waiting for CI results, typically caused by slow or unreliable pipelines that feel like an obstacle rather than a tool |
