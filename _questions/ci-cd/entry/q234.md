## The Intent
The interviewer is checking whether you treat a failing build as something to understand and act on, not something to dismiss or re-run until it passes. A good QE treats the pipeline as a diagnostic tool. They want to see a methodical approach and clear communication instincts.

## The STAR Script
**Situation:** I was doing my morning pipeline check and noticed that the PR pipeline for a colleague's feature branch was red. The developer had already asked in Slack whether it was "just flakiness" and was considering merging anyway.

**Task:** Investigate the failure properly before drawing any conclusions, and communicate findings clearly.

**Action:** I opened the pipeline run and navigated to the failing stage. The failure was in the API integration test stage, not the unit tests. I read the error message: it was a 500 response from the `/orders` endpoint when the test expected a 201. This was not a pattern I recognised as flaky — our flaky tests tended to fail on timing issues with specific error messages. I checked the test history for this specific test across the last 20 runs — it had passed cleanly on every previous run. Next I looked at what the PR had changed: a modification to the order validation logic. The 500 error and the validation change were directly connected. I reproduced the failure by running the test locally against the branch. It failed consistently — not intermittent. I wrote up my findings in the Slack thread: "This looks like a genuine regression, not flakiness. The test has passed on every previous run, and the failure correlates with the validation change in this PR. The endpoint is returning 500 where 201 is expected. Reproduces locally — not environment-related." I tagged the developer and the team lead.

**Result:** The developer reviewed the finding, confirmed the regression, fixed the validation logic, and the pipeline passed on rerun. The PR was not merged until the fix was in. My investigation took 12 minutes and prevented a defect from reaching staging.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Reads pipeline logs to find the failing test and error message; checks test history to distinguish flaky from genuine failures; communicates findings clearly before drawing conclusions |
| **Mid** | Analyses failure patterns across multiple runs; identifies root cause from logs and code diff; triages flakiness systematically and raises remediation tickets |
| **Senior** | Designs pipeline observability to make failure diagnosis faster; builds flakiness tracking dashboards; defines escalation paths for different failure types |

## Key Terms
| Term | Definition |
|------|------------|
| **Pipeline Log** | The recorded output of a CI pipeline run, showing the commands executed, their output, and any error messages, used to diagnose build and test failures |
| **Failing Stage** | The specific phase of a CI pipeline that did not pass, narrowing the scope of investigation to the jobs and tests within that stage |
| **Test History** | The record of a specific test's results across multiple pipeline runs, used to determine whether a failure is consistent (genuine regression) or intermittent (flakiness) |
| **Regression** | A defect introduced by a code change that breaks previously passing functionality, typically identified by comparing pipeline results before and after the change |
| **Flakiness** | The property of a test that fails intermittently without a code change, often due to timing, shared state, or environment issues |
| **Local Reproduction** | Running a failing test in a local development environment to confirm the failure is genuine and code-related, rather than a CI environment or infrastructure issue |
| **Code Diff** | The set of changes introduced by a pull request, reviewed alongside a failing test to identify whether the failure is causally connected to the change |
| **Escalation** | The act of raising a CI failure to the attention of the developer, team lead, or platform team when it requires action beyond the initial investigator's scope |
