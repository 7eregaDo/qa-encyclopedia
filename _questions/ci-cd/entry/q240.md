## The Intent
The interviewer is checking whether you approach releases with discipline and can articulate what QA's contribution to a release process looks like in practice. Releases without a checklist rely on memory under pressure, which is where quality failures happen. They want to see that you treat the release process as a quality artefact in its own right.

## The STAR Script
**Situation:** The team I joined had no formal release checklist — each release was done ad-hoc, with individuals checking what they remembered to check. Two recent releases had gone wrong: one because a database migration script had been forgotten, and one because nobody had checked whether the rollback procedure worked before deploying.

**Task:** Work with the team to define a QA-owned release checklist that would make releases consistently safe without being burdensome.

**Action:** I drafted a checklist by working backwards from recent release failures and near-misses. I organised it into four sections. Pre-release checks: CI pipeline green for the release candidate commit; all automated test stages passed (unit, API, E2E smoke); no open critical or high defects in the release scope; migration scripts reviewed and tested in staging. Deployment checks: correct artefact version confirmed before deployment; database migrations run in correct order; configuration values verified for the target environment (not staging values in production); rollback procedure documented and verified as executable. Post-deployment checks: health endpoint returning 200 and correct version; smoke test suite passed; error rate and latency stable on the monitoring dashboard for 10 minutes post-deployment; no unexpected ERROR log entries. Sign-off: QE sign-off recorded in the release ticket with timestamp; on-call engineer briefed on the release scope and known risks. I kept the checklist to one page and used checkboxes so it could be completed linearly without skipping. I also included a "stop and escalate" instruction — if any check failed, the deployment was paused rather than proceeding.

**Result:** The checklist was adopted for all subsequent releases. In the first three months of use, it caught two issues before they reached production — a misconfigured environment variable and a migration that had been run out of order in staging but not corrected in the release script. Release-related incidents in the following six months dropped to zero.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Follows and contributes to a release checklist; knows the key categories of verification; treats checklist items as genuine gates, not formalities |
| **Mid** | Owns and maintains the release checklist; adapts it to new deployment patterns; analyses post-release incidents to identify missing checklist items |
| **Senior** | Automates checklist items where possible; defines release readiness criteria as testable quality gates in the pipeline; designs the release governance process |

## Key Terms
| Term | Definition |
|------|------------|
| **Release Checklist** | A structured list of verification steps to be completed before, during, and after a deployment, reducing reliance on memory and ensuring consistent release quality |
| **Release Candidate** | A build artefact that has passed all pre-release quality gates and is being considered for production deployment |
| **Migration Script** | A versioned database change script that modifies schema or data as part of a deployment, which must be tested in staging and applied in the correct order in production |
| **Rollback Procedure** | A documented, tested plan for reverting a deployment to the previous version if the release causes issues, which must be verified as executable before deploying |
| **Sign-Off** | A formal recorded confirmation by a named QE that specified quality checks have been completed and the release is approved from a quality perspective |
| **Stop and Escalate** | A release process instruction that mandates pausing a deployment when a checklist item fails, rather than proceeding and hoping the issue resolves itself |
| **On-Call Engineer** | The engineer responsible for monitoring and responding to production issues during a defined period, who should be briefed on release scope before deployment |
| **Release Scope** | The set of changes included in a specific release, used to focus verification and communicate risk to stakeholders and the on-call team |
