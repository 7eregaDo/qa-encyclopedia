## The Intent
The interviewer is assessing whether you understand that database changes are often the highest-risk part of a deployment and require their own quality discipline. Many pipeline failures and production incidents stem from unvalidated migrations — running in the wrong order, breaking the current application before the new one is deployed, or having no tested rollback. Mid-level QEs should understand migration safety and be able to test it.

## The STAR Script
**Situation:** The team I worked with ran database migrations as part of the application startup — the service applied any pending migrations when it booted. This had caused two incidents: one where a migration took longer than expected and caused a startup timeout, and one where a column was renamed, breaking the running application before the new deployment was complete.

**Task:** Redesign the migration process to be safer and verify it in the CI pipeline before any migration reached production.

**Action:** I worked with the engineering team to introduce four practices, each with a CI verification step. Versioned migrations: we adopted Alembic (Python) with sequential version numbers and required every migration to have both an `upgrade` and `downgrade` function. The CI pipeline ran `alembic upgrade head` in a fresh test database to verify the migration applied cleanly, then `alembic downgrade -1` to verify rollback worked — both had to succeed. Backwards compatibility check: I introduced a convention that no migration could remove a column or rename it in a single step. The pattern was: add the new column (migration 1) → deploy the new application (which writes to both columns) → remove the old column (migration 2, in the next release). I documented this in the migration guide and added a SchemaSpy check to the pipeline that compared the current schema against the schema at the HEAD-1 migration — any column removal triggered a review gate. Separate migration job: migrations were moved out of application startup into a dedicated pipeline step that ran before the new application version was deployed. This meant the migration outcome was visible in the pipeline log, not hidden in application boot logs. Timing test: any migration that ran `ALTER TABLE` on a table with more than 100,000 rows in the test database was flagged with a warning and an estimated production execution time, based on the row count in production.

**Result:** The migration process contributed zero production incidents in the 12 months following the redesign. The backwards compatibility check caught two problematic column removal attempts before they were merged. The timing flag prompted one team to rewrite a migration using a non-locking approach, preventing a planned 45-minute table lock at peak hours.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that database migrations exist and are run as part of deployments; checks that migrations applied correctly after a deployment |
| **Mid** | Tests migration upgrade and downgrade in CI; enforces backwards compatibility conventions; separates migration execution from application startup; checks migration timing for large tables |
| **Senior** | Designs the migration safety framework; introduces zero-downtime migration patterns; tests migrations against production-scale data volumes in performance environments |

## Key Terms
| Term | Definition |
|------|------------|
| **Database Migration** | A versioned, tracked change to a database schema — adding tables, columns, or indexes — applied in order as part of an application deployment |
| **Alembic** | A Python database migration tool that manages schema versions through `upgrade` and `downgrade` functions, tracking which migrations have been applied |
| **Rollback Migration** | The `downgrade` function in a migration that reverses its changes, enabling safe recovery from a failed or problematic schema change |
| **Backwards Compatibility (Schema)** | The property that a schema change does not break the currently running version of the application — critical during rolling deployments where old and new app versions coexist briefly |
| **Zero-Downtime Migration** | A migration technique that avoids locking the database table during the change, typically by adding columns with defaults or using shadow tables |
| **Migration Timing** | The estimated duration of a migration based on the number of rows affected, used to identify potentially long-running operations that could cause application disruption |
| **Table Lock** | A database lock acquired during certain DDL operations (e.g. `ALTER TABLE`) that prevents reads or writes to the table for the lock's duration |
| **Separate Migration Job** | A CI/CD pattern where database migrations run as a dedicated pipeline step before the application is deployed, making migration outcomes visible and independently verifiable |
