## The Intent
The interviewer wants to see that you treat the pipeline as a product with its own quality criteria. A slow, flaky pipeline is a quality problem in itself — it erodes developer trust, increases bypass behaviour, and delays feedback. Mid-level QEs should be able to measure pipeline health, identify problems, and drive improvements with data.

## The STAR Script
**Situation:** The team I worked on had a sense that the CI pipeline "wasn't great" but had no data to back it up. Arguments about whether to invest in pipeline improvements kept stalling because there was no shared baseline to anchor the conversation.

**Task:** Establish pipeline health metrics, create a baseline, and use the data to drive a prioritised improvement programme.

**Action:** I identified four metrics and set up collection for each. Pass rate: the percentage of pipeline runs that completed green without any retry, measured weekly. I pulled this from GitHub Actions run history using the API and found it was 73% — meaning roughly one in four runs required investigation or re-run. Median duration: the p50 run time per stage. I found unit tests took 90 seconds (healthy), integration tests took 18 minutes (too slow), and E2E took 44 minutes (far too slow). Flakiness rate: the percentage of test cases that had passed and failed on the same commit SHA within the past 30 days. I built a simple script that queried the test result history and flagged any test with a non-zero fail count on a previously-green SHA — 11% of the test suite was flaky. Mean time to green: the average time from a push to a green pipeline. I measured this at 51 minutes. With these four numbers as a baseline, I ran a two-sprint improvement programme. I quarantined the 11% flaky tests into a non-blocking suite, immediately raising pass rate to 91%. I parallelised the integration tests across four runners, dropping from 18 to 6 minutes. I moved the full E2E suite to a nightly run, dropping the PR pipeline duration to under 14 minutes. I reported progress against the baseline in the weekly engineering update.

**Result:** Pass rate: 73% → 91%. Median PR pipeline duration: 51 minutes → 14 minutes. Flakiness rate: 11% → 2.8% (residual flakes under active remediation). Developer satisfaction with CI improved from 2.1/5 to 4.0/5 in the quarterly engineering survey.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Notices when the pipeline is slow or unreliable; re-runs flaky tests; reports issues informally |
| **Mid** | Defines and collects pipeline health metrics; builds a baseline; prioritises improvements by impact; reports progress against baseline to the team |
| **Senior** | Designs pipeline observability infrastructure; introduces automated flakiness detection and quarantine; defines pipeline health SLAs; presents pipeline health as a quality metric in engineering leadership reviews |

## Key Terms
| Term | Definition |
|------|------------|
| **Pipeline Pass Rate** | The percentage of pipeline runs that complete successfully without any manual re-run, used as the primary pipeline reliability metric |
| **Median Duration** | The p50 pipeline run time — half of runs complete faster than this value — used as a measure of typical developer wait time |
| **Flakiness Rate** | The proportion of test cases in a suite that have produced inconsistent results (pass and fail) on the same code within a measurement period |
| **Mean Time to Green** | The average elapsed time from a developer's push to a passing pipeline result, measuring the efficiency of the feedback loop |
| **Baseline** | The measured starting value of a metric before an improvement programme begins, used as the reference point for progress reporting |
| **Test Quarantine** | The practice of moving flaky tests to a non-blocking suite so they are investigated without polluting the main pipeline signal |
| **Pipeline Observability** | The infrastructure and tooling that makes pipeline health metrics visible and queryable, enabling data-driven improvement decisions |
| **Non-Blocking Suite** | A test suite whose results are reported but do not gate pipeline progression, used for flaky or experimental tests under investigation |
