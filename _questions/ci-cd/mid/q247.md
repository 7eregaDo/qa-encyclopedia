## The Intent
The interviewer wants to see that your automated tests are not silently brittle because of hidden data dependencies. Test data problems are one of the top causes of pipeline flakiness and false positives. Mid-level QEs should have a systematic approach that makes tests independent of pre-existing data and reproducible across environments.

## The STAR Script
**Situation:** The integration test suite I inherited was producing intermittent failures that were difficult to reproduce locally. After investigation I found the root cause: tests were relying on records created by previous test runs, and when those records were mutated or deleted by other tests, downstream tests failed. The tests also hardcoded specific IDs that existed in the staging database but not in CI's ephemeral environment.

**Task:** Redesign the test data approach to make every test self-contained and runnable in any environment without pre-existing data.

**Action:** I applied four principles. API-first data creation: every test that needed a resource created it via the application's API in its setup step, not via direct database insertion. This guaranteed the data was valid by construction and environment-agnostic — no database access credentials required in CI. Teardown registration: I built a teardown registry — a pytest fixture that recorded the ID of every resource created during a test and deleted them via the API after the test completed, even if the test failed. This prevented data accumulation across runs. Unique namespacing: all test-created entities used a naming convention incorporating the run ID and a UUID — `test_{run_id}_{uuid}` — so concurrent pipeline runs didn't conflict, and orphaned records from a failed teardown could be identified and cleaned by a weekly maintenance job. Environment bootstrap: shared reference data that every test depended on (product categories, currencies, country codes) was maintained as a versioned seed script in the repo, applied once at environment creation, not per-test. Any change to the seed script required a version bump, making seed state explicit and auditable.

**Result:** Intermittent data-related failures dropped from 15% to under 1% of runs. Tests ran cleanly in four different environments (local, CI ephemeral, shared staging, pre-production) without modification. The teardown registry pattern was adopted by the team as the standard approach and documented in the testing guide.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Creates test data manually or relies on pre-seeded databases; aware of the risk of shared state |
| **Mid** | Implements API-first data creation with teardown registration; uses unique namespacing for concurrent run isolation; maintains versioned seed scripts for reference data |
| **Senior** | Designs the data management strategy across all environments and test tiers; introduces data factory libraries; manages PII masking for production-derived data; defines data lifecycle governance |

## Key Terms
| Term | Definition |
|------|------------|
| **API-First Data Creation** | The practice of creating test data through the application's own API rather than direct database insertion, ensuring validity and environment portability |
| **Teardown Registry** | A mechanism that records all resources created during a test run and deletes them after the run completes, preventing data accumulation and cross-run interference |
| **Unique Namespacing** | Adding a run-specific prefix or suffix to test-created entity names to prevent ID or name collisions between concurrent pipeline runs |
| **Seed Script** | A versioned script that populates an environment with the baseline reference data required for the application and tests to function correctly |
| **Orphaned Record** | A test data record that was created during a test run but not deleted due to a teardown failure, accumulating in shared environments over time |
| **Data Accumulation** | The build-up of test records in a shared environment from repeated test runs without teardown, which can slow queries, cause uniqueness conflicts, and make environments unpredictable |
| **Environment-Agnostic Test** | A test that can run in any environment without modification, relying on environment variables for configuration and creating its own data rather than assuming pre-existing state |
| **Run ID** | A unique identifier assigned to each CI pipeline run, used in test data naming to ensure records from different runs are distinguishable |
