## The Intent
The interviewer wants to see whether you understand modern deployment patterns as quality mechanisms, not just operational concerns. Blue-green and canary deployments are tools for managing release risk, and QEs should understand how to test them, what can go wrong, and how to validate that rollback works before it's needed.

## The STAR Script
**Situation:** My team was adopting canary deployments for the first time to reduce the risk of releases to a high-traffic API. The platform team had configured the routing infrastructure, but nobody had defined what tests should run at each canary stage or how to verify that rollback worked correctly.

**Task:** Define the quality verification approach for canary deployments and write the tests that would gate each traffic increment.

**Action:** I designed quality checks across three canary phases. Phase 1 — 5% traffic: immediately after routing 5% of traffic to the new version, I ran the smoke test suite against the canary version directly (using a canary-specific header to force routing). I also monitored the error rate and p95 latency of the canary instances for five minutes via Grafana. If either exceeded 2x the baseline, the canary was marked as failed and rollback was initiated. Phase 2 — 25% traffic: after 15 minutes at 5% with stable metrics, traffic was incremented to 25%. I verified that the traffic split was correct by checking the access logs — 25% of requests should carry the canary version identifier in the response headers. Phase 3 — 100% cutover: the same monitoring criteria applied, with a final synthetic monitor run confirming all critical journeys were healthy. Rollback testing: I ran a planned rollback test in the staging environment before the first production canary — I deliberately introduced an error condition, confirmed the automated rollback trigger fired, and verified that 100% of traffic reverted to the stable version within 60 seconds. I also checked that no requests were dropped during the rollback.

**Result:** The canary approach caught one defect in the first production release — an increased p95 latency at the 5% stage triggered the automatic rollback before any meaningful customer impact. Without canary deployment, this would have been a full production incident. The planned rollback test revealed that a misconfigured rollback threshold had been set too aggressively — it would have triggered on normal traffic variance — and was corrected before the first production canary.

## Key Terms
| Term | Definition |
|------|------------|
| **Blue-Green Deployment** | A deployment strategy that maintains two identical production environments (blue and green), switching traffic between them at release to enable instant rollback |
| **Canary Deployment** | A deployment strategy that routes a small percentage of production traffic to the new version while the majority continues on the stable version, expanding incrementally as confidence grows |
| **Traffic Split** | The configured distribution of incoming requests between the stable and canary versions, typically expressed as a percentage |
| **Canary Analysis** | The automated comparison of the canary version's quality metrics against the stable baseline, used to determine whether the canary should advance or roll back |
| **Automatic Rollback** | A deployment mechanism that reverts traffic to the stable version when canary metrics exceed defined thresholds, without requiring manual intervention |
| **Rollback Threshold** | The metric value (e.g. error rate or p95 latency) at which an automatic rollback is triggered, calibrated from baseline data to avoid false positives |
| **Stable Version** | The current production release receiving the majority of traffic during a canary deployment, serving as the baseline for metric comparison |
| **Canary-Specific Header** | An HTTP request header used to force routing to the canary version during testing, enabling direct validation of the new deployment without changing traffic percentages |
