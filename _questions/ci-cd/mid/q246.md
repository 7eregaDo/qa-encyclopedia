## The Intent
The interviewer wants to see whether your quality thinking extends beyond application code to the infrastructure that runs it. Infrastructure defects — misconfigured resources, invalid Kubernetes manifests, incorrect environment variables — cause production incidents just as frequently as application bugs, but are often untested. Mid-level QEs who understand IaC testing are significantly more effective partners to DevOps and platform teams.

## The STAR Script
**Situation:** The team I worked with managed infrastructure using Terraform for cloud resources and Helm charts for Kubernetes deployments. Infrastructure changes were reviewed in pull requests but never automatically validated — a misconfigured Terraform module had previously caused a production incident when an S3 bucket was accidentally made publicly readable.

**Task:** Introduce automated quality checks for infrastructure changes in the CI pipeline to catch misconfigurations before they were applied.

**Action:** I introduced four checks as pipeline stages triggered on changes to infrastructure files. Terraform validation: I added `terraform validate` and `terraform fmt --check` to confirm the configuration was syntactically valid and consistently formatted. I also added `tfsec` — a static analysis tool for Terraform — which scanned for common security misconfigurations such as open security groups, unencrypted S3 buckets, and missing resource tags. Critical findings failed the pipeline. Terraform plan review: I configured `terraform plan` to run against a non-production environment and publish the plan output as a PR comment. This gave reviewers a human-readable summary of what would change — resources created, modified, or destroyed — before approving the PR. Helm chart linting: I added `helm lint` to validate chart syntax and `helm template | kubeval` to validate that the rendered Kubernetes manifests conformed to the Kubernetes API schema for the target cluster version. This caught a missing required field in a Deployment spec that would have failed silently at apply time. Configuration file testing: for application configuration files (YAML-based feature flags and service config), I wrote a schema validation step using `yamllint` and a custom JSON Schema check that verified required fields were present and correctly typed.

**Result:** The tfsec scan caught three security misconfigurations in the first two weeks — including an IAM policy with overly broad permissions. The `terraform plan` PR comment became popular with reviewers who had previously struggled to understand the impact of infrastructure PRs. Zero infrastructure-related production incidents occurred in the six months following the pipeline additions.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that infrastructure is defined in code; may manually review Terraform or Helm changes without automated validation |
| **Mid** | Implements linting, static analysis, and plan/diff review for IaC in CI; adds schema validation for configuration files; catches security misconfigurations automatically |
| **Senior** | Designs the full IaC quality strategy; introduces policy-as-code enforcement (e.g. OPA/Conftest); implements integration testing for infrastructure modules; defines infrastructure change risk tiers with corresponding gate criteria |

## Key Terms
| Term | Definition |
|------|------------|
| **Infrastructure as Code (IaC)** | The practice of defining and managing infrastructure (cloud resources, Kubernetes deployments, network configuration) through version-controlled code files rather than manual processes |
| **Terraform** | A widely used open-source IaC tool that declares cloud infrastructure resources in HCL files and manages their lifecycle through plan, apply, and destroy operations |
| **terraform validate** | A Terraform command that checks the syntax and internal consistency of configuration files without connecting to any provider |
| **tfsec** | An open-source static analysis tool for Terraform that identifies security misconfigurations such as open network ports, unencrypted storage, and over-permissive IAM policies |
| **Helm** | A Kubernetes package manager that templates and deploys applications as versioned "charts", enabling parameterised, repeatable Kubernetes deployments |
| **helm lint** | A Helm command that validates the syntax and structure of a Helm chart, checking for common errors before deployment |
| **kubeval** | A tool that validates Kubernetes manifest files against the official Kubernetes API schema, catching invalid resource specifications before they are applied to a cluster |
| **terraform plan** | A Terraform command that calculates and displays the changes that would be made to infrastructure if the current configuration were applied, without making any actual changes |
