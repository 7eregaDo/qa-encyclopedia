## The Intent
The interviewer wants to assess whether you've solved the practical problems of environment management that make pipelines unreliable — shared environment contention, configuration drift, and inconsistent provisioning. Mid-level QEs should have hands-on experience with at least one approach to environment isolation in CI.

## The STAR Script
**Situation:** My team's integration test stage was failing intermittently because multiple PR pipelines were sharing a single staging environment. Two concurrent pipelines would deploy different versions of the service simultaneously, causing one to test against the wrong deployment. Tests were also occasionally failing due to state left by previous runs.

**Task:** Design an environment strategy for the CI pipeline that eliminated contention between concurrent runs and ensured a clean state for each test execution.

**Action:** I implemented ephemeral environments using Docker Compose for integration tests and Kubernetes namespaces for the staging stage. For the integration test stage: I added a `docker-compose up -d` step at the start of each pipeline run that launched the service under test and its dependencies (a PostgreSQL container and a stub server) in isolation. Each run had its own containers on dynamically assigned ports — no sharing, no contention. A `docker-compose down -v` teardown step at the end (using a `post` job that ran even on failure) ensured no state persisted. For the staging environment: I worked with the platform team to provision a Kubernetes namespace per feature branch using a Helm chart and a cleanup CronJob that deleted namespaces older than 48 hours. The deploy step in the PR pipeline targeted `feature-{branch-name}` namespace, not the shared staging namespace. The shared staging namespace was reserved for main branch builds only. I also added a `wait-for-healthy` step after each deployment that polled the health endpoint until the service was ready, preventing tests from starting against a service still in its startup phase.

**Result:** Integration test flakiness caused by environment contention dropped to zero within two weeks of the rollout. Pipeline concurrency increased — teams could run multiple PRs simultaneously without blocking each other. The per-namespace feature environments also gave developers a live URL to share with stakeholders for feedback before merging.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests in a shared environment; aware that shared environments cause contention; follows environment setup steps in the pipeline |
| **Mid** | Implements ephemeral environments using Docker Compose or Kubernetes namespaces; configures teardown as a post-job step; adds health check waits; eliminates contention for concurrent runs |
| **Senior** | Designs the environment strategy across all pipeline stages; introduces environment templating, configuration management, and automated drift detection |

## Key Terms
| Term | Definition |
|------|------------|
| **Ephemeral Environment** | A short-lived, automatically provisioned environment created for a specific pipeline run or branch and destroyed after its purpose is served |
| **Docker Compose** | A tool for defining and running multi-container Docker applications, commonly used in CI to provision a service and its dependencies in isolation |
| **Kubernetes Namespace** | A logical partition within a Kubernetes cluster used to isolate resources, enabling multiple independent environments to coexist on shared infrastructure |
| **Helm Chart** | A package of Kubernetes configuration templates that can be parameterised and deployed to create consistent environments on demand |
| **Environment Contention** | The conflict that occurs when multiple pipeline runs share an environment, causing deployments or test data from one run to interfere with another |
| **Configuration Drift** | The gradual divergence of an environment's configuration from its intended state, caused by manual changes or failed teardown steps accumulating over time |
| **Wait-For-Healthy** | A pipeline step that polls a service's health endpoint after deployment until it returns a success response, preventing tests from starting before the service is ready |
| **CronJob** | A Kubernetes resource that runs a scheduled task at defined intervals, used here to clean up expired ephemeral namespaces automatically |
