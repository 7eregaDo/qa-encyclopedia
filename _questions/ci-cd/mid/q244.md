## The Intent
The interviewer wants to see that you can move beyond "tests pass = ship it" and define what quality actually means at each promotion point. Quality gates can cover test coverage thresholds, security scan severity limits, performance benchmarks, and more. They want to know you've designed and implemented these and managed the organisational dynamics when they fail.

## The STAR Script
**Situation:** The team I worked with had an informal standard — "all tests must pass" — as the only criterion for deployment. After a production incident caused by a known critical security vulnerability that had been in a dependency for three months, the engineering manager asked me to define more explicit quality gates for the deployment pipeline.

**Task:** Design and implement quality gates appropriate to each pipeline stage, covering not just test results but the additional quality dimensions we'd been ignoring.

**Action:** I defined gates for three pipeline stages. PR gate (required to merge): all unit and API tests pass; no new critical or high findings from the static analysis tool (SonarQube); no new critical severity dependency vulnerabilities (checked via `pip-audit` or `npm audit`). I configured SonarQube to fail the pipeline if the new code introduced issues — not if historical issues existed, which would have blocked all work. Staging promotion gate (required to deploy to staging): integration tests pass; contract tests pass for all registered consumers; E2E smoke suite passes. Pre-production gate (required to deploy to production): staging environment has been stable for a minimum of 30 minutes post-deployment (measured via the synthetic monitor); no open severity-1 or severity-2 defects in the release scope; explicit QE sign-off recorded in the release ticket. I implemented the gates as pipeline jobs with explicit failure messages — when a gate failed, the error message named the specific failing criterion and linked to the remediation guidance. I also introduced an override mechanism: any gate could be bypassed with a named approver's sign-off recorded in the pipeline run, which was logged and reported weekly.

**Result:** The dependency vulnerability gate caught two critical CVEs in the first month — both were patched before the release deployed. The override mechanism was used three times in the first quarter, all for the staging stability gate during urgent hotfixes; the override log gave leadership visibility into when standards were being waived and why.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Follows existing quality gates; understands that gates prevent deployment when criteria aren't met |
| **Mid** | Designs and implements quality gates across pipeline stages; defines criteria beyond test pass rate (security, coverage, performance); implements override mechanisms with audit logging |
| **Senior** | Designs the quality gate framework for the full delivery pipeline; defines SLO-based gates; integrates business risk into gate criteria; presents gate effectiveness data to engineering leadership |

## Key Terms
| Term | Definition |
|------|------------|
| **Quality Gate** | A measurable criterion a build must satisfy before progressing to the next pipeline stage or environment, making deployment standards explicit and automated |
| **Static Analysis** | Automated inspection of source code without executing it, identifying code quality issues, security vulnerabilities, and style violations |
| **SonarQube** | A widely used static analysis platform that measures code quality metrics including bugs, vulnerabilities, code smells, and test coverage |
| **Dependency Vulnerability** | A known security flaw in a third-party library used by the application, detected by tools like `npm audit` or `pip-audit` and classified by severity |
| **CVE (Common Vulnerabilities and Exposures)** | A publicly disclosed security vulnerability in software, assigned a unique identifier and severity score, used to prioritise dependency patching |
| **Staging Stability** | The property of a staging environment that has been running healthily for a defined period post-deployment, used as a gate criterion before production promotion |
| **Override Mechanism** | A documented process for bypassing a quality gate with explicit named approval and audit logging, providing an escape valve without silently eroding standards |
| **Gate Failure Message** | The specific, actionable error output displayed when a quality gate fails, naming the criterion and linking to remediation guidance |
