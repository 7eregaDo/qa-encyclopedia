## The Intent
The interviewer wants to see that you can translate the testing pyramid into a concrete pipeline structure. Mid-level QEs should be able to configure pipeline stages, justify the placement of each test type, and explain the trade-off between coverage and speed at each gate. They want practical implementation experience, not just conceptual knowledge.

## The STAR Script
**Situation:** I joined a team whose CI pipeline had a single stage that ran all tests sequentially — unit, integration, and E2E — taking 42 minutes on every push. Developers were not waiting for results and the pipeline had become effectively decorative.

**Task:** Redesign the pipeline into meaningful stages that gave fast feedback on common changes while maintaining full coverage before merge.

**Action:** I redesigned the pipeline into three stages using GitHub Actions. Stage one — fast feedback (every push, target: under 4 minutes): lint and static analysis, then unit tests run in parallel across four runners using a matrix strategy. Any failure cancelled remaining runners immediately via `fail-fast: true`. This stage ran on every push to any branch. Stage two — integration confidence (every PR, target: under 12 minutes): API integration tests and contract verification ran in parallel. I split the API tests into four groups by feature domain and distributed them across runners using dynamic test allocation. A service health check step ran first — if the dependent services weren't healthy, the stage failed immediately with a clear message rather than timing out after 12 minutes. Stage three — pre-merge validation (PR targeting main, target: under 20 minutes): the curated E2E suite (30 critical-path journeys) ran, plus a security scan and a dependency vulnerability check. I configured this stage to require manual approval for any E2E failure — a failed E2E was not auto-blocking on its own because of known environment variability, but it required a QE to explicitly mark it as flakiness before merge was permitted. Each stage published JUnit XML results to GitHub's test reporting, so failures showed inline on the PR with file and line context.

**Result:** Median time-to-feedback on a push dropped from 42 minutes to 3.8 minutes for the fast-feedback stage. Developer pipeline-bypass behaviour stopped within a week. The three-stage model was documented as the team's pipeline standard and reused for two new services onboarded that quarter.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Understands that tests should be split by speed; follows an existing multi-stage pipeline; adds tests to existing stages |
| **Mid** | Designs and configures a multi-stage pipeline; implements parallel execution with matrix strategies; defines stage gates and failure behaviour; publishes test reports inline on PRs |
| **Senior** | Architects the pipeline for an entire product area; introduces test impact analysis; defines pipeline duration SLAs; integrates quality metrics collection into pipeline runs |

## Key Terms
| Term | Definition |
|------|------------|
| **Multi-Stage Pipeline** | A CI pipeline organised into discrete phases of increasing test scope and duration, with each stage acting as a quality gate before the next begins |
| **Matrix Strategy** | A CI configuration that runs the same job in parallel across multiple parameter combinations (e.g. test groups, OS versions), reducing overall execution time |
| **fail-fast** | A pipeline configuration option that cancels all parallel runners in a stage as soon as one fails, preventing wasted execution time |
| **Dynamic Test Allocation** | The distribution of test files across parallel runners based on historical execution times, ensuring runners finish at roughly the same time |
| **Service Health Check** | A pre-stage step that verifies required dependent services are running and responsive before executing integration tests, preventing misleading failures |
| **JUnit XML** | A standard test result format supported by most CI platforms, enabling pass/fail reporting inline in pull requests with file and line context |
| **Manual Approval Gate** | A pipeline step requiring a named human to explicitly approve before execution continues, used for high-risk stages or ambiguous failures |
| **Stage Gate** | The pass/fail criterion that determines whether a pipeline stage's output permits progression to the next stage |
