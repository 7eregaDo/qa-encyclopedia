## The Intent
The interviewer wants to confirm you treat security as a continuous engineering concern embedded in the pipeline, not a periodic audit by a separate team. Mid-level QEs increasingly own security scanning integration as part of pipeline ownership. They want to see concrete tool knowledge and an understanding of how to configure scans without blocking all development on false positives.

## The STAR Script
**Situation:** The team I worked with had no security scanning in the pipeline. A dependency audit run manually by the security team had found seven high-severity CVEs in production dependencies — some months old. The engineering manager asked me to ensure vulnerabilities were detected in the pipeline rather than via periodic manual audits.

**Task:** Integrate three types of security scanning into the CI pipeline with appropriate severity thresholds and minimal false positive noise.

**Action:** I implemented three scanning types as pipeline stages. Dependency vulnerability scanning: I added `pip-audit` (for Python) and `npm audit --audit-level=high` (for the frontend) as early steps in the PR pipeline. I configured them to fail only on high and critical findings, allowing medium and low to generate warnings without blocking. I created a known-exceptions file for accepted risks with documented justification and review dates — without this, every team member would have to investigate the same false positives independently. SAST (Static Application Security Testing): I integrated Semgrep with a curated ruleset focused on the security patterns most relevant to our codebase (SQL injection, unsafe deserialisation, hardcoded secrets). I ran it on changed files only, not the full codebase, to keep it fast. New critical or high findings blocked the PR; findings in existing code created tickets rather than blocking. Container image scanning: I added Trivy to the pipeline's build stage, scanning the Docker image after it was built. Critical CVEs in the base image or installed packages failed the build, prompting a base image update. I set up a weekly scheduled scan of all production images in the registry to catch newly disclosed CVEs in already-deployed images.

**Result:** In the first month, the dependency scan caught three high-severity CVEs on PR creation — all were patched before merge. The scheduled image scan caught a critical CVE in a base image two weeks after the image had been deployed, triggering a patching sprint. The security team reduced their manual audit frequency from monthly to quarterly, citing the pipeline coverage.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that security scanning tools exist; runs them manually when asked; reads their output |
| **Mid** | Integrates dependency scanning, SAST, and container image scanning into CI pipeline; configures severity thresholds; manages known-exceptions; introduces scheduled scans for deployed images |
| **Senior** | Designs the security scanning strategy across the full pipeline; introduces DAST in pre-production; defines CVE response SLAs; integrates findings into the security defect backlog |

## Key Terms
| Term | Definition |
|------|------------|
| **SAST (Static Application Security Testing)** | Automated analysis of source code to identify security vulnerabilities without executing the application, run as a pipeline step on code changes |
| **Dependency Vulnerability Scanning** | Automated checking of third-party library versions against known CVE databases, identifying insecure dependencies before they reach production |
| **Container Image Scanning** | Analysis of a Docker image's layers and installed packages for known security vulnerabilities, run after the image is built in CI |
| **Trivy** | An open-source container and filesystem vulnerability scanner that checks images and dependencies for known CVEs |
| **Semgrep** | An open-source static analysis tool that uses pattern-matching rules to identify security vulnerabilities and code quality issues in source code |
| **Known-Exceptions File** | A configuration file that lists accepted security findings with documented justification and review dates, preventing repeated investigation of accepted risks |
| **Severity Threshold** | The minimum vulnerability severity level that causes a pipeline failure, allowing lower-severity findings to be reported as warnings without blocking development |
| **Scheduled Scan** | A CI pipeline run triggered by a time schedule rather than a code change, used to detect newly disclosed CVEs in already-deployed dependencies or images |
