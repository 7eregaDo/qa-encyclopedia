## The Intent
The interviewer wants to see architectural thinking about CI/CD — not just how to configure one pipeline, but how to design a pipeline system that remains coherent and performant as the number of engineers and services doubles or triples. They want trade-off reasoning and evidence of having solved scale problems in practice.

## The STAR Script
**Situation:** I joined a company at 40 engineers with 8 services and a single shared Jenkins instance everyone pushed to. As the company grew to 120 engineers and 30 services over 18 months, the pipeline became a bottleneck — queuing times exceeded 20 minutes, configuration was duplicated across services with slight variations, and there was no standard quality gate definition.

**Task:** Redesign the pipeline architecture to scale with continued growth without requiring per-service bespoke pipeline maintenance.

**Action:** I approached the design around three principles. Shared components, local execution: I migrated from Jenkins to GitHub Actions and introduced a shared reusable workflow library — a separate repository of composable workflow templates for common stages (unit tests, security scan, Docker build, deployment). Each service's pipeline was a thin caller file of 30 lines that referenced the shared workflows by version tag. Changing a quality standard in one place propagated to all 30 services on their next run. Self-hosted runners with auto-scaling: I replaced the single Jenkins instance with auto-scaling GitHub Actions runner groups on AWS EC2 using `actions-runner-controller`. Runners scaled from 0 to 40 instances in response to queue depth, eliminating the 20-minute wait. Large test jobs ran on beefy instances; lint and unit test jobs ran on smaller, cheaper ones. Tiered quality contracts: I defined three pipeline tiers — standard, elevated, and critical — based on service risk profile. Standard services ran unit, integration, and security scans. Elevated services added contract tests and performance benchmarks. Critical services (payments, auth) added full E2E suites and a manual approval gate before production. Each tier was a named reusable workflow; services opted into the appropriate tier in their caller file.

**Result:** Median pipeline wait time dropped from 20 minutes to under 90 seconds. Onboarding a new service to CI required creating a 30-line caller file — achieved by any engineer in under an hour. When I later needed to add a supply chain security step (SBOM generation) to all pipelines, it took one PR to the shared workflow library and all 30 services picked it up within 24 hours.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Works within a single pipeline; follows existing patterns |
| **Mid** | Configures pipelines for individual services; introduces reuse within a service; identifies bottlenecks |
| **Senior** | Designs the shared pipeline architecture; implements reusable workflow libraries; introduces auto-scaling execution; defines tiered quality contracts; enables one-PR-propagates-to-all updates |

## Key Terms
| Term | Definition |
|------|------------|
| **Reusable Workflow** | A GitHub Actions workflow defined in a shared repository that other repositories can call by reference, enabling one-place updates to propagate across all consumers |
| **actions-runner-controller** | A Kubernetes-based controller for GitHub Actions that manages self-hosted runner lifecycle, enabling auto-scaling runner pools that respond to queue depth |
| **Self-Hosted Runner** | A compute instance managed by the engineering team that executes GitHub Actions jobs, providing more control over performance, cost, and tooling than cloud-hosted runners |
| **Pipeline Tier** | A named quality contract defining which test stages and gates apply to a class of services, assigned based on the service's risk profile |
| **Caller File** | A thin pipeline configuration file in each service repository that references shared workflow components by version, delegating quality gate implementation to the shared library |
| **Queue Depth** | The number of pipeline jobs waiting for an available runner at a given moment, used as the auto-scaling signal for runner pool expansion |
| **SBOM (Software Bill of Materials)** | A structured inventory of all components and dependencies in a software artefact, used for supply chain security analysis and vulnerability tracking |
| **Auto-Scaling Runner** | A pipeline execution environment that automatically provisions additional runners when demand increases and terminates them when the queue clears, optimising cost and throughput |
