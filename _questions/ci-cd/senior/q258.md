## The Intent
The interviewer wants to see whether you apply the same observability thinking to the pipeline that you'd apply to a production service. A pipeline with no metrics is a black box — you know it's slow or unreliable, but you can't measure the improvement, find the bottleneck, or detect regression. Senior QEs who instrument their pipelines are significantly more effective at improving them.

## The STAR Script
**Situation:** After redesigning the CI pipeline for a 30-service platform, I had qualitative feedback that it was "much better" but no data to quantify improvement, present to leadership, or use to find remaining bottlenecks. The pipeline was also experiencing periodic slowdowns that nobody could pinpoint because we had no per-stage duration history.

**Task:** Design and implement observability for the pipeline that made its performance and reliability measurable, alertable, and historically queryable.

**Action:** I implemented three observability layers. Structured metrics collection: at the end of every pipeline run I added a step that posted a structured JSON event to a metrics API — containing pipeline name, branch, trigger type, overall duration, per-stage duration, pass/fail outcome, and the test result counts. I ingested these events into our existing Grafana/Loki stack. Dashboards: I built a Grafana dashboard with four panels — pipeline pass rate (7-day rolling), median and p95 duration by stage, flakiness rate by test suite (computed from pass/fail on the same commit SHA), and a top-10 slowest test cases table updated nightly from test result JSON. This gave the team a single view of pipeline health without querying raw logs. Alerting: I configured alerts for three conditions — pipeline pass rate dropping below 85% for any 24-hour window, any stage exceeding its duration SLA (unit: 5 minutes, integration: 15 minutes, E2E: 25 minutes), and any test appearing in the flaky list for more than 7 consecutive days without a fix ticket. Distributed tracing for pipeline jobs: for the integration test stage, I instrumented the test runner to emit OpenTelemetry spans — one span per test, with attributes including test name, suite, and outcome. This gave me a Jaeger trace of the full integration stage, making it trivially easy to find which specific test was the slowest on any given run.

**Result:** The dashboard identified that a single test in the integration suite accounted for 23% of the stage's total duration — a database-heavy test that was running full table scans. Optimising it reduced the integration stage from 14 to 9 minutes. The flakiness alert caught a test that had been quietly flaky for three weeks without any tracking ticket being raised.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Reads pipeline logs; checks pass/fail status manually |
| **Mid** | Tracks pipeline metrics in a spreadsheet or simple dashboard; identifies slow stages visually from pipeline run history |
| **Senior** | Instruments the pipeline with structured metrics and alerts; builds Grafana dashboards for pipeline health; uses distributed tracing to identify per-test bottlenecks; connects pipeline observability to engineering leadership reporting |

## Key Terms
| Term | Definition |
|------|------------|
| **Pipeline Observability** | The instrumentation of a CI/CD system to make its performance and reliability metrics queryable, visualisable, and alertable |
| **Structured Pipeline Event** | A JSON payload emitted at the end of each pipeline run containing metadata about its outcome, duration, and test results, ingested into a metrics or logging system |
| **Grafana** | An open-source data visualisation platform used to build dashboards from metrics stored in systems like Prometheus, Loki, or InfluxDB |
| **Pipeline Duration SLA** | A defined maximum acceptable runtime for each pipeline stage, used as the threshold for alerting when stages become unacceptably slow |
| **OpenTelemetry** | A vendor-neutral observability framework for generating and collecting traces, metrics, and logs, applicable to both applications and pipeline instrumentation |
| **Per-Test Span** | An OpenTelemetry trace span representing a single test case's execution, enabling bottleneck identification at individual test granularity |
| **Flakiness Alert** | A monitoring alert triggered when a test has been in the flaky quarantine list for a defined period without a remediation ticket, preventing silent accumulation |
| **Duration Hotspot** | A specific test or pipeline step that contributes disproportionately to the total stage runtime, identified through tracing or duration attribution analysis |
