## The Intent
The interviewer wants to see that you treat rollback as a first-class quality concern, not an afterthought. Many teams discover their rollback procedure doesn't work during an actual incident — the worst possible time. Senior QEs should be able to design a rollback strategy, account for the data compatibility challenges that make rollbacks complex, and advocate for regular rollback testing.

## The STAR Script
**Situation:** After a production incident where a rollback had taken 45 minutes — because the procedure was undocumented, the database migration was not reversible, and nobody had confirmed the previous Docker image tag before the deployment — I was asked to lead a rollback strategy improvement initiative.

**Task:** Design a rollback strategy that was fast, documented, tested, and accounted for the database migration problem.

**Action:** I addressed four components. Rollback mechanism: for application rollbacks, I standardised on redeploying the previous Docker image tag via the deployment pipeline — a named pipeline step `deploy:rollback` that took a target version as input. This was faster and safer than reverting code and rebuilding. The previous image was retained in the registry for 30 days, and the last three deployed versions were explicitly tagged as `stable-1`, `stable-2`, `stable-3` for instant reference. Database migration compatibility: I introduced the rule that every migration must be backwards-compatible with the previous application version for at least one release cycle. This meant rolling back the application code was always safe without needing to rollback the schema. For destructive migrations (column removals), they were deferred to a second release after the feature was fully deployed and confirmed stable. Rollback trigger criteria: I defined explicit, measurable triggers for initiating a rollback — error rate exceeding 1% for more than 5 minutes, p95 latency exceeding 2x baseline for more than 3 minutes, or a severity-1 defect confirmed in the release. These were in the runbook, not left to judgement under pressure. Planned rollback exercises: I introduced a quarterly "rollback drill" — we deployed a release to staging and then executed a rollback, timing the procedure and verifying that the previous version was healthy afterwards. The drill was documented and the outcome reported in the engineering newsletter.

**Result:** Mean time to rollback in production dropped from 45 minutes to under 8 minutes. The quarterly drill caught two issues — one where the `stable-1` tag had not been updated correctly, and one where a migration backwards-compatibility rule had been violated. Both were fixed before they could affect a production rollback.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that rollback is a recovery option; knows to escalate when a deployment needs to be reversed |
| **Mid** | Follows a documented rollback procedure; checks that the previous image tag is available before deploying; verifies health after rollback |
| **Senior** | Designs the rollback strategy including mechanism, data compatibility constraints, and trigger criteria; implements planned rollback exercises; defines rollback SLA; accounts for migration rollback complexity |

## Key Terms
| Term | Definition |
|------|------------|
| **Rollback** | The act of reverting a deployed system to a previous known-good version in response to a quality or reliability failure |
| **Rollback SLA** | A defined maximum time within which a rollback must complete, establishing an operational target for the speed of recovery |
| **Backwards-Compatible Migration** | A database schema change that the previous application version can operate against without errors, enabling application rollback without schema rollback |
| **Stable Tag** | An explicitly named Docker image tag (e.g. `stable-1`) pointing to the previous deployed version, maintained for instant rollback without searching the registry |
| **Rollback Trigger Criteria** | Explicitly defined, measurable conditions (error rate, latency, confirmed defect) that initiate a rollback, preventing ambiguous or delayed rollback decisions under incident pressure |
| **Rollback Drill** | A planned, non-emergency execution of the rollback procedure in a non-production environment, used to verify the procedure works and identify gaps before they matter |
| **Destructive Migration** | A database schema change that removes or renames data — such as dropping a column — which cannot be reversed without data loss and requires special release sequencing |
| **deploy:rollback** | A named, parameterised pipeline step that redeploys a specified previous image version, providing a single-command rollback mechanism with a full audit trail |
