## The Intent
The interviewer is testing whether your security thinking extends upstream of the application code to the pipeline itself and its inputs. Supply chain attacks (like the SolarWinds or XZ Utils incidents) compromise software through trusted dependencies or build tooling rather than the application directly. Senior QEs who understand supply chain security are increasingly valuable as this attack surface becomes more commonly exploited.

## The STAR Script
**Situation:** Following industry-wide awareness of supply chain attacks, our CISO asked engineering to assess the pipeline's exposure to dependency tampering and build tool compromise. I led the assessment and subsequent remediation as the senior QE responsible for the pipeline.

**Task:** Identify supply chain security gaps in the CI/CD pipeline and implement controls proportionate to the risk.

**Action:** I addressed four exposure areas. Dependency pinning: the Python and Node.js services were using version ranges in their dependency files (`^2.3.0`) rather than exact versions. A compromised package release within the range would be silently adopted. I introduced lockfile enforcement — `pip install --require-hashes` for Python and committed `package-lock.json` for Node — ensuring only cryptographically verified exact versions were installed. I also added a step to fail the pipeline if the lockfile was not up to date with the manifest. SBOM generation: I added `syft` to the build stage to generate a Software Bill of Materials for every Docker image, stored in the artefact registry alongside the image. This gave us an auditable inventory of every component in production. Artefact signing: I implemented `cosign` to sign every Docker image with a key stored in AWS KMS after the build. The deployment step verified the signature before pulling the image — an unsigned image would not deploy. GitHub Actions pinning: CI workflows were referencing third-party actions by version tag (`uses: actions/checkout@v3`), which could be silently updated by the action's author. I replaced all third-party action references with their full commit SHA (`uses: actions/checkout@abc1234`) so the exact code running in our pipeline was immutable.

**Result:** The CISO assessment rated our pipeline supply chain posture as significantly improved. The lockfile enforcement caught one PR where a developer had added a dependency without updating the lockfile, revealing a transitive dependency with a known vulnerability. The commit-SHA pinning of GitHub Actions was adopted as a company standard for all new workflows.

## Key Terms
| Term | Definition |
|------|------------|
| **Software Supply Chain** | The collection of components, dependencies, tools, and processes used to build and deliver software, each representing a potential attack surface |
| **SBOM (Software Bill of Materials)** | A structured inventory of all components and their versions in a software artefact, enabling vulnerability tracking and licence compliance |
| **Artefact Signing** | The process of cryptographically signing a build artefact (e.g. a Docker image) so that its origin and integrity can be verified before deployment |
| **cosign** | An open-source tool from the Sigstore project for signing, verifying, and storing signatures for container images |
| **Lockfile** | A file that records the exact resolved versions and cryptographic hashes of all dependencies, ensuring reproducible and tamper-evident builds |
| **Dependency Pinning** | The practice of specifying exact dependency versions (or hashes) rather than version ranges, preventing silent adoption of new releases |
| **Syft** | An open-source tool that generates SBOMs from container images and filesystems in standard formats such as CycloneDX and SPDX |
| **Commit SHA Pinning** | The practice of referencing a third-party GitHub Actions workflow by its full commit hash rather than a version tag, ensuring the exact code is immutable |
