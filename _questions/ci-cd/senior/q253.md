## The Intent
The interviewer wants to see end-to-end thinking about the CD half of CI/CD — how a build artefact travels from the build step through to production and what quality evidence is accumulated at each stage. Senior QEs should be able to design the full promotion chain, justify the gates at each boundary, and explain what happens when a gate fails.

## The STAR Script
**Situation:** At a SaaS company, the deployment process was manual and inconsistent — different team members ran different checks at different points. Some releases went directly from CI to production; others sat in staging for days without clear criteria for when they were ready to progress. The result was both quality incidents (from under-testing) and delivery delays (from over-caution without defined criteria).

**Task:** Design a formal promotion strategy with explicit, automated quality gates at each environment boundary.

**Action:** I defined four environments and the gate criteria between each. Dev → Integration: triggered automatically on merge to main. Gate: all unit, API, and contract tests pass; SAST and dependency scan pass with no critical findings; Docker image built and pushed to registry with commit SHA tag. Integration → Staging: triggered automatically when integration environment is healthy for 10 minutes post-deployment. Gate: integration environment smoke tests pass; all registered Pact consumer contracts verified against the new provider version; no open severity-1 defects. Staging → Pre-production: triggered manually by QE after staging sign-off. Gate: full E2E suite passes on staging; performance benchmark meets SLA thresholds (p95 latency under 500ms at 2x expected load); synthetic monitor shows 30 minutes of stable staging operation; release notes reviewed and approved. Pre-production → Production: triggered manually with dual approval (QE lead + engineering manager). Gate: pre-production environment healthy for 60 minutes; rollback procedure confirmed executable; on-call engineer briefed. Each gate was implemented as a GitHub Actions environment protection rule with required reviewers and automated status checks. Failed gates sent a Slack alert to the owning team with the specific failing criterion.

**Result:** Release-related production incidents dropped by 65% in the following six months. Release cadence increased — paradoxically, having defined criteria made releases faster because teams knew exactly what was required rather than defaulting to caution. The promotion chain became the team's shared mental model for "what does ready to ship mean."

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Follows existing promotion steps; checks that criteria are met before asking for sign-off |
| **Mid** | Defines and implements quality gates for individual environment transitions; automates promotion triggers; reports on gate failures |
| **Senior** | Designs the full promotion chain across all environments; defines gate criteria for each boundary; implements automated promotion with manual approval integration; connects promotion gates to SLOs |

## Key Terms
| Term | Definition |
|------|------------|
| **Release Promotion** | The act of advancing a build artefact from one environment to the next after meeting defined quality gate criteria |
| **Environment Protection Rule** | A GitHub Actions configuration that requires specific status checks and named reviewers to pass before a deployment to a protected environment is allowed |
| **Stability Window** | A defined time period during which an environment must remain healthy post-deployment before promotion to the next environment is triggered |
| **Dual Approval** | A release gate requiring two named individuals from different roles (e.g. QE lead and engineering manager) to explicitly approve before production deployment |
| **Promotion Trigger** | The event or condition that initiates a promotion check — either automatic (environment stability confirmed) or manual (QE sign-off) |
| **Pact Consumer Contract Verification** | A promotion gate check that confirms all registered consumer contracts pass against the new provider version before the provider is promoted |
| **Release Notes Review** | A gate step ensuring that the changes in a release are documented and reviewed before production deployment, informing stakeholders and the on-call team |
| **Promotion Chain** | The complete sequence of environments and gates through which a build artefact must pass from initial build to production deployment |
