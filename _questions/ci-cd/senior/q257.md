## The Intent
The interviewer wants to see whether you can resolve the tension between performance testing rigour and CI/CD speed. Full load tests can take hours; running them on every commit would destroy feedback loops. Senior QEs need a tiered approach that catches regressions early without making every PR an hour-long wait.

## The STAR Script
**Situation:** The platform I worked on had performance tests — a k6 suite — but they only ran manually before major releases. Three performance regressions had reached production between releases because nobody ran the suite on ordinary PRs. The tests took 35 minutes to run, which the team used as justification for not running them in CI.

**Task:** Design a performance testing strategy that caught regressions in CI without adding 35 minutes to every PR pipeline.

**Action:** I redesigned the performance testing into three tiers. Tier 1 — micro-benchmarks (every push, under 60 seconds): I identified the five most performance-sensitive functions — those handling high query volumes or complex calculations — and wrote pytest-benchmark tests for each. The benchmark ran on every push and compared results against the previous run's baseline stored in the pipeline cache. A regression of more than 15% in any benchmark failed the pipeline with a specific message. Tier 2 — component load tests (on merge to main, under 8 minutes): I wrote focused k6 scripts targeting individual API endpoints at 2x expected load for 3 minutes. These ran against the integration environment after merge, not on every PR. Results were compared against a 30-day rolling baseline stored in S3; a p95 latency regression of more than 20% or an error rate above 0.5% triggered a Slack alert and a rollback recommendation. Tier 3 — full suite (nightly and pre-release): the original 35-minute suite ran nightly and before every planned production release. This provided comprehensive soak and stress test data without blocking the daily development flow. I also introduced performance budget files — a YAML file per service defining the p95 latency and error rate SLAs — so the comparison logic was driven by documented targets, not hardcoded numbers.

**Result:** Two performance regressions were caught by the micro-benchmarks within the first month — one a 40% slowdown in a query function from an inefficient ORM change, one a memory allocation regression. Neither would have been caught before the next manual test cycle. The tier 2 tests caught one regression post-merge before it reached staging. The PR pipeline duration did not increase.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that performance testing exists; runs manual k6 or JMeter tests when asked |
| **Mid** | Writes and executes load tests; integrates a single performance test stage into CI; compares results against fixed thresholds |
| **Senior** | Designs the tiered performance testing strategy; implements micro-benchmarks for per-push feedback; uses rolling baselines for regression detection; maintains performance budget files as version-controlled service contracts |

## Key Terms
| Term | Definition |
|------|------------|
| **Micro-Benchmark** | A performance test targeting a specific function or code path, running in milliseconds and suitable for per-commit CI execution |
| **pytest-benchmark** | A pytest plugin that measures the execution time of test functions and compares results against stored baselines, failing if regressions exceed a defined threshold |
| **Component Load Test** | A focused performance test targeting a specific API endpoint or service component at elevated load, faster than a full system test and suitable for post-merge CI |
| **Rolling Baseline** | A performance reference value calculated from a moving window of recent measurements, adapting to legitimate performance changes while detecting regressions |
| **Performance Budget** | A version-controlled file defining the acceptable performance thresholds for a service — such as p95 latency and error rate — used to drive automated regression detection |
| **k6** | An open-source performance testing tool that uses JavaScript scripts to simulate virtual users and measure API response times and error rates |
| **Soak Test** | A performance test that applies sustained moderate load over an extended period to detect memory leaks and gradual performance degradation |
| **Latency Regression** | A measurable increase in response time compared to a baseline, used as the primary signal for detecting performance degradation introduced by a code change |
