## The Intent
The interviewer wants to see that you understand feature flags as both a delivery enabler and a quality challenge. Flags allow continuous deployment of incomplete features safely, but they introduce combinatorial testing complexity and accumulate technical debt if not governed. Senior QEs should have a view on the whole system — how flags are tested, how they're managed, and how their lifecycle is enforced.

## The STAR Script
**Situation:** I worked at a company that had adopted trunk-based development with feature flags as the mechanism for shipping incomplete features safely. Within a year, 60 active flags existed with no documented states, no testing requirements, and no removal process. Engineers were afraid to remove flags because they didn't know what would break.

**Task:** Design a feature flag quality strategy that ensured all flag states were tested, flags had a defined lifecycle, and the accumulation problem was arrested.

**Action:** I introduced a flag discipline across three dimensions. Testing contract: every new flag required a documented test matrix in its pull request — a table of flag states (on/off or variant values) and the test cases covering each. CI ran tests with the flag explicitly enabled and explicitly disabled using environment variable overrides. For flags affecting critical paths, both states were tested in the E2E suite. Flag classification and lifecycle: I classified all flags into three types. Experiment flags (max 30-day lifespan, tied to a specific feature or A/B test): these were created with a JIRA ticket for removal already in the backlog, due in 30 days. Release flags (used to progressively roll out a feature): these had a target activation date and a removal date 14 days after full rollout. Operational flags (kill switches for infrastructure): these were permanent but required annual review. I added a CI lint step that read flag metadata from our LaunchDarkly-exported config and warned when any experiment flag had exceeded its expiry date — making accumulation visible in every PR. Cleanup sprints: I ran quarterly flag removal sprints. I produced a flag age report from LaunchDarkly's API, sorted by age, and the team committed to removing the oldest 20% each quarter.

**Result:** The flag count dropped from 60 to 28 within two quarters. The lint step caught 11 expired flags that would otherwise have persisted indefinitely. Engineers reported more confidence in modifying flag-controlled code because the test matrix made flag interactions explicit.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests features in their flag-enabled state; may test the disabled state if reminded |
| **Mid** | Tests both flag states; documents flag coverage in test plans; raises flag removal as a task when features are fully rolled out |
| **Senior** | Designs the flag testing contract as a PR requirement; implements CI lint for flag expiry; classifies flags by lifecycle; runs flag cleanup sprints; integrates flag state management into the CD pipeline |

## Key Terms
| Term | Definition |
|------|------------|
| **Feature Flag** | A configuration mechanism that enables or disables application behaviour at runtime without a code deployment |
| **Trunk-Based Development** | A branching strategy where all developers commit to a single shared branch continuously, using feature flags to hide incomplete work rather than long-lived feature branches |
| **Flag State Matrix** | A table documenting all possible states of a feature flag and the test cases that cover each state, required as part of the flag's pull request |
| **Flag Classification** | The categorisation of a feature flag by its intended lifespan and purpose (experiment, release, or operational), governing its lifecycle and removal timeline |
| **Flag Expiry Lint** | A CI step that reads flag metadata and warns when a flag has exceeded its documented lifespan, making accumulation visible in pull request checks |
| **LaunchDarkly** | A feature flag management platform providing targeting, gradual rollout, A/B testing, and flag lifecycle management at scale |
| **Flag Removal Sprint** | A dedicated engineering effort to identify and remove expired or fully-rolled-out feature flags, reducing codebase complexity and testing surface |
| **Kill Switch** | A permanent operational flag that can instantly disable a feature or integration in production without a code deployment, used as a safety mechanism for high-risk components |
