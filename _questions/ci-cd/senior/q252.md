## The Intent
The interviewer is testing whether you can optimise pipeline speed through intelligent test selection rather than simply adding compute. TIA is a senior-level technique because it requires understanding the relationship between code and tests, managing the risk of the approach, and knowing when to fall back to a full suite run.

## The STAR Script
**Situation:** The unit test suite for a large Python monolith had grown to 3,200 tests taking 8 minutes to run. Most changes touched a small area of the codebase, yet every push waited for all 3,200 tests. A junior developer making a one-line change to a utility function waited the same 8 minutes as someone touching the core business logic.

**Task:** Implement test impact analysis to reduce the per-push feedback time for the common case while maintaining full coverage on higher-risk changes.

**Action:** I implemented TIA using pytest-cov and a custom selection script. During the full nightly suite run, I generated a coverage mapping — a JSON file recording which test covered which source file (using `--cov-report=json`). On each push, the pipeline ran `git diff --name-only HEAD~1 HEAD` to identify changed files, then queried the coverage map to find all tests that covered any of those files. Only those tests ran on push, plus a mandatory set of always-run tests (authentication, payments, data integrity — too critical to skip). The selected test set was passed to pytest via `-k` or as an explicit file list. I introduced two safety valves. Full suite fallback: if the changed file count exceeded 20 (indicating a broad change), the full suite ran — broad changes have a high probability of unexpected cross-cutting impact. Nightly full run: the complete suite always ran nightly and on any PR targeting main, ensuring no drift accumulated. I also added coverage map staleness detection — if the map was more than 24 hours old, TIA was bypassed and the full suite ran.

**Result:** Median per-push pipeline time dropped from 8 minutes to 1.4 minutes for single-file changes. Developer satisfaction with the push pipeline improved significantly. The nightly run continued to catch two to three regressions per month that TIA had correctly excluded from the push run — confirming the nightly run was the right safety net.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that not all tests need to run on every change; relies on existing test selection if configured |
| **Mid** | Separates fast and slow tests into stages; understands the concept of TIA; may implement basic file-based test selection |
| **Senior** | Implements coverage-map-based TIA with full suite fallbacks; manages coverage map freshness; introduces always-run test sets for high-risk areas; measures and reports TIA effectiveness |

## Key Terms
| Term | Definition |
|------|------------|
| **Test Impact Analysis (TIA)** | The technique of determining which tests are relevant to a given code change by analysing coverage or dependency relationships, enabling selective test execution |
| **Coverage Mapping** | A data structure that records which source files each test exercises, generated from a coverage instrumented test run and used as the input to TIA |
| **pytest-cov** | A pytest plugin that measures code coverage during test execution and generates reports in various formats, including JSON suitable for TIA |
| **Always-Run Test Set** | A defined subset of high-risk or cross-cutting tests that are always included regardless of TIA selection, providing a safety baseline |
| **Full Suite Fallback** | A TIA safety mechanism that bypasses selective execution and runs all tests when a change is broad enough that TIA's precision cannot be relied upon |
| **Coverage Map Staleness** | The condition where the coverage mapping data is outdated — because new tests or source files have been added since it was generated — making TIA unreliable |
| **Nightly Full Run** | A scheduled CI pipeline execution that runs the complete test suite regardless of TIA, serving as a safety net for any gaps in selective execution |
| **git diff** | A Git command that shows changed files between two commits, used in TIA to determine which source files a given change touches |
