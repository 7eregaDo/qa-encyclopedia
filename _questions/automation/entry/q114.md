category: automation

theory: "Good automation candidates share four characteristics: they are executed frequently enough to justify the build and maintenance investment, they are stable enough that the underlying functionality and UI do not change constantly, they have deterministic expected results that can be asserted programmatically, and they do not require human judgment to evaluate. Applying these criteria prevents wasted automation investment on tests that will either break constantly or never recoup their build cost."
---

## The Intent

The interviewer wants to see that you approach automation decisions analytically rather than defaulting to "automate everything." Entry-level automation engineers who cannot articulate selection criteria tend to build brittle, high-maintenance suites that consume more effort than they save. The ability to identify good candidates is as important as the ability to write the tests.

---

## The STAR Script

**Situation:** I was given a manual test suite of 200 test cases and asked to identify the highest-priority candidates for automation as part of a new automation initiative. There was no guidance on where to start — the entire suite was theoretically in scope.

**Task:** I needed to produce a prioritised shortlist of automation candidates with a clear rationale for each selection, so the team could focus build effort on the highest-value tests first.

**Action:** I scored each test case against four criteria on a simple 1–3 scale. **Execution frequency**: how often is this test run? Tests in the regression suite run every sprint scored 3; tests run only at major releases scored 1. **Stability**: how often does the underlying feature change? Core, mature features scored 3; rapidly evolving UI features scored 1. **Determinism**: does the test have a specific, assertable expected result? Tests with exact expected values scored 3; tests requiring visual or subjective assessment scored 1. **Complexity to automate**: is the test straightforward to script? Simple form interactions scored 3; tests requiring complex state setup or third-party interaction scored 1. Tests with a total score of 10 or higher were Tier 1 automation candidates. Scores of 7–9 were Tier 2. Below 7 were not recommended for automation at this stage. From the 200 cases, 47 scored Tier 1, 68 scored Tier 2, and 85 were below the threshold.

**Result:** I presented the scored matrix to the team. The 47 Tier 1 candidates were built first — covering the core regression suite for login, checkout, and account management. These were the most stable, most frequently executed tests in the suite. The automation paid back its build cost within three sprint cycles based on the time saved in manual regression. The scoring model was adopted as the team's standard automation selection framework going forward.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Apply clear criteria to identify automation candidates from a manual suite and explain the rationale for each dimension. |
| **Mid** | Build an automation ROI model incorporating build time, maintenance cost, and execution frequency — and use it to prioritise automation investment across a full product. |
| **Senior** | Design an organisational automation selection policy, integrate it with the test management system, and track automation coverage and ROI as standing engineering metrics. |

## Key Terms

| Term | Definition |
|------|------------|
| **Automation Candidate** | A test case that meets the criteria for being worthwhile to automate — frequent, stable, deterministic, and not judgment-dependent. |
| **Execution Frequency** | How often a test is run — higher frequency increases automation ROI. |
| **Stability** | How rarely the underlying feature or UI changes — higher stability reduces automation maintenance cost. |
| **Determinism** | The ability to specify an exact, programmatically assertable expected result. |
| **Automation ROI** | The return on investment from automating a test — time saved in execution minus time invested in build and maintenance. |
| **Tier Classification** | Grouping automation candidates by priority score — enables structured, highest-value-first build sequencing. |
