category: automation
level: entry
theory: "A locator is the mechanism used by an automation framework to identify and interact with a specific UI element on a page. Locator reliability is determined by how likely it is to break when the application changes — semantic, user-visible locators (such as accessible roles, labels, and text) are more stable than implementation-detail locators (such as auto-generated IDs or complex XPath expressions). The most reliable locator is the one that most closely reflects how a user identifies the element."
---

## The Intent

The interviewer is checking a fundamental practical skill — locator selection is one of the most common sources of test flakiness and maintenance cost in UI automation. Entry-level engineers who default to XPath or auto-generated IDs produce brittle suites. Those who understand locator hierarchy and prioritise semantic locators produce maintainable ones.

---

## The STAR Script

**Situation:** I was working on an automation suite where tests were failing after minor front-end deployments even when the functionality being tested had not changed. When I investigated, I found the tests were using XPath locators like `//div[@class='container']/form/div[2]/input` — which broke every time a developer added or removed a wrapper div.

**Task:** I needed to understand why these locators were fragile and replace them with more reliable alternatives.

**Action:** I learned to evaluate locators in a hierarchy from most to least reliable. First preference was **accessible locators** — ARIA roles, labels, and accessible names that reflect how assistive technologies and real users identify elements. In Playwright: `getByRole('button', { name: 'Submit' })` or `getByLabel('Email address')`. These are tied to semantics that change infrequently and also validate accessibility. Second preference was **test IDs** — `data-testid` attributes added specifically for automation, stable by convention and not affected by style or layout changes: `getByTestId('submit-button')`. Third preference was **stable CSS selectors** — IDs or specific class names that are intentionally stable: `#login-form-submit`. Fourth preference, used only as a last resort, was **XPath** — powerful but brittle when tied to DOM structure. For each fragile locator in the suite, I identified the most reliable available alternative. Where no stable option existed, I worked with a front-end developer to add `data-testid` attributes to the relevant elements.

**Result:** After the locator refactoring, the suite's flake rate from selector breakage dropped from 18% to 2% over the following month. The process of adding `data-testid` attributes also opened a collaborative relationship with the front-end team — they began proactively adding test IDs to new components as a standard practice.

## Key Terms

| Term | Definition |
|------|------------|
| **Locator** | A selector that identifies a UI element for interaction in an automated test. |
| **Accessible Locator** | A locator based on ARIA roles, labels, or accessible names — stable, semantic, and accessibility-validating. |
| **data-testid** | A custom HTML attribute added specifically for test automation — stable by convention and independent of style changes. |
| **CSS Selector** | A locator using CSS syntax to identify elements by ID, class, or attribute — moderately stable. |
| **XPath** | A locator using XML path syntax — powerful but brittle when tied to DOM structure. |
| **Locator Hierarchy** | The prioritised order for selecting locators: accessible > test ID > stable CSS > XPath. |
