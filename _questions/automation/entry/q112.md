theory: "The Page Object Model (POM) is a design pattern in UI test automation that creates a separate class or object for each page or component of the application under test. Each page object encapsulates the locators and interactions for that page, exposing them as methods to the test layer. This separates the test logic (what to test) from the UI interaction logic (how to interact with the page), reducing duplication and centralising the impact of UI changes."

## The Intent

The interviewer is checking that you understand automation as software engineering — not just scripting. The POM is the most fundamental design pattern in UI automation, and knowing it signals that you write maintainable, professional automation code rather than fragile, duplicated scripts. They also want to see that you understand the problem it solves, not just how to implement it.

---

## The STAR Script

**Situation:** I inherited an automation suite of 120 tests where locators and interaction steps were duplicated directly inside each test file. When the login page was redesigned and the email field's ID changed, I had to update 34 separate test files — each of which had the same `findElement(By.id("email"))` call.

**Task:** I needed to refactor the suite to eliminate this duplication and ensure that future UI changes required updates in only one place.

**Action:** I refactored the suite to use the Page Object Model. I created a `LoginPage` class that contained all the locators for the login page (email field, password field, submit button, error message) and methods representing user interactions (`enterEmail()`, `enterPassword()`, `clickSubmit()`, `getErrorMessage()`). The test files no longer contained any locators or raw Selenium interactions — they called `loginPage.enterEmail("user@example.com")` and `loginPage.clickSubmit()`. When the email field's ID changed again two weeks later, I updated it in exactly one place — the `LoginPage` class — and all 34 tests that used it were instantly fixed.

**Result:** Maintenance time for UI changes dropped dramatically. When a subsequent design change affected three pages simultaneously, updating all affected locators took 20 minutes instead of the hours it had previously taken. New test cases were also faster to write because the interaction methods were already built and reusable.

## Key Terms

| Term | Definition |
|------|------------|
| **Page Object Model (POM)** | A design pattern where each page or UI component is represented as a class encapsulating its locators and interactions. |
| **Locator** | The selector used to identify a UI element — e.g., ID, CSS selector, XPath. |
| **Encapsulation** | Hiding implementation details (locators, raw interactions) behind a clean interface (methods) — the core benefit of POM. |
| **Test Layer** | The layer containing test logic and assertions — separated from UI interaction logic in a POM architecture. |
| **Page Layer** | The layer containing page objects — encapsulates locators and interactions, called by the test layer. |
| **DRY Principle** | Don't Repeat Yourself — the core software engineering principle that POM enforces in automation code. |
