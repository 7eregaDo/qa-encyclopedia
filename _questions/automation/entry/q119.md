theory: "A flaky test is an automated test that produces inconsistent results — passing on some runs and failing on others — without any change to the application code or test code. Flaky tests are one of the most damaging quality problems in an automation suite because they erode trust in the test results: when developers see a failing test, they cannot tell whether it represents a real defect or another false alarm."

## The Intent

The interviewer is checking that you understand and take flakiness seriously as an engineering problem — not just an annoyance to be ignored or re-run away. Entry-level engineers who say "I just re-run it and it usually passes" are revealing a dangerous attitude. Those who can name specific causes and describe approaches to resolving them demonstrate the right mindset.

---

## The STAR Script

**Situation:** I joined a team whose CI pipeline had a flake rate of approximately 30% — nearly one in three pipeline runs had at least one test failure that disappeared on re-run. Developers had learned to ignore individual test failures and only investigate when three or more tests failed simultaneously. Real defects were being missed because the signal-to-noise ratio was too low.

**Task:** I was asked to investigate the most common flakiness causes in the suite and propose targeted fixes.

**Action:** I categorised all flaky tests over a two-week period by root cause. Five categories emerged. First, **timing and async issues** (40% of flakes): tests interacting with elements before async operations completed. Fixed by replacing hard sleeps with explicit condition-based waits. Second, **test interdependence** (25% of flakes): tests that passed when run in sequence but failed when run in parallel because they shared state — the same test user account, the same test data record. Fixed by making each test use isolated test data. Third, **environment instability** (15% of flakes): tests failing due to third-party service latency or test environment instability — not a test code problem but an infrastructure one. Addressed by adding retry logic for known-flaky third-party calls and improving environment reliability. Fourth, **overly precise assertions** (12% of flakes): tests asserting on content that legitimately varied — timestamps, animated values, dynamically generated IDs. Fixed by making assertions less brittle (e.g., asserting that a timestamp exists rather than asserting its exact value). Fifth, **application race conditions** (8% of flakes): genuine timing bugs in the application that tests were exposing. These were reported as defects rather than fixed in the test code.

**Result:** After addressing the first four categories, the flake rate dropped from 30% to 5%. Developer trust in the pipeline recovered — failing tests were investigated rather than dismissed. The genuine race conditions, once surfaced clearly as application defects, were fixed by developers, reducing the flake rate to under 2%.

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define flaky tests, name the most common causes, and describe basic fixes for timing and assertion flakiness. |
| **Mid** | Describe a flaky test investigation process — categorising causes, applying targeted fixes, and deciding when a flaky test represents an application defect rather than a test code problem. |

## Key Terms

| Term | Definition |
|------|------------|
| **Flaky Test** | A test that produces inconsistent results without code changes — passes sometimes, fails other times. |
| **Test Interdependence** | When tests share state and affect each other's results — eliminated by test isolation. |
| **Test Isolation** | The principle that each test should set up and tear down its own data and state independently. |
| **Overly Precise Assertion** | An assertion that fails due to legitimate, expected variation in output — e.g., asserting an exact timestamp. |
| **Flake Rate** | The percentage of test runs that produce inconsistent results — a key health metric for an automation suite. |
| **Quarantine** | Temporarily isolating flaky tests in a non-blocking pipeline job while they are investigated and fixed. |
