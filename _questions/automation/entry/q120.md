## The Intent

The interviewer is checking that you approach debugging systematically rather than randomly changing things until the test passes. Entry-level engineers who can only describe "running the test again to see if it passes" or "asking a developer to look at it" are revealing a gap in self-sufficiency. Systematic debugging is a core engineering skill.

---

## The STAR Script

**Situation:** A test in the CI pipeline that had been passing reliably for two weeks began failing consistently after a deployment. The test was verifying that a confirmed order displayed the correct order total on the order confirmation page.

**Task:** I needed to diagnose the failure quickly — determining whether it was a real application defect, a test code issue, or an environment problem — without blocking the team's deployment decision.

**Action:** I followed a structured debugging process. First, **read the failure message carefully**: the assertion error was `Expected "£45.00" but received "£45"` — the value was correct but the formatting was different. This immediately told me it was not a logic defect but a formatting change. Second, **check the recent deployment**: I looked at the deployment diff and found that a developer had reformatted currency output to remove the decimal places for round numbers. The application had changed; the test expected the old format. Third, **confirm in the application manually**: I opened the application and navigated to an order confirmation page — the displayed total was "£45" as reported. The application was behaving as intended by the developer; my test assertion was now incorrect. Fourth, **determine the right fix**: this was a test code problem, not a defect. I updated the assertion to match the new formatting, but also raised a question to the product owner: was the removal of decimal places intentional for all amounts, or only round numbers? (It turned out it was intentional, and I updated the test data to cover both cases.)

**Result:** The failing test was resolved within 20 minutes. More importantly, my question about the intentionality of the formatting change led to the product owner confirming that non-round amounts should still show two decimal places — a behaviour that had not been explicitly specified and was not yet implemented. A follow-up defect was raised, and the developer added the conditional formatting logic. The debugging process had surfaced an unspecified behaviour gap.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Describe a systematic four-step debugging process — read the error, isolate the cause, reproduce locally, determine whether it is a test or application issue. |
| **Mid** | Describe debugging complex failures — intermittent failures, environment-specific failures, and failures in parallel execution — using logging, screenshots, video recording, and network trace tools. |
| **Senior** | Build debugging infrastructure into the automation suite — automatic screenshots on failure, test step logging, network request capture, and integration with incident management for production-impacting failures. |

## Key Terms

| Term | Definition |
|------|------------|
| **Failure Message** | The output produced when an assertion fails — the first and most important source of diagnostic information. |
| **Assertion Error** | A failure caused by the actual result not matching the expected result — the most common type of test failure. |
| **Environment Issue** | A failure caused by the test infrastructure (browser, environment, network) rather than the application or test code. |
| **Test Data Problem** | A failure caused by the state of the test data (missing records, changed values) rather than application or test logic. |
| **Local Reproduction** | Running the failing test locally with debugging tools — the most effective way to investigate a CI failure. |
| **Screenshot on Failure** | An automatically captured screenshot at the moment of test failure — essential visual context for diagnosing UI test failures. |
