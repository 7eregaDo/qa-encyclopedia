theory: "A test assertion is a statement that verifies a specific expected condition is true at a given point in a test. If the assertion passes, the test continues; if it fails, the test stops and reports the failure. A good assertion is specific (tests exactly one condition), meaningful (its failure message immediately explains what went wrong), and placed at the right point in the test (after the action that produces the result being verified)."

## The Intent

The interviewer is checking that you write tests that actually verify something meaningful — not tests that execute a journey without confirming its outcome. Assertions are the core of any test, and poor assertions (too vague, too broad, placed incorrectly, or with unhelpful failure messages) are one of the most common quality issues in entry-level automation work.

---

## The STAR Script

**Situation:** During a code review, a senior engineer flagged several of my automated tests as "not actually testing anything useful." The tests were executing correctly but my assertions were weak — for example, after submitting a login form I was asserting `expect(page.url()).toContain('dashboard')` and nothing else.

**Task:** I needed to understand what made an assertion strong versus weak and rewrite my tests with assertions that gave genuine confidence in the feature's behaviour.

**Action:** I learned to evaluate assertions against three questions. First, **is it specific?** `toContain('dashboard')` in the URL was weak because it would pass even if the page had rendered an error state on the dashboard URL. A stronger assertion was `expect(await page.locator('h1').textContent()).toBe('Welcome back, Test User')` — confirming the correct page had loaded and the correct user session was active. Second, **is the failure message meaningful?** When an assertion fails, the error message should immediately tell a developer what went wrong without requiring them to read the test code. Playwright's built-in assertions produce messages like "Expected 'Welcome back, Test User' but received 'Session expired'" — immediately useful. Custom assertions should follow the same principle. Third, **is it in the right place?** I was asserting on outcomes before the async operation that produced them had completed — causing intermittent failures when assertions ran before the DOM had updated. Moving assertions to after the awaited action resolved the issue.

**Result:** After rewriting the assertions, the test suite caught two defects that the original weak assertions had missed: a logged-in user landing on a generic dashboard without personalisation (the URL was correct but the content was wrong), and an expired session redirecting to dashboard URL with an error modal rather than a proper re-login redirect. Neither would have been caught by the URL-only assertion.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define assertions, explain what makes a good assertion, and demonstrate the difference between weak and strong assertions with examples. |
| **Mid** | Discuss assertion strategies for different test types — UI assertions, API response assertions, database state assertions — and explain how to write assertion failure messages that are immediately actionable. |
| **Senior** | Design assertion standards for a team — including custom assertion helpers for domain-specific validations, assertion coverage review as part of code review, and strategies for asserting on complex or non-deterministic outputs. |

## Key Terms

| Term | Definition |
|------|------------|
| **Assertion** | A statement that verifies a specific expected condition — the core of any test's value. |
| **Strong Assertion** | An assertion that tests a specific, meaningful condition and produces a clear failure message when it fails. |
| **Weak Assertion** | An assertion that passes for too many conditions — gives false confidence by not verifying the actual behaviour of interest. |
| **Failure Message** | The error output when an assertion fails — should immediately explain what was expected versus what was received. |
| **Assertion Placement** | The position of an assertion within a test — must be after the action that produces the result being verified, and after any async operations complete. |
| **Soft Assertion** | An assertion that logs a failure but allows the test to continue — useful for verifying multiple conditions in a single test run. |
