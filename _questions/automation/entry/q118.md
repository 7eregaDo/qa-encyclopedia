Good project structure makes tests readable, reduces duplication, enables parallel development by multiple team members, and minimises the cost of maintaining tests as the application evolves."

## The Intent

The interviewer is checking that you treat automation as software engineering — with the same structural discipline applied to production code. Entry-level engineers who dump everything into a single file or folder produce suites that become unmaintainable within weeks. The ability to structure a project thoughtfully signals engineering maturity beyond just scripting.

---

## The STAR Script

**Situation:** I was building an automation project from scratch for a web application using Playwright and TypeScript. Rather than starting immediately with test files, I spent time designing the project structure before writing the first test.

**Task:** I needed to create a structure that a team could work in collaboratively, that would remain maintainable as the suite grew, and that applied clean separation of concerns.

**Action:** I organised the project into the following layers. **`/tests`** — test files only, containing test logic and assertions, no raw locators or interaction code. Each test file corresponded to a feature area (e.g., `login.spec.ts`, `checkout.spec.ts`). **`/pages`** — page object classes, one per page or major component, containing locators and interaction methods. **`/fixtures`** — test data files (JSON or TypeScript constants) containing the data used in tests, separated from test logic so data changes didn't require opening test files. **`/utils`** — shared helper functions (e.g., `generateRandomEmail()`, `formatCurrency()`, `waitForNetworkIdle()`). **`/config`** — environment configuration (base URLs, credentials, timeouts) managed via environment variables or a config file, so switching between environments required no code changes. **`playwright.config.ts`** — framework configuration including browser settings, parallelisation, and reporter settings. I also introduced a naming convention (feature-based file names), a linting rule set, and a README documenting the structure so new team members could navigate the project independently.

**Result:** When a second QA engineer joined the team two months later, they were writing their first test within 30 minutes of reading the README — because the structure was self-explanatory and consistent. When the application's login flow changed, updating the `LoginPage` class took 10 minutes and all 28 tests that used it were fixed without opening a single test file.

## Key Terms

| Term | Definition |
|------|------------|
| **Separation of Concerns** | The principle of organising code so that each part has a single, well-defined responsibility. |
| **Test Layer** | The layer containing test logic and assertions — calls page objects and utilities, contains no raw locators. |
| **Page Layer** | The layer containing page objects — encapsulates locators and interactions. |
| **Fixtures / Test Data** | Files containing the data used in tests — separated from test logic to reduce coupling. |
| **Utilities Layer** | Shared helper functions used across multiple tests or page objects. |
| **Configuration Layer** | Environment-specific settings (URLs, credentials, timeouts) — managed separately to enable environment switching without code changes. |
