title: "What is the difference between a test framework and a test library?"
category: automation
level: entry
theory: "A test library is a collection of pre-built functions and utilities that a tester calls to perform specific actions — such as browser interactions or API requests. A test framework is a broader structure that defines how tests are organised, executed, reported, and extended — it provides the skeleton within which tests and libraries operate. Libraries provide capability; frameworks provide structure."
---

## The Intent

The interviewer is checking that you understand the tooling ecosystem you work within — not just that you can use specific tools, but that you understand the distinction between different types of tools and how they fit together. Conflating frameworks and libraries signals shallow tooling knowledge. Understanding the distinction signals that you can make informed tooling decisions.

---

## The STAR Script

**Situation:** When onboarding onto a new team, I was asked to explain the automation stack to a developer who was unfamiliar with test tooling and was confused by the team using both Playwright and Jest — wondering why we had two separate tools.

**Task:** I needed to explain the distinction between a test framework and a test library clearly, using the team's actual tools as examples.

**Action:** I explained it this way: **Jest** was the test framework — it provided the structure for how tests were organised (describe blocks, test blocks), how they were executed (running suites, parallelisation), how results were reported (pass/fail output, coverage reports), and how setup and teardown were managed (beforeAll, afterEach). Jest controlled the shape of the test project. **Playwright** was the test library — it provided the specific capabilities needed to interact with a browser: launching browsers, navigating to URLs, finding elements, clicking, typing, and asserting on page content. Playwright gave the tests their power; Jest gave them their structure. I used a construction analogy: Jest was like the scaffolding and site management for a building project; Playwright was the tools and materials the builders used to do the actual work. Both were necessary; neither alone was sufficient.

**Result:** The developer understood the stack clearly and subsequently made a well-informed suggestion about adding a reporting library (Allure) without confusing it with either the framework or the interaction library. The analogy was reused in team onboarding documentation.

## Key Terms

| Term | Definition |
|------|------------|
| **Test Framework** | A structure that defines how tests are organised, executed, and reported — e.g., Jest, JUnit, Pytest, NUnit. |
| **Test Library** | A collection of utilities and functions for performing specific test actions — e.g., Playwright, Selenium, RestAssured. |
| **Assertion Library** | A library providing functions for verifying expected outcomes — e.g., Chai, AssertJ, built-in Jest matchers. |
| **Reporting Tool** | A tool that transforms test results into readable reports — e.g., Allure, ExtentReports, built-in framework reporters. |
| **Test Runner** | The component of a test framework responsible for discovering and executing tests — sometimes a separate tool (e.g., Mocha as runner with Chai as assertion library). |
| **Automation Stack** | The combination of framework, library, assertion, and reporting tools used in an automation project. |
