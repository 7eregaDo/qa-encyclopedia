Reliable parallelisation requires complete test isolation (no shared state between tests), careful management of shared resources (environments, databases, external services), appropriate sharding strategies (how tests are distributed across parallel workers), and monitoring for parallelisation-specific failure modes such as resource contention and race conditions introduced by concurrent execution."

## The Intent

The interviewer is assessing engineering depth in automation infrastructure. Parallelisation is not just a configuration setting — done poorly, it produces new classes of flakiness that are harder to diagnose than serial execution failures. Senior engineers must be able to design parallelisation that is both fast and reliable, with a clear understanding of the failure modes and mitigation strategies.

---

## The STAR Script

**Situation:** Our automation suite of 320 tests ran serially in 68 minutes — unacceptably slow for a team releasing daily. When I naively enabled 4-worker parallelisation in Playwright, runtime dropped to 22 minutes but flake rate jumped from 4% to 31%. The parallel failures were different from our existing flaky tests — they appeared to be caused by tests interfering with each other.

**Task:** I needed to diagnose the parallelisation-specific failure modes, address them systematically, and achieve fast, reliable parallel execution.

**Action:** I approached parallelisation as an architectural problem, not a configuration problem. First, **root cause diagnosis**: I ran the suite in parallel 10 times with detailed logging and identified three failure patterns. Shared database records — four tests were creating and then reading a "default admin user" record; in parallel, two tests would try to create it simultaneously, causing a unique constraint violation. Shared environment state — three tests modified application configuration (feature flags) without resetting them, causing subsequent parallel tests to run with unexpected configuration. Port contention — two tests were starting local mock servers on the same port. Second, **isolation architecture**: for database conflicts, I implemented unique-per-worker test data — each parallel worker was assigned a worker index (available in Playwright's `workerInfo.workerIndex`) and used it as a namespace prefix for all created records. Worker 1 created `w1_admin_user`, worker 2 created `w2_admin_user`. For configuration state, I added explicit teardown hooks that reset any modified configuration after each test. For port contention, I assigned mock server ports dynamically using the worker index as an offset. Third, **test sharding strategy**: I configured Playwright to shard the suite by test file rather than individual tests — ensuring tests within the same file ran on the same worker (preserving any file-level setup/teardown) while different files ran in parallel. Fourth, **resource monitoring**: I added CI runner resource monitoring to detect CPU and memory contention — the initial 4-worker configuration was hitting memory limits on the CI runner, causing test failures. I reduced to 3 workers and added a CI runner memory threshold alert.

**Result:** After the isolation architecture was in place, 3-worker parallelisation achieved 19-minute suite runtime with a 3.5% flake rate — lower than the original serial suite's 4%. The worker namespace pattern became the team standard for all new test data creation. The diagnostic process documented three parallelisation failure modes as a team reference guide, preventing future engineers from encountering the same issues.

## Key Terms

| Term | Definition |
|------|------------|
| **Test Parallelisation** | Running multiple tests simultaneously across parallel workers — reduces total suite execution time. |
| **Worker Index** | A unique identifier for each parallel execution worker — used as a namespace prefix for test data to prevent cross-worker interference. |
| **Test Sharding** | Distributing tests across parallel workers — by file, by tag, or by count. |
| **Resource Contention** | Competition between parallel workers for limited resources — CPU, memory, ports, or database connections. |
| **Isolation Architecture** | The structural design of test data, state management, and resource usage to ensure parallel tests do not interfere with each other. |
| **Parallelisation-Specific Flakiness** | Test failures that only occur during parallel execution — caused by race conditions, shared state, or resource contention between concurrent tests. |
