category: automation

theory: "Mutation testing evaluates the quality of a test suite by automatically introducing small defects (mutations) into the production code and checking whether the existing tests detect them. A mutation that is not caught by any test is a 'surviving mutant' — evidence that the test suite has a coverage gap or a weak assertion. Coverage metrics tell you which lines were executed; mutation testing tells you whether those executions actually verified correct behaviour."

## The Intent

The interviewer is assessing depth of testing theory and the ability to think critically about test suite quality. Mutation testing is a sophisticated technique that most teams have not implemented — knowledge of it signals genuine depth in test engineering. Senior engineers are expected to understand not just what mutation testing is but how to design a programme that is computationally practical and produces actionable results.

---

## The STAR Script

**Situation:** Our team had achieved 87% unit test coverage and was confident in the quality of the test suite. A production incident occurred where a boundary condition bug in a discount calculation function had passed all tests — the line was executed in tests, but the assertion was checking only that the function returned a value, not that the value was correct. Coverage had been misleading us into overconfidence.

**Task:** I wanted to introduce mutation testing to measure the actual effectiveness of our assertions — not just whether lines were being executed — and use the findings to improve test quality systematically.

**Action:** I designed the mutation testing programme in three stages. First, **tool selection and scope**: I used Stryker Mutator (the industry standard for JavaScript/TypeScript). Full-suite mutation testing was computationally expensive — I scoped it to the three highest-risk modules: the pricing engine, the eligibility calculator, and the discount rules engine. Running mutation testing on the full codebase would have taken 6 hours; scoping to these modules took 18 minutes. Second, **interpretation and triage**: Stryker generated a mutation score (percentage of mutations killed by the test suite) and a detailed report showing every surviving mutant — which line, what change was made, and which tests had failed to detect it. I triaged surviving mutants into three categories: assertion gaps (the right line was executed but the assertion wasn't specific enough), coverage gaps (no test exercised that path at all), and equivalent mutants (the mutation produced logically identical behaviour — acceptable survivors). Third, **improvement programme**: I presented the mutation report to the team in a workshop — using specific surviving mutants as case studies. For the pricing engine, a mutant that changed `>=` to `>` in a boundary condition survived all tests — because our tests used inputs that were clearly within or outside the boundary but never at the exact edge. The fix was not writing a new test but strengthening the assertion in an existing test to use a boundary value specifically.

**Result:** Initial mutation score for the three modules: 61% (meaning 39% of mutations survived). After two weeks of targeted assertion improvements — not new test cases but stronger assertions in existing ones — the mutation score rose to 84%. The two production bugs found in the following six months were in different modules — none in the mutation-tested modules. The exercise also permanently changed how the team wrote assertions: "does this test actually verify the behaviour it claims to?" became a standard code review question.

## Key Terms

| Term | Definition |
|------|------------|
| **Mutation Testing** | A technique that introduces small defects (mutations) into production code and checks whether tests detect them. |
| **Mutant** | A modified version of the production code with a single small change — e.g., `>` changed to `>=`. |
| **Killed Mutant** | A mutation detected by at least one test — evidence of effective assertion coverage at that code point. |
| **Surviving Mutant** | A mutation not detected by any test — evidence of an assertion gap or coverage gap. |
| **Mutation Score** | The percentage of mutants killed — the primary metric of test suite assertion effectiveness. |
| **Stryker Mutator** | The industry-standard mutation testing framework for JavaScript and TypeScript. |
| **Equivalent Mutant** | A mutation that produces logically identical behaviour to the original — cannot be killed and is excluded from the mutation score. |
