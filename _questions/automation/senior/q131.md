## The Intent

The interviewer is assessing whether you can operate at the level of system design — not just framework implementation. Senior automation engineers are expected to design architecture that scales across teams, time, and product complexity. They want to see decisions about tooling standardisation, shared libraries, ownership models, and pipeline integration — not just a description of one team's test suite.

---

## The STAR Script

**Situation:** I was the senior QA architect at a company with 12 product squads, each with embedded QA engineers using different automation tools — some using Cypress, some Playwright, some Selenium, and some with no automation at all. There was no shared reporting, no common standards, and no cross-team visibility of automation coverage or health. The CTO wanted a unified automation platform.

**Task:** I needed to design an automation architecture that provided organisational consistency without mandating a single approach for every squad, and that could be adopted incrementally without disrupting squads already running established automation.

**Action:** I designed the architecture across four layers. First, **standardisation layer**: I defined the non-negotiable organisational standards — test result reporting format (JUnit XML, consumed by a central dashboard), code style and review requirements for automation PRs, naming conventions, and the shared CI/CD integration pattern (all suites must run in GitHub Actions and produce artefacts on failure). Tooling choice within these standards was left to squads. Second, **shared library layer**: I built and published a shared npm package (`@company/qa-utils`) containing: authentication helpers (login flows for each user role), API client wrappers for common service calls, test data factories for shared domain objects, and custom assertion helpers for domain-specific validations. Squads could use the library or build their own — but using it reduced duplication significantly and ensured consistent behaviour for cross-cutting concerns. Third, **observability layer**: I deployed a central Allure test results server fed by all squad pipelines. Every squad's CI pushed results to the central server post-run — giving the CTO and engineering managers cross-squad visibility of pass rates, flake rates, and coverage trends from a single dashboard. Fourth, **governance layer**: I established a monthly automation guild — senior QA engineers from each squad — to review shared library updates, agree on new standards, and share learnings. Changes to the shared library required guild review before release. Squads contributed back to the library through pull requests reviewed by the guild.

**Result:** Within six months, all 12 squads were publishing results to the central dashboard. The shared library was adopted by 9 of 12 squads — reducing authentication-related test code duplication by an estimated 70%. The central dashboard allowed me to identify two squads with critical automation gaps and prioritise investment there. The guild became a professional community that accelerated skill sharing across the organisation — three automation patterns originating in individual squads were adopted organisation-wide through the guild process.

## Key Terms

| Term | Definition |
|------|------------|
| **Automation Architecture** | The high-level design of an organisation's automation system — layers, ownership, tooling standards, and infrastructure. |
| **Shared Library** | A centralised, versioned package of reusable automation utilities published for use across multiple teams. |
| **Central Observability** | A unified dashboard aggregating test results, health metrics, and coverage data from all teams. |
| **Automation Guild** | A cross-team community of senior automation engineers that governs shared standards and library evolution. |
| **Standardisation Layer** | The minimum set of non-negotiable automation standards applicable to all teams — reporting format, CI integration, code review. |
| **Incremental Adoption** | An architecture design principle enabling teams to adopt new patterns progressively without requiring big-bang migration. |
