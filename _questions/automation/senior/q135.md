## The Intent

The interviewer is assessing breadth beyond functional test automation. Senior automation engineers working on production systems are expected to have designed and implemented performance testing programmes — not just run JMeter scripts occasionally. They want to see tool selection rationale, metric definition, threshold setting, and CI/CD integration as a coherent discipline.

---

## The STAR Script

**Situation:** Our platform had experienced two production performance incidents in a quarter — a checkout flow degrading to 8-second response times under peak traffic, and an API endpoint timing out under concurrent load. Both had been invisible in functional testing. I was asked to build a performance testing programme that would catch regressions before they reached production.

**Task:** I needed to design a performance testing strategy covering the right scenarios, with appropriate tooling, defined thresholds, and practical CI/CD integration that didn't add excessive pipeline time.

**Action:** I built the programme across four stages. First, **scenario identification and baseline**: I identified the five highest-traffic API endpoints and the three most critical user journeys from production analytics. I ran initial baseline load tests against staging to establish current performance characteristics — p50, p95, and p99 response times, and maximum throughput before error rates increased. These baselines became the performance budget. Second, **tooling**: I chose k6 for API-level load testing — its JavaScript scripting model matched the team's skill set, it produced clean metrics output, and it had native Grafana integration for visualisation. For UI-level performance I used Playwright's performance APIs and Lighthouse CI for core web vitals. Third, **test types and thresholds**: I defined four test types with specific thresholds. Load tests (expected traffic: 100 concurrent users) — p95 response time must remain below 500ms. Stress tests (150% of expected traffic) — p95 below 800ms, error rate below 1%. Spike tests (sudden 300% traffic surge) — recovery within 60 seconds. Endurance tests (sustained load for 30 minutes) — no memory leak pattern, stable throughput. Fourth, **CI/CD integration**: load tests ran nightly against staging — not on every commit (too slow) but frequently enough to detect regressions before release. The nightly run published results to Grafana and failed the job if any threshold was breached, blocking the next day's release candidate pipeline until investigated. For each production deployment, a post-deploy smoke performance test (30 seconds, 20 concurrent users) confirmed no degradation in the production environment.

**Result:** The nightly performance gate caught a performance regression three weeks after implementation — a database query change that doubled response time for the product search endpoint under load. The regression was detected at 2am, the developer was notified by the CI alert, and the issue was resolved before the morning's release window. The post-deploy smoke test caught a CDN misconfiguration that was causing 40% increased latency for static assets — invisible to functional testing.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Run a basic load test with JMeter or k6 and interpret the output metrics. |
| **Mid** | Design a load testing scenario for a critical endpoint — virtual users, ramp-up, duration, and assertion thresholds. |
| **Senior** | Build a full performance testing programme — scenario selection, baseline establishment, tool selection, multiple test types, threshold definition, CI/CD integration, and production monitoring. |

## Key Terms

| Term | Definition |
|------|------------|
| **Load Testing** | Testing system behaviour under expected traffic levels — verifies performance meets requirements at normal load. |
| **Stress Testing** | Testing beyond expected limits — identifies the breaking point and recovery behaviour. |
| **Spike Testing** | Testing under sudden, extreme traffic surges — verifies the system can absorb and recover from unexpected load spikes. |
| **Endurance Testing** | Sustained load over extended periods — detects memory leaks and performance degradation over time. |
| **p95 / p99 Response Time** | The response time below which 95% / 99% of requests complete — more meaningful than average for user experience assessment. |
| **Performance Budget** | A defined maximum acceptable response time or resource usage — the threshold against which performance test results are assessed. |
