## The Intent

The interviewer is assessing the ability to solve a common and genuinely difficult automation engineering problem. Stateful workflows are everywhere — checkout flows, onboarding sequences, approval processes, multi-stage forms — and naive automation approaches (clicking through every step every time) produce slow, brittle test suites. Senior engineers are expected to design elegant solutions that reach the required state efficiently without sacrificing coverage.

---

## The STAR Script

**Situation:** I was automating tests for an insurance claims processing system with an eight-step workflow: Submitted → Documents Requested → Documents Received → Under Assessment → Assessment Complete → Approved/Rejected → Payment Scheduled → Paid. Each step had its own business rules, validations, and user actions. Testing step 6 (Approved/Rejected) through the UI required manually executing steps 1–5 first — taking 12–15 minutes of UI interaction per test.

**Task:** I needed to design an automation strategy that gave full coverage of every workflow state and transition without requiring UI navigation through all preceding steps for every test.

**Action:** I designed the strategy around three principles. First, **state modelling**: I created a complete state transition model — all eight states, all valid transitions, all invalid transitions, and the specific business rules governing each transition. This model became the master specification for test design, ensuring complete coverage without omission. Second, **API-driven state setup**: for tests targeting a specific workflow state (e.g., testing the "Under Assessment" state), I used the application's internal API to create a claim and advance it directly to the required state — bypassing all preceding UI steps. A claim in "Under Assessment" state could be set up in 400ms via API rather than 15 minutes of UI navigation. I wrote a `ClaimStateFactory` class with methods for each state: `createClaimInState('UNDER_ASSESSMENT', testData)`. Third, **layered coverage**: I covered the state machine at three levels. Unit tests (owned by developers) covered the business rules and transition logic in isolation. API tests covered the state transitions and validation rules at the service layer — verifying correct state changes, rejection of invalid transitions, and correct business rule application. UI tests covered the user experience at each state — what the user sees, what actions are available, and how the UI communicates state clearly. UI tests used the `ClaimStateFactory` to set up state via API before launching the browser, keeping them fast.

**Result:** Full workflow coverage — all eight states, all transitions, all invalid transitions — was achieved in 340 test cases. Average UI test setup time dropped from 12 minutes to under 30 seconds using API state setup. The complete suite ran in 22 minutes. The state model also proved valuable as a documentation artefact — the business analyst used it to identify two undocumented transition rules that had been implemented inconsistently between the UI and the API.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Automate a linear multi-step workflow using UI navigation, identifying the setup steps required for each test. |
| **Mid** | Apply state transition testing to identify all states and transitions, and begin using API calls to set up preconditions rather than navigating through all preceding steps. |
| **Senior** | Design a full stateful workflow automation strategy — state transition model, API-driven state factory, layered coverage across unit/API/UI, and explicit state verification at each transition. |

## Key Terms

| Term | Definition |
|------|------------|
| **Stateful Workflow** | A multi-step process where each step depends on state established by previous steps. |
| **State Transition Model** | A complete map of all workflow states, valid transitions, invalid transitions, and governing business rules. |
| **API-Driven State Setup** | Using the application's API to advance a system to the required state before UI testing — bypassing slow UI navigation. |
| **State Factory** | A test utility class providing methods to create objects in specific workflow states via API. |
| **Layered Coverage** | Testing a stateful workflow at unit, API, and UI levels — each layer covering the aspects it is best suited to verify. |
| **State Verification** | Explicit assertions confirming the system is in the expected state after each transition — not just that the action completed. |
