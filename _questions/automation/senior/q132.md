## The Intent

The interviewer is assessing architectural decision-making and the ability to navigate trade-offs under real-world constraints. Senior automation engineers are expected to have led framework creation — not just contributed to existing ones. They want to see structured decision-making, rationale for key choices, and a governance model that keeps the framework healthy over time.

---

## The STAR Script

**Situation:** I joined a company as their first senior QA engineer. There was no automation — just manual regression. The product was a complex B2B SaaS with a React frontend, a Node.js REST API, and a PostgreSQL database. The team consisted of two QA engineers (one junior, one mid-level) and twelve developers using TypeScript across the stack.

**Task:** I needed to design and build an automation framework from scratch that the whole team could contribute to, that would integrate into the CI/CD pipeline, and that would be maintainable by a small QA team over years.

**Action:** I made decisions in five categories. First, **language**: TypeScript — matching the team's existing skill set, eliminating a context-switch barrier, and enabling developers to contribute to test code. The alternative (Java with RestAssured/Selenium) would have created a knowledge silo. Second, **tooling**: Playwright for E2E and component testing (native TypeScript support, auto-waiting, cross-browser, built-in trace viewer), Supertest for API testing (lightweight, runs in Node without a test server), Jest as the test runner for API tests. I evaluated Cypress but rejected it due to limitations with multi-domain testing and less native TypeScript support at the time. Third, **architecture**: a three-layer structure — page objects for UI interaction, service clients for API interaction, and a shared test data factory layer. I documented the pattern in an ADR (Architecture Decision Record) so future engineers understood why the structure existed. Fourth, **data strategy**: dynamic test data creation via API before each test, with teardown after. No dependency on shared environment state. Fifth, **governance**: all automation PRs required review by at least one other QA engineer, the framework had its own CHANGELOG tracking breaking changes, and shared utilities were versioned semantically. I ran a monthly 30-minute "framework health" review to assess technical debt and plan maintenance.

**Result:** The framework was operational with 60 tests in four weeks. Within six months it had 280 tests across UI and API layers, contributed to by both QA engineers and three developers. The TypeScript choice proved correct — two developers submitted PRs for API tests covering their own services within the first month. The ADR documentation was referenced six months later when a new engineer joined and needed to understand why the architecture was structured the way it was.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Set up a basic automation project using an established framework with guidance. |
| **Mid** | Select a framework and tool stack for a team, implement the project structure, and integrate with CI. |
| **Senior** | Make and document all architectural decisions for a framework built from scratch — language, tooling, architecture pattern, data strategy, governance — with explicit trade-off reasoning for each choice. |

## Key Terms

| Term | Definition |
|------|------------|
| **Framework Architecture Decision** | A documented, reasoned choice about a fundamental aspect of the automation framework — language, tooling, pattern, or data strategy. |
| **ADR (Architecture Decision Record)** | A document recording a significant architectural decision, its context, and the rationale for the chosen option over alternatives. |
| **Knowledge Silo** | A situation where only one or a few people understand a system — a risk when framework language differs from the team's primary language. |
| **Semantic Versioning** | A versioning scheme (MAJOR.MINOR.PATCH) for shared libraries — communicates the nature of changes to consumers. |
| **CHANGELOG** | A file documenting all notable changes to a project per version — essential for shared frameworks used by multiple teams. |
| **Framework Governance** | The processes and ownership model for maintaining, evolving, and communicating changes to a shared automation framework. |
