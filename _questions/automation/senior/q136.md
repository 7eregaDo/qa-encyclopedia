## The Intent

The interviewer is assessing depth of UI testing knowledge and the ability to evaluate when a specific technique is worth its overhead. Visual regression testing has real costs — baseline management, false positive noise, and review process overhead — and senior engineers are expected to articulate both the value and the limitations, and describe how to implement it in a way that minimises the costs.

---

## The STAR Script

**Situation:** After a design system migration, three visual regressions had reached production — a button colour was slightly wrong on one page, a spacing issue in the data table caused content to overlap at certain viewport sizes, and a font weight change was applied inconsistently across similar components. All three had passed functional testing because they were purely visual — the functionality worked correctly. The product design team was frustrated by the recurring visual quality issues.

**Task:** I needed to design a visual regression testing programme that would catch these classes of issue automatically before release, without generating so many false positives that developers would disable or ignore it.

**Action:** I implemented visual regression testing using Playwright with the `@playwright/test` snapshot comparison, backed by Percy for cross-browser baseline management. I structured the programme around four decisions. First, **scope selection**: I limited visual regression to the 12 most-visited pages and the 8 design system component variations — not the entire application. Testing everything visually generated too many false positives from legitimate content changes. High-traffic, stable pages were the right target. Second, **baseline management process**: I established a clear workflow — when a developer made an intentional visual change, they ran `playwright test --update-snapshots` to update the baselines as part of their PR. The updated snapshots were reviewed in the PR diff by a QA engineer or designer who confirmed the change was intentional. No snapshot update was merged without explicit sign-off. Third, **tolerance thresholds**: I configured pixel difference tolerance at 0.1% — large enough to ignore sub-pixel rendering differences between runs (which caused false positives) but small enough to catch genuine layout shifts. Threshold tuning took two weeks of calibration against real runs. Fourth, **CI integration**: visual regression ran on PRs touching CSS, component files, or page layout files (identified via change detection). It did not run on every PR — only those that could plausibly affect visual output.

**Result:** Within the first month, visual regression caught four unintended visual changes — all introduced during CSS refactoring that passed functional testing. The false positive rate settled at 3% after threshold tuning — low enough that developers investigated failures rather than assuming they were noise. The design team's confidence in visual quality recovered, and no visual regressions reached production in the six months following implementation.

## Key Terms

| Term | Definition |
|------|------------|
| **Visual Regression Testing** | Automated detection of unintended visual changes by comparing current screenshots against approved baselines. |
| **Baseline Image** | The approved reference screenshot against which future runs are compared. |
| **Pixel Difference Tolerance** | The acceptable percentage of changed pixels before a visual test is marked as failed — calibrated to reduce false positives. |
| **False Positive** | A visual test failure caused by legitimate, expected changes (content updates, browser rendering variation) rather than regressions. |
| **Percy** | A cloud-based visual regression platform with cross-browser support and a review workflow for approving visual changes. |
| **Snapshot Comparison** | Playwright's built-in visual comparison mechanism — generates and compares PNG screenshots per test run. |
