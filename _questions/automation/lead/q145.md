## The Intent

The interviewer is assessing decision-making rigour and the ability to manage a complex, high-stakes organisational decision with many stakeholders. Lead-level professionals must be able to run a structured tool evaluation — not just read comparison articles and pick a favourite. They want to see evaluation criteria, stakeholder involvement, proof-of-concept methodology, and a decision process that produces organisational buy-in.

---

## The STAR Script

**Situation:** After two years of ad-hoc tooling decisions, the organisation had accumulated four different E2E frameworks (Cypress, Playwright, Selenium, and a custom Python solution) across 16 squads. Maintenance was fragmented, knowledge sharing was difficult, and the shared infrastructure team could not support four different CI integration patterns. The CTO asked me to lead a tooling standardisation initiative — selecting a single E2E framework for new projects while defining a migration path for existing suites.

**Task:** I needed to run a rigorous, transparent evaluation process that produced a defensible decision with broad stakeholder buy-in — not a decision that half the engineering organisation would resent.

**Action:** I ran the evaluation across five stages over eight weeks. First, **criteria definition with stakeholders**: I ran three workshops with representatives from different squad types — frontend-heavy squads, API-first squads, and mixed squads — to define evaluation criteria weighted by their relative importance. The top criteria were: developer experience and TypeScript support (weight: 25%), CI/CD integration and parallelisation (20%), cross-browser support (15%), community health and long-term viability (15%), migration effort from existing tools (15%), and licensing cost (10%). Second, **structured proof-of-concept**: I selected Playwright and Cypress as the two finalists (Selenium was excluded based on community trajectory; the custom Python solution on maintenance cost). I ran identical proof-of-concept implementations for each — the same 20 test scenarios, on the same application, by the same two engineers — measuring implementation time, code verbosity, CI integration effort, and failure diagnosis experience. Third, **total cost of ownership modelling**: I modelled the full 3-year cost of each option — licensing (both free, but cloud browser farm costs differed), training time for the 70 engineers who would use the tool, migration cost from existing Selenium suites (Playwright's API similarity reduced migration effort significantly), and expected maintenance cost based on community issue resolution rates. Fourth, **stakeholder review**: I presented the POC results, TCO model, and weighted criteria scores to the engineering leadership team and the Automation Guild. I did not present a recommendation — I presented the data and asked the group to reach a conclusion together. Fifth, **decision and migration plan**: the group chose Playwright. I published the decision as an ADR with full rationale, acknowledged the trade-offs explicitly (Cypress had better visual feedback tooling at the time), and defined a migration plan: new projects must use Playwright immediately; existing Cypress suites could continue until their next major refactor; Selenium suites had an 18-month migration target with support from my team.

**Result:** The decision was accepted with minimal resistance — because the process had been transparent and the criteria had been defined by the people who would live with the decision. Eighteen months later, 13 of 16 squads were using Playwright. The three remaining squads (still on Cypress) were mid-migration. The shared CI integration pattern, now possible with a single framework, reduced DevOps support overhead by 60%.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Use the framework your team has selected and provide feedback on its pain points. |
| **Mid** | Evaluate two or three automation tools for a team using defined criteria and recommend one with rationale. |
| **Senior** | Run a structured tool evaluation for a team — POC, criteria weighting, TCO modelling, and stakeholder presentation. |
| **Lead** | Lead an organisational tooling standardisation initiative — criteria workshops, structured POC, TCO modelling, stakeholder-driven decision process, ADR publication, and a phased migration plan with named support. |

## Key Terms

| Term | Definition |
|------|------------|
| **Total Cost of Ownership (TCO)** | The full cost of a tooling decision over time — licensing, training, migration, infrastructure, and maintenance. |
| **Weighted Criteria** | Evaluation dimensions assigned relative importance scores — enables objective comparison of tools across multiple factors. |
| **Proof of Concept (POC)** | A controlled, identical implementation using each candidate tool — produces comparable, evidence-based data for the decision. |
| **Architecture Decision Record (ADR)** | A document recording a significant tooling or architectural decision — its context, criteria, options considered, chosen option, and rationale. |
| **Migration Plan** | A phased roadmap for moving from legacy tooling to the selected standard — with timelines, support, and defined exceptions. |
| **Community Health** | The vitality of an open-source tool's community — issue response rate, release frequency, and contributor count — a proxy for long-term viability. |
