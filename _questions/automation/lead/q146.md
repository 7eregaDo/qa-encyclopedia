## The Intent

The interviewer is assessing software engineering maturity and the ability to build sustainable, scalable shared infrastructure. Inner source frameworks are the organisational answer to the choice between a centralised automation library (a bottleneck) and no shared library (fragmentation). Lead-level professionals must be able to design the ownership, contribution, governance, and versioning model — not just the technical implementation.

---

## The STAR Script

**Situation:** Our shared automation library (`@company/qa-utils`) had been built and maintained exclusively by my central QA team — a two-person bottleneck for 16 squads. Contribution requests from squads took two to four weeks to be reviewed and merged. Squads had started copying utilities directly into their own repos rather than waiting — creating divergent implementations of the same patterns. The library was becoming less useful the more successful it was.

**Task:** I needed to transform the library from a centrally-owned package into an inner source project — one that any engineer in the organisation could contribute to, governed by a transparent process that maintained quality without requiring my team as gatekeepers.

**Action:** I designed the inner source model across five dimensions. First, **open visibility**: I moved the repository from the `qa-team` GitHub organisation to the company's central `engineering` organisation and made it readable by all engineers. I added a comprehensive README, a CONTRIBUTING.md explaining the contribution process, and a CHANGELOG documenting all changes by version. Second, **contribution process**: I defined a clear contribution workflow — raise a GitHub issue describing the proposed utility and its use case (any engineer could do this), discuss in the issue thread (open to all engineers), implement in a feature branch, open a PR with tests and documentation, reviewed by at least two engineers (one from the contributing squad, one from the QA guild). My team was no longer the gatekeeper — the guild reviewed contributions collectively. Third, **semantic versioning and compatibility**: I introduced semantic versioning (MAJOR.MINOR.PATCH) and a deprecation policy — utilities could not be removed in a minor version, only deprecated with a migration guide. Breaking changes required a MAJOR version bump with a migration guide and a 90-day deprecation notice. This gave squads confidence that upgrading minor versions was safe. Fourth, **automated quality gates**: I added automated quality gates to the library's CI pipeline — all contributions required unit tests covering the utility, TypeScript type checking, ESLint compliance, and documentation strings. These gates ran automatically on every PR — reducing reviewer burden to logic and design, not style. Fifth, **contribution incentives**: I made contributions to the shared library explicitly visible in engineering performance conversations — contributing a utility that saved time across multiple squads was recognised as higher-impact than equivalent work within a single squad.

**Result:** Contribution rate to the library increased from 2 PRs per month (all from my team) to 14 PRs per month (11 from squads, 3 from my team) within four months. Average time from PR open to merge dropped from 18 days to 4 days — because the guild review was faster than waiting for my team, and automated quality gates eliminated the most time-consuming review feedback. The divergent utility implementations in individual squad repos were gradually unified as squads contributed their local patterns back to the shared library.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Use a shared automation library and raise issues when utilities are missing or broken. |
| **Mid** | Contribute a utility to a shared library — implementing, testing, documenting, and submitting a PR through the established process. |
| **Senior** | Design and govern a shared automation library — contribution guidelines, code review standards, versioning policy, and quality gates. |
| **Lead** | Transform a centrally-owned library into an inner source project — open visibility, distributed contribution model, guild-based governance, semantic versioning with compatibility guarantees, automated quality gates, and contribution incentives. |

## Key Terms

| Term | Definition |
|------|------------|
| **Inner Source** | Applying open-source collaboration principles within an organisation — open contribution, transparent governance, and distributed ownership. |
| **CONTRIBUTING.md** | A document defining how engineers can contribute to a project — the social contract of an inner source project. |
| **Semantic Versioning** | A versioning scheme (MAJOR.MINOR.PATCH) communicating the nature of changes — critical for shared library consumers. |
| **Deprecation Policy** | A defined process for removing or changing utilities — providing consumers time and migration guidance. |
| **Automated Quality Gates** | CI checks (type checking, linting, test coverage) that run automatically on every PR — reduce reviewer burden and enforce baseline standards. |
| **Contribution Bottleneck** | The state where a shared library is owned exclusively by a central team — limiting contribution rate and creating dependency. |
