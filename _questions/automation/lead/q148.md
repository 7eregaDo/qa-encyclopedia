## The Intent

The interviewer is assessing the ability to think about automation as a function of system architecture. Lead-level professionals overseeing automation during a major architectural transition must be able to design a strategy that serves both the current and future architecture simultaneously — avoiding the trap of either clinging to monolith-era testing patterns or prematurely abandoning working coverage. This requires architectural awareness, change management skill, and long-term planning.

---

## The STAR Script

**Situation:** I was Head of Quality Engineering at a company beginning a 24-month monolith decomposition. The existing automation suite consisted of 280 E2E tests written against the monolith — they tested the application through the UI, covering business logic that would progressively move into independent microservices. As services were extracted, the monolith tests began failing due to asynchronous service calls, API changes, and independently deployed components that the tests had never been designed to handle.

**Task:** I needed to design an automation strategy that would serve the organisation through the full 24-month transition — protecting existing coverage while building the microservices-native testing capability that would be required at the end state.

**Action:** I designed the strategy as a three-phase evolution. **Phase 1 — Stabilise the existing suite (months 1–6)**: before introducing new testing patterns, I needed to prevent the monolith E2E suite from becoming a liability as the first services were extracted. I added service boundary markers to the existing tests — tagging tests that covered functionality moving to a specific new service. When a service was extracted, its tagged tests were evaluated: tests covering user-visible journeys were retained as E2E tests (updated for the new architecture), tests covering pure service logic were migrated to API tests against the new service. I also added resilience to the E2E suite — retry logic and explicit waits for asynchronous service responses — to handle the mixed-architecture period. **Phase 2 — Build microservices-native testing (months 6–18)**: for each new service extracted from the monolith, I introduced a testing requirement as part of the service's definition of done: a component test suite (service-level tests with dependencies mocked), a contract test set defining the service's API contract with its consumers, and contribution to integration tests for the service's critical integrations. I trained each service team's embedded QA engineer in contract testing with Pact and component testing patterns. **Phase 3 — Rationalise the E2E suite (months 18–24)**: as microservices-native coverage matured, I progressively retired E2E tests that were now redundant — covered by the combination of component and contract tests. The E2E suite was reduced from 280 tests to 60 critical user journeys — the journeys that required end-to-end verification across multiple services and could not be covered at a lower layer.

**Result:** At the 24-month mark, the organisation had a three-layer automation estate: 60 E2E tests for critical journeys, 420 API/component tests across 12 services, and 180 contract tests across 28 service pairs. Total suite runtime had dropped from 90 minutes (monolith E2E only) to 28 minutes (all layers in parallel). The transition had been managed without a quality gap — no increase in production incidents during the migration period, which the CTO cited as evidence that the phased strategy had worked.

## Key Terms

| Term | Definition |
|------|------------|
| **Monolith-to-Microservices Migration** | An architectural transition from a single deployable unit to independently deployed services. |
| **Service Boundary Marker** | A test tag identifying which service's logic a test covers — enables targeted migration as services are extracted. |
| **Component Test** | A service-level test that tests the service in isolation with its dependencies mocked — the microservices equivalent of an integration test. |
| **Contract Testing** | Verifying that services agree on their interface contract independently — critical for independently deployable services. |
| **Mixed-Architecture Period** | The transition state where both monolith and microservices components are active — requires resilient testing that handles both. |
| **E2E Rationalisation** | The deliberate reduction of E2E tests as lower-layer coverage matures — preventing the inverted pyramid. |
