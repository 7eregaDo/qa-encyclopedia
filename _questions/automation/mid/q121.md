## The Intent

The interviewer is assessing whether you can design automation that scales efficiently. Mid-level automation engineers are expected to move beyond writing individual test cases toward building reusable frameworks. Data-driven testing is one of the most fundamental patterns for achieving coverage at scale, and the ability to implement it well — including handling data management and failure reporting — separates competent engineers from strong ones.

---

## The STAR Script

**Situation:** I was automating tests for a loan eligibility calculator that had over 30 distinct combinations of input values (applicant age, income, credit score, loan amount) each producing a specific eligibility outcome. Writing 30 individual test methods would have produced an unmanageable, duplicated codebase.

**Task:** I needed to design a data-driven approach that would allow all 30 combinations to be covered by a single test method, with data managed externally so that new scenarios could be added by the QA team without touching automation code.

**Action:** I implemented the framework in three parts. First, **data source**: I defined the test data in a JSON file — each entry containing `applicantAge`, `income`, `creditScore`, `loanAmount`, and `expectedOutcome`. The JSON was readable by non-technical stakeholders and editable without programming knowledge. Second, **parameterised test method**: I used Playwright's `test.each()` to iterate over the data file — the test method received each data object as its parameter and executed the same steps (navigate, fill form, submit, assert outcome) for every row. Adding a new test scenario required only adding a new JSON entry. Third, **failure isolation and reporting**: I ensured each data combination ran as an independently named test — when row 14 failed, the report showed "Loan eligibility - age 17, income £20000 - expected REJECTED" rather than a generic failure, making diagnosis immediate. I also handled data validation at load time — if a data entry was malformed, the suite flagged it before any tests ran rather than failing mid-execution.

**Result:** 30 coverage scenarios were implemented with a single test method and one data file — roughly 80% less code than individual methods would have required. When the business added four new eligibility scenarios in the next sprint, the QA analyst added four JSON entries and the coverage was complete with no code changes. One of the new scenarios immediately found a defect — an edge case where a 65-year-old applicant with maximum income was incorrectly rejected.

## Key Terms

| Term | Definition |
|------|------------|
| **Data-Driven Testing** | A framework pattern where test logic and test data are separated — the same test executes with multiple data sets. |
| **Parameterised Test** | A test method that accepts input parameters — enabling a single method to run with multiple data combinations. |
| **Test Data Source** | The external file or system containing the data used in a data-driven test — e.g., JSON, CSV, database, Excel. |
| **Failure Isolation** | Ensuring each data combination is reported as an independent test result — so one failure does not mask others. |
| **Coverage Scaling** | Increasing test coverage by adding data rows rather than writing new test methods — the core efficiency gain of data-driven testing. |
| **Data Validation** | Verifying that test data is valid and complete before test execution begins — prevents obscure mid-run failures. |
