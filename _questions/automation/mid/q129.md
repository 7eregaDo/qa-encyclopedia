title: "What is contract testing and how does it differ from integration testing in an automation context?"
category: automation
level: mid
theory: "Contract testing verifies that two services agree on the interface between them — the request format, response schema, and behaviour expectations — without requiring both services to be running simultaneously. Integration testing verifies that two or more services work correctly together in a real, deployed environment. Contract testing is faster and more isolated; integration testing is more realistic but slower and more environment-dependent."

## The Intent

The interviewer is assessing architectural awareness and knowledge of modern testing approaches for service-based systems. Contract testing is increasingly important in microservices and API-first organisations, and mid-level automation engineers are expected to understand why it exists, what problem it solves, and how to implement it — not just define it.

---

## The STAR Script

**Situation:** Our platform had eight microservices communicating via REST APIs. Integration tests ran in a shared staging environment where all services were deployed together — but they were slow (45 minutes total), environment-dependent, and when they failed it was often unclear which service had broken the interface. Services were also being deployed independently, meaning a provider service could change its API and break consumer services without anyone knowing until the next full integration test run.

**Task:** I was asked to introduce contract testing to catch interface breaking changes earlier and reduce dependence on the shared environment for interface validation.

**Action:** I implemented consumer-driven contract testing using Pact. First, I identified the three most critical service pairs — the order service consuming the inventory API, the payment service consuming the fraud detection API, and the notification service consuming the user profile API. For each pair, I had the **consumer** (e.g., order service) define a Pact contract — specifying exactly what requests it would send and what response structure it expected. These contracts were committed to the Pact Broker. On the **provider** side (e.g., inventory API), I added a contract verification step to its CI pipeline — on every build, the inventory service ran the order service's contract against its own endpoint to confirm it still honoured the agreed interface. If the inventory service changed its response schema in a way that broke the contract, the pipeline failed immediately — before the change was deployed to any shared environment. The key difference from integration testing: the services never needed to be running simultaneously — the consumer defined the contract from its perspective, and the provider verified it independently.

**Result:** Within four weeks of implementation, contract testing caught two breaking changes before they reached staging. A developer on the inventory team renamed a response field (`productId` to `itemId`) — a change that would have silently broken the order service's parsing logic. The contract test failed on the inventory service's pipeline, the developer was notified in seconds, and the change was coordinated before any downstream impact. The integration test suite was reduced to 20 high-value end-to-end scenarios — the contract layer handled interface verification, freeing integration tests to focus on realistic user journeys.

## Key Terms

| Term | Definition |
|------|------------|
| **Contract Testing** | Verifying that two services agree on their interface — independently, without requiring both to run simultaneously. |
| **Consumer** | The service that calls another service's API — defines the contract from its perspective. |
| **Provider** | The service that exposes an API — verifies the consumer's contract against its own implementation. |
| **Consumer-Driven Contract** | A contract defined by the consumer specifying what it needs from the provider — ensures provider changes don't break consumers. |
| **Pact** | A popular open-source consumer-driven contract testing framework. |
| **Pact Broker** | A central server storing and sharing contracts between consumer and provider teams. |
| **can-i-deploy** | A Pact CLI command checking whether a service version is compatible with all its consumers before deployment. |
