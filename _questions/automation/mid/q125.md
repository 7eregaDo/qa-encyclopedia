## The Intent

The interviewer is assessing depth in API testing — one of the highest-value, most underinvested automation layers in many organisations. Mid-level automation engineers are expected to design and implement comprehensive API test suites, not just verify that endpoints return 200 status codes. They want to see coverage of error paths, schema validation, authentication, and business logic — not just happy paths.

---

## The STAR Script

**Situation:** Our team had UI automation covering end-to-end journeys but almost no API-level testing. When defects occurred in the service layer, they were only detected by the UI tests — which meant slow diagnosis, as the failure appeared in the UI but the root cause was in the API.

**Task:** I was asked to build an API test suite from scratch using RestAssured (Java) that would provide fast, isolated coverage of the service layer.

**Action:** I structured the API test suite around five coverage areas. First, **happy path contract testing**: for every endpoint I verified the response status code, Content-Type header, and response body schema — using a JSON schema validator to confirm the structure rather than just individual fields. This caught schema breaking changes immediately. Second, **business logic verification**: for endpoints that performed calculations or transformations, I tested the logic directly — sending known inputs and asserting exact output values, including boundary conditions and edge cases. Third, **error handling**: for every endpoint I tested the response to invalid inputs — missing required fields, incorrect data types, out-of-range values, and invalid authentication tokens. I verified that error responses used appropriate status codes (400 for bad request, 401 for unauthorised, 404 for not found) and returned meaningful error messages. Fourth, **authentication and authorisation**: I verified that protected endpoints rejected requests without valid tokens, that tokens with insufficient permissions received 403 responses, and that token expiry was handled correctly. Fifth, **state changes**: for write operations (POST, PUT, DELETE), I verified the state change persisted correctly by following each write with a GET request to confirm the database had been updated.

**Result:** The API suite covered 34 endpoints with 180 test cases and ran in 4 minutes. Within the first two weeks, it caught three defects that the UI tests had not detected: an endpoint returning 200 with an empty body instead of 404 for missing resources, an authorisation bypass where a standard user could access an admin endpoint by manipulating the role parameter, and a schema breaking change where a field was renamed without a versioning strategy.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Write basic API tests verifying status codes and response body values for happy path scenarios. |
| **Mid** | Design a comprehensive API test suite covering contracts, business logic, error handling, authentication, and state verification — structured for maintainability and CI integration. |
| **Senior** | Build an API testing framework integrated with contract testing (Pact), OpenAPI schema validation, performance baseline testing, and security scanning — with coverage reporting feeding into the engineering quality dashboard. |

## Key Terms

| Term | Definition |
|------|------------|
| **API Test Automation** | Automated testing of service-layer endpoints — faster and more isolated than equivalent UI tests. |
| **Contract Testing** | Verifying that an API's request and response match an agreed interface contract. |
| **Schema Validation** | Asserting that the response body conforms to a defined JSON or XML schema structure. |
| **Status Code Verification** | Confirming the HTTP response code matches the expected value for a given request. |
| **Authorisation Testing** | Verifying that endpoints enforce access controls correctly — rejecting requests from users without sufficient permissions. |
| **State Verification** | Following a write operation with a read to confirm the change persisted correctly in the system. |
