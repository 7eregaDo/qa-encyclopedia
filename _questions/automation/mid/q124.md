## The Intent

The interviewer is assessing one of the most practically challenging aspects of automation engineering. Poor test data management is a leading cause of flaky tests, test interdependence, and parallel execution failures. Mid-level engineers are expected to have a coherent test data strategy — not just use whatever data is already in the environment and hope it doesn't change.

---

## The STAR Script

**Situation:** Our automation suite had been designed to run against a shared staging environment with a fixed set of test accounts. As the suite grew and we introduced parallel execution, tests began failing intermittently — two tests were trying to modify the same user account simultaneously, leaving it in an unexpected state for the second test.

**Task:** I needed to redesign the test data strategy to support parallel execution without test interference, while keeping test setup time manageable.

**Action:** I implemented a three-tier data strategy based on the nature of each test's data needs. **Tier 1 — Dynamic creation via API**: for tests that created, modified, or deleted data, I added a `beforeEach` hook that used the application's API to create a fresh, uniquely named test user or record before each test, and an `afterEach` hook to delete it afterwards. This gave every test its own isolated data state. Dynamic creation via API was fast (200–400ms per setup) and completely eliminated shared-state interference. **Tier 2 — Pre-seeded read-only data**: for tests that only read data (search results, catalogue pages, reporting), I maintained a dedicated read-only dataset seeded once per environment setup. These tests required no setup or teardown and ran the fastest. **Tier 3 — Environment-specific configuration**: credentials, base URLs, and environment-specific references were stored in environment variable files — one per environment — so the same test code ran against dev, staging, and production data configurations without modification.

**Result:** Parallel execution was enabled across four workers with zero cross-test data interference. Test setup time for dynamic creation averaged 280ms — negligible relative to the test execution time it replaced. Suite runtime with parallelisation dropped from 45 minutes to 14 minutes. The approach also improved test reliability in CI where environment state was unpredictable — each test brought its own data rather than depending on pre-existing state.

## Key Terms

| Term | Definition |
|------|------------|
| **Test Data Management** | The strategy for creating, maintaining, and cleaning up the data required by automated tests. |
| **Dynamic Data Creation** | Creating test data via API or database before each test — ensures isolation and supports parallel execution. |
| **Test Isolation** | The principle that each test is independent and does not rely on or affect the data state of other tests. |
| **beforeEach / afterEach** | Lifecycle hooks in test frameworks that run setup and teardown code before and after each individual test. |
| **Pre-Seeded Data** | Static data loaded into the environment once and used by multiple tests for read-only operations. |
| **Parallel Execution** | Running multiple tests simultaneously — requires fully isolated test data to prevent interference. |
