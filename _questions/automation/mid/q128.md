## The Intent

The interviewer is assessing engineering discipline around configuration management. Mid-level automation engineers are expected to design environment-portable test suites — tests that run on any environment by changing configuration, not code. They also want to see awareness of environment dependency challenges: tests that require specific services, feature flags, or data states that may differ between environments.

---

## The STAR Script

**Situation:** Our automation suite had been built to run against the staging environment only. When we tried to run the same suite against a new QA environment for a parallel workstream, 40% of tests failed immediately — base URLs were hardcoded, credentials were embedded in test files, and several tests depended on a third-party payment service that was not available in the QA environment.

**Task:** I needed to make the suite environment-portable and handle the third-party dependency in a way that allowed tests to run in environments where the live service was unavailable.

**Action:** I addressed environment portability in three steps. First, **externalise all environment-specific configuration**: I replaced every hardcoded URL, credential, and environment reference with environment variables loaded from a `.env` file (one per environment: `.env.staging`, `.env.qa`, `.env.production`). The test configuration file read from these variables at runtime. Switching environments required only setting `ENV=qa` — no code changes. Second, **feature flag awareness**: some tests covered features enabled only in certain environments via feature flags. I added feature flag checks to the relevant tests — if the flag was disabled in the current environment, the test would skip gracefully with a documented reason rather than fail confusingly. Third, **third-party service mocking for non-production environments**: for the payment service that was unavailable in QA, I implemented a mock server (using WireMock) that intercepted payment API calls and returned realistic, controlled responses. The mock was configured per environment — QA environments used the mock, staging used the real sandbox, production monitoring used the live service. The switch was controlled by an environment variable.

**Result:** The suite became fully environment-portable. Running against the QA environment required only `ENV=qa npx playwright test` — the correct URLs, credentials, and service configurations loaded automatically. The WireMock payment mock enabled payment flow testing in environments where the live service was unavailable, giving the QA environment full coverage parity with staging. Environment-related test failures dropped to zero.

## Key Terms

| Term | Definition |
|------|------------|
| **Environment Portability** | The ability to run the same test suite against any environment by changing configuration rather than code. |
| **Environment Variables** | Runtime configuration values injected from outside the test code — the standard mechanism for environment-specific settings. |
| **Feature Flag** | A configuration mechanism enabling or disabling features per environment — tests must handle gracefully when a feature is disabled. |
| **Service Virtualisation** | Simulating a third-party or unavailable service with a mock that returns realistic, controlled responses. |
| **WireMock** | A popular service virtualisation tool for stubbing HTTP services in test environments. |
| **Configuration File per Environment** | A separate `.env` or config file per environment containing its specific values — loaded at test runtime. |
