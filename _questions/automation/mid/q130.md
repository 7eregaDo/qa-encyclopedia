## The Intent

The interviewer is assessing practical experience with cross-browser automation challenges — infrastructure, configuration, parallelisation, and maintenance. Mid-level engineers are expected to have more than theoretical knowledge: they should have implemented cross-browser automation and managed the real-world trade-offs of speed, cost, and coverage.

---

## The STAR Script

**Situation:** Our Playwright test suite ran only against Chromium. The product supported Chrome, Firefox, Safari (WebKit), and mobile Chrome — and we had recurring cross-browser defects reaching production because we had no automated cross-browser coverage.

**Task:** I needed to extend the suite to cover the full browser matrix without making the CI pipeline unacceptably slow or expensive.

**Action:** I implemented a three-stage cross-browser strategy. First, **browser matrix definition**: I pulled three months of analytics to define the priority browser list. Chrome (including Chromium) represented 58% of sessions, Safari/WebKit 24%, Firefox 12%, and mobile Chrome 6%. I defined two tiers: Tier 1 (Chrome + WebKit — 82% of users) for every pipeline run; Tier 2 (Firefox + mobile Chrome) for release candidate runs only. Second, **parallelised multi-browser execution in CI**: Playwright's native multi-browser support made configuration straightforward — I defined projects in `playwright.config.ts` for each browser. In GitHub Actions I used a matrix strategy to run each browser project as a parallel job — Chromium, Firefox, and WebKit ran simultaneously rather than sequentially, keeping the total cross-browser run time close to single-browser time. Third, **browser-specific test tagging**: I identified tests that historically surfaced browser-specific issues — CSS layout tests, JavaScript interaction tests, and date/time picker tests — and tagged them `@cross-browser`. On every PR, only `@cross-browser` tagged tests ran against all Tier 1 browsers. The full suite ran against Chromium only on PRs. The full cross-browser suite (all tests, all browsers) ran only on merge to main and on release candidates.

**Result:** The tiered strategy added only 4 minutes to PR pipeline time (parallel Tier 1 cross-browser for tagged tests) while achieving meaningful cross-browser coverage on every merge. The first full cross-browser run caught a WebKit-specific CSS flex behaviour that was causing the checkout summary to overflow on Safari — previously invisible to Chromium-only testing. Release candidate runs subsequently found two Firefox-specific JavaScript timing issues over the following quarter.

## Key Terms

| Term | Definition |
|------|------------|
| **Cross-Browser Automation** | Running automated tests against multiple browsers to detect browser-specific rendering or behaviour differences. |
| **Browser Matrix** | The defined set of browsers and versions against which tests are executed — derived from real user analytics. |
| **Playwright Projects** | A Playwright configuration feature defining multiple browser targets within a single test suite. |
| **Matrix Strategy** | A CI configuration pattern running the same job in parallel with different parameters — used to parallelise cross-browser execution. |
| **Browser-Specific Tagging** | Marking tests that are most likely to surface cross-browser issues — enables selective cross-browser execution on PRs. |
| **WebKit** | The browser engine used by Safari — tested via Playwright's built-in WebKit support without requiring a macOS runner. |
