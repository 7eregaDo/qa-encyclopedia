## The Intent

The interviewer is assessing adaptability and investigative thinking. Third-party integrations with poor documentation are a common real-world challenge — payment gateways, logistics APIs, and communication platforms frequently have incomplete or outdated docs. Mid-level testers are expected to navigate this ambiguity productively rather than blocking on missing information.

---

## The STAR Script

**Situation:** I was asked to test an integration with a regional payment provider whose API documentation was sparse — a single-page PDF with endpoint names and basic parameter descriptions, but no error code definitions, no sandbox behaviour notes, and no documentation of edge cases or failure modes.

**Task:** I needed to build enough understanding of the integration's behaviour to design a meaningful test suite covering success paths, failure handling, and edge cases — despite the documentation gap.

**Action:** I worked through five stages. First, **extract what exists**: I read the available documentation thoroughly and extracted every piece of confirmed information — the endpoint structure, the required parameters, the authentication method, and the three success response examples provided. This gave me the confirmed happy path. Second, **collaborate with the developer**: the developer who built the integration had learned a significant amount through implementation that wasn't in the docs. I spent 30 minutes asking structured questions: What error responses have you seen? Are there any undocumented behaviours you discovered? What scenarios did you have to handle that weren't in the spec? This conversation doubled my understanding in less time than any amount of documentation reading would have. Third, **exploratory investigation in the sandbox**: I tested the integration directly in the provider's sandbox environment, deliberately sending malformed requests, missing parameters, and boundary values — observing and logging the actual responses. This reverse-engineered the error behaviour the documentation had omitted. I built a personal reference document of observed responses. Fourth, **risk-based prioritisation**: with limited knowledge, I prioritised test cases by business impact — successful payment processing first, payment failure handling second (users must receive clear feedback and not be charged), and edge cases (duplicate requests, session timeouts, currency edge cases) third. Fifth, **flag residual unknowns**: I documented explicitly what I could not test due to missing information — specific error codes whose trigger conditions were unknown, and behaviours specific to production that the sandbox did not replicate — and included this in my test summary as acknowledged risk.

**Result:** The exploratory investigation found that the sandbox returned a generic 500 error for any request with a missing optional parameter — which the documentation had not mentioned was required in certain contexts. This surfaced a gap in the developer's implementation that had not been caught because the happy path always included those parameters. The residual risk documentation gave the product owner an accurate picture of what remained unknown at release, enabling an informed go/no-go decision.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Describe using available documentation and asking the developer for clarification before writing test cases. |
| **Mid** | Combine documentation analysis, developer collaboration, sandbox exploration, risk-based prioritisation, and explicit documentation of residual unknowns into a coherent testing approach. |
| **Senior** | Design a third-party integration testing framework — contract testing, sandbox environment management, monitoring for provider-side changes, and a process for re-testing integrations when provider APIs are updated. |

---

## Key Terms

| Term | Definition |
|------|------------|
| **Third-Party Integration** | A connection between the application under test and an external service or API not owned by the development team. |
| **Reverse Engineering** | Deriving the expected behaviour of a system by observing its actual responses rather than reading its specification. |
| **Sandbox Environment** | A test instance of a third-party service that simulates production behaviour without processing real transactions. |
| **Residual Unknown** | A behaviour or scenario that cannot be tested due to missing information — must be documented and acknowledged as accepted risk. |
| **Contract Testing** | Verifying that the integration between two systems conforms to an agreed interface contract — reduces dependence on complete documentation. |
| **Developer Collaboration** | Working directly with the engineer who built the integration to extract implementation knowledge that supplements incomplete documentation. |
