title: "What is decision table testing and when is it more effective than other techniques?"
category: manual-testing
level: mid
theory: "Decision table testing is a black-box technique that organises combinations of input conditions and their corresponding expected actions or outputs into a table structure. It is most effective when business logic involves multiple conditions that interact to produce different outcomes — where equivalence partitioning and boundary value analysis test conditions in isolation but decision tables expose how combinations of conditions drive behaviour."
---

## The Intent

The interviewer is testing depth of test design knowledge. Mid-level testers are expected to select techniques based on the nature of the logic being tested — not apply the same technique to every problem. They want to see that you understand when decision tables add value over simpler techniques, and that you can construct one correctly for a realistic scenario.

---

## The STAR Script

**Situation:** I was testing a shipping cost calculation feature with the following rules: standard shipping is free for orders over £50, express shipping always costs £9.99, and for orders under £50, standard shipping costs £3.99. Additionally, loyalty members received free standard shipping regardless of order value.

**Task:** I needed to design test cases that covered all meaningful combinations of shipping type, order value, and loyalty status — and ensure no rule interaction was missed.

**Action:** I constructed a decision table with three conditions: **Shipping type** (Standard / Express), **Order value** (Under £50 / £50 or over), and **Loyalty member** (Yes / No). This produced eight possible combinations. I mapped each to its expected shipping cost outcome. Some combinations were immediately revealing — express shipping cost was £9.99 regardless of order value or loyalty status (two rules confirmed), loyalty members always paid £0 for standard regardless of order value (two rules confirmed), and non-loyalty members on standard shipping paid based on order value threshold (two rules confirmed). The remaining two combinations (express + loyalty and express + non-loyalty) both correctly resulted in £9.99, confirming express shipping was never free. I then reduced the eight cases by merging those with identical outcomes where the varying condition was provably irrelevant — reducing to six executable test cases without losing coverage. I also used the table construction process to surface an unspecified scenario: what happens when a loyalty member selects express shipping? The requirements didn't address whether loyalty status applied to express shipping. I flagged this as an ambiguity before execution.

**Result:** Six targeted test cases were executed. Two defects were found: the loyalty free-shipping rule was not being applied when the cart contained only sale items, and express shipping for orders over £50 was incorrectly showing as free for loyalty members — revealing the unspecified scenario had been implemented as "free express for loyalty members," which was wrong. The decision table had directly surfaced the ambiguity that led to the second defect.

## Key Terms

| Term | Definition |
|------|------------|
| **Decision Table** | A structured table mapping combinations of input conditions to expected system actions or outputs. |
| **Condition** | An input factor that influences the system's behaviour — a column in the decision table. |
| **Action / Output** | The expected system response for a given combination of conditions — the result row in the table. |
| **Column Merging** | Combining decision table columns where a condition's value is irrelevant to the outcome — reduces test case count without losing coverage. |
| **Don't Care Condition** | A condition whose value does not affect the outcome for a specific rule — represented as "-" in the table. |
| **Rule Gap** | A combination of conditions not covered by any existing business rule — an ambiguity that must be clarified before testing. |
