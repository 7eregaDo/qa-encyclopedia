## The Intent

The interviewer is checking whether you treat accessibility as a first-class quality concern rather than an optional extra. Mid-level testers are expected to know the WCAG standard, apply both automated and manual accessibility checks, and articulate why accessibility matters beyond compliance. They also want to see practical experience — not just theoretical knowledge.

---

## The STAR Script

**Situation:** I was assigned accessibility testing responsibility for a government-adjacent SaaS product. The product's enterprise contracts required WCAG 2.1 AA conformance, and an upcoming audit meant we needed to document our compliance status before the end of the quarter.

**Task:** I needed to conduct a systematic accessibility review of the product's core user journeys, identify and prioritise non-conformance issues, and produce a report that was useful to both developers and the compliance team.

**Action:** I structured the accessibility testing across three layers. First, **automated scanning**: I ran Axe DevTools and Lighthouse against all primary pages. Automated tools reliably catch approximately 30–40% of accessibility issues — things like missing alt text, insufficient colour contrast ratios, missing form labels, and invalid ARIA roles. I exported and deduplicated the findings, classifying each by WCAG criterion and severity. Second, **manual keyboard testing**: I navigated all core user journeys using only the keyboard — Tab, Shift+Tab, Enter, Space, and arrow keys. I verified that every interactive element was reachable, that focus order was logical, that focus indicators were visible, and that modal dialogs trapped focus correctly and returned it on close. This is an area automated tools cannot fully assess. Third, **screen reader testing**: I used NVDA on Windows with Chrome and VoiceOver on macOS with Safari to navigate key journeys. I listened for whether page structure made sense when read linearly, whether images had meaningful descriptions, whether form errors were announced, and whether dynamic content updates were communicated to the screen reader.

**Result:** The automated scan found 23 issues — missing alt text across image-heavy marketing pages was the most common. Manual keyboard testing found six additional issues, including a date picker that was completely keyboard-inaccessible. Screen reader testing found four issues, including a live search results area that updated silently without announcing the new content to the screen reader. The full report was delivered with 33 issues classified by WCAG criterion, severity, and recommended fix. Eleven were resolved before the audit, and the date picker issue was prioritised as a high-severity fix for the following sprint.

## Key Terms

| Term | Definition |
|------|------------|
| **WCAG** | Web Content Accessibility Guidelines — the international standard for web accessibility, published by W3C. |
| **WCAG 2.1 AA** | The most widely required conformance level — the baseline for legal compliance in most jurisdictions. |
| **Axe DevTools** | A browser extension and API for automated accessibility scanning — the industry standard tool. |
| **Keyboard Navigation** | Using Tab, Enter, and arrow keys to interact with a page without a mouse — essential for motor-impaired users. |
| **Screen Reader** | Assistive technology that reads page content aloud for visually impaired users — NVDA (Windows) and VoiceOver (macOS/iOS) are most common. |
| **Focus Indicator** | The visible outline showing which element is currently focused during keyboard navigation — must be clearly visible. |
| **ARIA** | Accessible Rich Internet Applications — HTML attributes that provide semantic information to assistive technologies. |
