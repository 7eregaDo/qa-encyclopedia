category: manual-testing
level: mid
theory: "Integration testing verifies that two or more components or systems work correctly together — focusing on the interface and data exchange between them. End-to-end testing verifies that a complete user journey works correctly across all components from start to finish, simulating real user behaviour. Integration testing focuses on component boundaries; end-to-end testing focuses on user outcomes."
---

## The Intent

The interviewer is testing conceptual precision and the ability to apply both types of testing strategically. Mid-level testers are expected to understand not just the definitions but when each type adds the most value — and to avoid conflating them, which leads to either gaps in coverage or duplicated effort.

---

## The STAR Script

**Situation:** I was testing a new feature that allowed users to export their order history as a PDF. The feature involved three components working together: the frontend export button, a backend API that queried the order database and generated the PDF, and a third-party PDF rendering service.

**Task:** I needed to design a test approach that covered both the component interfaces and the full user journey — without duplicating effort between the two.

**Action:** I applied integration and end-to-end testing as distinct, complementary layers. For **integration testing**, I focused on the boundaries between components. I tested the API directly using Postman — verifying it returned the correct data structure when given a valid user ID, handled an invalid user ID with a 404, and correctly passed the order data to the PDF rendering service in the expected format. I also tested the PDF rendering service boundary specifically: what happened when the order data was malformed, when the service was slow to respond, and when it returned an error. These tests were surgical — they didn't require the full frontend to be involved, and they isolated exactly which component was responsible for a failure. For **end-to-end testing**, I tested the complete user journey: log in as a user with order history, navigate to the account section, click Export PDF, confirm the download triggers, open the PDF, and verify the content matched the orders displayed on screen. I also tested edge cases at the user journey level: a user with no order history, a user with a very large number of orders, and the journey on a slow network connection.

**Result:** Integration testing found that the API was not handling orders containing special characters in product names correctly — they were being truncated in the data structure passed to the PDF service. This was an interface-level defect invisible to end-to-end testing with standard test data. End-to-end testing found that the PDF download triggered correctly but the downloaded file name defaulted to "document.pdf" rather than the expected "order-history-[username].pdf" — a user-facing issue invisible to API-level testing. Each layer found something the other would have missed.

## Key Terms

| Term | Definition |
|------|------------|
| **Integration Testing** | Testing the interface and data exchange between two or more components — focused on component boundaries. |
| **End-to-End Testing** | Testing a complete user journey from the UI through all components to the data layer — focused on user outcomes. |
| **Component Boundary** | The interface point where two components exchange data — the focus of integration testing. |
| **User Journey** | A complete sequence of steps a user performs to achieve a goal — the scope of end-to-end testing. |
| **Testing Pyramid** | A model recommending more unit tests, fewer integration tests, and even fewer E2E tests — based on speed and maintenance cost. |
| **Test Layer** | A distinct level of the test strategy with a specific scope and purpose — e.g., unit, integration, E2E. |
