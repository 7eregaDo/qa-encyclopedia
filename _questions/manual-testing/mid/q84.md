theory: "State transition testing is a black-box technique used to test systems whose behaviour depends on their current state. A state transition diagram or table maps all possible states, the events that trigger transitions between them, and the expected outcomes. Test cases are designed to exercise valid transitions (the system moves to the correct next state), invalid transitions (the system correctly refuses disallowed moves), and transition sequences (chains of valid states leading to a specific outcome)."

## The Intent

The interviewer is testing whether you can apply structured test design to workflow-based features — applications with multi-step processes, status machines, or sequential flows are extremely common, and state transition testing is the right tool for them. They want to see that you can model the state machine correctly, derive meaningful test cases, and think about invalid transitions — not just the happy path sequence.

---

## The STAR Script

**Situation:** I was testing an order management system where orders moved through the following states: Draft → Submitted → Under Review → Approved → Fulfilled, with the possibility of Rejected (from Under Review) and Cancelled (from Draft, Submitted, or Approved). Not all transitions were valid — for example, you could not move directly from Draft to Approved, and a Fulfilled order could not be Cancelled.

**Task:** I needed to design a test suite that verified all valid state transitions, confirmed that invalid transitions were correctly blocked, and tested the key multi-step sequences a real user would follow.

**Action:** I worked through three layers of test design. First, **state transition diagram**: I drew the complete diagram showing all states as nodes and all valid transitions as directed arrows. This immediately revealed a gap in the requirements — the specification mentioned Rejected state but did not define whether a rejected order could be resubmitted or only cancelled. I flagged this before writing test cases. Second, **valid transition tests**: I wrote one test case per valid transition — 8 test cases covering every arrow in the diagram. Each confirmed the system reached the correct next state and displayed the correct status label and available action buttons. Third, **invalid transition tests**: I attempted transitions that were not in the diagram — trying to Approve a Draft order directly, trying to Cancel a Fulfilled order, trying to move from Rejected back to Under Review. These confirmed the system blocked the transitions and displayed an appropriate error or simply did not offer the invalid action. Fourth, **sequence tests**: I designed three end-to-end sequence tests covering the most important user journeys — full happy path (Draft to Fulfilled), rejection and cancellation path, and the edge case of rapid status changes in quick succession.

**Result:** Valid and sequence tests all passed. Invalid transition testing found two defects: the API accepted a direct Draft-to-Approved transition when called directly (bypassing the UI), which would have been exploitable via API manipulation, and a Fulfilled order could be set back to Approved via a direct database-level action that was not blocked by application logic. Both were security-relevant findings that only emerged from testing the invalid transitions.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define state transition testing and identify the states and transitions for a simple two or three-state system. |
| **Mid** | Draw a full state transition diagram, derive test cases for valid and invalid transitions, design sequence tests, and surface specification gaps through the modelling process. |
| **Senior** | Apply state transition testing to complex distributed systems with concurrent state changes, integrate state machine verification into automated test suites, and use the state model as a living specification document. |

## Key Terms

| Term | Definition |
|------|------------|
| **State** | A distinct condition or mode the system can be in — e.g., Draft, Submitted, Approved. |
| **Transition** | A change from one state to another, triggered by an event or action. |
| **Valid Transition** | A state change that is permitted by the system's rules. |
| **Invalid Transition** | A state change that should be blocked — testing these confirms the system enforces its rules correctly. |
| **State Transition Diagram** | A visual map of all states and the transitions between them — the primary tool for state transition test design. |
| **Sequence Test** | A test case that exercises a chain of transitions end-to-end — covering a complete user journey through the state machine. |
