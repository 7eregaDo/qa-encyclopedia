## The Intent

The interviewer is using search as a practical canvas for observing how broadly and systematically you think about a feature. Search looks simple but has significant depth — indexing logic, ranking algorithms, edge cases, performance characteristics, and security considerations. They want to see that you think beyond "enter a term and check results appear."

---

## The STAR Script

**Situation:** I was tasked with testing a full-text search feature for a product catalogue application. The feature allowed users to search by product name, description, SKU, and category, with results ranked by relevance and filtered by in-stock status.

**Task:** I needed to design a comprehensive end-to-end test approach covering all meaningful dimensions of the feature before its release.

**Action:** I organised my test approach into six categories. **Functional — result correctness**: search for exact product name returns that product in results; search for a partial name returns all matching products; search is case-insensitive; search across all indexed fields (name, description, SKU, category) returns appropriate results. **Input handling**: empty search string behaviour (does it search everything or display an error?); single character query; very long query string (500+ characters); special characters (quotes, slashes, ampersands); non-Latin characters and emoji; numeric-only query; whitespace-only query. **Relevance and ranking**: exact name match appears above partial match; more recently added products do not unfairly outrank highly relevant older products; filtering by in-stock only removes out-of-stock items completely and immediately. **No results state**: a query with no matches displays a clear, helpful "no results" message rather than an empty page; the no-results message suggests related searches or a way to broaden the query. **Performance**: search response time for a common query is within the defined SLA (sub-500ms); response time under concurrent load (simulated using browser network throttling and timing tools). **Security**: SQL injection attempt in the search field; XSS payload in the search field; search query reflected in the URL is properly encoded to prevent injection via URL manipulation.

**Result:** Functional and input tests found two defects: a query containing a single apostrophe caused a 500 server error (SQL injection vulnerability — critical), and a whitespace-only query returned the full product catalogue rather than an error or empty state. The performance test identified that queries with very short terms (one or two characters) had response times exceeding 2 seconds — an indexing issue that was resolved before release.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Cover basic functional scenarios: matching, partial matching, case sensitivity, and empty state. |
| **Mid** | Test across all six dimensions — functional, input handling, relevance, no-results state, performance, and security — with specific test cases for each. |
| **Senior** | Design a search testing framework covering indexing pipeline verification, relevance scoring validation, A/B test coverage, and integration with a search analytics platform to measure real-world search quality. |

---

## Key Terms

| Dimension | Focus |
|-----------|-------|
| **Functional Correctness** | Does the search return the right results for standard queries? |
| **Input Handling** | Does the search handle all types of input gracefully without crashing? |
| **Relevance and Ranking** | Are the most relevant results surfaced first, consistently? |
| **No Results State** | Is the empty state clear, helpful, and non-confusing? |
| **Performance** | Is the search fast enough under normal and peak load? |
| **Security** | Is the search input sanitised against SQL injection and XSS? |
| **SQL Injection** | An attack inserting malicious SQL into an input field to manipulate the database — a critical risk in search inputs. |
