## The Intent

The interviewer is using a familiar, concrete example to observe your testing thought process in real time. They are not looking for an exhaustive list of every possible test case — they are looking for evidence that you think in categories, apply test design techniques, and consider dimensions beyond basic functionality. A candidate who only describes "enter valid credentials and check you're logged in" has revealed a significant gap.

---

## The STAR Script

**Situation:** I was asked to test the login form for a new web application before its beta release. The form had two fields — email address and password — and a "remember me" checkbox.

**Task:** I needed to design a comprehensive set of test cases that covered all meaningful scenarios efficiently, rather than testing every possible input combination.

**Action:** I organised my test approach into five categories. **Functional — positive**: valid email and correct password logs the user in successfully; redirect goes to the correct post-login page; "remember me" checkbox persists the session across browser restarts. **Functional — negative**: wrong password displays an appropriate error without revealing which field was incorrect; unregistered email address displays the same error message (not "email not found" — that would enable user enumeration); empty email or password fields trigger inline validation before form submission. **Boundary and input validation**: email field without "@" symbol, password at minimum length boundary, password below minimum length, very long input strings in both fields, special characters and emoji in both fields. **Security**: multiple failed login attempts trigger an account lockout or rate limiting (not unlimited attempts); password field masks input and does not appear in browser autocomplete in plain text; the form is served over HTTPS; the URL does not contain credentials after submission. **Usability**: error messages are clear and actionable without being too revealing; the form is accessible via keyboard navigation; "Forgot password" link is present and functional; the form is responsive and functional on mobile viewport.

**Result:** This structured approach produced 28 test cases from five categories. The security testing revealed that there was no rate limiting on login attempts — an unlimited brute force vulnerability. The usability testing flagged that the error message for a wrong password said "Email address not found" — a user enumeration vulnerability. Both were critical findings that would not have been discovered through functional-only testing.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Cover functional positive, functional negative, input validation, and at least mention security and usability dimensions. |
| **Mid** | Include security testing in depth (enumeration, rate limiting, session management), accessibility, and cross-browser/device coverage. |
| **Senior** | Add penetration testing considerations, OAuth/SSO flows, session token security, CSRF protection, and multi-factor authentication scenarios. |

---

## Key Terms

| Category | Examples |
|----------|---------|
| **Functional Positive** | Valid credentials, successful redirect, remember me persistence. |
| **Functional Negative** | Wrong password, unregistered email, empty fields. |
| **Boundary / Input Validation** | Min/max length, special characters, SQL injection attempt. |
| **Security** | Rate limiting, user enumeration, HTTPS enforcement, password masking. |
| **Usability** | Clear error messages, keyboard navigation, mobile responsiveness, forgot password flow. |
| **User Enumeration** | A security vulnerability where error messages reveal whether an email address is registered — enabling targeted attacks. |
