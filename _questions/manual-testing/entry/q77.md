title: "What is the difference between smoke testing and regression testing?"
category: manual-testing
level: entry
theory: "Smoke testing is a broad, shallow verification of a new build's core functionality to confirm it is stable enough for further testing. Regression testing is a comprehensive re-execution of existing test cases after a change, to verify that previously working functionality has not been broken. Smoke testing asks 'is this build worth testing?' while regression testing asks 'did anything we already had stop working?'"
---

## The Intent

The interviewer is checking precision in your use of testing terminology and your understanding of how different test types serve different purposes in the release cycle. Confusing these two — or using them interchangeably — signals shallow knowledge of testing practice. They also want to see that you understand how and when each fits into a deployment workflow.

---

## The STAR Script

**Situation:** Our team deployed multiple builds per day to a staging environment. Running a full regression suite after every deployment was impractical — it took two hours and would have consumed the team's entire capacity.

**Task:** I needed to design a tiered verification approach that gave rapid confidence after each deployment without burning regression time unnecessarily.

**Action:** I introduced two distinct verification levels. After every deployment — regardless of how small the change — I ran a **smoke test**: a 15-minute, 12-step suite covering the absolute core journeys: user can log in, dashboard loads, primary feature is accessible, and a basic end-to-end transaction completes. If smoke failed, the deployment was rolled back immediately and the development team was notified. No further testing was done on a build that failed smoke. **Regression testing** was reserved for end-of-sprint builds and release candidates — a full two-hour execution of all test cases covering every feature area. The regression suite was also run in full after any change touching shared infrastructure, authentication, or the payment flow — regardless of sprint timing. I documented this policy in the team's testing process guide so that every team member applied it consistently.

**Result:** The tiered approach reduced unnecessary regression runs by 60% without increasing the escaped defect rate. Smoke testing caught three broken deployments within minutes of deployment — before the development team had moved on to the next task — significantly reducing the cost of rollback. Regression testing remained thorough and trustworthy because it was not diluted by being run on every minor change.

## Key Terms

| Term | Definition |
|------|------------|
| **Smoke Testing** | Broad, shallow build verification confirming core functionality is stable enough for further testing. |
| **Regression Testing** | Comprehensive re-execution of existing tests after a change to confirm nothing previously working has broken. |
| **Build Acceptance Test** | Another name for a smoke test — the minimum bar a build must pass before investment in further testing is warranted. |
| **Regression Suite** | The curated set of test cases executed during a regression run. |
| **Rollback** | Reverting a deployment to a previous stable version when a critical failure is detected. |
| **Trigger Criteria** | The conditions that determine when a specific type of test should be run — e.g., run full regression on release candidates only. |
