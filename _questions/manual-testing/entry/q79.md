## The Intent

The interviewer is checking whether you understand testing as a user-centred discipline, not just a technical one. Many entry-level testers focus exclusively on functional correctness and miss usability issues that have significant real-world impact. They want to see that you actively look for friction, confusion, and poor communication in the user experience — not just broken functionality.

---

## The STAR Script

**Situation:** I was assigned to test a newly redesigned account settings page. All functional test cases had passed — every setting saved correctly, every toggle worked as expected. Before signing off, I conducted a usability review.

**Task:** I needed to evaluate the page not just for what it did, but for how easy and clear it was to use — approaching it as a user encountering it for the first time.

**Action:** I worked through a usability checklist organised around five dimensions. **Clarity**: is the purpose of each element immediately obvious? I found that a toggle labelled "Notifications" had no description of which notifications it controlled — a user could not make an informed decision about toggling it. **Error communication**: do error messages tell users what went wrong and what to do next? One error on the page said "Update failed" with no further context — not useful. **Consistency**: do interactive elements behave consistently with conventions established elsewhere in the product and on the web generally? A "Save" button was positioned on the left side of the form — unconventional and likely to be missed. **Efficiency**: can a user complete the most common task in a reasonable number of steps? I mapped the steps to change a password — seven steps, two of which required navigating away and back. **Accessibility basics**: can all interactive elements be reached and activated by keyboard alone? The account deletion confirmation dialog was not keyboard accessible.

**Result:** I raised five usability issues — none of which were functional defects. All five were accepted by the product owner and addressed before release. The most impactful fix was the notification toggle description — user research conducted post-release showed it was the most-used setting on the page, and the added description significantly reduced support queries about notification behaviour.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define usability testing, describe the dimensions you look for, and give concrete examples from a real or hypothetical scenario. |
| **Mid** | Discuss heuristic evaluation (e.g., Nielsen's 10 usability heuristics), how to prioritise usability findings by severity, and how to engage product and UX with usability defects constructively. |
| **Senior** | Design a usability testing programme — combining heuristic evaluation, user testing sessions, and accessibility audit — and integrate usability acceptance criteria into the definition of done. |

## Key Terms

| Term | Definition |
|------|------------|
| **Usability Testing** | Evaluating how easy and intuitive a system is to use from the perspective of its intended users. |
| **Heuristic Evaluation** | A usability review technique where a tester evaluates the interface against a set of established principles (heuristics). |
| **Clarity** | Whether the purpose and behaviour of UI elements is immediately understandable without instruction. |
| **Error Communication** | How clearly the system explains what went wrong and what the user should do next. |
| **Efficiency** | The number of steps and time required for a user to complete a common task. |
| **Accessibility** | Whether the system can be used by people with disabilities — including keyboard navigation, screen reader support, and sufficient colour contrast. |
