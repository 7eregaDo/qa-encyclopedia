## The Intent

The interviewer is checking whether your testing instincts extend beyond the happy path. Entry-level testers often focus heavily on positive scenarios — verifying that things work — and underinvest in negative testing, which is where many real-world defects hide. They want to see that you actively design both types of tests and understand why negative testing is equally important.

---

## The STAR Script

**Situation:** I was assigned to test a user registration form for a web application. The form included fields for first name, last name, email address, password, and date of birth.

**Task:** I needed to design test cases that covered both positive and negative scenarios thoroughly — not just confirm the happy path worked.

**Action:** For **positive testing**, I verified the expected behaviour with valid inputs: a correctly formatted email address, a password meeting the stated complexity rules, a valid date of birth within the accepted range, and standard name inputs. I confirmed the form submitted successfully and the user account was created. For **negative testing**, I designed a separate suite of tests targeting invalid and boundary inputs: submitting an email without an "@" symbol, a password below the minimum length, a future date of birth, a name field containing only spaces, SQL injection in the name field, an extremely long input string in each field, and submitting the form with each required field left empty one at a time. Each test case had a specific expected result — typically a clear, user-friendly validation error message displayed inline, with no form submission and no application crash.

**Result:** Positive tests all passed. Negative testing revealed three defects: the form accepted a future date of birth without error, a very long name input caused the page layout to break, and submitting with an empty password field produced a generic server error rather than an inline validation message. None of these would have been found through positive testing alone.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Define both clearly with concrete examples, and demonstrate that you actively design negative test cases — not just positive ones. |
| **Mid** | Discuss negative testing strategies systematically — error guessing, boundary conditions, injection attacks — and integrate them into test design as a discipline. |
| **Senior** | Build a negative testing framework that covers security (injection, authentication bypass), data integrity (corruption, truncation), and resilience (timeouts, service unavailability) across the full test suite. |

## Key Terms

| Term | Definition |
|------|------------|
| **Positive Testing** | Testing with valid, expected inputs to confirm the system behaves correctly on the happy path. |
| **Negative Testing** | Testing with invalid, unexpected, or boundary inputs to confirm the system handles errors gracefully. |
| **Happy Path** | The primary user journey with no errors, edge cases, or unexpected inputs — the simplest positive scenario. |
| **Error Guessing** | An experience-based technique for identifying likely negative test cases based on knowledge of common failure modes. |
| **Validation Message** | A user-facing error message displayed when input fails a validation rule — the expected result of most negative test cases. |
| **Graceful Failure** | A system's ability to handle invalid input or error conditions without crashing, exposing data, or producing misleading output. |
