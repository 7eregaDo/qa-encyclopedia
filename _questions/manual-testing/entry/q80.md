## The Intent

The interviewer is assessing whether you can think strategically about testing under real-world constraints. Every testing effort operates under time pressure — the ability to prioritise intelligently is one of the most practically valuable skills a QA engineer can demonstrate. A candidate who says "I would try to test everything" has not answered the question. They want a concrete prioritisation framework.

---

## The STAR Script

**Situation:** Thirty minutes before a scheduled release sign-off, a developer merged a last-minute bug fix that affected the checkout flow. I had planned to spend two hours on final regression, but the timeline had collapsed to 45 minutes.

**Task:** I needed to decide immediately which test cases to execute in the available time and which to defer — and communicate the coverage gap clearly before sign-off.

**Action:** I applied a three-tier prioritisation in under five minutes. **Tier 1 — Must execute**: test cases covering the area directly changed by the late fix (the checkout flow), any test cases covering revenue-critical paths (payment processing, order confirmation), and the smoke test suite confirming core application stability. These were non-negotiable regardless of time. **Tier 2 — Execute if time allows**: test cases for features adjacent to the changed area (cart management, promo code application), as changes in one area sometimes have side effects in connected features. **Tier 3 — Defer with documented risk**: test cases for unrelated features untouched by the fix and not in the critical path — account settings, help pages, search functionality. I executed Tier 1 in full (28 minutes), completed 60% of Tier 2 in the remaining time, and documented the deferred Tier 3 coverage in my release sign-off note with an explicit statement that those areas had not been tested in this cycle.

**Result:** The Tier 1 execution confirmed the bug fix was working correctly and no regressions had been introduced in the payment flow. One Tier 2 test caught a cosmetic issue with the cart item count after the fix — minor but worth logging. The deferred Tier 3 areas were tested the following morning with no findings. The product owner appreciated the transparent communication about what had and hadn't been tested — the decision to release was informed, not blind.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Apply a clear prioritisation framework based on risk and business impact, communicate deferred coverage, and execute the highest-value tests first. |
| **Mid** | Describe a more formal risk-based prioritisation model applied across a full sprint testing cycle, including impact analysis of specific code changes on test scope. |
| **Senior** | Design a test execution strategy that automatically prioritises based on code change impact, historical defect data, and risk tier — minimising manual triage decisions under pressure. |

---

## Key Terms

| Term | Definition |
|------|------------|
| **Risk-Based Prioritisation** | Ordering test execution by the combined likelihood and business impact of potential failures. |
| **Critical Path** | The sequence of functionality that must work for the core user journey and business operation to succeed. |
| **Tier 1 / Must Execute** | Test cases covering the changed area and highest business-risk functionality — non-negotiable under any time constraint. |
| **Deferred Coverage** | Test cases not executed in the current cycle — must be documented explicitly with an assessed risk statement. |
| **Impact Analysis** | Identifying which test cases are most relevant based on the specific changes made in a build. |
| **Coverage Communication** | Explicitly informing stakeholders what was and was not tested before a release decision is made. |
