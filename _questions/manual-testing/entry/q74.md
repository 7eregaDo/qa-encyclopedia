theory: "Boundary value analysis (BVA) is a black-box test design technique that focuses testing on the values at and immediately around the edges of valid input ranges. It is effective because defects disproportionately cluster at boundaries — where conditional logic changes state — making boundary values a higher-yield target than values in the middle of a range."

## The Intent

The interviewer wants to confirm you understand not just what BVA is, but why it works — the underlying principle that boundary conditions are where off-by-one errors, incorrect comparison operators, and range logic bugs hide. Candidates who can explain the "why" demonstrate deeper understanding than those who can only apply the technique mechanically.

---

## The STAR Script

**Situation:** I was testing a discount pricing rule for an e-commerce platform: orders totalling £50.00 or more qualified for a 10% discount. Orders below £50.00 received no discount.

**Task:** I needed to design test cases that would efficiently expose any defect in the boundary logic — specifically any off-by-one or comparison operator errors in the implementation.

**Action:** I applied boundary value analysis by identifying the exact boundary (£50.00) and testing the values immediately around it. My test cases were: £49.99 (just below the boundary — should receive no discount), £50.00 (exactly at the boundary — should receive the 10% discount), and £50.01 (just above the boundary — should receive the 10% discount). I also tested a clearly in-range value (£100.00) and a clearly out-of-range value (£10.00) as confirmatory cases. This gave me five focused test cases covering the boundary in both directions. I chose these specific values because boundary defects almost always appear at the exact edge or one unit either side — a developer who writes `> 50` instead of `>= 50` produces a bug that is invisible at £100 but visible at £50.00 exactly.

**Result:** The test at exactly £50.00 failed — the system was applying the condition as "greater than £50" rather than "greater than or equal to £50," meaning a £50.00 order received no discount. This was a direct off-by-one operator error that would have affected every customer whose order total was exactly £50.00. The boundary value test case found it precisely because it targeted the exact edge, where the logic was wrong.

## Key Terms

| Term | Definition |
|------|------------|
| **Boundary Value Analysis (BVA)** | A technique testing at the exact edges of valid input ranges and the values immediately adjacent. |
| **Boundary** | The point at which the system's behaviour changes — e.g., the minimum or maximum of a valid range. |
| **Off-by-One Error** | A defect where a boundary condition is implemented one unit off — e.g., `> 50` instead of `>= 50`. |
| **Two-Value BVA** | Testing the boundary value and the value just outside it — two test cases per boundary. |
| **Three-Value BVA** | Testing the value just below, at, and just above the boundary — three test cases per boundary edge. |
| **Comparison Operator Defect** | A bug caused by using the wrong comparison operator in conditional logic — e.g., `>` vs `>=`. |
