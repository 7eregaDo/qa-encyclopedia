## The Intent

The interviewer is assessing breadth of testing knowledge and the ability to think beyond English-language, Western-convention assumptions. Senior testers on products serving international markets must be able to design i18n and l10n testing strategies — covering not just text translation but date formats, currency, number formatting, right-to-left text, character encoding, and cultural appropriateness.

---

## The STAR Script

**Situation:** Our product was expanding from an English-only UK market into four new locales: French (France), German (Germany), Japanese (Japan), and Arabic (Saudi Arabia). The Arabic locale introduced right-to-left (RTL) layout requirements, and the Japanese locale introduced double-byte character set (DBCS) handling requirements — both were engineering challenges beyond simple text translation.

**Task:** I needed to design a systematic i18n and l10n testing strategy covering all four locales, prioritised by risk and executable within a team that did not speak all the target languages.

**Action:** I separated the strategy into two distinct phases. For **i18n testing** (can the application handle these locales?), I focused on engineering-level concerns that did not require language proficiency. First, **character encoding**: I tested every text input field with double-byte characters (Japanese kanji, Arabic script) and confirmed they were stored, retrieved, and displayed without corruption. Second, **RTL layout**: I enabled Arabic locale and verified that the entire layout mirrored correctly — navigation moved to the right side, text aligned right, form fields flowed right-to-left, and icons that implied directionality (arrows, chevrons) were mirrored appropriately. Third, **date, time, and number formatting**: I verified that dates displayed in the correct regional format (DD/MM/YYYY for France, YYYY/MM/DD for Japan), that currency symbols and decimal separators were locale-appropriate, and that large numbers used the correct thousands separator. Fourth, **string expansion**: translated strings are often 30–40% longer than English — I verified that the UI did not truncate, overflow, or break layout with expanded strings. For **l10n testing** (is the translation correct?), I worked with native-speaking reviewers for content accuracy. I defined a focused review scope — prioritising user-facing error messages, calls to action, onboarding flows, and legal text over static marketing copy. I created a structured review template for reviewers to flag mistranslations, culturally inappropriate content, and inconsistent terminology.

**Result:** i18n testing found six engineering defects: two fields stored Japanese characters as corrupted data (encoding issue), the Arabic layout had three icons that were not mirrored, and a date picker used MM/DD/YYYY format regardless of locale setting. l10n review found 14 translation issues — including a German error message that used informal "du" address form inconsistently with the formal "Sie" used elsewhere, and a Japanese product description that used gendered language inappropriate for the target audience. All were resolved before the locale launches.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Verify that a localised version displays the correct language and that dates and currency appear in the expected format. |
| **Mid** | Test string expansion, character encoding, and locale-specific formatting systematically across key user journeys. |
| **Senior** | Design a two-phase i18n/l10n strategy — engineering-level internationalisation testing and content-level localisation review — covering RTL layouts, DBCS encoding, cultural appropriateness, and a structured native-speaker review process. |

## Key Terms

| Term | Definition |
|------|------------|
| **Internationalisation (i18n)** | Building an application to support multiple languages and regions without code changes — the engineering foundation. |
| **Localisation (l10n)** | Adapting an application for a specific locale — translation, formatting, and cultural customisation. |
| **RTL (Right-to-Left)** | A text direction used in Arabic, Hebrew, and other languages — requires full UI mirroring. |
| **DBCS (Double-Byte Character Set)** | Character encoding for languages like Japanese and Chinese — requires specific database and input handling. |
| **String Expansion** | The increase in text length when translated — often 30–40% longer than English — which can break UI layouts. |
| **Locale** | A specific combination of language, region, and cultural conventions — e.g., French (France) vs French (Canada). |
| **Character Encoding** | The system used to represent text characters digitally — UTF-8 is the standard for international character support. |
