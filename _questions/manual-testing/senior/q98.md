## The Intent

The interviewer is assessing security awareness and systematic test design for access control. Role and permission testing is a common weakness in QA practice — many testers verify that the right options appear for the right roles but fail to verify that restricted options are genuinely inaccessible, not just invisible. Senior testers must close this gap.

---

## The STAR Script

**Situation:** I was testing a document management system with four user roles: Super Admin (full access), Editor (create, edit, publish), Reviewer (view and comment only), and Guest (view public documents only). A new bulk-delete feature had been introduced for Super Admin and Editor roles only.

**Task:** I needed to design a test strategy that verified correct behaviour for every role — including both what was permitted and what was explicitly blocked — without writing an unmanageable number of test cases.

**Action:** I built the strategy in three layers. First, **permission matrix construction**: I created a table with all feature actions as rows (view document, create document, edit document, publish document, bulk delete, access admin panel) and all roles as columns. Each cell was populated with the expected outcome: Permitted, Denied, or Not Applicable. This gave me the complete specification for what to test. Second, **positive permission testing**: for each Permitted cell, I verified the action was available, executable, and produced the correct outcome. For bulk delete as Super Admin — the action appeared in the UI, documents were selectable, deletion triggered a confirmation dialog, and confirmed deletion removed the documents from the system and the audit log recorded the action. Third, **negative permission testing — the critical layer**: for each Denied cell, I tested three access vectors, not just the UI. First, I confirmed the action was not visible or available in the UI for the restricted role. Second, I attempted the action via direct API call using the restricted role's authentication token — this is where permission enforcement must live, not just the UI. Third, I attempted URL manipulation — accessing admin panel URLs directly while logged in as a Reviewer. Fourth, I tested horizontal privilege escalation — could an Editor bulk-delete another Editor's documents? Could a Reviewer access documents in a project they had not been invited to?

**Result:** UI-level permission testing all passed — the correct actions were shown and hidden appropriately. API-level testing found a critical defect: the bulk delete endpoint did not validate the requesting user's role server-side. An authenticated Reviewer could send a direct API call with valid document IDs and the deletion would execute successfully — despite the UI hiding the option. This was a serious privilege escalation vulnerability invisible to UI-only testing. The URL manipulation test also found that a Guest user could access a shared document by modifying the document ID in the URL, bypassing the project invitation requirement.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Verify that the correct options are visible and functional for each role in the UI. |
| **Mid** | Build a permission matrix, test both positive and negative cases for each role, and include one or two direct access attempts beyond the UI. |
| **Senior** | Design a three-layer permission test strategy — UI, API, and URL/direct access — covering horizontal and vertical privilege escalation, with a permission matrix as the governing specification. |

---

## Key Terms

| Term | Definition |
|------|------------|
| **Permission Matrix** | A table mapping feature actions to user roles with expected outcomes — the governing specification for role-based testing. |
| **Positive Permission Testing** | Verifying that permitted actions are available, functional, and produce correct outcomes for each role. |
| **Negative Permission Testing** | Verifying that restricted actions are genuinely blocked — at the UI, API, and direct access levels. |
| **Vertical Privilege Escalation** | A vulnerability where a lower-privileged role gains access to actions reserved for a higher-privileged role. |
| **Horizontal Privilege Escalation** | A vulnerability where a user accesses resources belonging to another user of the same role. |
| **Server-Side Enforcement** | Permission validation performed at the API or backend level — the only reliable enforcement point, since UI-only enforcement can be bypassed. |
