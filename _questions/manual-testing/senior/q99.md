## The Intent

The interviewer is assessing the ability to think about testing in a high-stakes, irreversible scenario. Migration testing is one of the highest-risk activities in software delivery — data loss or corruption during a migration can be catastrophic and sometimes unrecoverable. Senior testers must be able to design a comprehensive, phased approach that catches problems before they affect real user data.

---

## The STAR Script

**Situation:** The company migrated 12,000 customer accounts from a legacy CRM to a new platform. The migration involved transferring customer profile data, historical order records (spanning five years), active subscription states, and user preference settings. The legacy system used a completely different data model, requiring complex field mapping and transformation logic.

**Task:** I needed to design a migration testing strategy that gave high confidence in data integrity, workflow continuity, and rollback safety — across all three migration phases.

**Action:** I structured the strategy across three phases. **Pre-migration testing**: First, I defined a data mapping specification — a document confirming exactly how every field in the legacy system mapped to the new system, including transformation rules for mismatched data types and deprecated fields. I used this as the test specification. Second, I created a golden record set of 150 representative accounts — covering standard customers, edge cases (accounts with zero orders, accounts with maximum order history, accounts with special characters in names, accounts with lapsed subscriptions, and accounts with multiple active addresses). I ran the migration script against a copy of production data in a staging environment and verified every golden record against the mapping spec — field by field. Third, I validated the rollback procedure: I confirmed the rollback script restored the legacy system to its pre-migration state without data loss, and that the rollback could complete within the defined RTO. **During migration testing**: I monitored the migration in real time — tracking progress, error logs, and any records flagged as failed or skipped by the migration script. I had a pre-agreed escalation threshold: if more than 0.5% of records failed, migration would be paused for investigation. **Post-migration testing**: I ran a targeted smoke test covering the five most critical user workflows in the new system for a sample of migrated accounts. I also verified that historical order data was accessible and accurate, that subscription states were correctly reflected, and that user preference settings had been preserved. I specifically tested the accounts I knew were edge cases from my golden record set.

**Result:** Pre-migration testing found three data mapping defects: subscription expiry dates were being migrated without timezone conversion (causing subscriptions to appear expired for users in GMT+1 and later zones), special characters in company names were being truncated at 50 characters, and accounts flagged as "dormant" in the legacy system were being migrated as "active" in the new system. All three were fixed before production migration. The production migration completed with a 0.08% failure rate — 10 records that were manually resolved post-migration. Post-migration smoke testing confirmed no additional issues. Zero user-reported data issues in the 30 days following migration.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Verify that specific data fields appear correctly in the new system after a simple data transfer. |
| **Mid** | Design pre and post-migration test cases for a moderate data migration, including golden records and rollback verification. |
| **Senior** | Design a three-phase migration testing strategy — pre-migration validation, real-time monitoring, and post-migration verification — with golden records, a data mapping specification, escalation thresholds, and rollback testing. |

## Key Terms

| Term | Definition |
|------|------------|
| **Migration Testing** | A three-phase testing approach covering pre-migration validation, during-migration monitoring, and post-migration verification. |
| **Data Mapping Specification** | A document defining how every field in the source system maps to the target system — the governing specification for migration testing. |
| **Golden Record** | A controlled account or data record with known values used to verify migration correctness field by field. |
| **Rollback Testing** | Verifying that the migration can be safely reversed without data loss — essential for risk management. |
| **RTO (Recovery Time Objective)** | The maximum acceptable time to restore the system to a working state — defines the rollback completion target. |
| **Escalation Threshold** | A predefined failure rate at which migration is paused for investigation — prevents a small error from scaling to all records. |
