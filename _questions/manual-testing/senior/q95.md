## The Intent

The interviewer is assessing the ability to think beyond UI testing into data flow and system integration. Senior testers working on enterprise products or data-critical systems must be able to design data integrity test strategies — verifying correctness at the data layer, not just the presentation layer. This is a technically demanding question that distinguishes testers with deep systems thinking from those who test only what they can see on screen.

---

## The STAR Script

**Situation:** I was testing a financial reconciliation workflow that spanned four systems: a CRM (where orders originated), an ERP (which processed billing), a data warehouse (which aggregated transaction records), and a reporting dashboard (where finance teams reviewed summaries). Data flowed through all four systems over a 24-hour batch processing cycle.

**Task:** I needed to design a data integrity test strategy that verified correctness at every stage of the workflow — not just that the dashboard showed numbers, but that the numbers were correct all the way back to their source.

**Action:** I structured the strategy around five principles. First, **define the data contract at each boundary**: before testing anything, I mapped what data should exist at each system boundary — field names, data types, value ranges, and transformation rules. For example, order values in the CRM were stored in pence (integer), transformed to decimal pounds in the ERP, and aggregated as rounded figures in the warehouse. Each transformation had a defined rule that I could assert against. Second, **golden record testing**: I created a set of 20 test orders with known, controlled values — covering standard transactions, edge cases (zero-value orders, maximum-value orders, orders with discounts), and boundary conditions. I tracked these golden records through the full workflow, asserting the exact expected value at each system stage. Third, **database-level verification**: I did not rely solely on UI presentation. For each stage, I queried the underlying database or API directly to confirm the stored value matched the expected transformed value — independently of what the UI displayed. Fourth, **error and exception path testing**: I tested what happened when data failed at a boundary — a malformed order from the CRM, a failed ERP processing event, a warehouse load failure. Did the system fail safely? Was the failure logged? Was data rolled back or left in a partial state? Fifth, **reconciliation verification**: I verified that summary totals at the reporting layer matched the sum of individual records at the source layer — testing for aggregation errors, rounding inconsistencies, and records that had been processed twice or not at all.

**Result:** Golden record testing found that orders containing a specific discount code were being truncated by one penny at the ERP-to-warehouse boundary due to a rounding difference between integer and float arithmetic. This was invisible at the reporting layer for individual transactions but created a cumulative discrepancy of £340 across the month's test data. Exception path testing found that a failed ERP processing event left an order in a "processing" state permanently — it was never retried or flagged for manual review. Both were data integrity defects with direct financial impact that no UI-level test would have detected.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Verify that data entered in one screen appears correctly on another screen in the same application. |
| **Mid** | Trace data through a two or three-system integration, verify transformations at each boundary, and test basic error paths. |
| **Senior** | Design a full data integrity strategy for a complex multi-system workflow — data contracts, golden records, database-level verification, exception path testing, and reconciliation verification. |

---

## Key Terms

| Term | Definition |
|------|------------|
| **Data Integrity** | The accuracy, completeness, consistency, and reliability of data throughout its lifecycle. |
| **Data Contract** | A defined specification of what data should exist at each system boundary — field names, types, ranges, and transformation rules. |
| **Golden Record** | A controlled test data record with known values used to trace and verify data through a multi-system workflow. |
| **Reconciliation Testing** | Verifying that summary totals match the sum of individual source records — detects aggregation errors and data loss. |
| **Database-Level Verification** | Querying the underlying data store directly to confirm stored values — independent of what the UI displays. |
| **Boundary Transformation** | A data conversion or calculation that occurs when data moves between two systems — a high-risk point for integrity failures. |
