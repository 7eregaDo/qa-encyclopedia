## The Intent

The interviewer is assessing the ability to manage a high-complexity, high-risk testing scenario that many mid-level testers have not encountered. A UX redesign touches everything visible in the product simultaneously, invalidates large portions of an existing test suite, and introduces usability risk that functional testing alone cannot address. Senior testers must be able to design a strategy that covers all three dimensions without being overwhelmed by the scope.

---

## The STAR Script

**Situation:** The product team undertook a complete redesign of the main application interface — a navigation restructure, a new design system, and a revised information architecture across 11 core screens. The backend logic was unchanged, but virtually every UI element had moved, been renamed, or been redesigned. The release was planned for eight weeks from design sign-off.

**Task:** I needed to design a testing strategy that preserved full functional regression coverage despite the UI changes, assessed usability of the new design, and was executable within the eight-week window.

**Action:** I structured the strategy around three parallel workstreams. First, **functional regression under the new UI**: I recognised that most of our existing test cases were still functionally valid — the same actions needed to produce the same outcomes — but the steps to get there had changed. I triaged the existing test suite into three groups: cases where only navigation steps had changed (update the steps, keep the assertion), cases where feature interaction had changed significantly (rewrite from scratch), and cases where the feature itself was unaffected by the redesign (execute as-is). This triage took three days and preserved 68% of existing cases with minor updates, avoiding a full rewrite. I also identified and flagged 34 test cases as permanently invalidated by the redesign. Second, **new interaction testing**: for every new UI pattern introduced by the design system — new modal types, a new navigation paradigm, new form components — I wrote net-new test cases covering the interaction behaviour, keyboard accessibility, and responsive behaviour at mobile and desktop breakpoints. Third, **usability evaluation of the redesign**: I conducted structured heuristic evaluation sessions using Nielsen's 10 usability heuristics on the five highest-traffic screens. I also facilitated three informal usability sessions with internal users unfamiliar with the redesign, observing where they hesitated or made errors. Usability findings were classified as blockers (would prevent task completion), significant (would cause frequent confusion), or minor (polish issues).

**Result:** Functional regression found four defects where backend behaviour had inadvertently changed during the frontend refactor — all caught before release. Usability evaluation found two blocker-level issues: the primary action button on the most-used screen was positioned in a location that all three usability session participants initially missed, and a new navigation pattern for nested settings was consistently misunderstood. Both were redesigned before release. The release went out on schedule with no post-launch user complaints about broken workflows — the most common concern with major redesigns.

## Seniority Shift

| Level | Expected Depth |
|-------|---------------|
| **Entry** | Update existing test cases when UI elements change and verify that the same outcomes are still achievable. |
| **Mid** | Triage an existing test suite for a redesign, update affected cases, and conduct heuristic usability evaluation on key screens. |
| **Senior** | Design a parallel three-workstream strategy — functional regression triage, new interaction testing, and structured usability evaluation — managing scope and capacity across an eight-week delivery window. |

## Key Terms

| Term | Definition |
|------|------------|
| **UX Redesign** | A significant change to the user interface, navigation, or interaction patterns of an existing product. |
| **Test Case Triage** | Classifying existing test cases by their validity under a new UI — update, rewrite, or retire. |
| **Functional Regression** | Re-executing tests to confirm existing backend behaviour is preserved despite frontend changes. |
| **Heuristic Evaluation** | A usability review technique where testers assess the interface against established usability principles. |
| **Nielsen's 10 Heuristics** | A widely used set of usability principles — including visibility of system status, consistency, and error prevention. |
| **Usability Session** | An informal session where real or representative users attempt tasks while a tester observes — revealing real interaction friction. |
