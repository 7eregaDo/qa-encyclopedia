## The Intent
The interviewer is testing whether you can build and run security at organisational scale — not just execute tests or coach a team, but design the programme architecture, secure executive sponsorship, and drive adoption across engineering teams with competing priorities. This is a leadership and change management challenge as much as a security one.

## The STAR Script
**Situation:** I was appointed Director of Application Security Engineering at a company with 200 engineers and 45 microservices. Security was owned entirely by a two-person security team focused on perimeter and infrastructure. Application security testing did not exist as a practice — no SAST, no DAST, no security requirements in stories, no penetration testing programme. The CISO had identified application security as the highest-risk gap and asked me to build the programme.

**Task:** Design, staff, and drive adoption of an application security testing programme across the full engineering organisation within 12 months.

**Action:** I structured the programme build across four workstreams. Foundations (months 1–3): I conducted a rapid risk assessment — surveying all 45 services to identify the highest-risk components (payments, authentication, PII data stores, external APIs). I hired two application security engineers with penetration testing and SAST tooling specialisations. I defined three programme tiers: automated baseline (all services), enhanced manual testing (high-risk services), and specialist penetration testing (critical services annually). Standards and tooling (months 3–6): I defined the application security standard — the minimum security requirements every service must meet. I published it as a versioned document and embedded it as testable criteria in the engineering definition of done. I integrated Semgrep SAST and OWASP ZAP DAST into the shared CI/CD pipeline library, so all 45 services inherited automated scanning with one PR to the shared library. Enablement (months 4–9): I ran a quarterly "AppSec Clinic" — a two-hour workshop for developers covering real findings from our own codebase as case studies. I embedded one application security engineer with each of the four highest-risk squads for one sprint to accelerate adoption and identify squad-specific risk patterns. Governance (ongoing): I established a monthly application security review — a 30-minute meeting with the CISO and VP Engineering reviewing the programme dashboard: coverage, open finding count by severity, mean time to remediate, and pen test status.

**Result:** At the 12-month mark: automated SAST and DAST coverage across all 45 services, 94 security findings remediated (23 High or Critical), three external penetration tests completed, and zero security-related production incidents in the second half of the year (versus four in the first half). The CISO presented the programme to the board as evidence of material security maturity improvement.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Applies security tests to assigned features |
| **Mid** | Integrates security testing into the team's delivery process |
| **Senior** | Designs the security testing strategy for a product area; builds team security capability |
| **Lead** | Designs and staffs the org-wide programme; defines standards and governance; secures executive sponsorship; drives adoption across all engineering teams; reports to CISO and board |

## Key Terms
| Term | Definition |
|------|------------|
| **Application Security Programme** | An organisational capability that systematically identifies, tracks, and remediates application security vulnerabilities through defined standards, tooling, and governance |
| **Programme Tier** | A classification of services by security risk profile, determining the depth of security testing applied — from automated baseline to specialist penetration testing |
| **Application Security Standard** | A versioned document defining the minimum security requirements every service must meet, embedded in the engineering definition of done |
| **AppSec Clinic** | A regular developer education session using real application findings as case studies, building security awareness through relevance rather than generic training |
| **Mean Time to Remediate** | The average time from a security finding being raised to its confirmed fix, used as a programme effectiveness metric |
| **Coverage (Security)** | The proportion of services or features that have received security testing at the required depth for their risk tier |
| **Security Maturity** | The degree to which an organisation's security practices are systematic, measurable, and embedded — used in board-level reporting and due diligence |
| **Board-Level Security Reporting** | Executive-facing communication of the security programme's status, framed in business risk and maturity terms for non-technical decision-makers |
