## The Intent
The interviewer is testing the same autonomy-versus-consistency tension from CI/CD governance, applied to security. Heavy-handed security governance creates bottlenecks and adversarial relationships with engineering teams; no governance produces inconsistent security outcomes. A lead must have a model that achieves organisational security coherence without becoming the single point of failure.

## The STAR Script
**Situation:** As Head of Application Security, I oversaw security across 25 product squads. Security reviews had historically been a centralised function — squads submitted features for security review to my team. With 25 squads and a two-person application security team, the queue was perpetually 3–4 weeks long. Squads were shipping without security review, and my team was overwhelmed.

**Task:** Redesign the security governance model to maintain security standards across all 25 squads without requiring central review of every feature.

**Action:** I implemented a three-layer governance model. Layer 1 — Automated enforcement (no human required): I embedded security gates directly in the CI pipeline — SAST, DAST, dependency scanning, and IaC security checks — with severity-based failure thresholds. Automated gates handled approximately 60% of the security coverage I had previously provided manually. Squads got instant feedback without queuing. Layer 2 — Self-service security (squad-owned): I published a security playbook for the seven most common feature types (authentication, file upload, payment integration, admin console, public API, third-party integration, data export). Each playbook contained a threat model, the applicable OWASP checklist, and example test cases. Squads used the playbook to self-assess their features and document the security test evidence in the story. My team reviewed the evidence asynchronously — a 20-minute review versus a 2-hour active assessment. Layer 3 — Risk-based central review (reserved for highest risk): I defined a short list of triggers requiring my team's active involvement — new authentication mechanisms, changes to cryptographic implementation, first-time third-party integrations processing PII, and any feature identified as critical in the company threat model. This reduced central review requests from approximately 80 per month to 12, all of which received dedicated attention. Accountability: squad security metrics — open critical/high finding age and SAST pass rate — were published in the monthly engineering dashboard, creating peer-visible accountability without penalty.

**Result:** Central review queue dropped from 3–4 weeks to 3 days for the 12 monthly critical reviews. Self-service playbook adoption reached 78% within two quarters. Automated gate coverage was consistent across all 25 squads. Squads reported in the engineering survey that security was now a manageable part of delivery rather than an external constraint.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Follows security procedures; not involved in governance design |
| **Mid** | Applies security standards within the squad; contributes to security playbooks |
| **Senior** | Designs security review processes for a product area; trains squads on self-service security; defines risk-based review triggers |
| **Lead** | Designs the three-layer governance model; eliminates the central bottleneck; defines risk-based triggers; publishes squad-level security metrics; maintains security coherence at org scale without central gatekeeping |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Playbook** | A feature-type-specific guide containing a threat model, applicable security checklist, and example test cases, enabling squads to self-assess security without specialist involvement |
| **Risk-Based Review Trigger** | A defined criterion that escalates a feature to central security review — such as a new authentication mechanism or cryptographic change — distinguishing high-risk from standard features |
| **Self-Service Security** | A governance model in which squads conduct and document their own security assessments using standardised playbooks, reviewed asynchronously by the central security team |
| **Asynchronous Review** | A security evidence review conducted at the reviewer's convenience after the squad has completed their self-assessment, replacing synchronous scheduled reviews |
| **Security Accountability Metric** | A squad-level measure — such as open critical finding age — published in engineering dashboards to create visible ownership without formal penalty |
| **Security Gate** | An automated CI pipeline check that enforces security standards at the point of code change, providing coverage without human reviewer involvement |
| **Central Bottleneck** | The governance anti-pattern where all security decisions or reviews flow through a single team, creating a throughput constraint that scales inversely with engineering growth |
| **Security Coherence** | The property that all engineering teams meet a consistent security standard, even though individual squads make autonomous feature decisions |
