## The Intent
The interviewer is checking whether you see security as part of quality, not a separate discipline owned entirely by a security team. Entry-level QEs sometimes assume security testing is outside their remit. They want to hear that you understand the overlap, can identify basic security defects in your day-to-day testing, and know when to escalate.

## The STAR Script
**Situation:** During a sprint review early in my career, I flagged that an API endpoint was returning more user data than the UI displayed. The developer said "the UI hides those fields" and considered it resolved. I wasn't sure whether this was a functional defect or a security concern — or both.

**Task:** Understand the security implication of what I'd observed and learn how to think about security as part of my regular QA work.

**Action:** I researched the pattern and discovered it was a well-documented security vulnerability called excessive data exposure — where the server returns sensitive fields and relies on the client to filter them, rather than protecting them server-side. An attacker intercepting the response or calling the API directly would see all the fields, not just what the UI showed. I raised this formally as a security defect rather than a UI inconsistency, documented it with the OWASP reference, and escalated it to the security team for severity assessment. Through this I learned three things I apply to every testing engagement now. First, I always check response payloads against what the spec says should be returned — not just what the UI displays. Second, I treat any response that contains fields marked as sensitive in the data dictionary as a potential over-exposure. Third, I know which security concerns I can assess myself (excessive data exposure, missing auth headers, information leakage in error messages) and which require escalation (injection vulnerabilities, cryptographic weaknesses, infrastructure-level issues).

**Result:** The defect was confirmed as a high-severity finding and fixed before release. I became the team's informal first responder for API response security checks. The clear escalation path I established — assess, document, escalate — was adopted by the wider QE team.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Identifies common surface-level security issues (excessive data exposure, missing auth, information leakage in errors); knows when to escalate; treats security defects with the same rigour as functional defects |
| **Mid** | Applies the OWASP Top 10 and OWASP API Security Top 10 systematically; performs basic authentication and authorisation testing; integrates security checks into test plans |
| **Senior** | Designs security testing strategy; leads threat modelling; integrates DAST tooling; performs manual penetration testing of application-layer vulnerabilities |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Testing** | The practice of verifying that a system resists unauthorised access, data exposure, and manipulation — treating security vulnerabilities as a category of defect |
| **Excessive Data Exposure** | A vulnerability where an API returns more data than the consumer is authorised to see, relying on client-side filtering rather than server-side access control |
| **OWASP** | The Open Web Application Security Project — a non-profit organisation that publishes widely used security risk frameworks including the Web Top 10 and API Security Top 10 |
| **Security Defect** | A vulnerability in an application that could be exploited by an attacker to gain unauthorised access, expose data, or disrupt service — treated with the same rigor as functional defects |
| **Information Leakage** | The unintended disclosure of sensitive information through error messages, response headers, or verbose API responses |
| **Escalation** | The process of raising a security concern to a specialist (security engineer or CISO) when it requires expertise beyond the QE's scope to assess or remediate |
| **OWASP API Security Top 10** | A ranked list of the most critical API security risks, used as a checklist for QEs performing API security testing |
| **Adversarial Thinking** | The practice of approaching a system from the perspective of an attacker — asking "how could this be misused?" rather than "does it work as intended?" |
