## The Intent
The interviewer wants to see whether you think about data protection as a testing concern. GDPR and similar regulations make data exposure a legal and financial risk. Entry-level QEs who can spot unexpected PII in API responses, unencrypted transmissions, or verbose error messages add significant security value without requiring deep security expertise.

## The STAR Script
**Situation:** I was assigned to test a new customer-facing order history feature that returned order details including delivery addresses and payment method information. Before functional testing, I decided to review the API responses for data exposure risks.

**Task:** Check the API responses for sensitive data that should not be returned to the client, and verify that transmission was appropriately secured.

**Action:** I tested across four areas. Response payload review: I called the `GET /orders/{id}` endpoint with a valid customer token and inspected every field in the response. I cross-referenced against the API specification and the data classification policy. The response included the full card number (masked in the UI as `****1234` but returned in full via the API as `4532123456781234`), the CVV stored at order time (a PCI-DSS violation), and the customer's date of birth — not needed for order display. Transmission security: I confirmed the endpoint was HTTPS-only by attempting HTTP — it correctly redirected. I used browser developer tools to confirm the certificate was valid and HSTS was present. Error message review: I sent invalid order IDs and inspected error responses for internal information — database table names, server paths, or stack traces. One 500 response contained a PostgreSQL error message with the table schema. Log exposure: I asked the development team whether any logging captured request bodies — in this case, payment details were being logged at DEBUG level in the application logs accessible to all engineers with server access.

**Result:** Three findings: full card number in API response (critical — PCI violation), CVV storage (critical — PCI violation), and payment details in debug logs (high). The date of birth in response was medium. All were raised with regulatory references and fixed before the feature launched.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Reviews API response payloads against data classification; checks for transmission security, error message leakage, and PII in obvious locations; references regulatory requirements |
| **Mid** | Tests data masking correctness, field-level encryption, data retention compliance, and cross-tenant data exposure; analyses log output for sensitive content |
| **Senior** | Designs data exposure testing strategy; performs data flow analysis to map sensitive data paths; tests encryption at rest; integrates with data classification frameworks |

## Key Terms
| Term | Definition |
|------|------------|
| **Sensitive Data Exposure** | The unintended disclosure of protected information — PII, credentials, financial data — through API responses, error messages, logs, or unencrypted transmission |
| **PII (Personally Identifiable Information)** | Data that can identify a specific individual, including name, address, date of birth, and financial details, subject to regulatory protection under GDPR and similar laws |
| **PCI-DSS** | The Payment Card Industry Data Security Standard, which prohibits storing CVV data and requires masking card numbers in responses |
| **Data Classification Policy** | An organisational framework that categorises data by sensitivity level (e.g. public, internal, confidential, restricted) and defines handling requirements for each level |
| **HSTS (HTTP Strict Transport Security)** | A response header ensuring browsers only communicate with the server over HTTPS, preventing protocol downgrade attacks |
| **Debug Log** | A verbose application log level capturing detailed operational information, which may inadvertently capture sensitive request payloads if not properly scoped |
| **CVV** | The Card Verification Value — a three or four digit security code on a payment card that PCI-DSS explicitly prohibits storing after transaction authorisation |
| **Response Payload Review** | The systematic inspection of every field in an API response against the data classification policy and API specification to identify unexpected sensitive data |
