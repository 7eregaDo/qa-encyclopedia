## The Intent
The interviewer wants to see whether you look beyond application code to the configuration layer, where many real-world vulnerabilities live. Security misconfigurations are often easy to test and fix but go undetected because they don't look like functional bugs. Entry-level QEs who check headers and error responses add meaningful security value.

## The STAR Script
**Situation:** I was preparing a pre-launch security checklist for a new web application and wanted to verify basic security configuration without access to the server infrastructure itself — using only what was visible from the browser and API responses.

**Task:** Check for common security misconfigurations using client-visible signals.

**Action:** I worked through a structured checklist. HTTP security headers: I sent a request to the application's main pages and inspected the response headers. I checked for: `Content-Security-Policy` (absent — a finding), `X-Content-Type-Options: nosniff` (present), `X-Frame-Options: DENY` (absent — a finding, leaving the site vulnerable to clickjacking), `Strict-Transport-Security` (present), and `Referrer-Policy` (absent). Server information disclosure: I checked whether the `Server` header revealed the web server type and version (it returned `nginx/1.18.0` — a finding, as version disclosure aids attackers in targeting known CVEs). Error messages: I sent requests designed to trigger errors — a 404 by requesting a non-existent path, and a 500 by sending a malformed request. The 404 returned the application's custom error page — good. The 500 returned a stack trace with internal file paths and the framework version. Directory listing: I requested `/static/`, `/uploads/`, and `/admin/` directly. The `/uploads/` path returned a directory listing of all uploaded files — a significant misconfiguration. Default paths: I checked whether `/admin`, `/phpmyadmin`, `/swagger-ui`, and `/.git` were accessible. The Swagger UI was accessible publicly with no authentication — exposing full API documentation.

**Result:** Five findings: missing CSP and X-Frame-Options headers (medium), nginx version disclosure (low), stack trace on 500 (high), directory listing on `/uploads/` (high), and unauthenticated Swagger UI (medium). All were raised with remediation steps and fixed before launch.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Checks HTTP security headers, server version disclosure, error verbosity, directory listing, and common default paths; uses browser developer tools and Postman |
| **Mid** | Checks TLS configuration (cipher suites, certificate validity), CORS policy, cookie security attributes, and cloud storage bucket permissions |
| **Senior** | Designs security configuration testing strategy; integrates automated header scanning into CI; performs infrastructure-level configuration review; defines security baseline configurations |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Misconfiguration** | An exploitable weakness arising from insecure default settings, unnecessary features, or missing security controls rather than code-level vulnerabilities |
| **Content-Security-Policy (CSP)** | An HTTP response header that restricts the sources from which the browser may load scripts, styles, and other resources, mitigating XSS attacks |
| **X-Frame-Options** | An HTTP response header that controls whether the page can be embedded in a frame, preventing clickjacking attacks |
| **Clickjacking** | An attack where a malicious page overlays a transparent frame containing the target site, tricking users into clicking on hidden controls |
| **Directory Listing** | A server misconfiguration that allows browsers to view the contents of a directory as a file list when no index file is present, potentially exposing sensitive files |
| **Version Disclosure** | The inclusion of software version information in response headers or error messages, enabling attackers to target known CVEs for that version |
| **Stack Trace** | A debug output in an error response showing the sequence of function calls that led to an exception — a security misconfiguration that exposes internal implementation details |
| **CORS (Cross-Origin Resource Sharing)** | A browser mechanism controlled by HTTP headers that determines which external origins are permitted to make API requests to the server |
