## The Intent
The interviewer is checking your communication and documentation discipline around security findings. A security finding that is vaguely described, unclassified, or missing reproduction steps creates investigation overhead and risks being deprioritised or misunderstood. They want to see that you treat security defects with the same rigour as functional defects — and handle the additional sensitivity appropriately.

## The STAR Script
**Situation:** During testing of an e-commerce platform, I discovered that the `GET /users/{id}/payment-methods` endpoint returned full payment card numbers for any authenticated user — not just the account holder. This was a BOLA vulnerability with significant financial and regulatory implications.

**Task:** Document and raise the finding in a way that clearly communicated the severity, reproduction path, and impact without inadvertently publicising sensitive details to the wrong audience.

**Action:** I documented the finding with the following structure. Title: clear and specific — "BOLA: Authenticated users can retrieve payment card numbers for any account via /users/{id}/payment-methods". Severity: High (using CVSS-informed reasoning — network-accessible, low attack complexity, no special privileges required beyond any authenticated user, high impact on confidentiality). Affected component: the endpoint, HTTP method, and environment. Description: a one-paragraph explanation of the vulnerability class (BOLA/IDOR), why it occurs here (no ownership check on the user ID parameter), and the regulatory implication (PCI-DSS — full card number exposure). Reproduction steps: step-by-step with exact requests — authenticate as User A, retrieve User A's user ID, authenticate as User B, call the endpoint with User A's ID using User B's token, observe the card number in the response. Evidence: screenshots and sanitised request/response logs with the card number partially redacted in the evidence but the finding described in full. Potential impact: any authenticated user can retrieve any other user's payment card numbers. References: OWASP API Security Top 10 API1:2023 — BOLA. I raised it directly to the security team and engineering lead via a private channel rather than the public defect board, given the sensitivity.

**Result:** The finding was triaged as P1 within two hours and fixed before the next release. The security team commented that the documentation quality enabled them to reproduce and confirm it without any follow-up questions — a 30-minute fix cycle rather than the usual multi-day back-and-forth.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Documents security defects with vulnerability type, reproduction steps, severity classification, impact statement, and OWASP reference; handles sensitive findings through appropriate channels |
| **Mid** | Applies CVSS scoring; maps findings to regulatory frameworks (GDPR, PCI-DSS); proposes remediation approaches in the defect report |
| **Senior** | Produces security finding reports suitable for CISO and executive audiences; prioritises findings by exploitability and business impact; tracks remediation SLA compliance |

## Key Terms
| Term | Definition |
|------|------------|
| **BOLA (Broken Object Level Authorisation)** | A vulnerability where an authenticated user can access another user's resources by manipulating the object identifier in a request |
| **CVSS (Common Vulnerability Scoring System)** | A standardised framework for rating the severity of security vulnerabilities based on exploitability and impact metrics |
| **Severity Classification** | The assignment of a priority level to a security finding based on the ease of exploitation and potential impact — typically Critical, High, Medium, Low |
| **Reproduction Steps** | A numbered sequence of actions that reliably reproduces a vulnerability, enabling the receiving engineer to confirm and fix it without investigation overhead |
| **Sensitive Finding Handling** | The practice of raising security defects through private or restricted channels rather than public issue trackers, preventing premature disclosure of exploitable vulnerabilities |
| **Regulatory Implication** | The compliance consequence of a security finding — such as a PCI-DSS or GDPR violation — which elevates its business priority beyond its technical severity alone |
| **OWASP Reference** | The citation of the relevant OWASP Top 10 or API Security Top 10 category in a security defect report, providing context and remediation guidance |
| **Remediation** | The fix or mitigation applied to resolve a security vulnerability, which a well-documented defect report can propose or guide |
