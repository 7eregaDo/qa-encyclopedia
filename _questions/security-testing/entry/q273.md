## The Intent
The interviewer is checking whether you understand that client-side validation is cosmetic from a security perspective — it improves UX but stops nothing. Security-aware QEs test at the server level, sending crafted inputs directly via API or by bypassing the UI. They want to see that you know the common attack payloads and what good vs bad responses look like.

## The STAR Script
**Situation:** I was testing a content management system with a rich text editor that allowed users to format posts. The UI had a character limit and stripped HTML tags in the preview. I wanted to verify whether the server enforced the same rules.

**Task:** Test the underlying API directly for input validation vulnerabilities, bypassing the UI controls.

**Action:** I sent requests directly to the API using Postman, bypassing the browser entirely. I tested three vulnerability classes. Cross-site scripting (XSS): I submitted post content containing a basic XSS payload — `<script>alert('xss')</script>` — and also a more evasive variant using an image tag with an onerror handler. I then retrieved the stored content via the API and checked whether the script tags were stored verbatim (stored XSS) or escaped. The first payload was escaped correctly; the image tag variant was stored unescaped — a stored XSS vulnerability. SQL injection: I submitted a single quote character (`'`) and a classic injection string (`' OR '1'='1`) in the post title field. I checked whether the response contained a database error (indicating unsanitised input reaching the query) vs a generic validation error. Path traversal: the API had an endpoint for uploading attachments referenced by filename. I submitted filenames containing `../` sequences such as `../../etc/passwd` and checked whether the response revealed any file system information. I documented each test case with the payload, the expected result ("input rejected or safely escaped"), the actual result, and the severity classification.

**Result:** The stored XSS via the image tag was confirmed and raised as a high-severity defect. The SQL injection and path traversal tests passed — the server was using parameterised queries and the attachment handler normalised paths correctly. The XSS finding was fixed by adding the onerror-specific variant to the server-side sanitisation ruleset.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests common input validation payloads (XSS, SQLi, path traversal) by sending them directly via API; distinguishes safe from unsafe server responses; documents findings with payloads and severity |
| **Mid** | Tests systematically across all user-controlled input fields; uses automated scanning for coverage; tests second-order injection scenarios; understands output encoding contexts |
| **Senior** | Designs input validation test strategy; performs manual testing for complex injection vectors; reviews sanitisation library configuration; integrates SAST and DAST for comprehensive coverage |

## Key Terms
| Term | Definition |
|------|------------|
| **Input Validation** | The process of verifying that user-supplied data conforms to expected format, type, and content before processing — a primary defence against injection attacks |
| **Cross-Site Scripting (XSS)** | A vulnerability where an attacker injects malicious scripts into content served to other users, executing in the victim's browser context |
| **Stored XSS** | A variant of XSS where the malicious payload is persisted in the server's data store and served to all subsequent users who view the affected content |
| **SQL Injection** | A vulnerability where untrusted input is incorporated into a database query, potentially allowing an attacker to read, modify, or delete data |
| **Path Traversal** | A vulnerability where user-controlled file path inputs containing `../` sequences allow access to files outside the intended directory |
| **Parameterised Query** | A SQL query construction technique that separates query structure from data, preventing SQL injection by treating user input as a literal value rather than executable SQL |
| **Output Encoding** | The transformation of special characters in user-controlled data before rendering in HTML, preventing the browser from interpreting them as executable markup |
| **Client-Side Validation** | Input validation implemented in the browser or client application, which can be bypassed by sending requests directly to the API and provides no security guarantee |
