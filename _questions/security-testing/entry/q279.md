## The Intent
The interviewer wants to see whether you can apply security thinking to one of the most common and highest-risk features in any application. Login and registration are the entry points to everything — if they fail, every downstream security control is undermined. They want concrete test cases, not general awareness.

## The STAR Script
**Situation:** I was assigned to test a new user registration and login flow for a healthcare application. The feature had passed functional QA. I was asked to perform a security-focused review before sign-off.

**Task:** Design and execute security test cases for the registration and login flows.

**Action:** I covered eight security areas across both flows. Registration: password policy enforcement — I tested passwords below the minimum length, without special characters, and matching the username; all should be rejected. I confirmed that the error message named the specific policy rule violated without revealing which part was correct (avoiding partial enumeration). Duplicate registration — I registered with an email that already existed and confirmed the error message did not confirm whether the email was registered, using a timing-neutral response. Password storage — I asked the developer to confirm bcrypt or Argon2 was used (not MD5 or SHA-1); I could not verify this directly but raised it as a review point. Login: account enumeration — error messages for unknown email vs wrong password were identical — a pass. Brute force protection — I sent 20 consecutive failed login attempts; the account locked after 10 — I verified the lockout notification was sent to the registered email. Secure password reset — I initiated a password reset, inspected the token in the email for length and randomness, used the token, then attempted to reuse it — it was correctly invalidated. HTTPS enforcement — I verified the login form was served over HTTPS and that submitting the form to HTTP was blocked. Post-login session — I verified the session token was regenerated after login (new value post-authentication) and that logout invalidated it server-side.

**Result:** One finding: the password reset token was a six-digit numeric code with a 24-hour expiry — insufficiently random and long-lived. I recommended at minimum a 32-character alphanumeric token with a 1-hour expiry. The token generation was updated before launch.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests account enumeration, brute force protection, password policy, reset token strength, HTTPS enforcement, and session regeneration on login |
| **Mid** | Tests multi-factor authentication flows, social login (OAuth), password change without re-authentication, and concurrent session limits |
| **Senior** | Designs the full authentication security test strategy; tests for account takeover chains; reviews cryptographic implementation; performs credential stuffing simulation |

## Key Terms
| Term | Definition |
|------|------------|
| **Brute Force Protection** | A control limiting repeated authentication attempts — via account lockout, CAPTCHA, or progressive delays — to prevent credential guessing attacks |
| **Account Enumeration** | The ability to determine whether a username or email is registered by observing different responses for valid vs invalid accounts |
| **Password Reset Token** | A temporary, single-use secret sent to a user's verified contact to authorise a password change — must be sufficiently random and short-lived |
| **Session Regeneration** | The server-side replacement of the session identifier immediately after login, preventing session fixation attacks |
| **Password Policy Enforcement** | Server-side validation of password complexity, length, and blacklist requirements — which must be enforced on the server even if also checked client-side |
| **Credential Stuffing** | An automated attack using lists of breached username/password pairs to attempt login, effective against sites that do not enforce brute force protection |
| **Timing Attack** | An attack that infers information from the time taken to process a request — for example, a longer response time for a valid username vs an invalid one reveals account existence |
| **Argon2/bcrypt** | Cryptographic hashing algorithms specifically designed for password storage, incorporating computational cost and salt to resist brute force attacks on stored credentials |
