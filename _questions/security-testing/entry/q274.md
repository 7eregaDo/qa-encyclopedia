## The Intent
The interviewer wants to know whether you test authentication as a security concern, not just a functional one. Functional testing confirms that valid credentials grant access; security testing asks what happens with invalid credentials, manipulated tokens, and repeated failures. They want to see that you apply adversarial thinking to authentication flows.

## The STAR Script
**Situation:** I was testing the authentication module of a healthcare portal. The functional tests confirmed that login worked with valid credentials. I was asked to perform additional security-focused testing on the same feature.

**Task:** Test the authentication flow for common security vulnerabilities beyond the happy path.

**Action:** I structured tests around five security concerns. Session token properties: after a successful login, I inspected the session cookie — I verified it had the `HttpOnly` flag (preventing JavaScript access), the `Secure` flag (HTTPS only), and `SameSite=Strict` (CSRF protection). The cookie lacked the `SameSite` attribute entirely — a finding. Session fixation: I captured a pre-login session ID, logged in, then checked whether the session ID changed after authentication. If it remained the same, the server was vulnerable to session fixation. It did change — a pass. Logout invalidation: I logged in, captured the session cookie, logged out, then manually sent a request with the captured cookie. The server should reject it with a 401. It returned a 200 with full account data — the server was not invalidating server-side sessions on logout. Concurrent session control: the spec stated a maximum of one concurrent session. I logged in from two browsers simultaneously and confirmed the first session was invalidated — it was not; both remained active. Account enumeration: I tested whether the error message for a non-existent username differed from a valid username with a wrong password. They were identical — a pass.

**Result:** Two security findings: missing `SameSite` cookie attribute (medium severity) and session not invalidated on logout (high severity). Both were raised with reproduction steps and OWASP references and fixed before the portal went live.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests session cookie properties, logout invalidation, account enumeration, and basic account lockout; knows what each finding means and how to raise it |
| **Mid** | Tests JWT claims, token expiry and refresh flows, privilege escalation scenarios, and password policy enforcement; understands OAuth flows |
| **Senior** | Designs authentication security test strategy; tests complex OAuth and SAML flows; reviews token signing implementation; performs account takeover scenario testing |

## Key Terms
| Term | Definition |
|------|------------|
| **Session Token** | A unique identifier issued to a user after authentication, used to maintain their logged-in state across subsequent requests |
| **HttpOnly Flag** | A cookie attribute that prevents JavaScript from accessing the cookie value, protecting session tokens from XSS-based theft |
| **Secure Flag** | A cookie attribute that instructs the browser to only send the cookie over HTTPS connections, preventing transmission over unencrypted channels |
| **SameSite Attribute** | A cookie attribute controlling when the browser sends the cookie with cross-site requests, protecting against CSRF attacks |
| **Session Fixation** | A vulnerability where an attacker sets a known session ID before a victim logs in; if the server does not regenerate the session ID on login, the attacker can hijack the session |
| **Logout Invalidation** | The server-side destruction of a session token when a user logs out, ensuring the token cannot be reused after the session ends |
| **Account Enumeration** | The ability to determine whether a username exists in a system from different error responses for valid vs invalid usernames |
| **Concurrent Session Control** | A security policy limiting how many simultaneous sessions a user can have, enforced server-side, often used in healthcare and financial systems |
