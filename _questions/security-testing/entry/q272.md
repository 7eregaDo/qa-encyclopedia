## The Intent
The interviewer is checking whether you have a framework for security testing rather than testing ad-hoc. Knowing the OWASP Top 10 by name is table stakes at entry level; they want to see that you can describe at least a few categories in concrete terms and explain how they translate into actual test cases.

## The STAR Script
**Situation:** My team was preparing for a security review of a new user authentication flow. The senior QA asked me to contribute security test cases to the test plan, and I needed a structured way to think about what to cover.

**Task:** Use the OWASP Top 10 as a framework to identify security test cases relevant to an authentication flow.

**Action:** I reviewed the current OWASP Top 10 and identified the categories most applicable to the authentication feature. Broken Access Control (A01): I wrote tests verifying that a logged-in user could not access another user's account data by manipulating the user ID in the URL or request body. Cryptographic Failures (A02): I confirmed that passwords were not returned in API responses and that the password reset token was sufficiently random and expired after use. Injection (A03): I tested the login form and API with SQL injection payloads in the username and password fields, verifying they returned a generic error rather than a database error or unexpected behaviour. Security Misconfiguration (A05): I checked that the authentication endpoints did not return stack traces or framework version headers, and that HTTPS was enforced with HSTS headers present. Identification and Authentication Failures (A07): I tested for account enumeration (whether the error message differed for "user not found" vs "wrong password"), brute force protection (whether repeated failed logins triggered lockout), and whether session tokens were invalidated on logout. For each category I wrote concrete test cases with expected results and included them in the test plan. I noted which findings were within my ability to assess and which (e.g. cryptographic implementation quality) required a security engineer's review.

**Result:** Five test cases based on the OWASP framework were added to the test plan. Two findings were discovered: the login endpoint returned different error messages for unknown username vs wrong password (account enumeration risk), and there was no brute force lockout. Both were raised as security defects and fixed before the feature launched.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Knows the OWASP Top 10 categories; can map 3–5 categories to concrete test cases for a given feature; knows when a finding requires escalation |
| **Mid** | Applies the full OWASP Top 10 and API Security Top 10 systematically across a feature or service; writes security test cases as a standard part of test planning |
| **Senior** | Uses OWASP as one input to a broader threat model; designs security testing strategy from first principles; introduces DAST tooling alongside manual OWASP-based testing |

## Key Terms
| Term | Definition |
|------|------------|
| **OWASP Top 10** | A ranked list of the ten most critical web application security risk categories, updated periodically based on industry data and used as a standard security testing framework |
| **Broken Access Control (A01)** | The most common web vulnerability — failures in access control that allow users to act outside their intended permissions |
| **Injection (A03)** | Vulnerabilities where untrusted data is sent to an interpreter (SQL, OS, LDAP) as part of a command, potentially allowing attackers to manipulate queries or commands |
| **Security Misconfiguration (A05)** | Insecure default configurations, missing security headers, verbose error messages, and unnecessary features that create exploitable weaknesses |
| **Account Enumeration** | A vulnerability where different error messages for valid vs invalid usernames allow an attacker to determine which accounts exist in the system |
| **Brute Force Protection** | A security control that limits repeated login attempts to prevent attackers from systematically guessing credentials |
| **HSTS (HTTP Strict Transport Security)** | A response header that instructs browsers to only communicate with the server over HTTPS, preventing protocol downgrade attacks |
| **Session Token Invalidation** | The server-side destruction of a session token on logout, preventing token reuse after a user has signed out |
