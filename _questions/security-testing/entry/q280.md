## The Intent
The interviewer wants to see that security testing is not an afterthought or a separate phase for you — it's woven into your normal working practice. Entry-level QEs who wait for a dedicated security review before considering security concerns miss the window where issues are cheapest to fix. They want to see security integrated into your workflow, not bolted on.

## The STAR Script
**Situation:** I noticed that my team's test plans had no security section — security was implicitly delegated to the security team, who performed a review once per quarter. This meant most features went weeks without any security consideration, and findings from the quarterly review were expensive to fix.

**Task:** Introduce a lightweight security testing practice that I could apply to every feature without requiring specialist security skills or significant time overhead.

**Action:** I introduced three changes to my personal testing workflow that I later shared with the team. Security risk classification at planning: for each feature I was testing, I spent five minutes classifying it by risk level. Features touching authentication, user data, payments, or admin functions were "elevated risk" and got a full OWASP-based security test pass. Features with no user input or data access were "low risk" and got a header and error message check only. This proportionality kept the overhead manageable. Security checklist per risk level: I maintained a two-page checklist — a short version for low-risk features (5 items: security headers, error verbosity, HTTPS, no PII in logs, no sensitive data in response) and a full version for elevated-risk features (20 items covering OWASP Top 10 and API Security Top 10 categories relevant to the feature type). I worked through the appropriate checklist during my regular test execution pass. Security defects as first-class: I raised security findings in the same defect tracker as functional defects, with a "security" label and severity classification. This made them visible in sprint planning and prevented them from being informally noted but never fixed.

**Result:** Over three months, I identified 11 security findings across features I tested — none of which had been raised by developers or in functional testing. The two highest-severity findings were fixed within the sprint. The team adopted my risk classification and checklist approach, and the quarterly security review began seeing fewer findings because issues were being caught earlier.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Adds a security risk classification step to test planning; applies a proportionate OWASP-based checklist during test execution; raises security findings as first-class defects |
| **Mid** | Writes security acceptance criteria into user stories; participates in threat modelling sessions; integrates DAST tool results into regular test reporting |
| **Senior** | Designs the security testing integration for the team's delivery process; leads threat modelling; defines security acceptance criteria standards; mentors QEs on security testing techniques |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Risk Classification** | The assignment of a risk level to a feature based on its data sensitivity and attack surface, used to determine the appropriate depth of security testing |
| **Security Test Checklist** | A structured list of security verification items, scaled by risk level, applied during regular test execution to ensure consistent security coverage |
| **Proportionate Security Testing** | The practice of applying security testing effort commensurate with a feature's risk profile — more thorough for high-risk features, lighter-touch for low-risk ones |
| **Security Acceptance Criteria** | Explicit, testable security requirements included in a user story alongside functional acceptance criteria |
| **Threat Modelling** | A structured analysis of a system's attack surface, data flows, and trust boundaries to identify potential security threats before implementation |
| **DAST (Dynamic Application Security Testing)** | Automated security testing of a running application, simulating attacker inputs to discover vulnerabilities |
| **Security Label** | A defect tracker tag that identifies a finding as security-related, enabling filtering, tracking, and reporting of security-specific issues |
| **Shift Left Security** | The practice of introducing security testing and considerations earlier in the development lifecycle, reducing the cost and complexity of fixing findings |
