## The Intent
The interviewer is checking whether you understand a common web vulnerability that affects state-changing operations. CSRF is easy to test for manually and is often found in legacy applications or APIs that assume same-origin requests. They want to see that you know the attack scenario, the common mitigations, and how to verify them.

## The STAR Script
**Situation:** I was testing a banking application's fund transfer feature. The feature required authentication and the developer had confirmed it used session cookies. I wanted to verify whether it was protected against CSRF attacks.

**Task:** Test the fund transfer endpoint for CSRF vulnerability.

**Action:** I tested for CSRF across three mitigations. Anti-CSRF token presence: I inspected the fund transfer form in the browser developer tools and confirmed a hidden field called `_csrf_token` was present with a unique value. I then sent the POST request to the transfer endpoint via Postman — simulating a cross-origin request — deliberately omitting the `_csrf_token` field. The server rejected the request with a 403. I also tested with a previously used (replayed) token — the server rejected it, confirming tokens were single-use. SameSite cookie attribute: I inspected the session cookie and confirmed it had `SameSite=Strict`, meaning the browser would not include it in cross-site requests at all — a defence-in-depth control. Origin header validation: I sent a request to the transfer endpoint with a crafted `Origin: https://evil.example.com` header and verified the server rejected it. Conversely, sending with `Origin: https://app.bank.com` was accepted. I also checked the API documentation to confirm any pure-API endpoints (not form-based) used token-based authentication (bearer tokens) rather than cookies — bearer tokens are not automatically sent by browsers and are therefore not vulnerable to CSRF.

**Result:** All three mitigations were functioning correctly. The feature passed the CSRF test cases. I documented the test cases and results in the security section of the test plan and noted that the SameSite cookie and CSRF token provided defence in depth.

## Key Terms
| Term | Definition |
|------|------------|
| **CSRF (Cross-Site Request Forgery)** | An attack where a malicious site causes a victim's authenticated browser session to make unintended requests to another site, exploiting automatic cookie inclusion |
| **Anti-CSRF Token** | A unique, secret, per-session value embedded in forms and verified server-side, preventing forged cross-site requests that lack the correct token |
| **SameSite=Strict** | A cookie attribute that prevents the browser from sending the cookie with any cross-site request, providing a strong defence against CSRF |
| **Origin Header** | An HTTP request header set by the browser indicating the origin (scheme, host, port) of the request, used by servers to validate that requests originate from expected domains |
| **Single-Use Token** | An anti-CSRF token that is invalidated after a single use, preventing replay attacks where an attacker captures a valid token |
| **Bearer Token** | An API authentication token passed in the Authorization header, not automatically sent by browsers in cross-site requests — making bearer token APIs inherently CSRF-resistant |
| **Defence in Depth** | A security principle of applying multiple independent controls for the same threat, so that the failure of one control does not expose the vulnerability |
| **State-Changing Operation** | An API request that modifies server state — such as a fund transfer, password change, or account deletion — which are the relevant targets for CSRF attacks |
