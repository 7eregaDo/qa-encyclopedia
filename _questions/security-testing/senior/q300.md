## The Intent
The interviewer is testing whether you can lead cultural and capability change, not just perform security testing yourself. At senior level, the highest-leverage contribution is often raising the security awareness of an entire team rather than finding all the vulnerabilities yourself. They want to see a structured, empathetic approach to capability building.

## The STAR Script
**Situation:** I joined a startup as the first QE with any security experience. The seven-person engineering team was talented but had no security background — passwords were stored with SHA-1, API keys were in environment variables on the CI server, and there were no security tests anywhere. The team was responsive and willing to learn but didn't know where to start.

**Task:** Build the team's security testing capability from near zero to a functional, self-sustaining practice over 12 months without overwhelming them or creating a security-is-QE's-job mentality.

**Action:** I sequenced the capability build in four phases. Phase 1 — Awareness (months 1–2): I ran a 90-minute "Attacker's Perspective" session using real examples from their own application — I showed them how I had found the SHA-1 password hashing and what an attacker would do with it. I made it concrete and non-shaming. I also ran a short session on the OWASP Top 10 using examples from public breaches. The goal was not to make developers into security experts but to make security feel real and relevant. Phase 2 — Tooling (months 2–4): I added Semgrep SAST to the PR pipeline and configured it to produce clear, linked messages explaining why each finding was a security issue — not just "vulnerability on line 47." I introduced the Semgrep findings as a team in the next sprint retrospective, treating them as learning opportunities. I also introduced a security checklist for the most common feature types — two pages, focused on the patterns most relevant to their stack. Phase 3 — Process (months 4–8): I introduced security acceptance criteria into the definition of done for elevated-risk stories. I ran paired security testing sessions — sitting with developers to test features we'd both worked on, explaining what I was testing and why. Three developers began independently adding security test cases to their own PRs. Phase 4 — Ownership (months 8–12): I facilitated the team's first STRIDE threat modelling session, with me as facilitator rather than expert. The team generated most of the threats themselves. I transitioned the security checklist ownership to the senior developer, who updated it based on two new issues we had found.

**Result:** After 12 months, three developers were writing security test cases independently. The STRIDE session became a standard part of significant feature planning. The security checklist was maintained by the team, not by me. The contracted annual pen test found two medium findings — compared to twelve the year before my arrival.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Learns security testing techniques from senior colleagues; applies checklist items as directed |
| **Mid** | Mentors one or two colleagues on specific security testing techniques; shares findings as learning opportunities |
| **Senior** | Designs and executes a team-wide security capability building programme; sequences education, tooling, and process changes; builds self-sustaining ownership; measures capability improvement through independent engineer behaviour |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Capability Building** | The sustained programme of education, tooling, and process change that raises a team's security testing capability from dependent to self-sustaining |
| **Attacker's Perspective Session** | A team education activity that demonstrates security vulnerabilities using real examples from the team's own application, making the threat concrete and relevant |
| **Paired Security Testing** | A technique where a QE and a developer test a feature together, with the QE explaining their security thinking in real time — building capability through observation and dialogue |
| **Self-Sustaining Practice** | A security testing practice that the team continues without the initiating QE's ongoing involvement, because ownership and knowledge have been transferred |
| **Security-Is-QE's-Job Mentality** | The anti-pattern where developers assume security is the QE's responsibility, reducing their own security awareness and creating a bottleneck |
| **Sequenced Capability Build** | A phased approach to building team security capability, where each phase (awareness, tooling, process, ownership) builds on the previous, preventing overwhelm |
| **Semgrep Finding Message** | The explanatory text in a Semgrep alert that describes why the pattern is a security issue and how to remediate it — critical for building developer understanding rather than compliance |
| **Capability Multiplier** | The role of a senior QE who builds the security capability of others, achieving greater collective impact than they could through individual security testing alone |
