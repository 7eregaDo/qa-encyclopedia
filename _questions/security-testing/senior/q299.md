## The Intent
The interviewer wants to see whether you can operate at the intersection of security testing and regulatory compliance — a context where test results are not just quality signals but audit evidence. Senior QEs working in regulated environments need to understand which tests satisfy which controls, how to generate and retain audit-ready evidence, and how to communicate programme status to auditors.

## The STAR Script
**Situation:** I was the senior QE at a B2B SaaS company preparing for its first SOC 2 Type II audit. The audit would assess whether security controls had been operating continuously for the preceding 12 months. The security testing programme needed to be designed not just to find vulnerabilities but to produce evidence satisfying specific SOC 2 control criteria.

**Task:** Design the security testing programme to satisfy SOC 2 Type II requirements, produce audit-ready evidence, and support the company through the audit process.

**Action:** I mapped the SOC 2 trust service criteria to specific security testing activities. CC7.1 (system monitoring): I implemented automated DAST scanning after every production deployment, with timestamped scan reports stored in an immutable S3 bucket with versioning. This evidenced continuous monitoring. CC6.8 (prevention and detection of malicious software): I documented the malware scanning pipeline for file uploads, the dependency vulnerability scanning in CI, and the container image scanning — each with dated scan logs. CC6.3 (authorisation of changes): I documented that every production deployment required a passing CI pipeline including security scans, with the scan results stored against the deployment event in the release audit log. For manual penetration testing, I conducted quarterly internal tests and one annual external test with a third-party pen tester, producing formal reports with findings, severity ratings, and remediation evidence. I created an evidence folder structure in a shared drive organised by control — auditors could navigate directly to the evidence for each control without needing to search. I also maintained a finding tracker showing every security defect raised, its severity, when it was remediated, and the evidence of remediation. Before the audit, I ran a pre-audit mock review with the CISO, walking through the evidence for each control to identify gaps.

**Result:** The SOC 2 Type II audit passed with no exceptions on the security testing controls. The auditor specifically noted the quality of the evidence organisation. Two minor observations were raised (both on controls outside security testing). The security testing programme continued post-audit as an embedded practice rather than a pre-audit scramble.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Follows the testing procedures defined by others; produces test results as directed |
| **Mid** | Understands that security test results serve as audit evidence; maintains organised evidence for their testing activities |
| **Senior** | Maps compliance control requirements to specific security testing activities; designs the evidence production and retention model; manages the programme through the audit cycle; presents security testing evidence to auditors |

## Key Terms
| Term | Definition |
|------|------------|
| **SOC 2 Type II** | A compliance framework assessing the operating effectiveness of security controls over a defined period (typically 12 months), requiring evidence of continuous operation |
| **Trust Service Criteria** | The SOC 2 categories against which controls are assessed — Security, Availability, Processing Integrity, Confidentiality, and Privacy |
| **Control Mapping** | The process of linking compliance requirements to specific technical activities — such as mapping CC7.1 (monitoring) to DAST scan evidence |
| **Audit Evidence** | Documented proof that a security control was operating as designed during the audit period — scan reports, deployment logs, penetration test reports, and finding remediation records |
| **Immutable Evidence Store** | A storage system configured to prevent modification of previously written records, ensuring compliance evidence cannot be altered after the fact |
| **Finding Tracker** | A document or tool recording every security finding, its severity, remediation status, and evidence of fix — used as audit evidence for vulnerability management controls |
| **Pre-Audit Mock Review** | An internal walkthrough of audit evidence against each control, conducted before the external auditor's assessment to identify gaps |
| **Third-Party Penetration Test** | An annual or periodic security assessment conducted by an external specialist, providing independent evidence of security testing for compliance purposes |
