## The Intent
The interviewer wants to confirm you have genuine hands-on penetration testing experience, not just familiarity with theory and tools. Senior security QEs should be able to describe a real engagement: what they tested, how they identified vectors, which techniques they applied, what they found, and how findings were documented and remediated. They want specificity, not generality.

## The STAR Script
**Situation:** I was asked to perform an internal penetration test of a B2C mobile banking API before an external audit. The scope covered the authentication flow, account management, payment initiation, and the admin API. I had authenticated access as a test customer account and the OpenAPI specification.

**Task:** Perform a structured application-layer penetration test, documenting all findings with reproducible proof of concept and CVSS scores.

**Action:** I followed a five-phase methodology. Reconnaissance: I read the OpenAPI spec, mapped all endpoints by sensitivity (payments and admin rated highest), and identified the authentication mechanism (OAuth 2.0 with JWT). I noted that the JWT included a `role` claim and a `user_id` claim — both prime candidates for manipulation. Mapping: I proxied the mobile app through Burp Suite to capture all API calls, including undocumented endpoints the spec omitted. I found a `/internal/debug/users` endpoint not in the spec that returned full account data. Exploitation — authentication: I decoded the JWT using jwt.io and attempted to modify the `role` claim from `customer` to `admin`, re-signing with `alg: none` (the none algorithm attack). The API accepted it — a critical JWT validation failure. Exploitation — IDOR: I systematically substituted user IDs in account, payment history, and document endpoints using Burp Intruder. Payment history was IDOR-vulnerable — any user ID returned that user's transaction history. Exploitation — business logic: I tested the payment initiation endpoint for race conditions by firing 10 simultaneous payment requests for the same idempotency key from two different sessions — both processed, creating a duplicate payment. Chain: I documented a realistic attack chain: authenticate as customer → exploit JWT none algorithm → access admin API → enumerate all accounts → extract PII.

**Result:** Six findings including two criticals. The JWT `alg: none` vulnerability and the undocumented debug endpoint were P0s, patched within 48 hours. The external auditor, engaged three weeks later, found no new critical issues — a direct result of the internal test scope.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests known vulnerability patterns manually; uses Postman for basic security checks |
| **Mid** | Uses OWASP ZAP and Burp Suite in intercepting proxy mode; tests IDOR, injection, and authentication systematically; documents findings with reproduction steps |
| **Senior** | Conducts structured penetration tests with methodology phases; uses Burp Suite Pro for active scanning and intruder attacks; chains vulnerabilities; discovers undocumented endpoints; calculates CVSS scores; produces professional pentest reports |

## Key Terms
| Term | Definition |
|------|------------|
| **Penetration Testing** | An authorised, simulated attack on a system designed to identify security vulnerabilities before malicious actors do |
| **Burp Suite** | The industry-standard web application penetration testing tool providing an intercepting proxy, scanner, intruder, repeater, and extensibility platform |
| **JWT alg:none Attack** | An attack exploiting JWT implementations that accept the `none` algorithm, allowing an attacker to forge tokens without a valid signature |
| **Burp Intruder** | A Burp Suite tool that automates the sending of payloads into defined request positions, used for IDOR testing, brute force, and parameter fuzzing |
| **Attack Chain** | A sequence of individual vulnerabilities combined to achieve a higher-impact goal — such as escalating from customer access to full admin control |
| **Undocumented Endpoint** | An API path not present in the published specification, often overlooked in security reviews and access control audits |
| **CVSS Score** | A numerical score (0–10) from the Common Vulnerability Scoring System reflecting a vulnerability's severity based on exploitability and impact |
| **Proof of Concept (PoC)** | A reproducible demonstration of a vulnerability — typically a request and response showing the security failure — included in penetration test findings |
