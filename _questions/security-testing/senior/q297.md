## The Intent
The interviewer wants to see whether you can contribute to security at the code level, not just the testing layer. Senior QEs who can perform meaningful security code reviews multiply the team's security capability — catching issues before a commit enters the codebase, in the context where developers are most receptive to feedback. They want a structured approach, not just "I look for SQL injection."

## The STAR Script
**Situation:** My team had automated SAST (Semgrep) in the pipeline but was finding that it missed context-dependent security issues — particularly around access control logic, sensitive data handling, and security control removal. I was asked to establish a lightweight security code review process for high-risk PRs.

**Task:** Define and implement a security code review practice that complemented the automated SAST layer without adding unacceptable review overhead.

**Action:** I created a risk-based security code review process with three components. Trigger criteria: a PR triggered a security code review if it touched authentication or authorisation logic, user input processing, cryptographic operations, payment or PII handling, security configuration (CORS, CSP, cookie settings), or deleted or modified an existing security control. For changes outside these categories, SAST coverage was considered sufficient. Review checklist by change type: I built a focused review guide — for authorisation changes, I checked that the permission check was server-side (not just client-side), applied to every affected endpoint, and could not be bypassed by parameter manipulation. For input handling, I checked that all user-controlled values were validated, sanitised, and escaped in the correct output context. For dependency changes, I checked for known CVEs in new dependencies and reviewed the library's security track record. Review technique: rather than reading the diff linearly, I read the spec or PR description first to understand the intended behaviour, then asked "what would an attacker do with this change?" I commented directly in the PR with specific references — e.g. "This endpoint modifies user state but has no ownership check on line 47 — User B could call this with User A's ID." I distinguished between blocking issues (security vulnerabilities) and informational comments (security hygiene suggestions).

**Result:** Over 10 weeks, security code review on triggered PRs caught 14 security issues that SAST had not flagged — predominantly authorisation logic gaps and insecure direct object references. The average review added 20 minutes per triggered PR. The process was documented and adopted by two other senior engineers who were trained to apply it.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Participates in code reviews; raises obvious security concerns informally |
| **Mid** | Reviews code changes for common vulnerability patterns; references OWASP in review comments; distinguishes between blocking and informational security feedback |
| **Senior** | Defines the security code review process with risk-based triggers and change-type checklists; applies threat-informed reasoning to authorisation logic; trains others; integrates security review with automated SAST coverage |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Code Review** | A human examination of source code changes specifically looking for security vulnerabilities, complementing automated SAST with contextual reasoning |
| **Risk-Based Trigger** | A criterion that determines when a security code review is required, based on the risk profile of the code area being changed |
| **Ownership Check** | A server-side validation that the requesting user is the authorised owner of the resource being accessed or modified |
| **Output Context** | The rendering environment into which user-controlled data is inserted — HTML body, HTML attribute, JavaScript, URL — each requiring a different escaping technique to prevent XSS |
| **Security Control Removal** | A code change that deletes or modifies an existing security mechanism — such as removing a permission check or disabling CSRF protection — which should always trigger a security review |
| **Blocking Comment** | A code review comment identifying a security vulnerability that must be fixed before the PR is merged, as opposed to an informational suggestion |
| **Threat-Informed Reasoning** | The practice of reading code with an adversarial mindset — asking what an attacker could do with the change — rather than only verifying that it implements the intended functionality |
| **Parameter Manipulation** | The ability to modify request parameters to bypass security controls — such as changing a user ID or role value in a request body — which ownership checks and server-side validation must prevent |
