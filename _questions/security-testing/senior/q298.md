## The Intent
The interviewer wants to see whether your security thinking extends to internal trust boundaries — a frequently overlooked area. Many microservices architectures assume all internal traffic is trusted, which means a single compromised service becomes a pivot point for the entire system. Senior QEs who test internal trust models prevent the most serious post-breach escalation scenarios.

## The STAR Script
**Situation:** I was performing a security assessment of a microservices platform with twelve services communicating via REST over a service mesh. The platform had a mature API gateway securing public traffic, but the internal service-to-service communication had never been security tested. A threat model review suggested that a compromised internal service could call any other service without restriction.

**Task:** Design and execute a security test of the service-to-service communication model, focusing on authentication, authorisation, and input trust assumptions.

**Action:** I tested four internal security properties. Service-to-service authentication: I attempted to call internal service endpoints directly (bypassing the API gateway, using the internal cluster DNS) without any authentication header. Seven of twelve services returned data — they had no service-to-service authentication at all, relying entirely on network segmentation. I documented this as a critical finding: if any service was compromised (e.g. via SSRF or a vulnerable dependency), it could freely call all other services. Service account privilege: I reviewed the Kubernetes service accounts assigned to each service. Four services had no service account (defaulting to the `default` service account with access to the Kubernetes API) and two had `cluster-admin` role bindings — massively over-privileged. Mutual TLS verification: I checked whether the service mesh (Istio) was configured in strict mTLS mode, requiring both sides of every connection to present a valid certificate. It was in permissive mode — mTLS was optional rather than enforced. Internal input validation: I tested the hypothesis that internal services trusted data from other internal services without validation. The order service accepted a `user_id` field from the cart service without verifying the value — I sent a modified request substituting a different user ID and the order service created an order for that user. This was an internal trust assumption creating an IDOR vulnerability.

**Result:** Three critical findings: no inter-service authentication, Istio in permissive mode, and internal IDOR via trusted user_id. The architecture team implemented mutual TLS in strict mode and JWT-based service-to-service authentication. The internal IDOR was fixed by having the order service look up the user ID from the authenticated token rather than accepting it as a parameter.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests the public API security; may not be aware that internal services exist as an attack surface |
| **Mid** | Tests whether internal endpoints are accessible without authentication; checks service account permissions |
| **Senior** | Designs the internal service security test programme; tests mTLS enforcement, service account privilege, inter-service authentication, and internal input trust assumptions; assesses lateral movement potential |

## Key Terms
| Term | Definition |
|------|------------|
| **Service Mesh** | A dedicated infrastructure layer managing service-to-service communication in a microservices architecture, providing traffic management, observability, and security features |
| **Mutual TLS (mTLS)** | A TLS mode where both the client and server present and verify certificates, providing two-way authentication for service-to-service communication |
| **Istio** | A widely used open-source service mesh providing mTLS, traffic control, and observability for Kubernetes-based microservices |
| **Service Account** | A Kubernetes identity assigned to a pod, used to authenticate the pod to the Kubernetes API and to other services — should follow least privilege |
| **Lateral Movement** | A post-compromise technique where an attacker moves from one compromised service to other internal services, escalating access within the system |
| **Internal Trust Assumption** | The implicit assumption that traffic arriving from another internal service is trustworthy and does not require the same validation as external traffic — a dangerous pattern in microservices |
| **SSRF (Server-Side Request Forgery)** | A vulnerability where the application makes HTTP requests to attacker-controlled URLs — in a microservices context, exploitable to pivot to internal services |
| **Strict mTLS Mode** | An Istio configuration that requires all service-to-service traffic to use mutual TLS, rejecting plain-text connections — as opposed to permissive mode where mTLS is optional |
