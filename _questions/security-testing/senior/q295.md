## The Intent
The interviewer is testing whether you can assess cryptographic security without needing to be a cryptographer. Senior QEs should know which algorithms are approved, which are deprecated, what key management looks like in practice, and how to identify the most common cryptographic failures — most of which are configuration and implementation issues, not fundamental mathematics.

## The STAR Script
**Situation:** I was performing a security review of a healthcare platform that encrypted patient records at rest and used end-to-end encrypted messaging between clinicians. The development team had implemented the cryptography themselves rather than using a managed service. I was asked to assess the cryptographic implementation.

**Task:** Review the cryptographic implementation for common weaknesses without conducting a full cryptographic audit (which would require a specialist).

**Action:** I assessed six areas. Algorithm selection: I reviewed the code for encryption algorithm usage. Patient records were encrypted with AES-128-ECB — a finding. ECB mode is deterministic and pattern-revealing; AES-256-GCM was the correct choice providing both confidentiality and integrity. The messaging used RSA-1024 — deprecated below 2048 bits. IV/nonce reuse: the AES-CBC implementation for file encryption was reusing a fixed IV stored as a constant in the code — a critical finding. IV reuse with the same key completely undermines confidentiality. Key storage: I searched the codebase for hardcoded encryption keys using Semgrep rules for hex strings of 16, 24, or 32 bytes and base64-encoded strings in crypto contexts. Two hardcoded AES keys were found in configuration files committed to git. Key rotation: I asked the team whether a key rotation process existed. No process was documented — if a key was compromised, there was no mechanism to re-encrypt existing data with a new key. Password hashing: I verified that passwords were hashed with bcrypt (cost factor 12) — correct. The password reset token was hashed with MD5 — a finding; MD5 is not collision-resistant and is unsuitable for security tokens. TLS configuration: using `testssl.sh`, I verified the API server's TLS configuration — no deprecated TLS 1.0 or 1.1, strong cipher suites, valid certificate, HSTS present.

**Result:** Five findings including two criticals (ECB mode and IV reuse). Both required code-level changes to the encryption library usage. The hardcoded keys required rotation (new keys, re-encryption of existing data) — the most complex remediation, completed over two weeks. The RSA key size upgrade required new certificate generation. All were addressed before the platform processed real patient data.

## Key Terms
| Term | Definition |
|------|------------|
| **AES-GCM (Galois/Counter Mode)** | An authenticated encryption mode for AES that provides both confidentiality and integrity, and requires a unique nonce per encryption operation |
| **ECB Mode (Electronic Codebook)** | An insecure AES mode that encrypts identical plaintext blocks to identical ciphertext blocks, revealing data patterns — deprecated for all use cases |
| **IV (Initialisation Vector)** | A random value used in block cipher modes to ensure that identical plaintexts produce different ciphertexts; must be unique per encryption operation and never reused with the same key |
| **RSA-2048** | The minimum recommended RSA key size; RSA-1024 is deprecated and considered breakable with sufficient resources |
| **Key Rotation** | The process of replacing cryptographic keys on a defined schedule or after a suspected compromise, requiring re-encryption of data protected by the old key |
| **Hardcoded Key** | An encryption key embedded in source code or configuration files rather than retrieved from a key management system — a critical finding that exposes the key to any code reader |
| **testssl.sh** | An open-source tool that assesses a server's TLS configuration for weak cipher suites, deprecated protocol versions, and certificate issues |
| **Key Management Infrastructure** | A dedicated system (e.g. AWS KMS, HashiCorp Vault) for generating, storing, rotating, and auditing cryptographic keys, separate from the application |
