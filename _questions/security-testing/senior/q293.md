## The Intent
The interviewer wants to see that you can architect a security automation pipeline, not just run individual tools. They want to understand how you choose tools, configure them to minimise noise, integrate them into CI/CD without adding prohibitive overhead, and manage the findings as an ongoing programme rather than a one-off scan.

## The STAR Script
**Situation:** I joined a company with no automated security testing in the pipeline. Security findings were only discovered during quarterly manual penetration tests or, worse, in production incidents. I was asked to design and implement continuous automated security testing as part of the CI/CD pipeline.

**Task:** Design a SAST and DAST pipeline that provided meaningful security coverage without creating unacceptable false positive noise or pipeline duration overhead.

**Action:** I designed the pipeline in two complementary layers. SAST layer (every PR, target under 3 minutes): I selected Semgrep over SonarQube for its speed and precision on security-specific rules. I configured a custom ruleset combining the Semgrep security-audit rules with three custom rules for our specific patterns (a known insecure internal library usage, hardcoded connection strings, and a framework-specific XSS vector). I ran Semgrep on changed files only — not the full codebase — to maintain speed. Findings were categorised: Critical and High blocked the PR; Medium created a non-blocking annotation; Low was logged but not surfaced in the PR. I maintained a false positive suppression file reviewed quarterly. DAST layer (post-merge to main, target under 15 minutes): I used OWASP ZAP in API scan mode with the OpenAPI spec. For authenticated scanning, I wrote a ZAP Python script that obtained a short-lived token from the test identity provider and injected it as a bearer token. I configured an active scan policy excluding denial-of-service payloads (which disrupted the test environment) and upload fuzzing (handled manually). ZAP generated an HTML report as a pipeline artefact and failed the pipeline only on newly introduced High or Critical findings — not on pre-existing known issues tracked in a baseline file. I introduced a weekly DAST full scan with a more aggressive policy for broader coverage outside the critical path.

**Result:** The SAST layer caught eight new security issues in the first month that would previously have reached the quarterly pen test. The DAST layer caught two API configuration findings within three weeks of deployment. The security team reduced their quarterly pen test engagement from five days to two days, focusing exclusively on business logic and the areas DAST cannot cover.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Runs SAST or DAST tools when asked; reads their output |
| **Mid** | Configures ZAP for authenticated API scanning; triages SAST findings; integrates both into CI with severity thresholds |
| **Senior** | Designs the combined SAST/DAST pipeline architecture; writes custom SAST rules; configures active scan policies; manages false positive baselines; defines the complementary scope of automated and manual testing |

## Key Terms
| Term | Definition |
|------|------------|
| **SAST (Static Application Security Testing)** | Security testing that analyses source code without executing the application, identifying vulnerability patterns at the point of code change |
| **DAST (Dynamic Application Security Testing)** | Security testing against a running application, sending crafted inputs to discover exploitable vulnerabilities in the application's actual behaviour |
| **Custom Semgrep Rule** | A pattern-based detection rule written in Semgrep's YAML syntax targeting a specific vulnerability pattern relevant to the codebase's technology stack |
| **False Positive Suppression File** | A configuration file that lists known acceptable findings, preventing them from appearing in pipeline results and reducing noise for developers |
| **Active Scan Policy** | A ZAP configuration defining which vulnerability checks are run during an active scan, used to exclude disruptive payloads from automated testing |
| **DAST Baseline File** | A ZAP configuration file recording pre-existing known findings, so the pipeline only fails on newly introduced issues rather than accumulating historical noise |
| **Changed Files Only** | A SAST execution scope limiting analysis to files modified in the current PR, maintaining fast feedback while ensuring new code meets security standards |
| **Complementary Coverage** | The combined attack surface coverage achieved by using both SAST (code patterns) and DAST (runtime behaviour) — neither provides complete coverage alone |
