## The Intent
The interviewer is testing whether you can design a coherent, scalable security testing approach rather than applying ad-hoc checks. A senior security QE should be able to justify the methodology, explain the complementary roles of different techniques, and demonstrate that the strategy covers the specific risk profile of the application rather than following a generic template.

## The STAR Script
**Situation:** I was the senior QA at a fintech company building a B2B lending platform — a web application processing credit applications, storing financial documents, and integrating with credit bureaus and banking APIs. The platform was approaching its first regulated launch. There was no existing security testing strategy; it needed to be designed from scratch.

**Task:** Design a comprehensive security testing strategy proportionate to the platform's risk profile and regulatory environment (FCA, GDPR).

**Action:** I built the strategy around four layers. Layer 1 — Design-time: for every new feature, I ran a STRIDE threat modelling session during story refinement. Threats were translated into security acceptance criteria. For the highest-risk components (credit scoring engine, document storage, bank API integration), I produced formal threat models with data flow diagrams. Layer 2 — Build-time (CI pipeline): SAST using Semgrep with finance-specific rulesets (detecting insecure cryptography, hardcoded secrets, and injection patterns), dependency scanning with Snyk blocking on High+ CVEs, and IaC security scanning with tfsec for the Terraform configuration. Layer 3 — Test-time: for every feature, I applied a security test checklist scaled to the feature's risk tier. High-risk features (authentication, document access, financial calculations) received a manual OWASP API Security Top 10 pass. I ran authenticated ZAP API scans on each release candidate against the full OpenAPI spec. Quarterly, I ran a targeted manual penetration test focusing on the areas of highest business logic complexity — the credit decision engine and the multi-party document sharing workflow. Layer 4 — Production: synthetic monitors checked that sensitive endpoints remained authenticated. Anomaly detection alerts fired on unusual access patterns (bulk downloads, off-hours admin access). I also defined a vulnerability disclosure policy so that external researchers could report findings responsibly. I documented the strategy in a security testing policy and reviewed it with the CISO and head of engineering.

**Result:** The FCA authorisation process included a technical assessment of security controls — the documented strategy and its coverage evidence were cited as materially contributing to the successful authorisation. Three critical vulnerabilities were found and fixed during pre-launch testing, none during the post-launch period. The strategy was adopted as the company's security testing standard.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Applies a checklist to individual features; raises findings through the defect process |
| **Mid** | Integrates security testing into the delivery process; applies OWASP frameworks systematically; runs ZAP scans |
| **Senior** | Designs the multi-layer strategy tailored to the application's risk profile and regulatory environment; leads threat modelling; designs the automated pipeline layer; defines the manual testing scope; integrates production monitoring; presents the strategy to CISO and regulators |

## Key Terms
| Term | Definition |
|------|------------|
| **Defence in Depth** | A security principle applying multiple independent security controls, so that the failure of any single control does not expose the system |
| **Risk Tier** | A classification of features or components by their security risk profile, used to determine the appropriate depth and type of security testing |
| **Semgrep** | An open-source static analysis tool using pattern-matching rules to detect security vulnerabilities and code quality issues in source code |
| **Snyk** | A developer security platform that scans dependencies, containers, and IaC for vulnerabilities and provides remediation guidance |
| **Vulnerability Disclosure Policy** | A published process allowing external researchers to report security findings to an organisation in good faith, with defined response commitments |
| **FCA (Financial Conduct Authority)** | The UK financial services regulator, whose authorisation process includes assessment of technical security controls for regulated financial platforms |
| **Anomaly Detection** | A production monitoring technique that alerts on statistically unusual access patterns — such as bulk data downloads or off-hours privileged access — that may indicate a security incident |
| **Security Testing Policy** | A formal document defining the organisation's approach to security testing — the methods, tools, frequency, and ownership — reviewed and approved by the CISO |
