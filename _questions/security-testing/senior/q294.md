## The Intent
The interviewer wants to see whether you understand that multi-tenancy isolation is one of the most critical and failure-prone properties in SaaS applications — and that verifying it requires systematic, exhaustive testing across every data access path, not just the obvious ones. A senior QE should have a methodology that accounts for the non-obvious paths where isolation is hardest to enforce.

## The STAR Script
**Situation:** I was the senior QE for a SaaS HR platform serving multiple enterprise clients. Each client's employee data, payroll, and documents were strictly confidential from other clients. The platform had passed functional testing. I was asked to lead a multi-tenant isolation security test before the first enterprise customer went live.

**Task:** Design and execute a systematic multi-tenant isolation test across all data access paths.

**Action:** I provisioned three test tenants (A, B, C) with distinct, identifiable datasets — each with unique employee names prefixed with the tenant name so any cross-tenant leak was immediately apparent in test results. I then mapped and tested nine data access categories. Direct resource access: I called every resource endpoint authenticated as Tenant B, substituting Tenant A's resource IDs. Expected: 403 or 404. I found one endpoint (`GET /reports/{id}`) returning Tenant A's data to Tenant B. Search and filter: I ran search queries as Tenant B that I expected would match only Tenant A's data (using Tenant A's unique employee names). One search endpoint returned cross-tenant results — a missing tenant filter in the query. Bulk exports and reports: I triggered a full data export as Tenant B and verified the CSV contained only Tenant B's employees. Passed. Background jobs: I reviewed the job scheduler configuration. One job (payroll processing) processed all tenants in a single job without tenant filtering — a potential cross-tenant data mutation risk. Webhooks and events: I verified that event payloads sent to Tenant B's webhook contained only Tenant B's data. Passed. Admin API: I tested whether a Tenant A admin could query Tenant B's data using the admin API. A missing tenant scope check allowed it — a finding. Shared infrastructure: I checked whether the application used shared database tables with a `tenant_id` column (row-level security) or separate schemas. It used shared tables; I verified that every ORM query included the tenant filter using a database query log review.

**Result:** Four isolation failures found: the report endpoint, the search query, the payroll job, and the admin API. All were fixed before the enterprise customer launch. The systematic test matrix became the standard for isolation testing for new features.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests obvious cross-tenant resource access by substituting IDs; may not test search, exports, or background jobs |
| **Mid** | Tests direct access, search, exports, and basic admin API; uses test tenant accounts with identifiable data; documents findings systematically |
| **Senior** | Designs the full isolation test matrix across all data access paths including background jobs, webhooks, and shared infrastructure; reviews ORM query patterns; tests row-level security enforcement; verifies isolation at the database query level |

## Key Terms
| Term | Definition |
|------|------------|
| **Multi-Tenant Isolation** | The property that one tenant's data is inaccessible, invisible, and unmodifiable by any other tenant — a contractual and regulatory requirement in SaaS platforms |
| **Tenant ID** | A unique identifier associated with each customer organisation, used as a mandatory filter on every data access query in a shared-database multi-tenant architecture |
| **Row-Level Security** | A database feature that enforces access control at the row level based on the executing user's identity, used to enforce tenant isolation at the database layer |
| **Cross-Tenant Leak** | An isolation failure where one tenant's data is returned in, or accessible from, another tenant's session |
| **Shared Table Architecture** | A multi-tenancy database design where all tenants' data coexists in shared tables differentiated by a `tenant_id` column — requiring disciplined query-level filtering |
| **ORM Query Filter** | A filter applied by the Object-Relational Mapper to scope every data query to the current tenant, preventing inadvertent cross-tenant data access |
| **Isolation Test Matrix** | A structured test plan mapping every data access path (direct access, search, export, background job, webhook, admin API) against the cross-tenant isolation requirement |
| **Background Job Tenant Scope** | The configuration of a scheduled or asynchronous job to process data for a single tenant at a time, preventing cross-tenant data mutation in batch processing |
