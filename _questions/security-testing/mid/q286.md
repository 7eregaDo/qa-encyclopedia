## The Intent
The interviewer wants to see a comprehensive security mindset applied to a feature area that has a dense attack surface. File handling is consistently exploited in real breaches — malicious file uploads, path traversal in filenames, stored XSS via SVG files, and missing content-disposition headers. Mid-level security QEs should have a systematic checklist for this attack surface.

## The STAR Script
**Situation:** I was performing security testing on a document management system that allowed users to upload files and share download links. The feature had passed functional testing. I was asked to assess its security posture.

**Task:** Systematically test the file upload and download functionality for security vulnerabilities.

**Action:** I tested across six areas. File type validation by content: I uploaded a PHP script renamed as `malicious.pdf` and a HTML file renamed as `image.jpg`. The server was checking file extensions only — both were accepted. A properly configured server should inspect the file's magic bytes (content). I escalated this as a high finding. Path traversal in filename: I submitted filenames including `../../../etc/passwd` and `....//....//etc/passwd`. Both were sanitised to a flat filename — the server was normalising paths correctly. Malicious content — SVG XSS: I uploaded an SVG file containing an embedded `<script>alert('xss')</script>`. When served, the browser executed the script — a stored XSS via file upload. SVG files must be served with `Content-Type: text/plain` or blocked entirely, not as `image/svg+xml`. Content-Disposition header: I requested a download and checked the response headers. The server was serving files without `Content-Disposition: attachment`, meaning the browser rendered HTML and JavaScript files inline rather than downloading them. Storage location: I asked the engineering team to confirm files were stored outside the web root and on object storage (S3), not the application server filesystem. Confirmed — a pass. Malware scanning: the system had no antivirus scanning on uploaded files. I raised this as a medium finding with a recommendation to integrate ClamAV or a cloud scanning service.

**Result:** Three findings: extension-only validation (high), SVG XSS via inline serving (high), and missing Content-Disposition header (medium). All were fixed before the feature launched. The extension-only validation fix required migrating to a MIME type detection library that read file headers.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests obvious wrong file types; checks for basic path traversal; may not test SVG XSS or content-disposition |
| **Mid** | Tests magic byte validation, SVG XSS, Content-Disposition headers, path traversal, storage location, malware scanning, and file size limits; understands the full attack surface |
| **Senior** | Designs file handling security architecture; tests archive extraction (zip bombs, zip slip); reviews storage permissions and CDN configuration; defines the file handling security standard |

## Key Terms
| Term | Definition |
|------|------------|
| **Magic Bytes** | The first few bytes of a file's binary content that identify its true format, used for secure server-side file type validation independent of the file extension |
| **SVG XSS** | Cross-site scripting delivered via an SVG file containing embedded JavaScript, executed by the browser when the file is served with an SVG content type |
| **Content-Disposition: attachment** | An HTTP response header that instructs the browser to download a file rather than render it inline, preventing execution of uploaded HTML or JavaScript |
| **Path Traversal** | A vulnerability where a user-controlled filename containing `../` sequences allows access to files outside the intended storage directory |
| **Zip Slip** | A path traversal attack specific to archive extraction, where a malicious archive contains filenames with `../` paths that extract files to unexpected locations |
| **Zip Bomb** | A maliciously crafted compressed file that expands to an enormous size when extracted, designed to exhaust server storage or memory |
| **ClamAV** | An open-source antivirus engine that can scan uploaded files for known malware signatures as part of a file upload security pipeline |
| **Web Root** | The directory served publicly by a web server; storing uploaded files outside the web root prevents direct URL access to uploaded content |
