## The Intent
The interviewer wants to see whether you can make security testing a natural part of the delivery rhythm without creating a separate, heavyweight security phase that slows delivery. Mid-level QEs should be able to describe specifically where security fits in sprints, who does what, and how findings are tracked and remediated.

## The STAR Script
**Situation:** I joined a team that had a "security sprint" every three months — a dedicated sprint where QE and a contracted pen tester reviewed the application. Between security sprints, no security testing happened, and features built in months one and two were never reviewed until month three. The security sprint consistently found issues too late and too expensive to fix elegantly.

**Task:** Redesign the team's security testing approach to make it continuous and proportionate, eliminating the boom-and-bust cycle.

**Action:** I introduced security activities at four points in the delivery flow. Story refinement: for any story touching authentication, user data, payments, or file handling, I added a five-minute "security check-in" to refinement — asking "what could go wrong from a security perspective?" and capturing answers as security requirements in the acceptance criteria. This took the STRIDE categories as a prompt. Definition of ready: stories rated "elevated security risk" required at least two security acceptance criteria before entering a sprint. Definition of done: all stories had three security DoD items — no critical or high OWASP ZAP findings on new endpoints, no sensitive data in new API responses beyond what the spec documents, and security test cases documented in the test plan. Automated scanning in CI: ZAP API scan ran on every merge to main in the integration environment, with High findings blocking promotion to staging. Sprint retrospectives: I added a standing question — "did we introduce any security debt this sprint?" — which made the team think about security as a sprint-level concern, not a quarterly one.

**Result:** In the following quarter, the contracted pen tester's engagement found two medium-severity issues rather than the previous average of twelve. The pen tester commented that the findings were notably fewer and less critical than previous engagements. The team found it easier to fix security issues raised in the same sprint they were introduced than to retrofit them three months later.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Applies a security checklist during test execution; raises security findings through the standard defect process |
| **Mid** | Integrates security activities into the full delivery flow — refinement, DoR, DoD, CI scanning, retrospectives; defines the security risk classification for stories; tracks security findings alongside functional defects |
| **Senior** | Designs the team's security testing integration model; defines security acceptance criteria standards; leads threat modelling; measures security debt over time; presents security posture to engineering leadership |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Sprint** | An anti-pattern where security testing is batched into a dedicated sprint, creating a periodic review cycle that misses issues introduced between reviews |
| **Security Acceptance Criteria** | Testable security requirements added to a user story alongside functional criteria, ensuring security is considered from the point of story writing |
| **Definition of Ready (DoR)** | A team agreement specifying the conditions a story must meet before it enters a sprint — which can include the presence of security acceptance criteria for elevated-risk stories |
| **Definition of Done (DoD)** | A team agreement specifying the criteria a story must meet before it is considered complete — which should include security checks proportionate to the story's risk level |
| **Security Debt** | The accumulated security weaknesses that arise from taking shortcuts, deferring security work, or building features without security requirements — tracked and paid down like technical debt |
| **Security Risk Classification** | The assignment of a risk level (low/elevated/high) to a story based on its data sensitivity and attack surface, used to determine the appropriate security testing depth |
| **Continuous Security Testing** | The integration of security checks throughout the delivery lifecycle rather than in a separate periodic phase |
| **Pen Tester** | A security professional who performs authorised simulated attacks on a system to identify vulnerabilities, typically engaged periodically to complement internal security testing |
