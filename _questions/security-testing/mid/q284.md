## The Intent
The interviewer wants to see whether your injection testing extends beyond the one-liner SQLi check to a structured understanding of injection contexts. Modern applications use many interpreters — and each has its own payload syntax and risk profile. Mid-level security QEs should know the most common injection types, how to identify which interpreter is in use, and how to test each.

## The STAR Script
**Situation:** I was performing security testing on a Node.js application that used MongoDB as its database, a templating engine for email generation, and a search feature that called a third-party LDAP directory. I wanted to test injection vulnerabilities appropriate to each technology rather than applying SQL injection payloads uniformly.

**Task:** Identify the injection contexts in the application and test each with the appropriate payloads.

**Action:** I identified and tested four injection contexts. NoSQL injection (MongoDB): MongoDB query operators use JSON syntax. I sent the login endpoint a JSON body with `{"username": {"$gt": ""}, "password": {"$gt": ""}}` — a NoSQL injection payload that evaluates to "username greater than empty string AND password greater than empty string", potentially matching all records. The endpoint returned a 200 with a valid session — confirmed NoSQL injection. I also tested `{"$where": "sleep(5000)"}` as a blind timing test. Server-side template injection (SSTI): the email template system accepted a user-controlled "message" field in a notification. I submitted `{{7*7}}` — a Jinja2/Twig detection payload. The notification email contained `49` instead of the literal string — confirmed SSTI, a high-severity finding enabling potential remote code execution. LDAP injection: the search endpoint queried LDAP with the user's input. I submitted `*)(uid=*))(|(uid=*` — a classic LDAP injection payload designed to manipulate the filter logic. The response returned all directory entries — confirmed. OS command injection: the application had an export feature that called a shell command with a user-provided filename. I submitted `report; ls -la` as a filename. The response contained the directory listing appended to the report output — confirmed command injection.

**Result:** All four injection types were confirmed as vulnerabilities. SSTI was the most critical — potential RCE — and was patched within 24 hours. The NoSQL injection and command injection were patched in the same sprint. The LDAP injection required a vendor library update and was tracked as a P1 with a two-week remediation deadline.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests basic SQL injection with single-quote payloads; knows that injection vulnerabilities exist beyond SQL |
| **Mid** | Identifies injection contexts (NoSQL, SSTI, LDAP, OS command) from technology stack; applies context-appropriate payloads; uses timing-based blind injection tests; documents each finding with the interpreter context |
| **Senior** | Designs injection test strategy across the full application stack; uses Burp Suite for systematic injection testing; tests second-order injection; reviews ORM and query library configurations for injection prevention |

## Key Terms
| Term | Definition |
|------|------------|
| **NoSQL Injection** | An injection attack targeting document or key-value databases (e.g. MongoDB) by manipulating query operators in JSON payloads rather than SQL syntax |
| **Server-Side Template Injection (SSTI)** | A vulnerability where user input is evaluated within a server-side template engine, potentially allowing arbitrary code execution on the server |
| **LDAP Injection** | Manipulation of LDAP directory query filters through unsanitised user input, potentially exposing directory entries or bypassing authentication |
| **OS Command Injection** | A vulnerability where user input is incorporated into an operating system command, allowing an attacker to execute arbitrary commands on the server |
| **Blind Injection** | An injection variant where the application does not return query results directly but reveals vulnerability through timing differences, error messages, or out-of-band channels |
| **Jinja2/Twig** | Popular server-side template engines for Python and PHP respectively; `{{7*7}}` is a standard detection payload for template injection in these engines |
| **Injection Context** | The specific interpreter (SQL engine, template engine, LDAP server, OS shell) that processes user-controlled input at a given point in the application |
| **Remote Code Execution (RCE)** | A critical vulnerability class allowing an attacker to execute arbitrary code on the server, typically the most severe outcome of successful SSTI or command injection |
