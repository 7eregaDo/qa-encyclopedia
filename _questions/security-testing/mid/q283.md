## The Intent
The interviewer wants to see a structured, methodical approach to one of the most common and impactful vulnerability classes. IDOR and privilege escalation are consistently in the OWASP API Security Top 10 because they require deliberate adversarial test design — they never appear in functional test suites. They want concrete methodology and evidence of findings.

## The STAR Script
**Situation:** I was performing a security review of a project management SaaS application with three roles: viewer, editor, and admin. Each user belonged to one organisation (tenant), and projects and tasks were scoped to an organisation. I was tasked with systematically testing authorisation across the application.

**Task:** Design and execute authorisation tests covering horizontal privilege escalation across tenants, vertical privilege escalation across roles, and IDOR via direct object reference manipulation.

**Action:** I created a test matrix and three test accounts: User A (editor, Org 1), User B (viewer, Org 1), and User C (editor, Org 2). I tested three authorisation planes. Horizontal (cross-user, same role): I created a project as User A, captured its ID, then authenticated as User B and attempted GET, PUT, and DELETE on User A's project ID. All should return 403. GET returned 200 — User B could read User A's project despite being in the same org with viewer-only rights. Cross-tenant (most critical): I authenticated as User C (Org 2) and attempted to access User A's project ID. GET returned 200 — a full tenant isolation failure. I repeated this across tasks, comments, and file attachments. Vertical (role escalation): as User B (viewer), I sent POST and DELETE requests that required editor or admin role. The UI hid these controls, but the API accepted them — a viewer could create and delete tasks by calling the API directly. I also tested parameter tampering: the `POST /projects` body accepted a `role` field — I sent `"role": "admin"` and verified it was rejected. It was applied — a mass assignment vulnerability granting the created resource admin-level permissions. I documented every finding with the exact request, the expected response (403), and the actual response.

**Result:** Four findings: viewer reading editor-scoped projects (medium), cross-tenant project access (critical), viewer performing editor actions via API (high), and mass assignment privilege escalation (high). The cross-tenant finding was treated as a P0 and patched within 24 hours. The others were resolved within the sprint.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tests basic 401/403 scenarios; verifies that one user cannot access another's resource when obvious |
| **Mid** | Designs a systematic authorisation test matrix across roles and tenants; tests horizontal and vertical escalation; tests mass assignment and parameter tampering; bypasses UI controls by calling APIs directly |
| **Senior** | Designs the full authorisation security test strategy; tests complex permission inheritance; performs JWT claims manipulation; integrates automated BOLA scanning; models multi-tenant isolation as a security contract |

## Key Terms
| Term | Definition |
|------|------------|
| **IDOR (Insecure Direct Object Reference)** | A vulnerability where a user can access or modify another user's resource by manipulating a predictable object identifier in a request |
| **Horizontal Privilege Escalation** | Gaining access to resources or actions of another user at the same permission level — e.g. one editor accessing another editor's private data |
| **Vertical Privilege Escalation** | Gaining access to actions or resources reserved for a higher permission level — e.g. a viewer performing editor or admin actions |
| **Cross-Tenant Access** | A multi-tenancy isolation failure where a user in one organisation can access resources belonging to another organisation |
| **Mass Assignment** | A vulnerability where an API accepts and applies request fields not intended to be user-controlled — such as `role` or `is_admin` — because the server does not whitelist accepted fields |
| **Parameter Tampering** | The manipulation of request parameters — IDs, role fields, scope values — to attempt to access or modify resources outside the user's authorised scope |
| **Authorisation Test Matrix** | A structured table mapping every sensitive endpoint and action against every role combination, used to ensure systematic authorisation test coverage |
| **UI Bypass** | The technique of calling an API endpoint directly rather than through the application's UI, circumventing client-side controls that hide restricted actions from lower-privilege users |
