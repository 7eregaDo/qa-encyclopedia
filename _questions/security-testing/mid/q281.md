## The Intent
The interviewer wants to see whether you can contribute to security design, not just test after the fact. Threat modelling is a mid-level expectation because it requires you to think from an attacker's perspective about a system that doesn't yet exist. They want to see a concrete method (not just "I think about security risks") and evidence that it influenced the feature's design or testing.

## The STAR Script
**Situation:** My squad was building a new file-sharing feature that allowed users to upload documents, share links with external users, and set access expiry times. Before development began, I was asked to participate in the threat modelling session alongside the developer and product manager.

**Task:** Identify the security threats relevant to the file-sharing feature and translate them into security requirements and test cases.

**Action:** I used a simplified STRIDE approach to structure the session. I started by drawing a data flow diagram on a whiteboard: user uploads a file → file stored in S3 → share link generated → external user clicks link → file served. For each component and data flow, I applied STRIDE categories. Spoofing: could an external user impersonate an internal user? The share link needed a signed token with the recipient's email bound to it — not a guessable ID. Tampering: could the file be replaced or modified after upload? The storage layer needed write-once semantics; the download link needed to serve the version at share time. Repudiation: could a user deny sharing a file? Upload and share events needed audit logging. Information Disclosure: could a user enumerate other users' files by manipulating the file ID? File IDs needed to be non-sequential UUIDs; server-side ownership checks were required on every access. Denial of Service: could the upload endpoint be used to fill storage? Per-user upload quotas and file size limits were required. Elevation of Privilege: could a viewer-role user re-share or delete a file? Share actions needed to be restricted to the file owner. Each STRIDE finding became either a security requirement in the story's acceptance criteria or a security test case. I documented them in a shared threat model document and linked it to the feature epic.

**Result:** Three security requirements were added to the user story before a single line was written: non-sequential UUIDs for file IDs, server-side ownership validation on all file operations, and per-user upload quotas. Two of these were directly tested in my security test pass and passed cleanly because they had been built from the requirement.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Aware that threat modelling exists; may contribute observations when prompted |
| **Mid** | Facilitates or contributes to threat modelling using STRIDE or similar; translates threats into security requirements and test cases; documents findings in a threat model linked to the feature |
| **Senior** | Leads threat modelling for complex systems; applies PASTA or other risk-based frameworks; integrates threat model outputs into security test strategy and architecture decisions |

## Key Terms
| Term | Definition |
|------|------------|
| **Threat Modelling** | A structured analysis of a system's attack surface, data flows, and trust boundaries to identify potential security threats before implementation |
| **STRIDE** | A threat categorisation framework covering Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege |
| **Data Flow Diagram (DFD)** | A visual representation of how data moves through a system, used in threat modelling to identify where data crosses trust boundaries |
| **Trust Boundary** | A point in a system where data passes between components with different levels of trust — such as from the internet into the application — where security controls must be applied |
| **Attack Surface** | The sum of all inputs, endpoints, and interfaces through which an attacker could attempt to interact with a system |
| **Security Requirement** | A testable, specific security property a feature must implement, derived from threat modelling and included in acceptance criteria |
| **Non-Sequential UUID** | A universally unique identifier that cannot be guessed or enumerated by incrementing, used to prevent insecure direct object reference attacks |
| **PASTA (Process for Attack Simulation and Threat Analysis)** | A risk-centric threat modelling framework that aligns threats to business objectives and technical implementation |
