## The Intent
The interviewer wants to see hands-on experience with the most widely used free DAST tool. Mid-level security QEs should be able to run ZAP scans, configure them appropriately, interpret results, and integrate them into a CI pipeline — not just know that ZAP exists. They want practical knowledge and an understanding of what ZAP covers and what it misses.

## The STAR Script
**Situation:** My team was preparing for a security review of a REST API before a public launch. Manual testing had covered authentication and authorisation, but I wanted automated scanning coverage for the full API surface, which had 40+ endpoints.

**Task:** Run an authenticated OWASP ZAP scan against the API, interpret the results, and integrate a scan into the CI pipeline for ongoing coverage.

**Action:** I used ZAP's API scan mode with the OpenAPI specification as the input. Setup: I downloaded the application's OpenAPI spec and imported it into ZAP using the `activeScan` API with the spec file. For authenticated scanning, I configured a ZAP script that obtained a JWT from the token endpoint and added it as a bearer token to all subsequent requests — without this, ZAP would test only unauthenticated paths. Scan execution: I ran ZAP in a Docker container using `ghcr.io/zaproxy/zaproxy:stable` with the `zap-api-scan.py` script. The scan ran active checks (injection attempts, header checks, CORS misconfiguration) across all documented endpoints. Result triage: ZAP returned a report with alerts categorised as High, Medium, Low, and Informational. I triaged each alert: High alerts required investigation regardless of whether they were confirmed vulnerabilities or false positives. I confirmed three Medium alerts as genuine — missing rate limiting on the password reset endpoint, a CORS misconfiguration allowing credentials from a wildcard origin, and a missing `X-Content-Type-Options` header. CI integration: I added ZAP to the pre-release pipeline stage as a Docker job. I configured it to fail the pipeline only on High-severity confirmed findings, and to generate an HTML report as a pipeline artefact. I documented the known false positives in a ZAP rules file to suppress them from future runs.

**Result:** The three Medium findings were fixed before launch. The CI integration caught one new High finding in the following sprint — a path that ZAP discovered by following a link in an API response that was not in the OpenAPI spec, exposing a debug endpoint. This endpoint was not known to the QE team and was immediately disabled.

## Key Terms
| Term | Definition |
|------|------------|
| **OWASP ZAP** | An open-source dynamic application security testing tool that scans running applications for common vulnerabilities through proxy interception or direct scanning |
| **DAST (Dynamic Application Security Testing)** | Security testing performed against a running application by sending crafted inputs and observing responses, in contrast to SAST which analyses source code |
| **Active Scan** | A ZAP scan mode that actively sends attack payloads to the target to identify vulnerabilities, as opposed to passive scanning which only observes traffic |
| **Passive Scan** | A ZAP mode that analyses application traffic without sending additional requests, identifying vulnerabilities detectable from normal request/response pairs |
| **ZAP API Scan** | A ZAP scan mode that uses an OpenAPI, SOAP, or GraphQL specification to guide testing of all documented endpoints |
| **False Positive** | A ZAP alert for a vulnerability that does not actually exist in the application — requires triage and suppression to prevent noise from obscuring genuine findings |
| **Alert Triage** | The process of reviewing ZAP's reported alerts to confirm which are genuine vulnerabilities, which are false positives, and what the correct severity is |
| **False Positive Suppression** | A ZAP configuration that marks specific alerts as acceptable, preventing them from appearing in future scan reports and reducing noise |
