## The Intent
The interviewer is assessing your ability to communicate security testing work in a structured, data-driven way that informs team decisions. Security testing without visibility creates a false sense of assurance — teams think they're covered when large parts of the attack surface have never been examined. They want to see that you track coverage, not just findings.

## The STAR Script
**Situation:** My team had been performing ad-hoc security testing for several months, but when the engineering manager asked "how much of our application has been security tested?", nobody had a clear answer. Security findings were tracked in the defect system but there was no view of coverage — which features and risk categories had been assessed.

**Task:** Design a security testing coverage tracking approach that gave the team and management visibility into what had been tested and where the gaps were.

**Action:** I introduced a security coverage matrix maintained as a shared document, updated after each feature's security review. The matrix had two axes. Feature axis: every significant feature or API grouping in the application (authentication, user management, file handling, payments, admin console, public API). Risk axis: the OWASP Top 10 categories plus three application-specific risk categories identified from our threat model (business logic abuse, multi-tenant data isolation, third-party integration trust). Each cell in the matrix had four possible states: Not Tested (red), Tested — No Findings (green), Tested — Open Finding (amber), and Tested — Finding Resolved (light green). I populated the matrix retrospectively from our existing defect history and test plans, then updated it going forward as part of every security test pass. I generated a monthly one-page summary for the engineering manager: percentage of cells in each state, the trend over the previous three months, and the top three untested risk areas by business priority. I also colour-coded the matrix by feature risk level — payments and authentication cells were highlighted as highest priority, so any red cell in those rows was immediately visible.

**Result:** The matrix immediately surfaced that the admin console had never been security tested across any OWASP category — a significant gap. An admin console security review was scheduled for the following sprint and found two high-severity findings. The engineering manager began citing the coverage percentage in engineering leadership updates as a quality metric alongside functional test coverage.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Tracks individual security findings; may not have a view of broader coverage |
| **Mid** | Maintains a security coverage matrix; maps findings to risk categories; produces monthly coverage summaries for the engineering team and management |
| **Senior** | Designs the security coverage measurement framework; integrates automated scanning coverage into the matrix; presents coverage trends to CISO and engineering leadership |

## Key Terms
| Term | Definition |
|------|------------|
| **Security Coverage Matrix** | A structured view mapping features or components against security risk categories, showing the test status and finding state for each combination |
| **Coverage State** | The classification of a security test coverage cell as Not Tested, Tested — No Findings, Tested — Open Finding, or Tested — Finding Resolved |
| **Risk-Based Coverage** | The prioritisation of security testing effort towards higher-risk features and categories, ensuring the most impactful areas receive the most scrutiny |
| **Application-Specific Risk Category** | A security risk category derived from the application's particular threat model — such as multi-tenant isolation — that supplements standard frameworks like OWASP |
| **Security Coverage Gap** | A feature or risk category that has not been tested, representing unknown security exposure that should be prioritised for assessment |
| **Coverage Trend** | The change in security test coverage percentage over time, used to demonstrate progress and identify stagnating areas |
| **False Assurance** | The mistaken belief that a system is secure because no findings have been raised — which may reflect untested coverage rather than genuine security |
| **Security Metric** | A quantitative measure of security posture — such as coverage percentage, open finding count, or mean time to remediate — used to track and communicate security programme effectiveness |
