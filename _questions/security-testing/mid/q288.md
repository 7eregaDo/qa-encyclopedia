## The Intent
The interviewer wants to see whether your security thinking extends to the application's external dependencies, which represent a significant and often under-tested attack surface. Third-party integrations introduce trust assumptions — if those assumptions are wrong, the security of the whole application can be undermined. They want to see methodical coverage, not just "we trust the vendor."

## The STAR Script
**Situation:** I was testing a SaaS platform that integrated with four external services: a payment processor, an email delivery service, an identity provider (SSO), and a cloud storage provider. I was asked to assess the security of these integrations beyond verifying that they functioned correctly.

**Task:** Design and execute security-focused tests for the four third-party integrations.

**Action:** I organised tests around five security concerns per integration. Credential storage: I checked that API keys, OAuth secrets, and service account credentials were stored in a secrets manager (AWS Secrets Manager) and not in environment variables baked into the Docker image, config files committed to git, or returned in any API response. A `git log -p` search for common secret patterns found one historical commit containing a SendGrid API key — which had since been rotated, but the commit remained in history. I flagged the need to rewrite git history. Transmission security: I confirmed all outbound calls to third-party services used HTTPS and that SSL certificate verification was not disabled — a common insecure shortcut I verified by reviewing the HTTP client configuration. Minimum privilege: I reviewed the permissions granted to each integration. The cloud storage integration had `s3:*` permissions on the entire account rather than the specific bucket — excessive privilege. The payment processor API key had refund capability despite the integration only requiring charge capability. Response validation: I tested what happened when third-party services returned unexpected responses. Injecting a mock 500 from the payment processor via WireMock — did the application handle it gracefully without exposing internal state? Webhook signature verification: for the payment processor's webhook callbacks, I verified the application was checking the HMAC signature header before processing the event, and tested that unsigned or incorrectly signed webhooks were rejected.

**Result:** Three findings: API key in git history (medium — key was rotated but history needed rewriting), excessive S3 permissions (high — least privilege violation), and payment processor API key with unnecessary refund capability (medium). All were remediated within the sprint. The git history rewrite was scheduled for the following week.

## Seniority Shift
| Level | Expected Depth |
|-------|---------------|
| **Entry** | Checks that third-party credentials are not hardcoded; verifies HTTPS for external calls |
| **Mid** | Tests credential storage in secrets managers, minimum privilege configuration, response handling for third-party failures, and webhook signature verification; searches git history for exposed secrets |
| **Senior** | Designs third-party integration security standards; assesses vendor security posture; tests supply chain integrity of third-party SDKs; performs security review of vendor access agreements |

## Key Terms
| Term | Definition |
|------|------------|
| **Secrets Manager** | A secure, auditable store for API keys, passwords, and certificates that provides access control, rotation, and audit logging — preferred over environment variables or config files |
| **Minimum Privilege** | The security principle of granting the least permissions necessary for a function to operate, reducing the blast radius of a compromised credential |
| **SSL Certificate Verification** | The validation that an HTTPS server's certificate is valid, unexpired, and signed by a trusted authority — disabling it creates vulnerability to man-in-the-middle attacks |
| **Webhook Signature Verification** | The validation of an HMAC signature in a webhook callback header to confirm the event originated from the expected sender |
| **Git History Secret Exposure** | The persistence of secrets in a repository's commit history even after they have been removed from the current codebase, detectable by searching historical commits |
| **Excessive Privilege** | A permission configuration granting more access than required — such as `s3:*` instead of `s3:GetObject` on a specific bucket — creating unnecessary risk if credentials are compromised |
| **Defensive Response Handling** | The practice of treating third-party API responses as untrusted input — validating response schema, handling errors gracefully, and not relying on third-party data without validation |
| **Blast Radius** | The extent of potential damage if a specific credential or component is compromised — minimised by applying the minimum privilege principle |
